Third-party Hey packages

Hey packages are ordinary public source repositories with immutable versioned releases. GitHub is the first supported publishing host; Hey does not require a central registry to begin sharing packages.

How publishing works

A package repository contains `hey-package.json`, the declared source files, documentation, and specs. For each release, publish two immutable assets:

  • `NAME-VERSION.zip` — the exact package archive.
  • `NAME-VERSION.release.json` — the archive, manifest, and canonical-content SHA-256 values.

Create them with:

hey package verify .
hey package pack . --out dist

Upload both files to a GitHub Release whose tag matches the package version. Consumers pin the exact version, release URL, and hashes in `hey.lock.json`. The archive hash verifies transport bytes; the manifest hash verifies the exact declaration; the content hash verifies the installed files independently of zip metadata.

The catalog below is discovery metadata, not a replacement for lockfiles. A catalog edit cannot silently change already locked package bytes.

Known packages

`hey_web`

Reusable application-facing HTTP server/client conveniences and command-line helpers above Hey's official `stdlib:web`, `stdlib:http`, and `stdlib:net` foundations.

  • Repository: `https://github.com/JayTeeSF/hey_web`
  • Package import examples: `pkg:hey_web@0.1.0/server`, `pkg:hey_web@0.1.0/client`
  • Release assets: attach `hey_web-0.1.0.zip` and `hey_web-0.1.0.release.json` to the GitHub release tagged `v0.1.0`.
  • Trust: copy all three SHA-256 values from the generated release JSON into the consuming project's lockfile.

`hey_web` should not contain its own competing package manager. Package manifests, checksums, packing, locked imports, and verification belong to the Hey toolchain. The package should contain only its modules, binaries/wrappers, docs, examples, and specs.

Adding a package to this catalog

Open a documentation change that includes the repository, package name, latest immutable version, module names, release asset URLs, and the release JSON hashes. Do not list moving branch archives or mutable `latest` URLs as verified releases.
