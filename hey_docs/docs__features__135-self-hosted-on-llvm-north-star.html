North Star: Hey-written tooling targeting LLVM/object code

The long-term direction is **self-hosted-on-LLVM**, with two guardrails:

1. **Do not abandon native-C early.** Native-C remains the default/oracle until LLVM reaches full backend-world parity, has zero mismatches/build failures, and is easier to debug when it fails.

2. **Do not make the self-hosted compiler the everyday compiler until build time is under control.** A Hey-written compiler that takes tens of seconds on common rows is proof of capability, not yet the daily path.

Axes, not one tangled choice

"Self-hosted" means the compiler, tools, scripts, and stdlib infrastructure are written in Hey. It does **not** have to mean the compiler emits C forever.

| Axis | Options |

| --- | --- |

| Compiler implementation language | C bootstrap, Hey self-hosted |

| Backend target | native-C, typed-ir-C, LLVM, later maybe direct object |

| Distribution strategy | requires system C/clang, bundled toolchain, standalone object/native path |

The strategic target is:

Hey compiler written in Hey -> typed IR -> LLVM backend -> native executable

That target is stronger than self-hosted-C as an end state. Self-hosted-C remains the dogfooding proof; self-hosted-on-LLVM is the product direction.

Backend roles

| Backend | Main benefit | Why we keep it |

| --- | --- | --- |

| native-C | Reliable oracle and default shipping path | Proves semantics, gives portable C output, and is easiest to debug. |

| self-hosted-C | Dogfooding proof | Proves Hey can parse, analyze, compile, test, and run itself. |

| LLVM | Long-term performance/platform backend | Gives optimization, lower-level control, object-code path, and a plausible no-toolchain route. |

Current interpretation of the latest report

The report shape supports this strategy: native-C and self-hosted are at parity, while LLVM has zero output mismatches and zero build failures but still has unsupported rows. That means LLVM is currently **clean but incomplete**, not fundamentally buggy.

Priority implications

During the current v0.99.98a release-readiness gate, ergonomics come first: CLI, Result/error handling, collections, text/regex, package workflow, debug, and real sockets/web runtime. After that gate has real stdlib/spec coverage, return to the compiler path in this order:

1. **Canonical symbols/imports.** Required so self-hosted, LLVM, native-C, and REPL agree on every name/module.

2. **Typed IR.** The stable compiler contract. LLVM should not rediscover semantics from source text.

3. **LLVM full backend-world compatibility.** The biggest remaining compiler/backend blocker is unsupported rows, not mismatches.

4. **No-toolchain distribution.** Aim at Hey-written tooling plus LLVM/object output, not C forever.

5. **Self-hosted compiler emits LLVM/object path.** Once LLVM supports the full language, add a Hey-written LLVM emitter.

Anti-goals

• Do not add benchmark-specific LLVM shortcuts.

• Do not hide native-C as an oracle fallback inside LLVM or self-hosted paths.

• Do not remove specs to make parity look better.

• Do not optimize for the final architecture before the daily language is ergonomic enough to write real programs.

v0.99.99a alignment

The ergonomics gate is intentionally before the next broad LLVM-compatibility push: the language must be worth self-hosting before we spend the next tranche of effort making every backend consume the expanded surface. Once the ergonomics gate is usable, return to the North Star path: canonical symbols -> typed IR -> LLVM full parity -> no-toolchain/object distribution -> Hey-written compiler targeting LLVM/object code.
