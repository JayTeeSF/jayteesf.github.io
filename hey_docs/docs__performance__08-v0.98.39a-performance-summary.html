v0.98.39a performance summary

This note summarizes the latest JayTeeSF-reported performance observations around

v0.98.38a and the interpretation that should guide the next slices.

Full suite speed recovered

The earlier v0.98.36a full suite had a slowest file around 3.6s because

`specs/language/520_dual_suite_benchmark_coverage.hey` was running the heavy dual

suite. v0.98.37a/v0.98.38a moved that spec back to the fast coverage path.

Latest reported full suite:

Finished in 6.905s
Files: 182, failures: 0
Slowest files:
  0.950s heyc spec specs/language/500_self_hosted_supported_subset_release.hey
  0.944s heyc spec specs/language/460_bootstrap_handoff_gate.hey
  0.942s heyc example examples/self_hosted_supported_subset.hey

That is good news: ordinary examples did not remain doubled. The suite is back in

the same general range as the pre-bridge reports while covering more files.

Dual timing is now a heavier compatibility matrix

Latest dual timing:

supported=12 unsupported=0
note=self-hosted language-compatibility bridge; direct self-hosted compiler parity still pending

The important distinction:

• `supported=12 unsupported=0` means every tracked fixture works through the

self-hosted front door.

• It does **not** mean direct self-hosted compiler parity is complete.

• Use `./bin/subcmds/hey-selfhost-coverage` to see direct vs bridged counts.

• Any fixture with `compiler_mode=native_oracle_bridge` is compatibility coverage,

not direct Hey-written parser/emitter parity.

The latest reported `self_ms` values were higher than earlier direct-subset

results, mostly around 185ms-226ms for smaller fixtures and 354ms for

`jobs_production`. This should be treated as dual-suite/front-door cost, not as

runtime-only app performance. The priority is still language compatibility first,

then moving bridged fixtures to direct, then optimizing self-hosted performance.

Native-C direct timings are noisy but not clearly regressed

Latest reported Native-C dual timings were higher for several tiny fixtures:

| Fixture | Latest native_ms |

|---|---:|

| hello | 11.680 |

| variable_greeting | 15.244 |

| math_and_bool | 19.813 |

| function_control_flow | 20.846 |

| stdout_io | 23.521 |

| string_function | 20.679 |

| string_len | 24.296 |

| tiny_probe | 21.407 |

Earlier reports had many tiny fixtures around 11ms-14ms. These are small enough

that machine load, thermal state, compiler cache, filesystem state, and the amount

of work inside the benchmark wrapper can move them around. Track them, but do not

optimize based on one run.

Useful next check:

./bin/test --dual-timing
./bin/test --dual-timing
./bin/test --dual-timing

Compare medians rather than a single run.

1BRC full proof

Latest full proof with progress disabled:

workers=8
runtime_ms=38026
rows_per_sec=26297568
est_1b_s=38
sub60=true
sub5=false
rows_match=true

English interpretation: the native Hey runtime still processes the full 1B-row

file correctly in under one minute on JayTeeSF's M3, but it is not close to the

under-5s aspirational goal. A historical best around 29s existed, so the 38s run

should be tracked as a possible regression or environment variance until bisected.

Progress logging was not the cause of the 38s result because this run used

`--no-progress`.

1BRC scale matrix

The latest high-confidence scale run tested 1M, 10M, and 100M rows across workers

1, 2, 4, and 8 for `native-c`, `self-hosted`, and `llvm`.

Important reading rules:

• `runtime_ms` is measured inside the compiled Hey app around `one_brc_summary`.

• `build_s` and `proc_s` are useful, but informational.

• `proj_1b` is a linear projection from that row count. Treat 100M as much more

trustworthy than 1M or 10M.

• LLVM is currently expected to be `unsupported` for 1BRC because runtime

aggregate/map/string lowering has not landed.

Best reported row in that run:

rows=10000000 implementation=native-c workers=4 rows_per_second=90909090.91 projected_runtime_1b_seconds=11.00 confidence=medium

That 10M/4-worker row is encouraging, but it is only medium confidence and did

not match the 100M or real 1B proof. The 100M rows are more predictive:

| Implementation | Workers | 100M runtime_ms | Projected 1B |

|---|---:|---:|---:|

| native-c | 1 | 4212 | 42.12s |

| native-c | 2 | 2041 | 20.41s |

| native-c | 4 | 1523 | 15.23s |

| native-c | 8 | 1485 | 14.85s |

| self-hosted bridge | 1 | 6540 | 65.40s |

| self-hosted bridge | 2 | 3476 | 34.76s |

| self-hosted bridge | 4 | 2202 | 22.02s |

| self-hosted bridge | 8 | 2384 | 23.84s |

English interpretation: on generated 100M-row data, Native-C scales well from 1 to

4 workers and only slightly from 4 to 8 workers. The self-hosted bridge path is

slower, but it remains correct. LLVM is not participating in 1BRC yet.

The real 1B file remains slower than the best 100M projection, so full-file I/O,

station distribution, partition boundaries, and merge behavior still matter.

Next performance work

1. Keep full-suite tests fast by separating coverage checks from explicit timing

benchmarks.

2. Move self-hosted bridged fixtures to direct one construct at a time.

3. Add LLVM runtime/data-model lowering before expecting LLVM 1BRC rows to work.

4. Use `1brc-scale` for frequent confidence and the real full proof only for

1BRC-specific slices or release-quality checks.

5. For Native-C 1BRC, profile parse loops, station hash-table layout, worker

partitioning, and merge behavior; LLVM alone will not automatically solve those.
