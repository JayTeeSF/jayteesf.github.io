Processes, command-line arguments, and environment tutorial

Use `stdlib:process`, `stdlib:cli`, and `stdlib:env` to write Hey-native command tools.

import 'stdlib:process'
import 'stdlib:cli'
import 'stdlib:env'

Read arguments

let args = cli.args()
says len(args)

Read environment values

let home = Env.default('HOME', '')
says home

Execute a process with explicit arguments

let result = process.run_argv(['git', '--version'])
says result['ok']
says result['stdout']

Prefer argument arrays over shell command strings. Shell strings reintroduce quoting bugs and hidden dependencies. Maintained Hey tools should move their command bodies into compiled Hey programs and leave only thin launchers where necessary.

Host facts without shell commands

Use `stdlib:env` instead of spawning `pwd` or `uname`:

import 'stdlib:env'

says Env.cwd()
says Env.os()
says Env.arch()

These values come from the runtime and are available to vendoring and tooling without hidden shell utilities.

Required option values

Use `cli.missing_value(args, name)` before expensive work when an option requires a value:

if cli.missing_value(args, 'out')
  fail '--out requires a value'
end

This catches both `--out` without a following value and `--out=`. The docs generator uses this guard before scanning or rendering a site.

Hey-owned command bodies

`tools/hey_selfc.hey`, `tools/hey_dist_stage.hey`, `tools/check_source_archive.hey`, and `tools/hey_1brc_download_data.hey` demonstrate the intended command architecture: a tiny launcher may locate the checkout, but argument parsing, filesystem policy, process calls, and diagnostics belong in Hey.
