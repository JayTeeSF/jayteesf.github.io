Hey-lang Architecture Standard

This document describes the design boundary used when deciding whether a capability belongs in the language, runtime, standard library, or a third-party package.

Core rule

Syntax is scarce. Plumbing is stable. Porcelain belongs in libraries.

Do not add language syntax when the feature can be implemented as a library over existing plumbing.

Language plumbing

The language/runtime owns only the primitives that libraries cannot safely or efficiently implement themselves:

values
functions
modules
imports
maps/records
arrays
strings
bytes
numbers
booleans
nil
shape metadata
actors
mailboxes
send/receive
links
monitors
supervisors
I/O targets
scheduler hooks
allocation/performance introspection

These are the building blocks for the standard library and third-party packages.

Standard-library porcelain

The standard library should provide batteries-included workflows without closing off alternate implementations:

json.encode / json.decode
jsonlines.encode / jsonlines.decode
jobs.queue / jobs.perform_later / jobs.perform_at
workflow chains/rings
retry/backoff/dead-letter policies
stream processors
1BRC helpers
HTTP client/server
process pipes
collection helpers

A third party should be able to build a different JSON parser, job system, HTTP framework, queue backend, scheduler policy, or workflow package without fighting special syntax.

Data model rule

Hey-lang owns core values. JSON is a codec.

The current runtime still uses legacy `HeyJsonObject` names internally, but the language direction is:

core map/record value  -> encode/decode through JSON codecs
core array value       -> encode/decode through JSON codecs
core string/bytes/etc  -> encode/decode through JSON codecs

Convenient object/array literals stay, but they are map/array literals in the language model rather than a permanent commitment to one JSON standard.

Shape rule

Shapes are plumbing. They describe stable map/record layouts and key sets.

Shapes support:

fast key lookup
record-like optimization
JSON codec implementation
job payload conventions
actor message dispatch
future object/record systems

Shape APIs may be exposed for introspection, but most code should not depend on exact numeric shape IDs across processes or versions.

Immutability rule

Core values are immutable from user code. Runtime implementations may use private mutation while constructing a new value, but user-visible operations such as `put`, `delete`, `merge`, and `push` must return new values and must not mutate existing values.

Shapes must preserve this rule. A shape is immutable metadata about a value's structure, not a mutable hidden class that can change an existing value in place.

let user = {name: 'Ada', visits: 1}
let original = shape_id(user)
let next = {...user, city: 'SF'}

# user is unchanged; next has a different shape.

Iteration rule

Collection iteration starts as snapshot plumbing: `keys`, `values`, and `entries` return arrays. Future loops and enumerable helpers should build on this rather than adding special JSON-specific syntax.
