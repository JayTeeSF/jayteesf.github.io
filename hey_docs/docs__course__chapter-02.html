Chapter 2 — The Shape of the Compiler

A **compiler** is a program that translates source code into a form a

machine can execute. Hey's compiler does this in *stages*, where each

stage consumes the previous stage's output and produces something a

little closer to machine code. This chapter is the map; Chapters 6–10

walk each stage in detail.

The pipeline in one picture

flowchart LR
    SRC["Hey source<br/>(.hey files)"] --> P["Parser<br/><i>HeyParser</i>"]
    P --> SEM["Semantic analysis<br/><i>HeySemantics + HeyHir</i>"]
    SEM --> HIR["HIR<br/>(typed program IR)"]
    HIR --> LOW["Lowering<br/><i>HeyMir</i>"]
    LOW --> MIR["MIR<br/>(textual ABI IR)"]
    MIR --> EMIT["LLVM emission<br/><i>HeyLlvm</i>"]
    EMIT --> LL[".ll file<br/>(LLVM IR text)"]
    LL --> LINK["Object + link<br/>(thin native step)"]
    LINK --> EXE["Executable"]
    RT["Runtime library<br/>(hey_runtime.c)"] -.->|linked in when needed| LINK

Modules written in *italics* above are ordinary Hey source files in

`stdlib/`. Only the last step (turning LLVM IR into an executable) and

the runtime library are native code — and neither one understands the

Hey language.

What each stage is, in plain words

**Parsing** (Chapter 6). The **parser** reads your source text and

checks that it is grammatically well-formed — that `fn`, `end`, quotes

and brackets all line up — and builds a structured, machine-friendly

description of the program. Along the way it resolves **imports**:

statements like `import 'stdlib:Text'` that pull other modules into

your program. Hey resolves the entire import graph up front, so later

stages see one self-contained program. (Many compilers split this

stage into a *tokenizer* that groups characters into words and a

parser that groups words into structure; Chapter 6 shows how Hey

organizes that work.)

**Semantic analysis** (Chapter 7). Grammar is not meaning. *Semantic*

checks answer questions grammar cannot: Does this function exist? Is

it called with the right number of arguments? What is the type of this

variable? The result of this stage is Hey's first **intermediate

representation** — an **IR** is simply a data format a compiler uses

internally between source and machine code. Hey's first IR is called

**HIR** ("high-level IR"): the program with its structure resolved and

every expression annotated with a type.

**Lowering to MIR** (Chapter 8). **Lowering** means translating a

representation into a less abstract one — closer to what the machine

does, with the conveniences compiled away. Hey lowers HIR into **MIR**

("mid-level IR"), and here Hey makes an unusual, load-bearing choice:

MIR is *plain text* with a stable, documented format. You can print

it, read it, diff it, and write tests against it. MIR is Hey's **ABI

boundary** — an **ABI** (application binary interface) is a contract

about representation that both sides of a boundary agree never to

break casually. Everything above MIR is free to evolve; everything

below it can trust the format.

**LLVM emission** (Chapter 9). **LLVM** is a widely used, open-source

compiler infrastructure; its own textual IR (a `.ll` file) can be

turned into fast machine code for essentially every mainstream

processor. Hey's back end, the `HeyLlvm` module, translates MIR text

into LLVM IR text. By targeting LLVM, Hey gets decades of optimization

and portability work for free — while keeping every *decision* about

what code to generate in Hey source.

**Linking** (Chapter 10). The final native step assembles the LLVM IR

into an object file and **links** it — combines it with the runtime

library and the operating system's startup code — producing the

executable you run.

Why stages at all?

Three reasons, and they recur throughout this book:

1. **Each stage is testable alone.** Hey's test suite contains

thousands of *specs* (executable expectations, Chapter 11), many of

which pin the exact output of a single stage — "this source must

produce exactly this MIR."

2. **Stages localize change.** A new syntax feature touches the

parser and HIR; a new optimization touches emission. The MIR

contract in the middle keeps the halves honest.

3. **Stages make self-hosting tractable.** Chapter 12 shows that a

compiler that can print its own intermediate output at every stage

is a compiler you can *prove* rebuilt itself correctly.

Discussion questions

1. MIR being *text* costs some compile-time performance (printing and

re-parsing between stages). List two engineering benefits that Hey

buys with that cost, and name a compiler you know that made the

opposite choice.

2. Why is it valuable that the native launcher and runtime contain no

language semantics? What could go wrong in a project where "just a

little" parsing lived in C?

3. Sketch the pipeline for a language you use. Which of Hey's stages

does it merge or skip?
