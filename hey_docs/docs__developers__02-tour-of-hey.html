Tour of Hey

The Tour is a guided introduction backed by eight maintained programs. Every chapter has a self-testing source file under `examples/tour`, and the repository exposes one command surface for listing, running, and verifying them.

./bin/hey tour list
./bin/hey tour run 01
./bin/hey tour verify

Use an ID such as `03` or a slug such as `actors`. Running without a selector executes every lesson. `verify` runs each embedded spec; `run` prints the program's ordinary output.

01 — Values and immutable updates

Source: `examples/tour_01_values.hey`

Objects and arrays are values. Spread syntax returns a new value rather than mutating the original.

let user = {name: 'Ada', visits: 1}
let next = {...user, visits: user.visits + 1}
let names = ['Ada', 'Grace']
let expanded = [...names, 'Katherine']

Run it:

./bin/hey tour run values

Continue with object field access, collection method syntax, and JSON codecs.

02 — Functions and callable values

Source: `examples/tour_02_functions.hey`

Functions are first-class values. They can be assigned, stored in collections, passed to Jobs and streams, and invoked through a local binding.

let formatter = greet
let operations = [add]
let operation = operations[0]
says formatter('Ada')
says operation(40, 2)

Run it:

./bin/hey tour run functions

Continue with functions and callable values.

03 — Actors and private state

Source: `examples/tour_03_actors.hey`

Actors own identity and persistent state. Only the actor changes that state, one received message at a time.

actor Counter(label)
  state count = 0
  receive message

  if message == 'increment'
    set count = count + 1
    says label + '=' + count
  end
end

Run it:

./bin/hey tour run actors

Continue with actors and mailboxes and actor scalability.

04 — Stateless worker pools

Source: `examples/tour_04_workers.hey`

Workers process immutable messages without retaining state. A bounded pool replicates one worker definition behind one queue.

worker Square(message)
  send message.reply_to, message.value * message.value
end

let pool = workers(Square, count: 2, capacity: 8)

Run it:

./bin/hey tour run workers

Continue with stateless workers.

05 — Jobs, receipts, and results

Source: `examples/tour_05_jobs.hey`

Jobs are the managed application layer over workers. Enqueueing returns a receipt; callers can await a result and inspect bounded operational metrics.

let Thumbnails = jobs.define('TourThumbnails', Thumbnail,
  count: 2,
  capacity: 8
)
let receipt = Thumbnails.enqueue({id: 'cover'})
let result = jobs.result(Thumbnails, receipt, 5000)

Run it:

./bin/hey tour run jobs

Continue with concurrency and Jobs, worker and Job tutorial, and `stdlib:jobs`.

06 — Bounded parallel streams

Source: `examples/tour_06_streams.hey`

Streams keep familiar collection chains while using bounded Jobs-backed execution internally.

let values = [1, 2, 3, 4]
  .map(double, workers: 2)
  .filter(greater_than_four, workers: 2)
  .collect()

Run it:

./bin/hey tour run streams

Continue with `stdlib:streams`.

07 — I/O handles and Result values

Source: `examples/tour_07_io_and_results.hey`

Files, mappings, processes, pipes, and sockets use explicit opaque handles. Expected failures and lifecycle operations return Result-shaped values rather than hiding ownership.

let writer = files.open(path, 'write').value
let wrote = writer.write_all("hello from a file\n")
let writer_closed = writer.close()

Run it:

./bin/hey tour run io

Continue with input and output, stateful I/O handles, and `stdlib:result`.

08 — Web routes as ordinary callables

Source: `examples/tour_08_web.hey`

Web routes preserve ordinary callable identity. Pure dispatch is a fast composition and test boundary; `web.serve` runs the maintained live socket server.

let routes = web.routes([
  web.get('/hello/:name', hello)
])
let response = web.dispatch(routes, request)

Run it:

./bin/hey tour run web

Continue with `stdlib:web`, static files, and the task-oriented application tutorial.

Verify the whole Tour

./bin/hey tour verify

A successful run ends with:

tour verify ok lessons=8

The documentation smoke gate runs the same verification. A Tour chapter cannot be published as maintained guidance unless its linked program and embedded spec pass.
