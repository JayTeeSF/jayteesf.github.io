Historical roadmap audit ledger

> This file is a chronological record. Statements that a backend was “default,” “selectable,” or “planned” describe the cited historical alpha only. Current truth lives in `docs/humans/north-star.md` and `docs/humans/roadmap.md`; as of v0.99.239a the LLVM-hosted Hey -> HIR -> MIR -> LLVM route is the product default.

v0.99.215a roadmap audit

The main compiler goal is no longer a broad backend matrix. It is full repository closure through two independent Hey hosts and one direct LLVM MIR consumer. The examples producer/build milestone is complete at 88 equivalent MIR units, 87 LLVM application builds, and one library. Remaining open work is execution/effect parity, language-spec closure, tooling/stdlib closure, explicit block MIR, source diagnostics, LLVM-hosted public tool builds, and Stage-2/Stage-3 fixed-point bootstrap.

Implementation-independence north star (v0.99.211a)

Hey must have a standalone, versioned, backend-opaque conformance suite that

can evaluate the native reference compiler, the Hey-written compiler, and

future Rust, Java, JVM, WebAssembly, or other implementations solely through

observable language behavior. The native and self-hosted implementations may

eventually split into separate repositories, but they must continue consuming

the same conformance artifact.

Canonical typed IR is a separate optional interoperability seam. It enables

multiple frontends to share LLVM and future backend consumers, but it must not

leak backend structure into the language specification. Repository-specific

pipeline assertions live under `specs/compiler/`; LLVM structural goldens live

under `specs/ir/`; portable language behavior lives under

`specs/conformance/`.

Current priority order

Compiler implementation independence north star

The Hey language must be specified by a standalone, backend-opaque conformance

suite: source acceptance/rejection and documented observable behavior only.

Native C, the Hey-written compiler, and future Rust/Java implementations should

be independently evolvable against that same suite. Compiler-specific tests may

add parser, semantic, canonical-IR, LLVM, object, and optimization contracts,

but those are never requirements of language conformance.

Canonical `hey-ir` is the optional interoperability layer. The official native

and Hey frontends should both emit it so their LLVM output can be compared

through one consumer. Third-party implementations may either adopt that seam or

own a completely different backend while remaining conforming.

JayTeeSF's current top priorities are the formerly numbered goals **2-5, 7, 14, 13, 9, and 8**, followed by the real-app blockers surfaced by the HTTP/Redis/Rack-style server targets. Class/object syntax and user-facing async/await are deliberately not top priorities.
