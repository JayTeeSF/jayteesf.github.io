Actor scalability

Hey actors are lightweight isolated mailbox processes scheduled as one-message turns on a bounded native worker pool. Ordinary compiled actors do not own operating-system threads. Actor state and FIFO mailbox order remain private while workers execute runnable turns in parallel.

Resident actor capacity

`benchmarks/hey/actor_capacity.hey` creates dormant actors and reports creation time, resident-memory growth, sample-based capacity, and observed RSS per actor. Its memory percentage is capped at 80%. Linux runs respect effective cgroup limits as well as available memory, so the benchmark stops before exhausting the host or container.

Use this benchmark for resident actor density. A billion actors is not a fixed language guarantee: capacity depends on actor state, mailbox contents, allocator behavior, and physical memory.

Parallel fan-out

`benchmarks/hey/actor_fanout.hey` keeps many actors runnable and reports elapsed time and memory growth. Use it for scheduler throughput. A process ring usually has one runnable actor and primarily measures handoff latency, not all-core parallelism.

Compile it normally:

./bin/heyc build benchmarks/hey/actor_fanout.hey -o build/actor-fanout

Then compare the same workload across worker counts:

for workers in 1 2 4 8 16
do
  HEY_WORKERS="$workers" ./build/actor-fanout 20000 200
done

Scheduler diagnostics

Detailed counters are deliberately opt-in because measurement must not add permanent hot-path synchronization:

HEY_WORKERS=4 HEY_SCHEDULER_STATS=1 \
  ./build/actor-fanout 20000 200

At shutdown the runtime writes one `HEY_SCHEDULER_STATS` JSON object to stderr with worker count, ready-depth high-water mark, enqueue/dequeue counts, processed turns, steals, sleep events, and wake signals. Compare counters only across identical actor workloads. Use profiling-disabled runs for promoted performance numbers.

Current evidence

Million-actor residence and fast bulk shutdown are proven in the maintained Linux diagnostic. Per-worker ready queues and stealing are also correct. However, tiny actor turns still become slower as workers increase. This means the next bottlenecks are outside the former global ready queue: actor registry operations, mailbox mutexes, message allocation/recycling, reference counting, and generic boxed-value operations.

A naive bounded-spin experiment increased steal traffic and was removed. Future worker spinning or parking changes must improve repeated results across several worker counts rather than merely reducing one internal counter.
