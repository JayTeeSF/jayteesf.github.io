v0.50: Evented I/O and Real C10K

Type: runtime plumbing.

v0.50 starts the real evented-I/O foundation by adding a native nonblocking TCP loopback benchmark primitive:

program
  let result = c10k_ping(10000)
  says result['ok']
  says result['completed']
end

`c10k_ping(client_count)` opens a real loopback TCP listener, creates nonblocking client sockets, accepts server-side sockets, drives readiness through `poll(2)`, sends `ping`, replies `pong`, and returns an immutable result map.

Returned fields include:

kind: evented_io_c10k_ping
backend: poll-nonblocking-tcp-loopback
ok
clients
connect_started
accepted
completed
poll_loops
read_events
write_events
error

This is deliberately runtime plumbing, not final public socket syntax. It proves that Hey can drive many real nonblocking descriptors from a single event loop before exposing a stable socket API.

Benchmark command

ulimit -n 30000
bin/subcmds/hey-c10k-benchmark --clients 10000

The smoke suite uses tiny client counts. True C10K depends on the host file-descriptor limit and OS loopback behavior. On macOS, raise `ulimit -n` before trying 10,000 clients.

What is real in this slice

• Real `socket`, `bind`, `listen`, `connect`, `accept`, `send`, and `recv` calls.

• Nonblocking descriptors via `fcntl(O_NONBLOCK)`.

• Readiness through `poll(2)`.

• Loopback TCP clients and accepted server sockets.

• A workload benchmark path that uses the socket implementation instead of actor-only simulation.

What remains

• General-purpose `tcp.listen`, `tcp.connect`, and actor-owned descriptor APIs.

• Per-descriptor owner actors and mailbox-delivered readiness/completion messages.

• Bounded write queues and backpressure policies.

• kqueue/epoll backends behind the same event-loop shape.

• TLS and process/pipe integration.
