v0.49/v0.50.2: Jobs and Queues as Standard-Library Porcelain

v0.49 implemented the first jobs/queues slice without adding language syntax. v0.50.2 tightens the supporting data model so jobs pools can use actor handles directly or object-shaped pid refs.

Jobs are standard-library porcelain over existing plumbing:

functions
modules
actors
mailboxes
send / receive
links / monitors
supervisors
local pid registry
immutable core map/array values

The goal is to make a simple default job system possible while still allowing a third-party package to implement a Sidekiq-like, SolidQueue-like, Oban-like, or custom distributed queue system in Hey-lang.

No special job syntax

This is intentionally **not** language syntax:

job queue: 'imports'

The direction is normal library code:

import '../stdlib/jobs.hey'

let queue = jobs.queue({
  name: 'local/imports',
  min_workers: 1,
  max_workers: 8,
  retries: 5,
  scheduler: 'round_robin',
  scale_on: {latency_ms: 500}
})

Queue values

`jobs.queue(config)` returns an immutable queue configuration value with defaults:

let queue = jobs.queue({name: 'local/imports'})

Fields include:

kind: queue
name / namespace
min_workers
max_workers
scheduler
scale_on: {metric: "latency_ms", latency_ms: ...}
retries
priority
backend: actor-supervisor-runtime
persistence: none

Queue names reserve distributed space from the start. Prefer names such as:

local/default
local/imports
node-a/imports
cluster/us-east/imports

start_pool / perform

New code should create supervised workers, start a queue runtime, and perform against that runtime:

let max_restarts = queue['max_restarts']
let w1 = supervise worker max max_restarts
let w2 = supervise worker max max_restarts
let runtime = jobs.start_pool(queue, [w1, w2])
let receipt = jobs.perform(runtime, {name: 'Ada'})

The scheduler actor forwards immutable job envelopes to workers. Round-robin dispatch is selected with `scheduler: 'round_robin'`.

Worker refs

Source object/array literals are now core values, not strict JSON text, so `[w1, w2]` can hold actor handles. Jobs also accepts pid-based local refs:

let ref = jobs.worker_ref(w1)
let runtime = jobs.start_pool(queue, [ref, jobs.worker_ref(w2)])

`jobs.worker_ref(w1)` returns `{kind: 'actor_ref', pid: ...}`. Internally the stdlib can use `send_pid(pid, message)` or `actor_by_pid(pid)` to cross from an object-shaped local ref back into the runtime actor registry.

These pid refs are local runtime ids. They are not durable distributed actor references.

Compatibility helpers

`jobs.perform_later(queue, worker, payload)` and `jobs.perform_at(queue, worker, at, payload)` remain for older examples. `worker` may be an actor handle, integer pid, or `{kind: 'actor_ref', pid: ...}` object.

let job_id = jobs.perform_later(queue, worker, {name: 'Ada'})
let job_id = jobs.perform_at(queue, jobs.worker_ref(worker), '2026-07-01T09:00:00Z', {name: 'Ada'})

Why this is useful

This proves the architectural boundary:

• jobs can be expressed as ordinary library functions;

• actors remain the low-level worker plumbing;

• queues are immutable config values;

• scheduler ownership lives in a queue runtime actor;

• worker references can be actors, pids, or object-shaped local refs;

• retry/priority/scaling policy live in data, not syntax.

What remains

The jobs-owned-pools prerequisite remains future work in v0.52. The production-ready jobs entry is v0.57:

• first-class actor specs so `jobs.start(queue, WorkerActor)` can create supervised workers internally;

• runtime-owned worker pools;

• strong monitor/link exit reasons suitable for job failure policy;

• real timer wheel for `perform_at`;

• backoff/retry/dead-letter mechanics;

• durable queue backends and recovery;

• queue latency/depth measurement and worker health;

• cancellation, status, join, and result retrieval;

• production examples/specs.

Those should remain library/runtime work, not syntax, unless a missing plumbing primitive blocks library implementation. See v0.57 production-ready jobs stdlib.
