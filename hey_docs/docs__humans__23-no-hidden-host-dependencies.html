v0.99.236a TLS dependency update

Hey no longer hides HTTPS semantics behind `curl`: `stdlib:tls` performs the handshake and certificate/hostname validation, and `stdlib:http` uses that transport for bounded text responses. Linux currently links the explicit system OpenSSL development/runtime boundary; Darwin links the explicit Security/CoreFoundation platform boundary. The local TLS regression uses the `openssl s_server` executable only as a test fixture server, not as product transport.

The remaining `curl` use is now specifically a byte-preserving streaming-download migration target, not a TLS implementation. Provider reproducibility is also still open: linking a system OpenSSL is secure validation but is not equivalent to shipping pinned provider bits.

v0.99.216a distribution rule

Public distribution commands are compiled from `.hey` source. `hey` itself supports `#!/usr/bin/env hey`, creates/reuses content-addressed HIR/MIR sidecars, compiles through LLVM, and runs the cached application with its original arguments. The trusted C bootstrap is private implementation machinery under `libexec/hey/heyc-bootstrap`, not a public product command or scripting-language dependency. Shell remains acceptable only for source-checkout build/release orchestration until equivalent Hey commands replace it; no Python/Ruby/Perl/Node implementation is added.

v0.99.210a dependency note

The backend timing body moved from shell plus `awk`/`sed` parsing into `tools/hey_compiler_matrix.hey`; `hey-dual-timing` and `hey-backend-timing` are now thin compatibility launchers. The matrix still invokes explicit compiler/linker boundaries. The LLVM closure remains shell-orchestrated and is still a future Hey-rehome target.

The v0.99.213a semantic audit covers **175 maintained files and 1,318 executable-position dependencies**. HIR/MIR production, validation, lowering, golden maintenance, and sidecar cache policy are implemented in Hey. Implementation scope remains deliberately narrow: one migration-required `curl`, plus Darwin-only `codesign` and `xattr` trust boundaries. Command and verification scopes still contain shell/text utilities and remain migration work; no Python, Ruby, Perl, Node, host ZIP, or host tar implementation body was added.

v0.99.208a dependency note

The refreshed semantic audit covers **164 maintained files and 1,297 executable-position dependencies**. The only migration-required implementation command remains one `curl`; Darwin `codesign` and `xattr` remain explicit trust boundaries. No Python, Ruby, Perl, Node, host `zip`, or host `tar` implementation was added.

The generic LLVM stdlib delegation added in this slice is intentionally classified as transitional because it invokes `./bin/heyc` at application runtime. It removes compiler unsupported classifications but does not satisfy the no-hidden-host-dependency goal. Direct runtime ABIs must replace it before LLVM-built applications or distributions claim independence.

v0.99.206a dependency note

The v0.99.206a audit covers **163 maintained files and 1,283 executable-position dependencies** in about **7.36 seconds**. This slice adds no host implementation command. Reserved-word documentation, import normalization, reachability, and tooling facades are implemented in Hey/runtime/compiler code. `curl` remains the single migration-required binary-HTTPS transport; `codesign` and `xattr` remain explicit Darwin trust boundaries; compiler/linker and Docker remain declared build boundaries.

v0.99.204a dependency update

No new host implementation was introduced. The LLVM link command now exposes whether the shared Hey runtime object is `linked` or `elided`; pure programs use only the declared Clang/linker boundary. Docker and older target Clang remain explicit cross-build boundaries, not hidden implementations. The semantic audit was rerun after the slice: 163 maintained files and 1,275 executable-position dependencies. Implementation scope remains one migration-required `curl` plus Darwin-only `codesign` and `xattr`; no Python, Ruby, Perl, Node, host `zip`, or host `tar` implementation was added.

v0.99.201a build-dependency and repeated-work progress

The promotion/backend-world path now shares a versioned runtime object among native-C, LLVM, and self-hosted application builds. The Hey-written self-host driver also reuses one LLVM-built emitter executable rather than generating a new compiler program per source. This does not remove `cc`/clang—the explicit bootstrap/link boundaries remain—but it removes hundreds of redundant invocations and makes the remaining dependency cost visible.

Promotion focused mode skips only native spec bodies already completed by the full suite. It does not suppress backend program builds or failure rows. `--exhaustive` remains available when repeated specs and unlimited builds are useful for diagnosis.

v0.99.200a dependency progress

  • The remaining `curl` call is narrower: URL parsing, HTTP wire format, response parsing, bounded text reads, and redirects now belong to `stdlib:http`. `curl` remains solely because the 1BRC payloads are redirected HTTPS binary archives and Hey lacks TLS plus byte-preserving net reads.
  • `codesign` and `xattr` are not removal targets. They are declared Darwin-only local-trust/signing boundaries and are bypassed for non-Darwin targets.
  • Linux container vendoring no longer fails by interpreting shell payload lines as Hey source.
  • Source handoff enumeration is Hey-owned and now uses canonical repository roots, preventing an overlaid application checkout from entering release archives.
  • `.hey/` caches are ignored recursively; generated dependency state no longer appears as accidental source.

Implementation-scope external commands therefore remain one migration target (`curl`) plus two accepted platform boundaries (`codesign`, `xattr`). Compiler/linker, Docker, and Git remain separately declared bootstrap/workflow boundaries.

No hidden host dependencies

Hey is intended to build and operate its maintained tools in Hey. A command that silently requires Python, Node, Ruby, Bash logic, `awk`, `sed`, `zip`, or `curl` is unfinished language/toolchain work unless it is an explicitly declared platform boundary.

Current status at v0.99.200a

Completed:

  • `hey package` no longer has a Python implementation. Its command body is Hey and release distributions carry an LLVM-built target executable.
  • `hey-zip-source` no longer invokes host `zip`; it delegates archive enumeration and deterministic packing to `tools/hey_source_archive.hey` plus `stdlib:archive`.
  • The dependency audit scans executable positions, preserves shell quoting, and ignores comments, prose, and heredoc bodies. It no longer turns `--nodes` option assignments into a false `node` dependency.

Still migration work:

  • Current implementation scope has one migration target, `curl`, plus the explicit macOS `codesign` and `xattr` boundaries.
  • Shell-owned release and performance orchestration.
  • Repeated `grep`, `awk`, `sed`, `find`, `sort`, `tail`, `rm`, `cp`, `mv`, and similar behavior inside maintained scripts.
  • `curl` in data-download tooling.
  • Test-only `unzip` inspection, which should eventually be expressible through a Hey archive-inspection tool even though it no longer owns production archive creation.

Allowed explicit boundaries

These may remain only when surfaced in help/distribution metadata and covered by skip/error behavior:

  • C compiler for current native bootstrap.
  • Clang/LLVM linker until no-host-toolchain executables are complete.
  • Docker as a temporary target toolchain container.
  • Git for source-control workflows.
  • macOS `codesign` and `xattr` for local trust.

Audit commands

./bin/heyc tools/hey_script_audit.hey --run
./bin/heyc tools/hey_script_audit.hey --run -- --details

Counts are an inventory, not a score. Each location must be classified as a thin launcher, migration-required behavior, or an explicit boundary.

Migration rule

Do not replace one hidden host language with another. When a script exposes a missing capability, add the reusable primitive at the correct core/runtime or stdlib layer, build the Hey command, switch the launcher, add a release prohibition, and then delete the old implementation.
