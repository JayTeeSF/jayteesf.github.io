Native heyc

`heyc` is the native Hey compiler/front door implemented in C. It runs source, specs, the REPL, the persistent test suite, and native C backend builds.

Current backend status

./bin/heyc --backends
./bin/heyc toolchain

Expected shape:

backend        status             selectable  notes
native-c       active/default     yes         optimized benchmark path; build default
typed-ir-c     proving-subset     yes         compiler-owned typed IR consumer for scalar/control/string/mixed user-call plus boxed data-model parse/update/collection/spread/control-flow subsets
self-hosted-c  full-parity-alt    yes         selectable with --backend self-hosted-c; native-C remains default/oracle
llvm           supported-subset   yes         scalar/float/string/control-flow/stdout + checked data-model/file/text/process/Result/text/collections/net/http runtime subset; requires clang today

`native-c` is the active/default backend. `typed-ir-c` is the compiler-owned proving backend for typed-IR consumption and currently covers scalar/control/string/module-call/mixed scalar-string mini-spec subsets plus an initial boxed data-model subset. `heyc ir` and `heyc build --emit-ir` expose the shared typed IR/ABI handoff for native-C, LLVM, and self-hosted convergence. LLVM is selectable for a checked subset with `--backend llvm`, including actor registry/join bridges and boxed numeric actor/job values through `value.i64`, but it is not the default and still requires clang until the no-toolchain distribution slice is complete. The self-hosted compiler is now selectable with `--backend self-hosted-c` after parser/build/spec/REPL parity and backend-world matched native-C; native-C remains the default/oracle while performance and distribution gates continue.

Build native apps

./bin/heyc build examples/hello.hey -o build/hello
./bin/heyc build --backend native-c examples/hello.hey -o build/hello
./bin/heyc build --backend self-hosted-c examples/hello.hey -o build/hello-selfhosted
./bin/heyc build --backend llvm examples/llvm_runtime_call_smoke.hey -o build/llvm-runtime-call
./bin/heyc build --cnative examples/hello.hey -o build/hello-cnative
./bin/heyc build --selfhosted examples/hello.hey -o build/hello-selfhosted
./bin/heyc build --llvm examples/llvm_runtime_call_smoke.hey -o build/llvm-runtime-call
./bin/heyc build --emit-c examples/hello.hey -o build/hello.c
./bin/heyc build --object examples/hello.hey -o build/hello.o
./bin/heyc build-dir examples -o build/examples-native

`heyc build` defaults to optimized native C backend apps. `--backend native-c` / `--cnative` is the explicit native default. `--backend self-hosted-c` / `--selfhosted` selects the promoted Hey-written compiler path. `--backend llvm` / `--llvm` is a checked subset path for scalar/float/string/control-flow/stdout, module-qualified helper functions, data-model, file/process/text facade fixtures, actor registry/join, boxed numeric actor/job values via `value.i64`, and requires clang in maintainer/source checkouts today.

Environment backend selection

For one-binary workflows, set `HEY_BACKEND` instead of using wrapper scripts:

HEY_BACKEND=native-c ./bin/heyc build examples/hello.hey -o build/hello-cnative
HEY_BACKEND=selfhosted ./bin/heyc build examples/hello.hey -o build/hello-selfhosted
HEY_BACKEND=llvm ./bin/heyc build examples/llvm_runtime_call_smoke.hey -o build/llvm-runtime-call

Explicit `--backend`, `--cnative`, `--selfhosted`, and `--llvm` flags still win over `HEY_BACKEND`.

Distribution intent

The product direction is one installed Hey distribution that can compile Hey programs without users manually managing Ruby, make, `cc`, or LLVM. Maintainers may still need development toolchains while building Hey itself.

LLVM toolchain status

`heyc toolchain` reports the current host/backend truth. Today the LLVM backend is selectable for the checked scalar/float/string/control-flow/stdout plus growing data-model/file/text/process subset, but executable LLVM builds still require maintainer clang. Release bundles include a `lib/hey/toolchain/STATUS.md` marker so users can see that the no-toolchain distribution gate is still pending rather than silently depending on a hidden runner or undocumented compiler.

`heyc build-dir --backend llvm <dir> -o <out_dir>` batch-builds the same supported subset and fails honestly on unsupported Hey constructs.

Canonical symbols

Use `heyc symbols <file.hey>` to print the compiler-facing canonical symbol table. Module-qualified source names keep `source_name=Module.fn` while the canonical key uses the backend-safe `Module__fn` spelling. This is the Track B integration surface for native-C, self-hosted, LLVM, and REPL convergence.

typed-ir-c backend

`heyc build --backend typed-ir-c <file.hey> -o <app>` builds the current compiler-owned typed IR consumer subset. The subset is intentionally small: unboxed scalar locals/functions/control-flow plus typed string user-call paths, including module-qualified calls that consume canonical backend-safe symbols. It now also has a boxed data-model path for parse/update/collection/spread/control-flow subsets through the runtime ABI.
