Object field access

Hey objects use JavaScript-style dot access for ordinary fixed field names. Bracket access remains the general form for dynamic keys, numeric indexes, punctuation, and names that collide with built-in value properties or methods.

Ordinary fields use dots

let user = {
  name: 'Ada',
  profile: {city: 'San Francisco'},
  friends: [{name: 'Grace'}],
}

says user.name
says user.profile.city
says user.friends[0].name

For an identifier-like field, `object.key` is syntax sugar for `object['key']`. The forms have the same missing-field behavior and preserve immutable value semantics.

Dot access may follow any supported object-producing expression:

says ({name: 'Ada'}).name
says users[0].name
says load_user().profile.city

This is object lookup syntax, not JavaScript prototype lookup. Hey does not walk prototypes or bind a JavaScript-style `this` value.

When brackets are required

Use brackets when the key is computed:

let field = 'name'
says user[field]

Use brackets for keys that are not identifiers:

let response = {'display-name': 'Ada', 'content type': 'text/plain'}
says response['display-name']
says response['content type']

Array indexes remain bracketed:

says user.friends[0].name

Built-in properties and methods take precedence

Hey already uses property and method syntax for collection/value operations. These names retain their built-in meaning after a dot:

len  length  size  count
keys  values  entries  items  pairs
type  to_json  json_canonical_text
get  at  put  push  has  has?  del  delete  sort

Additional standard runtime methods may also reserve their dot-call spelling. This keeps expressions such as these unambiguous:

says user.length
says user.keys.length
says user.to_json
says user.has?('name')

A data object may still contain a field with one of those names. Use the canonical bracket form to access it:

let record = {name: 'Ada', length: 'full', type: 'person'}

says record.name       # Ada
says record.length     # 3, the object length
says record['length']  # full
says record.type       # object
says record['type']    # person

The bracket form is therefore the explicit escape hatch for collisions. A future built-in may claim another dot spelling, but bracket access remains stable for data keys.

Immutable updates

Field access does not make objects mutable. Use object spread to produce an updated value:

let user = {name: 'Ada', visits: 2}
let next = {...user, visits: user.visits + 1}

says user.visits  # 2
says next.visits  # 3

Assignments such as `user.visits = 3` and `user['visits'] = 3` are not object mutation syntax.

Compiler contract

Dot field access has one source meaning. The product compiler records that meaning in semantic HIR, lowers it to ABI MIR, and emits direct LLVM without exposing generated symbols, LLVM slots, or runtime object helpers to application code.

See Reserved words and target-safe names for the distinction between Hey syntax words and implementation-only reserved identifiers.

Preferred lookup order

For a fixed ordinary field, write `object.attribute`. Use `object['attribute']`

when the key is dynamic, punctuated, numeric, or collides with built-in

property/method sugar. Keep `get(object, 'attribute')` for compatibility and

low-level generic code, not as the public teaching style.

says user.name
says user['method_with_reserved_word_name']
