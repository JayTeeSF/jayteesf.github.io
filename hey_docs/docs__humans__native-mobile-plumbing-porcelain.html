Native mobile plumbing and porcelain

This document decides where reusable iOS, iPadOS, and Android capabilities

belong. It is deliberately conservative about language-core changes.

Decision rule

language core      only semantics that libraries and host tooling cannot express
runtime plumbing   target-neutral ABI, lifecycle, scheduling, and opaque handles
stdlib             stable application-neutral workflows with multiple consumers
Hey CLI porcelain  build, sign, install, launch, inspect, and receipt workflows
third-party package reusable platform or UI behavior with an independent cadence
application        product UI, schema, credentials, policy, store metadata, releases

Syntax is scarce. Native-app work must first try existing functions, modules,

actors, workers, Jobs, streams, packages, and opaque native capabilities. A

mobile feature does not justify new syntax merely because the host SDK exposes a

new class, callback, annotation, or lifecycle method.

Language core: no new mobile syntax

The current iOS evidence and Android foundation do **not** justify new language

syntax. Hey already has the necessary semantic building blocks:

  • functions and modules for application policy;
  • immutable values for manifests, commands, and messages;
  • actors for private long-lived state and identity;
  • stateless workers for parallel pure or isolated handlers;
  • Jobs for retries, deadlines, backpressure, partitioning, durability, and

supervision;

  • worker-backed streams for bounded transformations;
  • opaque native handles and the package ABI for host resources;
  • semantic HIR, ABI MIR, and direct LLVM target lowering.

A future core change requires an application-neutral compiler/runtime problem

that cannot be expressed through these facilities. UIKit, Android Views,

Jetpack Compose, Keychain, Keystore, biometrics, notifications, and store policy

are not language semantics.

Compiler and runtime plumbing

Hey owns the narrow target-neutral machinery that packages cannot safely fake:

  • target triples, pointer width, calling convention, object format, and ABI MIR;
  • direct LLVM object emission for device and simulator/emulator targets;
  • deterministic native symbol export and callback descriptors;
  • target framework/library metadata carried through package manifests;
  • opaque host handles with explicit ownership and lifecycle;
  • safe host-to-Hey callback entry and managed-runtime attachment;
  • scheduler wake-up and cancellation hooks that preserve actor/worker/Job rules;
  • main-thread handoff as a general host-executor capability if both iOS and

Android prove the same requirement;

  • application lifecycle events only as generic messages/capabilities after both

platforms expose a common contract;

  • machine-readable build, link, signing, install, launch, and publication

evidence boundaries.

Platform SDK objects must remain outside the Hey value model. Plumbing may carry

an opaque capability or validated descriptor; it must not expose Objective-C,

Swift, Java, Kotlin, JNI, or raw pointers as ordinary Hey values.

Standard library: promote only proven cross-platform workflows

Existing application-neutral stdlib facilities should be reused on mobile:

  • `stdlib:Crypto`, `stdlib:Http`, `stdlib:Tls`, `stdlib:Time`;
  • actors, workers, Jobs, streams, cancellation, and results;
  • bytes, JSON/JSONL, files where the platform sandbox permits them;
  • synchronization primitives such as field revisions and conflict metadata.

Do not add `stdlib:uikit`, `stdlib:android`, or store-specific modules. A future

`stdlib:app` or lifecycle facade is possible only after at least iOS and Android,

and preferably a second application, demonstrate one stable target-neutral

contract. Until then, lifecycle and UI remain package APIs.

Hey CLI porcelain

The language repository may provide small target-neutral developer workflows:

hey mobile TARGET doctor
hey mobile TARGET sdk
hey mobile TARGET native-app
hey mobile TARGET device-app
hey mobile TARGET archive

These commands may discover SDKs, select sysroots, invoke the ordinary compiler,

link, sign, install, launch, and emit receipts. They must not contain product

screens, product schemas, credentials, privacy copy, App Store/Play listing

text, or application release policy.

The public command remains porcelain. Detailed staging, compiler flags, SDK

copying, signing inspection, and device-tool invocation stay hidden below

`bin/subcmds` and `tools` as plumbing.

Third-party package candidates

The preferred extraction order is package-first:

`hey_mobile`

A target-neutral package candidate for:

  • application lifecycle messages;
  • navigation and screen/view descriptors;
  • commands, actions, validation, and accessibility metadata;
  • state restoration contracts;
  • platform capability interfaces;
  • test doubles and deterministic host receipts.

It should be ordinary Hey wherever possible and use actors/Jobs rather than

inventing another event/channel model.

`hey_ios`

An independently versioned native package candidate for:

  • UIKit host/view rendering;
  • iOS/iPadOS lifecycle integration;
  • Keychain, LocalAuthentication, protected files, notifications, clipboard, and

background-task adapters;

  • Apple framework link metadata and entitlement helpers.

It may use Objective-C/Swift behind the native package ABI. It must expose

opaque capabilities and ordinary Hey values, not Apple objects or pointers.

`hey_android`

An independently versioned native package candidate for:

  • Activity/application lifecycle integration;
  • Android Views or Compose host rendering;
  • Keystore, BiometricPrompt, protected storage, notifications, clipboard,

WorkManager/background adapters, and intent handling;

  • NDK/JNI/Gradle link and packaging metadata.

It must not require language annotations or Kotlin/Java classes in application

source. Those remain package-owned host implementation details.

Additional packages

Authentication middleware, encrypted storage, synchronization protocols, and

store/deployment automation should remain separate packages when they are

reusable. `vss_sync` is the model: protocol logic is independently versioned,

while VSS product policy stays in the application and server repositories.

Application repository ownership

A native application owns:

  • product screens, navigation, copy, icons, and accessibility decisions;
  • persistence schema, account model, security policy, and recovery UX;
  • bundle/application identifiers, entitlements, privacy declarations, and store

metadata;

  • signing identities, credentials, release cadence, TestFlight/internal testing,

Play tracks, and review receipts;

  • application-specific native code until it is reduced to a reusable package.

Generated application output remains disposable. The source repository is the

application, not the generated `.app`, APK, AAB, IPA, Gradle, or Xcode output.

Concurrency on mobile

Use Hey's existing model on every target:

  • actor: application session, navigation coordinator, account, vault, or other

persistent private state;

  • partitioned Job: per-account/per-document mutation and synchronization;
  • worker: validation, decoding, encryption-independent transformations;
  • bounded stream: import, migration, backup, restore, and batch synchronization;
  • Job policy: retries, deadlines, idempotency, backpressure, dead letters, and

cancellation.

Native UI callbacks should enqueue ordinary messages or Jobs and return quickly.

Do not expose public channels or execute blocking platform work on the UI thread.

Promotion criteria

Move a capability inward only with executable evidence:

1. one application proves the problem and keeps the first solution locally;

2. a second consumer proves the reusable boundary;

3. extract an independently versioned package;

4. promote only the irreducible target-neutral plumbing to Hey;

5. promote package porcelain to stdlib only when stability and broad demand are

demonstrated;

6. change language syntax only when every library/package approach is inadequate.

Current native target evidence

  • direct native iOS application through Mach-O/UIKit: passed;
  • development-signed iOS physical-device porcelain: structurally passed, target receipt pending;
  • ordinary Hey direct LLVM Android application: passed;
  • native Android navigation/list/form/secure-field/accessibility host: passed;
  • Android multi-ABI ELF matrix and JNI callback registration: passed;
  • Android Gradle/APK/device execution: target-host receipt pending;
  • store signing and publication: application-owned and pending.

Next implementation order

1. Keep the retired non-native mobile surface deleted.

2. Run the generated Android view-stack project through NDK/Gradle and retain a target receipt.

3. Prove equivalent iOS and Android lifecycle/callback/main-thread needs.

4. Extract `hey_mobile`, `hey_ios`, and `hey_android` package candidates with

their own source-ZIP, tests, docs, and release cadence.

5. Add only package-neutral compiler/runtime fixtures to Hey.

6. Consider stdlib promotion after multiple applications use the package APIs.
