Hey developer documentationAudience-specific Hey documentation

Vendor stdlib and command

`stdlib:Vendor` owns Hey distribution vendoring. The command-line entry point is:

./bin/hey vendor --repo ../my_app

The launcher is intentionally thin. It sets the Hey source root and runs the stdlib module:

./bin/heyc stdlib:Vendor --run -- "$@"

That keeps argument parsing, target selection, copying, verification, and container cross-build planning in Hey instead of a growing shell script.

What vendoring does

For a target app repository:

./bin/hey vendor --repo ../elders_prayer_app

Hey copies the current platform dist into:

../elders_prayer_app/vendor/hey/<target>/

For an exact vendor root:

./bin/hey vendor --vendor-dir ../elders_prayer_app/vendor/hey

Hey copies into:

../elders_prayer_app/vendor/hey/<target>/

A successful run prints:

vendored Hey 0.99.xa for darwin-arm64
source: /path/to/hey/dist/hey-0.99.xa-darwin-arm64
dest:   /path/to/app/vendor/hey/darwin-arm64
stdlib:Web present

Options

--repo PATH             Target app repo root.
--vendor-dir PATH       Exact vendor root.
--target TARGET         linux-x64, linux-arm64, darwin-arm64, darwin-x64.
--no-build              Copy an existing dist instead of running make dist-host.
--minimal               Copy only bin, stdlib, Version.hey, and README.md.
--container VALUE       auto, docker, podman, or off.
--container-image IMG   Default debian:bookworm-slim.
--help                  Print usage.
--version               Print Hey version.

Use `--repo` or `--vendor-dir`, not both.

Current host vendoring

On an M3 Mac:

cd "$HOME/dev/hey-lang-bootstrap-plan"
./bin/hey vendor --repo "$HOME/dev/elders_prayer_app"

cd "$HOME/dev/elders_prayer_app"
vendor/hey/darwin-arm64/bin/heyc --version
make check
git status --short
git add vendor/hey
git commit -m "Revendor Hey $(vendor/hey/darwin-arm64/bin/heyc --version) darwin-arm64"
./bin/update_llm

This populates:

vendor/hey/darwin-arm64/

Linux x64 vendoring from macOS

When `--target` differs from the host target and the target is Linux, `stdlib:Vendor` can run the vendoring command inside Docker or Podman with a matching `--platform`.

cd "$HOME/dev/hey-lang-bootstrap-plan"
./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target linux-x64 \
  --container auto

`auto` prefers Docker and falls back to Podman. Linux x64 uses `linux/amd64`; Linux arm64 uses `linux/arm64`.

The container installs the source-build dependencies in `debian:bookworm-slim`, builds the Linux distribution with `make dist-host`, verifies that distribution inside the target container, and only then publishes it to the host app repository.

The release build uses the same current LLVM pointer mode as an ordinary Hey build. Vendoring does not force an older LLVM compatibility mode into `hey-selfc`; the release compiler, semantic HIR emitter, ABI MIR lowering, and target verification all use one tested LLVM contract.

This is good for producing a runnable vendored Linux binary. It is not a trustworthy performance benchmark environment.

Verify from the app repo:

cd "$HOME/dev/elders_prayer_app"
ls vendor/hey/linux-x64/bin
file vendor/hey/linux-x64/bin/heyc
find vendor/hey/linux-x64/stdlib -maxdepth 1 -type f | sort | tail

git status --short
git add vendor/hey
git commit -m "Revendor Hey linux-x64"
./bin/update_llm

The `file` output should say `ELF 64-bit ... x86-64` for `linux-x64`.

Stdlib API

import 'stdlib:Vendor'

Vendor.usage()
Vendor.host_target()
Vendor.host_target_from('Darwin', 'arm64')
Vendor.platform_for_target('linux-x64')
Vendor.plan(['--repo', '../app'])
Vendor.run(['--repo', '../app'])

`Vendor.plan(args)` is pure apart from host detection. It returns a Result object containing the target, source dist, destination, copy mode, and container settings.

`Vendor.run(args)` performs the build/copy/verification workflow and returns a Result object with printable messages.

Verification

After copying, vendoring writes a tiny program that imports `stdlib:Json` and runs it with `HEY_STDLIB_PATH` pointing at the vendored stdlib. This catches broken launcher, stdlib, or import-path layouts before the command reports success.

Should vendoring copy docs?

Yes by default. A full vendored dist includes `docs/`, examples, specs, runtime files, tools, stdlib, and binaries. That is the best default for app repos that will be edited by humans and LLMs because the vendored tree is self-explaining.

Use `--minimal` only when you intentionally want the smallest runnable footprint:

./bin/hey vendor --repo ../app --minimal

Minimal mode copies only `bin/`, `stdlib/`, `Version.hey`, and `README.md`.

macOS Gatekeeper and local trust

`Known Developer` is an Apple distribution property. To get that label for people outside your machine, sign with an Apple Developer ID certificate and notarize the distributed artifact with Apple. Local vendoring cannot fake Developer ID or notarization.

For app repos on your own Mac, `hey-vendor` can still make the vendored Darwin binaries easier to run:

./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target darwin-arm64 \
  --macos-trust local

`--macos-trust local` removes the `com.apple.quarantine` attribute from the vendored tree and ad-hoc signs the vendored Hey command-line binaries with:

codesign --force --sign - PATH

For real Developer ID signing, pass the certificate identity explicitly:

./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target darwin-arm64 \
  --macos-sign-identity "Developer ID Application: Your Name (TEAMID)"

That signs the vendored binaries with the provided identity. Full public distribution should still add notarization and stapling in a release packaging workflow.

Use `--no-macos-trust` when you want the vendored bits copied exactly without local quarantine or signing changes.

macOS local trust without Developer ID

Apple's public "Known Developer" path requires a Developer ID certificate and notarization. For JayTeeSF's local app repos, use local trust instead:

./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target darwin-arm64 \
  --macos-trust local

`--macos-trust local` removes `com.apple.quarantine` from the vendored Darwin tree and ad-hoc signs command-line binaries with `codesign --sign -`. This identifies the binaries as locally signed for that machine, but it does not make them appear as an Apple "Known Developer" to other Macs.

If JayTeeSF later creates or installs a local/private signing certificate in Keychain, pass its exact identity with `--macos-sign-identity`. If he later joins the Apple Developer Program and obtains a Developer ID certificate, pass that same option with the Developer ID identity and notarize the distributed artifact before sharing it publicly.
