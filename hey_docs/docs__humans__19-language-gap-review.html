v0.99.211a language/conformance update

The repository now distinguishes the Hey language from official compiler implementation details. `hey-language-conformance-v1` is outside-in and backend opaque. Current gaps in that suite are breadth, negative diagnostics, actors/jobs, modules/packages, I/O, and concurrency—not LLVM structure.

Canonical typed IR is tracked separately as an optional implementation contract. Its core tier is executable; imports/modules, complete boxed typing, actors/jobs, source spans, and effects remain.

v0.99.210a language/tooling update

No new language syntax is introduced. The compiler CLI now exposes two previously implicit dimensions: `hey-selfc --compiler-backend native-c|llvm` selects how the Hey-written compiler executable runs, while `--output-backend c` names its current output. LLVM output is an explicit unsupported architectural gap rather than a hidden native delegation.

v0.99.209a language-surface update

Functions and actors may now end in `?` or `!` across top-level,

module-qualified, REPL, native-C, self-hosted-C, typed-IR, and LLVM paths. The

suffix is part of the name and carries convention rather than implicit

behavior. Importing `stdlib:collections` enables the receiver-oriented

collection surface while preserving direct module calls and

`collections.range(...)`.

v0.99.208a language-surface update

Fixed object fields now have first-class `object.key` syntax with brackets retained for dynamic, punctuated, numeric, and built-in-collision keys. This closes the prior ergonomic gap without changing immutable update semantics. On LLVM, all 58 rows classified unsupported in the v0.99.205a report now emit/link in the focused build set; direct runtime ABIs and execution parity remain open.

v0.99.201a ergonomics and report-quality update

The ergonomic syntax is now the public teaching contract, not an optional appendix. Public Markdown code fences and shipped examples use bracket access, spread literals, `.length`, `.keys`, and `.to_json`; a Hey-owned docs-smoke checker rejects reintroduction of standalone collection `get(...)`, `len(...)`, or `to_json(...)` calls. HTTP APIs named `get` remain ordinary module calls.

Compiler/report work also closed three facade-drift gaps found by the promotion report and replaced per-row runtime/emitter compilation with versioned shared artifacts. The next language gap is self-host import reachability: the Hey-written emitter currently emits every imported function, so an unused unsupported function can reject an otherwise supported program.

v0.99.187a documentation boundary

Public language guides now describe current behavior only. Version telemetry, backend evidence, defects, and unfinished work remain in maintainer records such as this file and are excluded from the generated HTML site.

v0.99.186a

  • Fixed the REPL/language inconsistency where `says expr` worked as a statement but `says(expr)` fell through to ordinary function lookup and failed with `unknown function: says`.
  • `says(expr)` is now a parenthesized statement alias in direct REPL input, functions, actors, program blocks, the native-C compiler, self-hosted-C parser/compiler path, typed IR, and LLVM lowering. No import is required.
  • Added language spec 854 and REPL transcript 210, and extended release C to prove native-C, self-hosted-C, LLVM, function, actor, conditional, and program-block behavior. Existing `says expr` syntax remains canonical and unchanged.

v0.99.185a

  • `web.serve(routes, options)` is now the live HTTP service boundary: it binds a loopback/TCP listener, accepts HTTP/1.1 requests, dispatches through ordinary `stdlib:web` routes, writes responses, and supports finite `max_requests` specs. The runtime uses a poll-backed worker pool with default workers based on available CPU cores so I/O is not pinned to one core.
  • LLVM now lowers the same live `web.serve` path through an ordinary runtime ABI shim with generated handler dispatch, pinned by specs 851-852.
  • `./bin/hey-zip-source --prompt` now reminds JayTeeSF how to regenerate the public docs site and revendor both `darwin-arm64` and Linux x64 Hey distributions into `elders_prayer_app`.

v0.99.184a

Cleaned README and docs README so version history stays in release notes instead of user entry docs; fixed hey-doc-site default curation, Markdown list/table rendering, and generated Markdown links; added macOS vendoring trust options for quarantine removal, ad-hoc signing, and optional Developer ID signing; refreshed the LLVM runtime docs catalog so docs.has/docs.symbols match stdlib:hey_docs; pinned docs/vendor/catalog LLVM coverage with specs 847-850.

v0.99.183a - docs-site links and LLVM net facade slice

  • Fixed `hey-doc-site` Markdown link rewriting: generated HTML now turns relative `.md` references such as `tutorial.md` and `00-quick-start.md` into links to the generated static pages.
  • `stdlib:docs` now resolves links relative to the current Markdown document and includes directly linked user-facing docs when they are not part of the internal exclusion list.

  • Added specs 844, 845, and 846 to pin docs-site link generation, LLVM `net.ping_clients` parity, and Jobs-backed docs-site rendering.

v0.99.182a update: fixes the v0.99.181a follow-up bugs and moves LLVM coverage forward in clusters. LLVM now lowers triple-quoted interpolation as real interpolated string expressions, delegates the expanded docs catalog through stdlib:hey_docs instead of stale static catalog data, and lowers the pure stdlib:web dispatch/response helpers, stdlib:vendor planning helpers, stdlib:docs site-planning helpers, and process.run_env_argv through ordinary runtime ABI shims rather than treating them as unsupported.

v0.99.181a update: adds language visibility and docs-site publishing before returning to LLVM. `private fn` hides same-source/module implementation helpers from importers, and `private import` lets a file use an implementation dependency without leaking that transitive module to downstream importers. `stdlib:docs` and `./bin/hey-doc-site` generate user-facing static HTML docs from README/tutorial/stdlib docs with maintainer/LLM/backend handoff docs excluded by default. `docs/developers/stdlib/web.md` now explains route dispatch internals, service/concurrency descriptors, Jobs/actor guidance, and an EHY form example.

The slice also adds `process.run_env_argv`/`command_env_argv` so verification can pass environment overrides without shell env-prefix tricks, and `heyc` can run stdlib entrypoints such as `stdlib:vendor` directly.

v0.99.179a update: promotes `hey/web` into the standard library as `stdlib:web`. The slice adds pure web request/router/response helpers, route params, JSON body helpers, file responses via `static_files`, middleware descriptors with request IDs, recovery, max-body, JSON validation, secure headers, service/serve descriptors defaulting to port 3072, docs/signature metadata, and a web-first tutorial link to `docs/developers/stdlib/web.md`. It also keeps the scheme import resolver path (`stdlib:name`, `std:name`, `pkg:name@version/module`) visible for stdlib/package/local modules.

v0.99.179a language gap note: web tutorials now expose one first-class handler-reference pressure point. Native-C and interpreter paths support module function references such as `HealthController.show` as handler symbols for `web.dispatch`; full language-level first-class functions/closures, custom middleware invocation, aliases/from-imports, and re-exports remain deliberate future design work.

v0.99.177a update: continues LLVM compatibility/performance by lowering ordinary C10K (`c10k_ping`/`net.c10k`), file_partitions, file-backed 1BRC station formatting, and boxed get-plus aggregation through normal runtime/data-model ABI paths. Specs/IR 822 and 823 pin the C10K/file-partition/file-backed-1BRC and 1BRC-style aggregate cases; `benchmarks/hey/one_brc_style_aggregate.hey` now builds/runs under LLVM without benchmark program-shape shortcuts. Latest pasted baseline before this slice was LLVM correctness=523/538 and good=507/538 with 15 unsupported rows; rerun promotion-report to measure the new denominator.

v0.99.177a LLVM gap note: file-backed 1BRC exposed a real language-surface gap, not a benchmark special case. Object spread updates containing `stat['sum'] + temp`, dynamic string keys, `sort(keys(stats))`, and string-returning helpers with mixed numeric/string concatenation now lower under LLVM. The remaining gap review should focus on the final tooling rows and full promotion-report deltas.

v0.99.176a LLVM/challenge note: challenge workloads must not be pre-rejected by policy. Public runtime functions such as one_brc_summary/generate/sample are part of the language runtime surface and may lower through ordinary ABI shims; what remains forbidden is recognizing benchmark-specific source shapes or substituting canned challenge answers.

Language gap review for CLI usability

This is a working review for making Hey comfortable as a fast command-line language that can replace common Perl/Awk/Sed/Ruby/Python scripts without losing Hey's compiled-runtime and actor strengths.

Snapshot

Hey already has useful CLI foundations: `argv()`/`cli.args()`, safe argv process execution, env helpers, JSON/array/object values, Result records, filesystem/path/text helpers, regex-lite, streaming file primitives, jobs, actors, modules, and native/self-hosted parity. The remaining concern is less "can Hey express programs?" and more "can a user write short, obvious, safe scripts without dropping into plumbing?"

Language concerns to address or discuss

1. **Constants / immutable bindings.** Hey has `let` plus `set`/assignment, but no dedicated constant declaration. Add a roadmap item for `const NAME = value` and/or module-level immutable exports, with backend/parser/REPL semantics and diagnostics for reassignment. This is important for scripts, package manifests, configuration, and generated docs.

2. **Import ergonomics and scope clarity.** Current imports are explicit and module-qualified, which is good for readability and backend lowering. Before beta, evaluate Python-like conveniences (`from`, `as`, explicit re-export lists) with canonical-symbol collision diagnostics. Do not introduce mutable globals or implicit root-scope injection as a shortcut.

3. **Pipelines and streaming text ergonomics.** Hey has file/process/runtime primitives, but a practical CLI needs pleasant line iteration, stdin/stdout filters, process pipelines, glob/walk, and low-allocation transforms over large files.

4. **Regex-lite vs. full scripting regex.** Regex-lite is useful, but Perl/Ruby/Python replacement needs captures, capture-aware replacement, escape classes, flags, and a clear compromise document if full regex is deferred.

5. **Exit/status ergonomics.** Result values are good for library boundaries, but CLIs need first-class usage errors, exit codes, stderr formatting, and `--help` generation that stays tied to docs metadata.

6. **Numeric and byte/string lanes.** Fast CLI work needs predictable int/fixed/string/bytes performance under LLVM without accidental boxing in hot loops. The language can keep boxed values as the universal data model, but typed lanes need a visible roadmap.

7. **Syntax for common data access.** Bracket access works; dot-field access remains a possible readability win for object-like records if it can be kept explicit and non-magical.

8. **Package/app distribution.** A script replacement needs single-binary or no-toolchain packaging, fast startup, pinned package locks, and safe local/remote install behavior.

9. **Error recovery syntax.** Result boundaries are explicit today. Before adding rescue/try syntax, keep proving whether Result helpers plus CLI usage helpers are enough.

10. **Tooling/introspection.** `./bin/hey doc SYMBOL` and `./bin/heyc --describe SYMBOL` are a good start. REPL help and generated API references should consume the same metadata so examples show signatures, argument shapes, and return shapes.

Roadmap implications

  • Add constants/module immutable exports as a language-design item before beta, with specs across interpreter/native-C/self-hosted-C/LLVM and REPL behavior.
  • Add an import-ergonomics slice before beta: `from`/alias/re-export syntax, canonical-symbol collision diagnostics, and examples showing when explicit module prefixes remain preferable.
  • Build a CLI batteries track around `stdin.lines`, `files.walk/glob`, `process.pipeline`, capture-aware regex replacement, usage/exit helpers, and docs-derived help.
  • Keep performance gates honest: prove fast startup, streaming throughput, and low-allocation hot loops under native-C and LLVM before declaring Hey a serious replacement for mature scripting languages.
  • Avoid syntax churn unless a porcelain stdlib shape has proven insufficient.

v0.99.200a transport and property update

Closed language ergonomics gap: read-only core operations now support property notation. `value.len`, `.length`, `.size`, and `.count` lower to `len(value)`; collection views, type, and JSON text have equivalent read-only properties and may chain. Argument-taking and mutating operations remain normal calls.

Partially closed transport gap: `stdlib:http` now owns URL parsing, request serialization, response parsing, bounded plain-HTTP text requests, and redirects. Still missing before external binary HTTPS tools can be removed:

  • certificate-chain and hostname-validating TLS;
  • byte-preserving socket reads/writes;
  • bounded streaming directly to temporary files;
  • checksum verification and atomic destination publication around that stream.
