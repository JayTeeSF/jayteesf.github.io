Profile the managed scheduler

Hey's actors, stateless workers, Jobs, and parallel streams share one bounded managed scheduler. Performance work must begin with repeated measurements of the same ordinary Hey workloads rather than with workload-specific runtime shortcuts.

One command

./bin/hey-scheduler-profile \
  --max-workers 8 \
  --runs 5 \
  --out build/scheduler-profile

The command compiles four ordinary Hey workloads once and runs each across `1..max-workers`:

  • actor fan-out;
  • stateless worker fan-out;
  • worker-backed Jobs;
  • Jobs-backed streams.

It writes:

build/scheduler-profile/matrix.tsv
build/scheduler-profile/summary.tsv

`matrix.tsv` contains every run and every opt-in scheduler counter. `summary.tsv` contains the median elapsed time and item throughput for each workload/worker-count pair.

What is measured

Set `HEY_SCHEDULER_PROFILE=1` to emit one `HEY_SCHEDULER_PROFILE` JSON record at process shutdown. The record separates:

  • actor-registry lock wait, acquisitions, and scanned nodes;
  • mailbox lock wait, enqueues, and dequeues;
  • message-pool lock wait, allocations, reuse, and returns;
  • ready-queue lock wait, local pops, steal attempts, and successful steals;
  • actor turn count and handler runtime;
  • value clone/release calls, deep clones, and reference-counted clones;
  • scheduler sleeps, wake signals, queue depth, and processed turns.

The ordinary runtime path does not maintain these counters unless profiling is enabled.

Profile only selected workloads

./bin/hey-scheduler-profile \
  --workloads actor,worker \
  --max-workers 4 \
  --runs 7 \
  --actor-items 20000 \
  --worker-items 20000

Supported names are `actor`, `worker`, `job`, and `stream`.

Reuse already-built profile applications

Use `--skip-build` only when the output directory already contains `actor`, `worker`, `job`, and `stream` applications produced by this tool or its regression gate:

./bin/hey-scheduler-profile \
  --skip-build \
  --out build/scheduler-profile \
  --max-workers 8 \
  --runs 5

Interpret the results

Compare repeated medians, not a single best run. A higher worker count is not automatically better. For very small actor turns, extra queue locks, steals, wakeups, and value ownership work can exceed useful handler time.

Only promote a scheduler change when:

1. the profile identifies a specific dominant cost;

2. the change improves repeated results on relevant workloads;

3. one-worker behavior does not regress materially;

4. actor, worker, Job, and stream semantics remain unchanged;

5. sanitizer and two-host compiler gates remain green.

Guarded repository verification

Long pasted command blocks can interleave with a still-running build in an interactive terminal. Run the maintained sequence as one guarded command instead:

./bin/hey-verify-current --full

For a shorter build/profile/docs/archive check:

./bin/hey-verify-current --focused

Each phase runs synchronously and the script stops on the first failing command.

v0.99.255a contention policy

The v0.99.254a matrices showed that empty victim queues were still locked and that idle workers repeatedly scanned peers while useful turns were tiny. v0.99.255a retains four general changes:

1. each ready queue publishes an atomic depth hint;

2. a stealer skips a victim whose hint is zero, while the queue mutex remains the authoritative ownership boundary;

3. ordinary steal rounds inspect at most four peers in randomized order;

4. every eighth unsuccessful bounded round performs a full sweep, and failed rounds yield before retrying.

The profile record now distinguishes:

steal_attempts
steal_lock_attempts
steal_empty_skips
steal_successes
steal_probe_rounds
steal_full_sweeps
park_events
yield_events

`HEY_SCHEDULER_STEAL_PROBES` can temporarily lower the internal probe count for diagnostics. It is not a public language or deployment contract.

Retained alternating evidence

The old and new binaries were alternated for seven runs on the same Linux host with 2,000 items and ten rounds:

actor118.03 ms18.13 ms+0.6%

One-worker movement is within ordinary run noise; the multi-worker reductions are large enough to retain. Target-Mac evidence is still required before making broad platform claims.

Generation-2 managed scheduling — v0.99.256a

External submissions enter one global injection queue. A worker drains at most 32 actors, executes one, and appends the rest to its local queue. Worker-owned rescheduling never returns to the injection queue.

A stealer removes approximately half of one nonempty victim queue, capped at 32 actors. It executes one actor and publishes the remainder locally. Empty-depth hints, bounded randomized probes, periodic fairness sweeps, and failed-probe yielding remain in effect.

Scheduler status now exposes:

queue_model
injection_depth
injection_enqueued
injection_dequeued
running_workers
spinning_workers
parked_workers
max_spinning_workers
steal_batches
stolen_actors
injection_drains
state_transitions

The state counts are observational metrics, not a user-facing scheduling API. `HEY_SCHEDULER_PROFILE` remains opt-in.

Alternating seven-run medians on the current Linux host showed approximately flat one-worker actor time, a 1.5 percent one-worker worker regression, a 1.7 percent four-worker actor improvement, and a 1.3 percent four-worker worker improvement versus v0.99.255a. These are local measurements, not broad platform claims.
