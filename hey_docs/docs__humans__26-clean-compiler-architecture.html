v0.99.216a implementation checkpoint

The clean architecture is now enforced around two official producer hosts and one product consumer: C-hosted Hey and LLVM-hosted Hey must emit equivalent HIR/MIR, then direct LLVM consumes MIR without reopening source. The examples tree is producer-complete (88/88) and direct-LLVM-build complete (87 applications plus one library). The next architecture gate is complete execution/effect parity and explicit block-based MIR.

Clean compiler architecture: source, sidecars, HIR, MIR, and backends

This document is the implementation plan for correcting Hey's compiler

architecture while the language is still alpha. It is a maintainer contract,

not part of the backend-opaque language specification.

Decision summary

Starting from scratch, Hey should use this pipeline:

.hey source
  -> lexer/parser
  -> resolved AST
  -> semantic HIR
  -> verified MIR
  -> C consumer / LLVM consumer / future consumers
  -> object and executable

The rules are:

1. `.hey` source is the only human-authored semantic source of truth.

2. Optional source annotations describe Hey-language semantics, not LLVM or C

representation details.

3. Generated sidecars may accelerate compilation or expose module interfaces,

but deleting them must never change acceptance, diagnostics, or output.

4. Frontends stop after semantic HIR. They do not contain private LLVM

lowering rules.

5. HIR-to-MIR lowering is shared and verified.

6. C, LLVM, and future backends consume MIR. They do not reparse Hey source or

re-infer source semantics.

7. The portable language conformance suite stays outside-in and backend-opaque.

8. Canonical IR is an optional interoperability contract, never the language

definition.

What “sidecar” means

The term is overloaded. In this repository, it has referred to a **metadata

file associated with a source module but stored outside the `.hey` file**.

For example:

src/cards.hey
.hey/interfaces/<content-hash>/cards.hyi
.hey/cache/<compiler-id>/<content-hash>/cards.hir

A sidecar can contain generated information such as:

  • exported symbol names and signatures;
  • source and dependency hashes;
  • resolved module identities;
  • inferred semantic facts;
  • serialized HIR or MIR;
  • compiler/version/target cache keys.

This is different from a “sidecar process” or container. The clean compiler

plan does not require a compiler daemon or auxiliary process.

Safe sidecar rule

A generated sidecar is safe only when all of these are true:

  • it is produced from source plus explicit compiler inputs;
  • it is content-addressed or otherwise strongly invalidated;
  • it is validated before use;
  • it can be deleted at any time;
  • a cache miss causes re-analysis, not different semantics;
  • it is never hand-edited to make a program compile;
  • it is never required to distribute ordinary source packages.

Unsafe sidecar rule

Do **not** put required type declarations or semantic overrides in an invisible

`.hey/` file. That creates two source languages: the visible `.hey` program and

an implicit metadata program. It makes builds non-reproducible, confuses tools,

and allows stale facts to silently change code generation.

If a fact affects program meaning, overload resolution, public API, or an error

message, it belongs in source syntax or in a declared package/module manifest.

Architecture options

Option 1: each backend compiles Hey source directly

Hey source -> native-C frontend/backend
Hey source -> LLVM frontend/backend
Hey source -> another backend

**Advantages**

  • fastest way to bootstrap one backend;
  • few intermediate data structures at first.

**Disadvantages**

  • parser and semantic rules are duplicated;
  • every backend invents its own type inference and diagnostics;
  • source bugs appear differently by backend;
  • compiler implementations cannot be compared at a stable boundary;
  • deleting obsolete lowering becomes very difficult.

**Decision:** transitional only. Do not add new source-specific LLVM inference

when the equivalent HIR/MIR path exists.

Option 2: one canonical IR does everything

Hey source -> one textual IR -> LLVM/C

**Advantages**

  • creates a source-independent handoff quickly;
  • easier to inspect than hidden compiler state;
  • useful for early producer parity.

**Disadvantages**

  • semantic concepts and machine/ABI concepts become mixed;
  • one format is forced to serve parsing, type analysis, optimization,

serialization, and code generation;

  • changes needed by LLVM can leak backward into source annotations;
  • structured control flow, actors, effects, and diagnostics become awkward.

**Decision:** `hey-ir 0.2` is a valuable bootstrap artifact, but it is a

transitional combined tier. Split it before widening it into the entire

language.

Option 3: required inferred-type sidecars

Hey source + hidden inferred-type file -> backend

**Advantages**

  • avoids writing types in source;
  • can reduce repeated analysis.

**Disadvantages**

  • hidden second source of truth;
  • stale or missing sidecars change behavior;
  • poor portability for third-party implementations;
  • difficult editor, package, and CI semantics.

**Decision:** reject as a language architecture. Inferred facts may be cached,

but the compiler must be able to reproduce them from source.

Option 4: semantic source annotations, generated interfaces, HIR, and MIR

Hey source
  -> implementation-specific frontend
  -> shared semantic HIR
  -> shared verified MIR
  -> independent backend consumers

**Advantages**

  • source meaning is explicit and stable;
  • multiple frontend implementations can be compared at HIR;
  • multiple backends can be compared at MIR;
  • generated interfaces permit fast incremental compilation without hidden

semantics;

  • LLVM details stay out of the language specification;
  • Rust, Java, JVM, WebAssembly, or other implementations can either own the

entire pipeline or optionally reuse official IR consumers.

**Decision:** this is the target architecture.

Source types versus compiler types

The current alpha implementation accepts provisional annotations such as:

fn emit(tree: boxed) -> str
fn count(xs: boxed) -> i64

Those names helped expose ABI ambiguity, but `boxed`, `str`, and raw machine

widths are compiler-oriented terms. Starting from scratch, source annotations

should use language-semantic names, for example:

fn emit(tree: Value) -> String
fn count(xs: Array) -> Int
fn ready?(value: Bool) -> Bool

The source type and its lowering are separate decisions:

Hey source type     semantic HIR type     MIR/ABI representation
Int                 int                   i64 initially
Float               float                 f64 initially
Bool                bool                  i1/i64 according to ABI
String              string                runtime string reference
Value               dynamic               boxed Hey value
Nil                 nil                   canonical nil value

The language may later add fixed-width numeric types deliberately, but they

must be specified as user-visible semantics rather than introduced merely to

make an LLVM signature compile.

Required alpha correction

Before optional annotations spread through public packages:

1. freeze expansion of the provisional lowercase annotation surface;

2. decide and document semantic source type names;

3. migrate compiler implementation code and conformance fixtures;

4. keep low-level names in MIR and runtime ABI schemas, not ordinary source;

5. add an explicit compatibility/removal note because this is alpha software.

Annotations remain optional unless Hey deliberately adopts a static typing

mode. Unannotated functions use ordinary semantic analysis and dynamic `Value`

operations where facts cannot be proven.

The two IR levels

Hey HIR: semantic compiler boundary

HIR represents resolved Hey meaning without committing to a target ABI.

It must preserve:

  • module and package identity;
  • canonical symbols and visibility;
  • source spans;
  • functions, actors, specs, and program entry points;
  • semantic source types and inferred facts;
  • structured `if`, loops, match/receive, and returns;
  • immutable collection operations;
  • actor, mailbox, supervision, job, I/O, and error effects;
  • calls to named standard-library/runtime capabilities;
  • diagnostics sufficient to reject invalid programs before backend lowering.

HIR should be structured rather than LLVM-like SSA. It is the correct optional

interoperability boundary for third-party frontends that want official backend

support.

Hey MIR: backend contract

MIR is produced only after HIR validates. It makes execution requirements

explicit:

  • control-flow graph and blocks;
  • explicit local/value types;
  • ownership/lifetime or retain/release operations where needed;
  • boxed/unboxed conversions;
  • actor/runtime ABI calls;
  • target-independent integer and floating operations;
  • explicit error and effect edges;
  • source-span references;
  • runtime ABI version;
  • target feature requirements without target-specific syntax.

MIR may use SSA when useful. LLVM and C consumers must lower MIR rather than

reconstructing source semantics.

Serialization

The in-memory HIR/MIR model is primary. Serialized artifacts are versioned,

deterministic views used for:

  • producer/consumer boundaries;
  • debugging and golden tests;
  • cross-language implementations;
  • incremental build caches;
  • bootstrap fixed-point comparisons.

A serialized artifact must contain schema version, language version, runtime ABI

version, source/module identity, dependency hashes, and a stable validation

result. Byte identity is required only for the canonical serializer; semantic

identity is otherwise determined by the schema.

Compiler implementation boundaries

The native-C and Hey-written compilers should become genuinely independent

frontends:

native C lexer/parser/resolver/analyzer -> HIR
Hey lexer/parser/resolver/analyzer      -> HIR

They share:

  • the backend-opaque conformance suite;
  • the HIR schema and verifier contract;
  • the HIR-to-MIR semantic lowering contract;
  • MIR validators;
  • backend-consumer suites;
  • runtime ABI specifications.

They do **not** need to share parser code, AST layout, diagnostics wording beyond

what conformance requires, or internal optimization choices.

Future Java/Rust implementations have two valid choices:

1. satisfy language conformance and own every later stage;

2. additionally emit validated HIR or MIR to reuse official consumers.

Specification layout

Use these boundaries:

specs/conformance/
  Backend-opaque source programs and observable behavior.

specs/frontend/
  Shared optional parser/resolution/diagnostic fixtures.
  These may define source spans and semantic symbols, but never LLVM.

specs/hir/
  HIR schema, validation, round-trip, module, type, and effect contracts.

specs/mir/
  MIR validation, CFG, conversion, runtime ABI, and deterministic lowering.

specs/backend/c/
  MIR-to-C lowering and C toolchain integration.

specs/backend/llvm/
  MIR-to-LLVM lowering, verifier, object, link, and target tests.

specs/implementation/native-c/
specs/implementation/hey/
  Implementation-specific regressions and bootstrap tests.

`specs/conformance` is the only mandatory suite for an independent Hey

implementation. HIR/MIR suites are required only for implementations claiming

official compiler interoperability.

Clean bootstrap sequence

The self-host proof should be staged and reproducible:

Stage 0: trusted native-C compiler
  compiles the Hey frontend and HIR/MIR libraries

Stage 1: Hey compiler executable built by Stage 0
  compiles the same compiler sources -> HIR -> MIR -> LLVM object

Stage 2: LLVM-built Hey compiler executable
  recompiles the same compiler sources -> HIR -> MIR -> LLVM object

Stage 3: Stage-2 compiler repeats the build

Required checks:

  • Stage 1, 2, and 3 pass the same backend-opaque conformance suite;
  • Stage 1 and Stage 2 emit semantically equivalent HIR for the maintained

bootstrap corpus;

  • Stage 2 and Stage 3 reach a deterministic fixed point after stripping

declared non-semantic fields such as timestamps or build IDs;

  • the original source is deleted before serialized HIR/MIR consumers run;
  • no stage invokes a source-language fallback after the frontend boundary;
  • all compiler executables and caches are content-addressed by compiler source,

schema version, runtime ABI, target, and build mode.

Rules for the next responder

1. Read this document, `language-conformance.md`, and

`compiler-interoperability.md` before changing compiler architecture.

2. Do not add new LLVM behavior to the language conformance suite.

3. Do not make sidecars required semantic inputs.

4. Do not add another direct source-to-LLVM heuristic when the fact belongs in

HIR or MIR.

5. Do not regenerate structural goldens until each change is classified as HIR,

MIR, or backend behavior.

6. Add a mini-spec at the lowest correct layer for every bug found.

7. Preserve the native-C compiler as the behavioral oracle during migration,

but do not let it silently analyze source for a supposedly source-independent

consumer.

8. Treat optional source signatures as provisional until semantic names are

decided; never add source syntax merely to expose an LLVM ABI detail.

9. Keep caches removable and content-addressed; add a cold-build test whenever

a warm-cache path is introduced.

10. Delete superseded lowering paths once their HIR/MIR replacement passes the

full maintained gates. Alpha status is permission to remove bad architecture,

not to keep two permanent compilers inside one executable.
