Visibility and private imports

Visibility lets a module expose a small public API while keeping helpers and implementation dependencies private.

Private functions

Functions are public by default. Add `private` when a helper is only for the current file or module:

module Sessions
  private fn normalized_id(request)
    return trim(request['params']['session_id'])
  end

  fn show(request)
    return {id: Sessions.normalized_id(request)}
  end
end

Code that imports this module may call `Sessions.show(...)`, but not `Sessions.normalized_id(...)`.

Top-level helpers may also be private:

private fn clean_name(name)
  return trim(name)
end

Private imports

A normal import contributes its public modules to the importing file's reachable dependency surface. Use `private import` when the dependency is an implementation detail:

private import './internal/token_codec.hey'

module Sessions
  fn decode(token)
    return TokenCodec.decode(token)
  end
end

Downstream code sees `Sessions.decode(...)`, not `TokenCodec` through this import.

Choosing a boundary

Use `private fn` for a helper implemented in the same source unit. Use `private import` for a helper implemented in another source unit. It is common to use both:

private import './internal/token_codec.hey'

module Sessions
  private fn token_from(request)
    return request['headers']['x-session-token']
  end

  fn current(request)
    return TokenCodec.decode(Sessions.token_from(request))
  end
end

What private does not mean

`private` defines a source-level API boundary. It does not encrypt code, isolate memory, or create a security sandbox. Private code is still compiled and can be inspected like the rest of a distributed executable.

See Modules, packages, and visibility for import schemes and package resolution.
