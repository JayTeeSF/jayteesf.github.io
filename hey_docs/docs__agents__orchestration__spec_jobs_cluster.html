jobs cluster implementation spec

Group: tour_05_jobs, jobs_adaptive_scaling, jobs_backoff_idempotency, jobs_declaration,

jobs_durable_replay, jobs_partitioned, jobs_reliability, jobs_worker_pool.

Reference material used: survey/<name>.mir (full read, all 8), HeyLlvm.hey (full-text grep +

targeted reads of hey_llvm_emit_text_value's sub-functions), and the ONLY jobs-flavored native

pin available, p18_jobs_partitioned.ll (for jobs_partitioned). No other example in this group has

a native pin anywhere in the scratchpad (verified: only p18_jobs_partitioned.ll contains an actual

`call ptr @hey_llvm_jobs_define_worker_value(ptr ...)` call site; every other .ll only carries the

boilerplate `declare` line, which is emitted unconditionally by `hey_llvm_prelude_declares_*` and

proves nothing about call-site shape).

Good news confirmed by reading, not assumed: the runtime-side `declare`s for essentially

everything this cluster needs (`hey_llvm_jobs_define_worker_value`, `hey_llvm_jobs_named_result_value`,

`hey_llvm_jobs_named_cancel_value`, `hey_llvm_jobs_named_dead_letters_value`,

`hey_llvm_jobs_named_status_value`, `hey_llvm_jobs_named_stop_value`, `hey_llvm_jobs_test_clock_set_value`,

`hey_llvm_jobs_test_clock_advance_value`, `hey_llvm_path_ensure_dir_value`, `hey_llvm_path_remove_tree_value`,

`hey_llvm_result_error_value`, `hey_llvm_result_error_with_value`, `hey_llvm_join_value_string`,

`hey_llvm_fail_diagnostic_value`, `hey_llvm_warn_value`) already exist in

`hey_llvm_prelude_declares_1..4` (HeyLlvm.hey:1989-2569). **No `Jobs.await_all` runtime symbol is

declared anywhere** — first pin for jobs_declaration must confirm its C symbol name.

All work in this cluster is *emitter-side pattern recognition* — a giant textual-MIR-to-LLVM

function (`hey_llvm_emit_text_value`, HeyLlvm.hey:2576-5386) that currently has **zero** branches

recognizing `Jobs.`, `Files.`, or `Result.` as special namespaces (verified: `grep -c "Jobs\."

HeyLlvm.hey` = 0 outside this spec's own citations). The established idiom for adding a new

namespaced/method intrinsic is a `hey_llvm_starts_with?(clean, '<prefix>(')` branch inside

`hey_llvm_boxed_expr_value` (HeyLlvm.hey:373-900) — see existing siblings `'file('` (~774),

`'monitor_to('` (~718), `'json('` (~649), `'.stop()'` suffix match (~858), `'.status().'` fused

suffix match (~869) as templates to copy.

Per-example unsupported constructs (file order)

tour_05_jobs.mir

1. L18 `Jobs.define("TourThumbnails", Thumbnail, count: 2, capacity: 8)` — new handler (Jobs.define, see Cluster item 1).

2. L19 `Thumbnails.enqueue({id: "cover"})` in a `local ... :boxed =` assignment — new handler (Cluster item 2). Note: the *negated* form `!X.enqueue(...)` used only inside `if bool` conditions is already supported (hey_llvm_cmp_expr_value, HeyLlvm.hey:1257-1276) but that code path is not reachable from a plain assignment.

3. L20 `if bool !receipt.accepted` — **already supported**, don't touch (hey_llvm_cmp_expr_value negation branch, HeyLlvm.hey:1278-1287, via hey_llvm_boxed_int_term_value dotted-field read).

4. L21 `fail str "tour job queue is full"` — **already supported** (plain string literal fail, HeyLlvm.hey:3810-3822).

5. L23 `Jobs.result(Thumbnails, receipt, 5000)` — new handler (Cluster item 1c).

6. L24 `if bool !result.ok` — **already supported** (same as #3).

7. L25 `fail boxed result.error` — new handler (Cluster item 4): only `fail str '<literal>'` exists (HeyLlvm.hey:3810); `fail boxed <expr>` has no branch at all. Runtime symbol `hey_llvm_fail_diagnostic_value(ptr)` is already declared (HeyLlvm.hey:1998) — pattern is: evaluate `result.error` as boxed (existing single-dot boxed_expr_value branch, ~880) then `call void @hey_llvm_fail_diagnostic_value(ptr <t>)`.

8. L27 `say boxed result.value.path` — new handler (Cluster item 3): two-level dotted chain. The single-dot case (`say boxed result.value`) already works via hey_llvm_boxed_expr_value's generic `.field` branch (~880-891); a second `.field` on the result is unsupported (`hey_llvm_c_name_value(dot_key)` rejects a key containing a `.`). Fix: chain two `json_get_key_value` calls instead of one when the tail after the first dot itself contains a dot.

9. L28 `say str "completed=" + Thumbnails.status().completed` — **already supported**: `hey_llvm_str_part_or_access_value` has a dedicated `.status().<field>` fused branch (HeyLlvm.hey:1483-1496) that emits `workers_status_value` + `json_get_key_value` + `value_to_raw_string_cstr`, and the surrounding `+`-splitter (hey_llvm_str_expr_value) already left-folds through it.

10. L29 `Thumbnails.stop()` in `local ... :boxed =` — **already supported** as an expression (hey_llvm_boxed_expr_value:858-866, `.stop()` suffix branch, maps to `hey_llvm_workers_stop_value`).

jobs_adaptive_scaling.mir

1. L14 `sleep i64 25` — **already supported** (HeyLlvm.hey:3824, literal sleep).

2. L15 `ret boxed page * 2` — worth double-checking against hey_llvm_boxed_expr_value's arithmetic support; `page` is the boxed worker param, `page * 2` is boxed-int multiplication. Not traced in depth here; likely routes through the same `!<term>` /or plain-term paths used elsewhere for boxed arithmetic (`hey_llvm_boxed_int_term_value` handles the read but not necessarily the multiply-and-rebox). Flag as **needs verification**, not confirmed unsupported — low priority, check after Jobs.define lands since nothing else in this file reaches L15 today.

3. L19 `Jobs.test_clock_set(0)` — new handler (Cluster item 1f): `@hey_llvm_jobs_test_clock_set_value(ptr)`, arg shape (int-literal wrapped via `hey_llvm_int_value(i64 N)`?) unconfirmed — needs a pin.

4. L20 `Jobs.define("Previews", RenderPreview, min_workers: 1, max_workers: 4, target_queue_latency_ms: 1, capacity: 16)` — new handler (Cluster item 1a). This is the pin to nail down the min_workers/max_workers/target_queue_latency_ms argument positions (inferred from p18 as slots 5/6/7, all-zero defaults there).

5. L21-24 `Previews.enqueue(1)` x4 (bare int payload, not an object literal) — new handler (Cluster item 2); note payload here is a bare int literal, not `{...}`, so the payload-building side of the new enqueue handler must delegate to the *existing* int-literal-to-json path already used inside object/array literal building (`hey_llvm_json_push_i64`-style), not just boxed_expr_value's object-literal branch.

6. L25/27/35/37 `Jobs.test_clock_advance(5|200)` — new handler (Cluster item 1f sibling), `@hey_llvm_jobs_test_clock_advance_value(ptr)`.

7. L26/36 `Previews.status()` (bare, no chained field) in `local ... :boxed =` — **already supported**? Not directly — the `.status().field` fused branch requires a trailing `.field`; a bare `.status()` with nothing after needs the plain `.status()`→`workers_status_value` mapping. Not found as a standalone branch in hey_llvm_boxed_expr_value (only `.stop()` has a bare-suffix branch); **new small addition** — mirror the existing `.stop()` branch (~858-866) for `.status()`.

8. L28/38 `Previews.status().pool` — one level of chaining on top of `.status()`: the existing `.status().field` fused branch (1483 in str-context, and its boxed_expr_value twin ~869-879) covers exactly this shape — **already supported** once reached.

9. L29/39 `say str "workers=" + scaled.current_workers + " peak=" + scaled.peak_workers` — **already supported**: multi-term `+` concatenation with dotted-boxed-field terms is handled generically by `hey_llvm_str_part_or_access_value`'s single-dot branch (HeyLlvm.hey:1498-1510) plus `hey_llvm_str_expr_value`'s left-fold splitter (1517-1609).

10. L30-33 `Jobs.result(Previews, one, 1000).value` — new handler (Cluster item 1c) plus a trailing `.value` field access, which should fall out for free once Jobs.result returns a boxed handle through the same code path as any other boxed local (reuse the existing single-dot chain machinery).

11. L34 `say str "results=" + first + "," + second + ...` — **already supported** (bare boxed-local concatenation, same machinery as #9, plain locals not even needing the dot branch).

jobs_declaration.mir

1. L18 `Jobs.define("Doubles", Double, count: 2, capacity: 8, result_capacity: 8)` — Cluster item 1a; confirms result_capacity slot (p18 has it explicitly too, at slot 10 = 32).

2. L19 `local b3 receipts:boxed = []` — **already supported** (empty array literal, HeyLlvm.hey:419-424).

3. L20 `control boxed for b4 value in [3, 4, 5]` — **already supported**: all-int-literal array for-loop unrolling exists (HeyLlvm.hey:3605-3650, `flit_all_int` branch).

4. L21 `store receipts:boxed = [...receipts, Doubles.enqueue(value)]` — **two** gaps stacked: (a) Cluster item 2 (`.enqueue(` in assignment position), and (b) array-literal element parsing (HeyLlvm.hey:426-471) only accepts int-literal / nested `[`/`{` / bare-i64-local / single-quoted-string elements — a call-expression element (`Doubles.enqueue(value)`) has no branch and falls through to the `return {ok:false...}` at line ~470-471. New: extend the element loop to recognize a trailing element that is itself a recognized boxed_expr_value form (call/method), pushing via `hey_llvm_json_push_value` the way nested `[`/`{` elements already do (444).

5. L23 `say str join(Jobs.await_all(Doubles, receipts, 1000), ",")` — **two** new pieces: (a) `Jobs.await_all(...)` (Cluster item 1g — **no runtime symbol found anywhere in the declares; first thing to confirm via the native pin**), and (b) `join(<boxed>, "<sep>")` builtin — `hey_llvm_join_value_string(ptr,ptr)` is declared (HeyLlvm.hey:2563) but **never invoked** anywhere in the file (grep confirms zero call sites) — needs a new `'join('`-prefix branch, most naturally added to `hey_llvm_str_part_or_access_value` (HeyLlvm.hey:1362) alongside its other prefix intrinsics, since it's used in `say str` context here.

6. L24 `Doubles.stop()` — **already supported**.

jobs_partitioned.mir

1. L14 top-level `fn ByCustomer(message:boxed) -> boxed` / L14 `ret boxed message.customer` — plain fn (not worker/actor); single-dot `ret boxed <local>.<field>` return. Check: the `ret boxed <expr>` path (HeyLlvm.hey:4554+) delegates to hey_llvm_boxed_expr_value, whose generic single-dot branch (880-891) covers `message.customer` — **already supported**. The `fn`-vs-`worker` distinction itself doesn't need new code; the header-parsing pass already treats `fn '`/`worker '` identically for body emission (HeyLlvm.hey:3379-3390) — `ByCustomer` just needs to be resolvable as a function-pointer target for `partition_by:` (see item below).

2. L19 `sleep boxed message.delay_ms` — new handler (Cluster item 5): only `sleep i64 <literal-or-int-chain>` exists (HeyLlvm.hey:3824-3835); `sleep boxed <expr>` has zero branches and hits `return {ok:false, error:'unsupported MIR operation...'}` at line ~4946 (this is the group's live, reported blocker). Fix is small and low-risk: reuse `hey_llvm_boxed_int_term_value` (HeyLlvm.hey:1083-1105) verbatim — it already emits exactly the native shape (`load ptr b.obj` + `json_get_key_i64`) confirmed against p18_jobs_partitioned.ll lines 613-615.

3. L20 `ret str message.customer + ":" + message.sequence` where `fn_ret == boxed` — new handler (Cluster item 3b): the `ret_type=='str' and fn_ret=='boxed'` branch (HeyLlvm.hey:4802-4809) currently requires the whole expr to be a single-quoted literal and fails otherwise; needs the general concatenation-chain path (see Cluster item 3).

4. L24 `Jobs.define("CustomerEvents", SendCustomerEvent, count: 4, capacity: 32, result_capacity: 32, partition_by: ByCustomer, partitions: 4)` — Cluster item 1a; **this is the one call site with a full native reference** (p18_jobs_partitioned.ll:644) — use it as the primary template. Also requires resolving `ByCustomer` (a plain `fn`, not `worker`) to `@hey_fn_ByCustomer` plus its literal name string `@.str.7` — same fn-pointer-and-name-string pattern already used for the worker argument, applied to a second fn.

5. L25-27 `Events.enqueue({customer: "alpha", sequence: 1, delay_ms: 20})` x3 — Cluster item 2.

6. L28 `Jobs.result(Events, first, 1000).value` x3 inside an array literal `[...]` — Cluster item 1c **plus** "call-expression array element" (same array-literal gap as jobs_declaration #4).

7. L29 `say str join(values, ",")` — Cluster item "join(" (see jobs_declaration #5b); this is the simpler case since `values` is a bare boxed local, not a nested call, and is directly confirmed against p18_jobs_partitioned.ll:687-689 (`join_value_string` + `puts`).

8. L30 `say bool first.partition == second.partition` — new handler: **not** an int/bool comparison despite the MIR shape — confirmed against p18 (lines 690-699) that native compares via `strcmp` on `value_to_raw_string_cstr` of each `.partition` field, not `json_get_key_i64`/icmp. `hey_llvm_cmp_expr_value`'s op-branch (HeyLlvm.hey:1298-1311) always does `hey_llvm_int_expr_value` + `icmp`, which is the wrong shape here. Needs a new pre-check in cmp_expr_value: if both operands are single-dot boxed-field accesses (and neither is an int-literal RHS), do the `json_get_key_value` + `value_to_raw_string_cstr` + `strcmp` + `icmp eq/ne` dance instead of the int path — same output shape (`i1`), just a different construction.

9. L31 `Events.status().partition_count` in `say boxed` — **already supported** (`.status().field` fused branch, boxed_expr_value ~869-879).

10. L32 `Events.stop()` — **already supported**.

Cluster-level summary (highest leverage first)

1. **`Jobs.*` namespace as a whole** — appears in every one of the 8 examples, always as the very

first blocker today. Zero support exists. Sub-pieces, roughly in the order that unblocks the

most examples fastest:

  • 1a. `Jobs.define(name, WorkerFnOrFnName, kw...)` → `@hey_llvm_jobs_define_worker_value` (19

fixed args). **Highest single-item leverage** — blocks all 8 examples. Only 5 of the ~13

named kwargs have a confirmed argument slot from the one available pin (count, capacity,

result_capacity, partition_by, partitions); min_workers/max_workers/target_queue_latency_ms/

retries/backoff_ms/max_backoff_ms/idempotency_capacity/journal_path/journal_compact_after/

dead_letter_capacity and one unexplained always-zero slot (position 9 in the p18 call) are

inferred defaults only — **do not guess the rest; pin jobs_adaptive_scaling (min/max_workers,

target_queue_latency_ms), jobs_backoff_idempotency (retries, backoff_ms, max_backoff_ms,

idempotency_capacity), jobs_reliability (dead_letter_capacity), and jobs_durable_replay

(journal_path, journal_compact_after) before finalizing the arg table.**

  • 1b. `<boxed handle>.enqueue(<payload-expr>)` in assignment/array-element position →

`@hey_llvm_workers_enqueue_value(handle, payload)`. Confirmed symbol name from p18. Blocks 7/8

examples (all but jobs_adaptive_scaling... no, that one uses it too — blocks all 8).

  • 1c. `Jobs.result(handle, receipt, timeout)` → wrap timeout via `hey_llvm_int_value(i64 N)`

then `@hey_llvm_jobs_named_result_value(handle, receipt, timeout_ptr)`. Confirmed from p18.

Blocks 7/8 (all but jobs_adaptive_scaling's simpler paths — actually also used there).

  • 1d. Bare `<handle>.status()` / already-fused `<handle>.status().field` — only the bare form

  • 1f. `Jobs.test_clock_set(n)` / `Jobs.test_clock_advance(n)`. Blocks only jobs_adaptive_scaling

2. **`<boxed>.enqueue(...)` as a general expression** (not just the already-supported

`if bool !X.enqueue(...)` negation special-case) — needed in all 8 examples, always paired with

Jobs.define. Implement 1a/1b/1c together; they're used in lockstep in every example.

3. **Dotted-boxed-field string concatenation, two call sites**:

  • 3a. Inside object-literal values (`{output: "a" + x.field + "b"}`) — tour_05_jobs,

jobs_worker_pool.

  • 3b. In `ret str <expr>` when the function's declared return type is `boxed`
