Great Command-Line Scripting with Hey

A practical guide to structuring, rendering, parallelizing, compiling, caching, and benchmarking a real Hey command-line program. The running example generates a static package index, while its Hey and Ruby implementations intentionally produce byte-for-byte-identical HTML.

Complete source files
  • Executable Hey implementation
  • Ruby parity implementation

The example now has exactly two source files. The Hey file contains its own shebang, CLI entry point, timestamp generation, rendering logic, and program block; no shell launcher or separate generate-packages-index.hey file is required.

Current implementation contract

The two-file implementation consists of one executable Hey program and one Ruby parity program. They use the same natural ordering, escaping rules, compact linked-heading layout, and optional fixed timestamp. With the same PACKAGES_INDEX_GENERATED_AT value, their HTML must be byte-for-byte identical.

1. The shape of a good Hey script

The current example deliberately uses only two files:

  1. generate-packages-index.hey, the complete executable Hey program.
  2. generate-packages-index.rb, the independent Ruby parity implementation.

The old shell-launcher plus generate-packages-index.hey split is gone. A Hey shebang script can own its imports, behavior, timestamp fallback, CLI handling, and entry point directly: 

#!/usr/bin/env hey

import 'stdlib:Cli'
import 'stdlib:Collections'
import 'stdlib:Env'
import 'stdlib:Files'
import 'stdlib:Streams'
import 'stdlib:Text'
import 'stdlib:Time'

module PackagesIndex
  fn run(args)
    # command behavior
  end
end

program
  PackagesIndex.run(Cli.args())
end

This keeps the executable surface obvious: run ./generate-packages-index.hey. The Ruby file is not a launcher; it is a separately implemented correctness and performance reference.

2. Command-line arguments and help

Cli.args() returns application arguments. Cli.positionals(args) extracts positional values, while Cli.flag?, Cli.flag_int, and Cli.missing_value? support readable option handling. 

let args = Cli.args()

if Cli.flag?(args, 'help') or Cli.flag?(args, 'h')
  PackagesIndex.usage()
  return
end

if Cli.missing_value?(args, 'workers')
  fail '--workers requires an integer value'
end

let paths = Cli.positionals(args)
let workers = Cli.flag_int(args, 'workers', cpu_count())

Use a raw squiggly heredoc for static help text:

private fn usage_text()
  return <<~'USAGE'
    Usage: ./generate-packages-index.hey [OPTIONS] [PACKAGES_DIR]

    Options:
      --workers N   Number of parallel workers
      -h, --help    Show help
  USAGE
end

3. Filesystem discovery

The stdlib:Files module owns traversal, metadata, path manipulation, and publication.

let packages = Files.entries('./packages')
let all_files = Files.files('./packages/hey_web', '')
let markdown = Files.files('./packages/hey_web', '.md')
  • Files.entries(path) returns immediate children.
  • Files.files(root, suffix) recursively returns matching regular files.
  • An empty suffix returns every regular file below the root.
  • Files.directory?, Files.file?, and related helpers provide filesystem checks.
  • Files.write_atomic(path, content) publishes through a sibling temporary file and rename.

For this index generator, one recursive scan per package is preferable to rescanning the same tree for every extension:

let package_files = Files.files(package_dir, '')
let docs_dirs = PackagesIndex.docs_directories(package_dir, package_files)
let linkable_files = PackagesIndex.linkable_files(package_files)

The implementation also checks immediate version directories directly so an empty versioned docs/ directory is still indexed even though it contains no file discovered by the recursive scan.

4. Ergonomic collection style within the LLVM subset

Hey supports receiver-oriented collection operations, but the executable source must also stay inside the current direct LLVM lowering subset. The package index therefore uses the receiver chains already proven by the LLVM benchmark, while leaving stateful traversal and accumulation as explicit loops. 

private fn sorted_unique(paths)
  return paths
    .uniq
    .sort(PackagesIndex.compare_paths)
end

The parallel rendering path uses the supported stream pipeline:

set sections = tasks
  .map(PackagesIndex.render_package, workers: workers)
  .collect()

A broader refactor to synchronous .map, .filter, and .reduce was tested, but the current direct LLVM backend rejected that application with unsupported-lowering diagnostics. Shipping code that actually compiles is more important than forcing every available collection helper into this example. Those forms can replace the remaining loops after the LLVM lowerer supports them end to end.

The natural comparator also remains imperative because it advances independently through two strings. That loop is a clearer representation of the state machine than an artificial collection pipeline.

5. Rendering deterministic HTML

Small pure functions render one link, while explicit loops append those links to line arrays. This form is currently LLVM-compatible, keeps indentation obvious, and still joins each completed fragment only once. 

let lines = [
  '      <h3>Markdown and HTML files</h3>',
  '      <ul>'
]

for path in linkable_files
  set lines = [
    ...lines,
    PackagesIndex.file_link(
      path,
      packages_dir,
      package_dir
    )
  ]
end

set lines = [...lines, '      </ul>']
return Text.join(lines, "
")

This preserves the compact linked-heading format and keeps the Hey and Ruby outputs byte-for-byte comparable. Once receiver reduce is supported by the direct LLVM path for this program, this accumulation loop can be reconsidered without changing the output contract.

6. Escape at the boundary

Structured path data should remain ordinary strings until rendering. Escape it only when it crosses into HTML text or a URL attribute. 

private fn html_escape(value)
  let escaped = Text.replace_all(value, '&', '&amp;')
  set escaped = Text.replace_all(escaped, '<', '&lt;')
  set escaped = Text.replace_all(escaped, '>', '&gt;')
  set escaped = Text.replace_all(escaped, '"', '&quot;')
  set escaped = Text.replace_all(escaped, "'", '&#39;')
  return escaped
end

URL paths are encoded separately. The generator preserves path separators while escaping characters such as spaces, percent signs, fragments, quotes, and angle brackets.

7. Parallel work without manual threads

One package is a natural independent work unit. Package trees can therefore be scanned and rendered with a bounded parallel stream: 

let sections = tasks
  .map(PackagesIndex.render_package, workers: workers)
  .collect()

Parallel stream stages preserve input order by default, so work may finish in any internal order while the generated page remains deterministic.

Parallelism is not automatically faster.

The original native-C benchmark used seven small package trees and found that worker overhead dominated. The updated 10-package LLVM benchmark stayed near 3 ms across all tested worker counts, with 10 workers recording the best mean.

The program therefore keeps an explicit sequential fast path:

if workers == 1
  for task in tasks
    set sections = [...sections, PackagesIndex.render_package(task)]
  end
else
  if !tasks.empty?
    set sections = tasks
      .map(PackagesIndex.render_package, workers: workers)
      .collect()
  end
end

More workers may become useful for larger trees, Markdown rendering, archive inspection, hashing, link validation, or network-bound work. Worker count and backend should both be measured rather than assumed.

8. Atomic output and reproducible timestamps

Do not truncate the live index before its replacement is complete:

let written = Files.write_atomic(
  output_path,
  PackagesIndex.render_page(
    sections,
    PackagesIndex.generated_at()
  )
)

if !written.ok
  fail "could not write #{output_path}: #{written.error}"
end

Readers see either the previous complete index or the newly generated complete index, rather than a partially written page.

The single Hey file reads an optional deterministic timestamp and otherwise obtains the current UTC time itself: 

private fn generated_at()
  let configured = Env.default(
    'PACKAGES_INDEX_GENERATED_AT',
    ''
  )

  if configured != ''
    return configured
  end

  return Time.utc_iso8601()
end

Supplying the same environment value to Ruby and Hey makes parity tests meaningful without requiring a separate shell launcher.

9. A minimal build-up version

A deliberately smaller version teaches the current APIs before the complete program adds validation, escaping, natural sorting, parallelism, and exact Ruby parity:

import 'stdlib:Cli'
import 'stdlib:Files'
import 'stdlib:Text'

module SimplePackagesIndex
  fn run(args)
    let paths = Cli.positionals(args)
    let root = './packages'

    if paths.length == 1
      set root = paths[0]
    end

    let links = Files.files(root, '.md')
      .map(SimplePackagesIndex.relative_link)

    let lines = [
      '<!doctype html>',
      '<html lang="en">',
      '<body>',
      '  <h1>Packages</h1>',
      '  <ul>',
      Text.join(links, "
"),
      '  </ul>',
      '</body>',
      '</html>',
      ''
    ]

    Files.write_atomic(
      Files.join(root, 'index.html'),
      Text.join(lines, "
")
    )
  end
end

program
  SimplePackagesIndex.run(Cli.args())
end

The complete linked implementation adds:

  • argument validation and --help;
  • Markdown and HTML links;
  • clickable package-name headings;
  • versioned docs/ links, including empty directories;
  • HTML and URL escaping;
  • natural numeric sorting;
  • optional package-level parallelism;
  • a sequential fast path;
  • atomic publication;
  • a generated UTC timestamp;
  • byte-for-byte output parity with the Ruby implementation.

10. How the benchmark was run

The updated benchmark separates LLVM-compiled runtime, native-C-compiled runtime, Ruby runtime, warm application-cache hits, application-artifact misses, and an evaluator control.

The test ran on an Apple M3 MacBook Pro using macOS Tahoe 26.5.2 (25F84), with 18 GB of RAM and 12 CPU cores: 6 performance cores and 6 efficiency cores. The package tree contained 10 top-level package directories.

Both Hey backends were compiled once before the runtime benchmark:

export HEY_ROOT="$HOME/dev/hey-lang-bootstrap-plan"
export HEY_STDLIB_PATH="$HEY_ROOT/stdlib"

"$HEY_ROOT/bin/heyc" build   --backend llvm   ./generate-packages-index.hey   -o ./build/generate-packages-index-llvm

"$HEY_ROOT/bin/heyc" build   --backend native-c   ./generate-packages-index.hey   -o ./build/generate-packages-index-native-c

One observed LLVM build took 9.301 seconds. One observed native-C build took 4.452 seconds. LLVM therefore cost more to build, but produced a much faster executable for this workload.

Runtime measurements used three warm-up runs followed by thirty measured runs. A fixed timestamp ensured that LLVM, native-C, and Ruby produced byte-for-byte-identical HTML.

Application-cache behavior

The first bin/hey invocation missed both the source index and application artifact:

hey-run: cache=miss key=13512060f760334a3dfc3490
hey-run: cache=published backend=llvm

The second invocation hit both:

What the updated numbers show
  • LLVM compilation is the dominant runtime improvement for this workload.
  • The fastest LLVM result was 44.43 times faster than Ruby.
  • LLVM with one worker was still 34.68 times faster than Ruby.
  • native-C with one worker was only 1.35 times faster than Ruby.
  • native-C parallel execution degraded sharply, reaching 415.8 ms with 10 workers.
  • A warm bin/hey cache hit remained 2.59 times faster than Ruby.
  • The application cache avoided a roughly 670 ms LLVM artifact rebuild.
  • heyc --run is an evaluator control, not the application-runtime benchmark.

These numbers describe one machine, one package tree, and one workload. They should not be generalized to every Hey or Ruby program. They do strongly support benchmarking LLVM when LLVM is the production execution path, and measuring cached and uncached application launches separately.

11. Verification workflow

Run the source through the launcher:

chmod +x \
  ./generate-packages-index.hey \
  ./generate-packages-index.rb

./generate-packages-index.hey --help
./generate-packages-index.hey ./packages

Compile once for runtime benchmarking:

export HEY_ROOT="$HOME/dev/hey-lang-bootstrap-plan"
export HEY_STDLIB_PATH="$HEY_ROOT/stdlib"

"$HEY_ROOT/bin/heyc" build \
  --backend native-c \
  ./generate-packages-index.hey \
  -o ./build/generate-packages-index

Verify that Ruby and Hey produce exactly the same output:

STAMP='2000-01-01T00:00:00Z'

PACKAGES_INDEX_GENERATED_AT="$STAMP" \
  ./generate-packages-index.hey ./packages
cp ./packages/index.html /tmp/packages-index-from-hey.html

PACKAGES_INDEX_GENERATED_AT="$STAMP" \
  ./generate-packages-index.rb ./packages
cp ./packages/index.html /tmp/packages-index-from-ruby.html

cmp /tmp/packages-index-from-hey.html \
    /tmp/packages-index-from-ruby.html

Check the generated page:

test -s ./packages/index.html
grep -n '<h2><a href=' ./packages/index.html
grep -n 'docs/' ./packages/index.html
grep -nE '\.(md|html)</a>' ./packages/index.html
grep -n 'Generated .*Z' ./packages/index.html

The complete implementations linked at the top support normal documentation trees, nested documentation, multiple versions, natural sorting such as 0.2.2 versus 0.2.10, empty versioned docs/ directories, clickable package headings, atomic output, and deterministic cross-language rendering.

Performance verification should additionally compare the LLVM and native-C compiled artifacts, the direct cached LLVM executable, a warm bin/hey application-cache hit, an application-artifact miss, and an evaluator control. Those are different execution paths and should not be collapsed into one number.
