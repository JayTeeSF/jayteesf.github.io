Authoring a third-party Hey package

This page shows the current package shape for a small third-party package. It is intentionally file-first and cache-first: package metadata is ordinary Hey data, modules are copied into a local cache, and trust is enforced with SHA-256 checksums before install.

Suggested layout

flashcards_tools/
  src/
    flashcards/main.hey
    flashcards/format.hey
  docs/
    README.md
  tests/
    format_spec.hey

The package source path points at the directory containing the module files named in the manifest. Nested module names map to nested `.hey` files.

Manifest, lock, and verified local install

import 'stdlib:packages'
import 'stdlib:result'

program
  let manifest = packages.manifest('flashcards_tools', '0.1.0', ['flashcards/main', 'flashcards/format'])
  let dep = packages.dependency('stdlib/http', '=0.99.114a')
  let package = packages.add_dependency(manifest, dep)
  let source = packages.file_source('src', ['flashcards/main', 'flashcards/format'])
  let trusted = result.value(packages.trust_source(source, package))
  let lock = packages.lockfile(package, [packages.lock_dependency(dep, '0.99.114a', packages.sha256('known-dependency-digest'))])
  let installed = packages.install_verified_local('build/pkg-cache', package, trusted, lock)
  says result.is_ok(installed)
end

Use `packages.install_verified_local` for local development whenever you want the same trust boundary as a fetched package. It recomputes the SHA-256 digest over the declared source modules and rejects untrusted or mismatched sources before copying files into the cache.

File and remote fetch primitives

For package archives or registry responses, fetch the bytes first, verify SHA-256, then install the unpacked source directory.

import 'stdlib:packages'
import 'stdlib:crypto'
import 'stdlib:result'

program
  let expected = packages.sha256(crypto.sha256_text('abc'))
  let local = packages.fetch_file('fixtures/pkg.txt', 'build/pkg.txt', expected)
  says result.is_ok(local)

  let remote = packages.remote_source('http://packages.example.test/flashcards_tools-0.1.0.zip', ['flashcards/main'])
  says remote['transport']
  # packages.fetch_remote(remote['url'], 'build/flashcards_tools.zip', expected)
end

`packages.fetch_remote` performs a real `http://` GET through the runtime networking primitive and verifies the downloaded body with SHA-256. HTTPS/TLS policy remains explicit: use a reverse proxy or signed/checksummed artifacts until native TLS support is added.

Dependency isolation and import scope

Modules remain namespaces. Package isolation is a tooling/manifest contract layered on top of modules rather than a new object system. Use top-level imports for app-owned modules; use package manifests, locks, cache paths, and scoped namespaces for third-party code. A locked dependency resolves inside its own versioned namespace, so the app and two packages can depend on different versions without stomping each other.

import 'stdlib:packages'

program
  let app = packages.manifest('cards', '1.0.0', ['main'])
  let dep = packages.dependency('shared-tools', '=2.1.0')
  let locked = packages.lock_dependency(dep, '2.1.0', packages.sha256('digest'))
  let scope = packages.dependency_scope(app, locked)
  says packages.namespace(app)
  says packages.scoped_module(app, 'main')
  says packages.scoped_import_path('pkg-cache', app, 'main')
  says scope['dependency_namespace']
end

This is the current answer to “imports inside a module/package”: they are private implementation imports for that owner scope. Public app imports stay above the module/program. Future import-generation tooling should write cache paths and canonical symbols from the lockfile instead of consulting the app's dependency versions.

Import aliases

Aliases are metadata today, pinned so tooling can generate readable imports later without inventing a new shape.

import 'stdlib:packages'

program
  let alias = packages.import_alias('flashcards/main', 'cards')
  says alias['alias'] + '/' + alias['module']
end

Semver ranges and generated docs

Use semver records for tooling decisions instead of comparing version strings by hand. `packages.caret_range(...)` and `packages.range_requirement(...)` let a registry choose the newest compatible package, while `packages.docs_markdown(...)` gives package authors a generated README starting point.

import 'stdlib:packages'
import 'stdlib:result'

program
  let parsed = result.value(packages.parse_semver('1.4.0'))
  let range = packages.caret_range(parsed)
  let dep = packages.dependency('flashcards_tools', packages.range_requirement(range))
  let registry = packages.registry([
    packages.manifest('flashcards_tools', '1.3.0', ['flashcards/main']),
    packages.manifest('flashcards_tools', '1.4.0', ['flashcards/main'])
  ])
  let selected = result.value(packages.resolve_best(registry, dep))
  let docs = packages.docs_markdown(selected, [packages.module_doc('flashcards/main', 'Reusable flashcard helpers.')])
  says packages.id(selected)
  says docs
end
