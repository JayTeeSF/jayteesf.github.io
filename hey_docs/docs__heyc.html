Using `heyc`

`heyc` runs specs, executes source files, builds native applications, inspects symbols, and selects compiler backends.

Run and test

./bin/heyc app.hey --run
./bin/heyc app.hey --test

A file may contain both executable specs and a `program` block. `--test` runs the specs; `--run` runs the program.

Run the repository suite through the maintained wrapper:

./bin/test --full

Build an executable

./bin/heyc build app.hey -o build/app
build/app

The default backend is `native-c`. Select another backend explicitly when testing backend compatibility:

./bin/heyc build --backend self-hosted-c app.hey -o build/app-selfhost
./bin/heyc build --backend llvm app.hey -o build/app-llvm

List the backends recognized by the current checkout:

./bin/heyc --backends

`native-c` is the behavioral oracle. `self-hosted-c` is selectable and expected to preserve the same behavior. LLVM is selectable but may still report unsupported language or tooling surfaces honestly.

Select a backend with the environment

Commands that honor backend selection can use `HEY_BACKEND`:

HEY_BACKEND=llvm ./bin/heyc app.hey --run

An explicit `--backend` option should be preferred in scripts because it makes the chosen compiler path visible at the command site.

Inspect compiler-facing symbols

./bin/heyc symbols app.hey

Module-qualified source names retain their readable source form while the compiler also emits a backend-safe canonical symbol name. This command is useful when diagnosing imported functions or comparing backend output.

Read API documentation

./bin/heyc --describe web.serve
./bin/hey doc web.serve

Both commands read the standard documentation catalog and print signatures and value-shape information for known language and stdlib functions.

Use the REPL

./bin/hey-repl

The REPL shares the language runtime and supports expressions, bindings, functions, actors, imports, specs, and program blocks. See the REPL guide.

Toolchain and distributions

./bin/heyc toolchain

This reports host and backend toolchain requirements. Source checkouts build local binaries with `make`; released distributions are intended to provide runnable Hey commands without requiring the user to rebuild the compiler.

For application-local distributions, see Vendoring Hey.

Release verification

Maintainers use the release suites rather than assembling ad hoc command lists:

./bin/test --release-c
./bin/test --release-a
./bin/test --release-b

The full backend-world promotion proof is run separately:

time HEY_BACKEND_WORLD_JOBS=8 ./bin/test --promotion-report

Backend plans and historical compatibility measurements are maintainer records and are intentionally not included in the public documentation site.
