Workers and Jobs, step by step

Hey separates **parallel execution** from **stateful identity**:

  • a `worker` handles one immutable message and retains no state;
  • `workers(...)` creates a low-level bounded pool;
  • `Jobs.define(...)` adds receipts, results, retries, cancellation, metrics, idempotency, and optional durability;
  • an `actor` remains the right tool when a long-lived identity owns private state.

This tutorial uses complete programs. Every name shown below is defined, and every example can be compiled or pasted into the REPL.

A worker pool with an actor collector

import 'std:Workers'

worker Square(message)
  send message.reply_to, message.value * message.value
end

actor Collector(expected)
  state seen = 0
  state total = 0

  receive value

  set seen = seen + 1
  set total = total + value

  if seen == expected
    says total
    exit
  end
end

program
  let collector = spawn Collector(4)
  let pool = workers(Square, count: 2, capacity: 8)

  for value in [1, 2, 3, 4]
    let accepted = pool.enqueue({
      reply_to: collector,
      value: value
    })

    if accepted == false
      fail 'worker queue is full'
    end
  end

  join collector

  let metrics = Workers.await_idle(pool, 1000)
  says metrics.completed

  pool.stop()
end

Output:

30
4

What each declaration and value means

`worker Square(message)` declares a stateless handler. `message` is the one immutable value assigned to one available worker instance.

`message.reply_to` is an actor reference supplied by the caller. The worker does not know or retain a global collector.

`message.value` is the number to square. Object fields keep related immutable inputs together without positional argument conventions.

`send message.reply_to, ...` places the result in the collector's private mailbox. The worker does not mutate the collector.

`actor Collector(expected)` owns persistent private state. `expected` is the number of results required before the actor can print and exit.

`state seen` and `state total` belong only to that actor. Other actors and workers cannot read or modify them directly.

`receive value` defines the message handled by one serialized actor turn. The runtime never executes two turns for the same actor at once.

`spawn Collector(4)` creates one collector identity expecting four replies. The returned `collector` value is the actor reference placed into each work message.

`workers(Square, count: 2, capacity: 8)` creates a pool with two stateless worker instances and a bounded eight-message queue. The runtime schedules the instances on its managed OS-thread pool; this does not create two dedicated threads.

`pool.enqueue(...)` returns `true` when the message entered the bounded queue. It returns `false` when the pool is closed or full, making backpressure explicit.

`join collector` waits for the collector to print the total and exit. It does not stop the worker pool.

`Workers.await_idle(pool, 1000)` waits up to 1,000 milliseconds for queued and active work to reach zero, then returns the same metrics object exposed by `pool.status()`.

`metrics.completed` is the number of worker messages completed by this pool.

`pool.stop()` rejects new work, releases queued work that never started, lets active turns finish, and closes the pool.

Use a bare pool when the caller already owns result routing. Use a Job when the runtime should own result records and reliability.

Declaring a static Job

For a Job whose name and policy are fixed by the application, use the thin declaration form:

worker Double(message)
  return message * 2
end

job Doubles
  worker Double
  count: 2
  capacity: 8
  result_capacity: 8
end

program
  let receipt = Doubles.enqueue(21)
  says Jobs.result(Doubles, receipt, 1000).value
  let stopped = Doubles.stop()
end

`job Doubles` introduces the application binding. `worker Double` identifies the real worker declaration, not a string. Every remaining line is passed as the corresponding `Jobs.define` option. The compiler desugars this declaration before HIR, so its runtime behavior, durability, scaling, partitioning, metrics, and shutdown are identical to the explicit form.

A named Job with a result

worker Double(message)
  return message * 2
end

program
  let Doubles = Jobs.define(
    'Doubles',
    Double,
    count: 2,
    capacity: 8
  )

  let receipt = Doubles.enqueue(21)
  let outcome = Jobs.result(Doubles, receipt, 1000)

  says receipt.accepted
  says outcome.status
  says outcome.value

  Doubles.stop()
end

Output:

true
completed
42

What the Job adds

`Jobs.define(...)` creates a named runtime object around the `Double` worker declaration.

`'Doubles'` is the operational name used by metrics and diagnostics. It is not a function-name string used for dispatch; the second argument is the actual worker value.

`count: 2` fixes the number of worker instances. Fixed Jobs remain useful when reproducibility matters more than elasticity.

`capacity: 8` bounds work waiting to start. The bound prevents an overloaded producer from consuming memory indefinitely.

`Doubles.enqueue(21)` returns a receipt object immediately. Important receipt fields are:

  • `id`: the Job-local work identity;
  • `accepted`: whether the bounded Job accepted the message;
  • `status`: normally `queued` or `rejected` at enqueue time;
  • `duplicate`: whether an idempotency key reused an earlier receipt.

`Jobs.result(Doubles, receipt, 1000)` waits up to one second for that receipt's result record. It returns an outcome rather than throwing away a worker return value.

Important outcome fields are:

  • `status`: `completed`, `failed`, `cancelled`, `rejected`, `timeout`, or `unknown`;
  • `ok`: `true` only for successful completion;
  • `value`: the worker's returned value on success;
  • `error`: the failure reason when `ok` is false;
  • `attempts`: how many handler attempts were made.

The `1000` timeout bounds the caller's wait. It does not currently kill a native handler that is already running.

Adaptive queue-latency scaling

Use the complete adaptive option trio instead of `count:` when a Job should respond to queue pressure:

let Previews = Jobs.define(
  'Previews',
  RenderPreview,
  min_workers: 1,
  max_workers: 8,
  target_queue_latency_ms: 25,
  capacity: 256
)

`min_workers` is both the initial size and the idle floor. `max_workers` is a hard ceiling. `target_queue_latency_ms` is the oldest queued-work age that asks the pool to grow. The runtime also considers backlog per live worker, but starts or retires at most one lightweight worker per rate interval. These workers run on the existing managed scheduler rather than owning operating-system threads.

When the queue drains and all turns remain idle, the pool retires workers gradually until it reaches `min_workers`. Fixed `count:` and adaptive options are intentionally mutually exclusive.

`Previews.status().pool` exposes `current_workers`, `desired_workers`, `peak_workers`, `oldest_queue_wait_ns`, `scale_up_events`, `scale_down_events`, `workers_started`, `workers_retired`, and `last_scale_reason`.

Retries and explicit failure acknowledgements

import 'std:Result'

worker ImportRecord(message)
  if message.valid == false
    return Result.error('invalid record')
  end

  return Result.ok(message.value * 2)
end

program
  let Imports = Jobs.define(
    'Imports',
    ImportRecord,
    count: 4,
    capacity: 256,
    retries: 3,
    backoff_ms: 100,
    max_backoff_ms: 2000,
    dead_letter_capacity: 128
  )
end

Returning an ordinary value and returning `Result.ok(value)` both acknowledge success. `Result.ok` is useful when success and failure are chosen in nearby branches.

`Result.error(message)` tells the Job runtime that the attempt failed in a retryable way.

`retries: 3` permits three attempts after the first attempt, for at most four total attempts.

`backoff_ms: 100` delays the first retry by 100 milliseconds. Later delays double.

`max_backoff_ms: 2000` caps that exponential growth.

The delayed work sits in one runtime timer queue. It does not occupy a worker while waiting.

After the final failed attempt, the runtime stores a bounded dead-letter entry containing the receipt id, attempts, error, and original payload.

Cancellation and dead letters

let receipt = Imports.enqueue({valid: false, value: 21})
let cancellation = Jobs.cancel(Imports, receipt)

if cancellation.cancelled == false
  warn 'the handler may already be running'
end

for letter in Jobs.dead_letters(Imports)
  warn letter.id
  warn letter.attempts
  warn letter.error
  warn letter.payload
end

`Jobs.cancel(Job, receipt)` requests cancellation. Queued work and delayed retries can be cancelled immediately. A handler already executing may finish before it observes the request; the runtime does not terminate an arbitrary native thread.

`cancellation.cancelled` reports whether the request moved the receipt into the cancelled state. `cancellation.status` reports the current receipt state.

`Jobs.dead_letters(Job)` returns the Job's bounded final-failure records. Each `letter` contains the receipt `id`, the number of `attempts`, the final `error`, and the original immutable `payload`. Dead letters exist so failed work can be inspected, alerted on, or explicitly replayed instead of disappearing.

Idempotent enqueue

let first = Imports.enqueue({
  idempotency_key: 'customer-42-import',
  value: 21
})

let duplicate = Imports.enqueue({
  idempotency_key: 'customer-42-import',
  value: 999
})

The `idempotency_key` is ordinary message data with one runtime-recognized purpose. While the key remains within the Job's `idempotency_capacity`, the second enqueue returns the first receipt instead of executing the worker again.

`duplicate.duplicate` is `true`, and `duplicate.id` equals `first.id`.

Idempotency is essential for durable at-least-once replay: a process may crash after an external side effect but before recording completion.

Durable replay

worker DoubleRecord(message)
  return message.value * 2
end

program
  let DurableDoubles = Jobs.define(
    'DurableDoubles',
    DoubleRecord,
    count: 2,
    capacity: 32,
    result_capacity: 128,
    idempotency_capacity: 128,
    journal_path: 'var/jobs/durable-doubles.jsonl',
    journal_compact_after: 4096
  )

  let receipt = DurableDoubles.enqueue({
    idempotency_key: 'double-21',
    value: 21
  })

  let outcome = Jobs.result(DurableDoubles, receipt, 1000)
  says outcome.value

  DurableDoubles.stop()
end

`journal_path` enables an append-only canonical-JSON journal. The parent directory must already exist. Without this option, the Job remains process-local.

Before accepted work is scheduled, the runtime appends and synchronizes an `enqueue` record containing the immutable payload and idempotency key.

After completion, failure, or cancellation, the runtime appends a terminal record. On the next process start, `Jobs.define` reads records for the same Job name:

  • completed results become available again;
  • idempotency keys continue to reuse the original receipt;
  • accepted work without a terminal record is enqueued again.

That final rule provides **at-least-once** execution. It cannot prove that a side effect did not happen immediately before a crash, so durable handlers should use idempotency keys or make their external operations idempotent.

`journal_compact_after` bounds journal growth. After that many records, the runtime atomically replaces the journal with a current snapshot of retained receipts and terminal states.

Job status adds:

  • `durable`: whether journaling is enabled;
  • `journal_events`: records read or written since definition;
  • `journal_errors`: failed reads, parses, writes, synchronizations, or compactions;
  • `replayed`: enqueue records restored from the journal.

Deterministic retry and scaling tests

Reliability and scaling tests should not sleep merely to age retry, queue, or idle deadlines. The Jobs test clock controls retry deadlines and adaptive scaling decisions:

let started_at = Jobs.test_clock_set(0)
let advanced_to = Jobs.test_clock_advance(100)

`Jobs.test_clock_set(milliseconds)` enables the manual Jobs clock and sets its absolute value.

`Jobs.test_clock_advance(milliseconds)` moves it forward, wakes the retry timer thread, and makes the new time visible to scaling decisions.

These helpers make retry/backoff and scale-up/scale-down specs deterministic. They do not change file, network, process, or general `Time.now_ns()` behavior.

Choosing the abstraction

Use a function when the caller should wait synchronously.

Use an actor when one identity owns persistent private state.

Use a bare worker pool when the caller owns result routing and only needs bounded parallel execution.

Use a Job for receipts, result records, retries, cancellation, idempotency, durability, metrics, and runtime lifecycle.

Use worker-backed Streams—once their roadmap slice lands—for familiar bounded `map`, `filter`, `each`, and `.through(Job)` pipelines.
