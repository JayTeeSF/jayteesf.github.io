v0.98.39a: Source handoff and slice reporting

This slice makes `bin/hey-zip-source` a clearer handoff tool for long-running

Hey development threads.

The helper still creates a clean source zip, excluding `.git`, `build`, `dist`,

`.hey`, dSYM bundles, and generated ignored cruft. The important change is the

continuation prompt: it now tells the next LLM how to produce cumulative alpha

slices and how to report them back to JayTeeSF.

Source zip workflow

Create the current source handoff zip:

./bin/hey-zip-source --downloads

That writes:

~/Downloads/hey-lang-bootstrap-plan-<version>.zip

Apply a returned zip from the root of the existing git checkout:

cd ~/dev/hey-lang-bootstrap-plan
unzip -o ~/Downloads/hey-lang-bootstrap-plan-<version>.zip

`unzip -o` does not remove files that were deleted or renamed in the source tree.

If a slice deletes or renames files, the assistant must report exact `rm` commands.

If no files were deleted or renamed, the final answer should explicitly say:

rm commands needed: No.

Reporting multiple builds

When multiple alpha builds are returned in one answer, every zip should be

cumulative. The final zip should contain all prior changes. Present commands in

four copy/paste-friendly blocks, in this order:

1. all download/source-zip links or download commands, in version order;

2. all unzips plus `git status --short`, `git add .`, and `git commit` for each

slice, in version order;

3. all final verification/testing commands except the expensive full backend gate,

run only after the latest commit;

4. the expensive `backend-world-full` command last, in its own block, so the

summary and failure rows can be copied back together.

For intermediate builds, a compact apply/status/add/commit block is enough:

unzip -o ~/Downloads/hey-lang-bootstrap-plan-v0.98.40a.zip
git status --short
git add .
git commit -m "v0.98.40a - concise slice description"

For the final build, include the full verification and check-in commands:

git status --short
git add .gitignore Makefile README.md bin benchmarks docker docs examples runtime specs stdlib test tools
git status --short
git commit -m "Release hey vNEXT concise description"

The final build response should include the relevant small checks plus the

standard gates:

make clean

cd "$dir"
./bin/heyc --version
./bin/heyc --backends
printf 'let xs = ["Ada"]\nsays to_json([...xs, "next"])\nquit\n' | ./bin/hey-repl --quiet
cd -

The real 1B-row 1BRC proof should only be run when JayTeeSF asks for it or when a

1BRC-specific slice needs validation:

./bin/perf 1brc-full \
  --data ../1brc/measurements.txt \
  --rows 1000000000 \
  --workers "8" \
  --no-progress \
  --out build/m3-1brc-proof-no-progress

Backend-world full evidence block

Place this expensive gate last in final answers. Use `HEY_BACKEND_WORLD_JOBS=8` on JayTeeSF's M3 for the fast parallel full-world proof.

./bin/test --full
./bin/test --dual-timing
HEY_BACKEND_WORLD_JOBS=8 ./bin/test --backend-world-full

Current handoff facts

• Native-C is still the active/default backend.

• LLVM is selectable only for its checked subset, including generated 1BRC

scale runtime lowering. It is not yet the user-facing no-toolchain

distribution and does not have full runtime/data-model parity yet.

• Self-hosted compatibility coverage is `supported=12 unsupported=0`, currently

`direct_count=12` and `bridged_count=0`; primary-compiler promotion is still

pending until parser/build/spec/REPL, diagnostics, backend handoff, and

performance gates pass.

• Backend-world parity work should favor many tiny implementation-independent

language specs and general lowering contracts. Do not turn challenge harnesses

such as C10K into benchmark-specific compiler shortcuts.

• `1brc-scale` is the preferred confidence ramp before expensive full-file proof

runs.
