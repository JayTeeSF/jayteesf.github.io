v0.99.213a HIR/MIR consumer boundary

The LLVM-hosted Hey compiler now emits semantic HIR and ABI MIR rather than a single durable combined artifact. The current LLVM consumer still receives a temporary Hey-owned `hey-ir 0.2` translation, so the next parity milestone is direct verified-MIR consumption. New LLVM semantics must be added to HIR/MIR first; do not widen the temporary adapter or create another backend source analyzer.

The compiler contract proves source deletion before consumption, byte-identical HIR/MIR across native-C-hosted and LLVM-hosted producer executions, cache corruption recovery, and no native C sidecar schema implementation. These are implementation contracts, not language conformance requirements.

v0.99.212a LLVM-hosted frontend milestone

The shared LLVM consumer now receives canonical IR from the native frontend, the native-C-hosted Hey frontend, and the LLVM-hosted Hey frontend. Optional function signatures and corrected string map updates remove the ABI corruption that appeared when compiler implementation code was lowered directly. Remaining gaps are canonical-IR surface coverage and the stale full-world run-failure denominator, not the existence of the self-hosted LLVM-output lane.

v0.99.211a shared-consumer boundary

LLVM parity is now evaluated through two implementation paths:

  • transitional direct source-to-LLVM lowering for the broad existing surface;
  • canonical `hey-ir 0.2` consumption for the growing portable core tier.

Both the native C frontend and the Hey-written frontend emit the canonical tier

and use the same source-independent LLVM consumer. Language conformance remains

outside-in and does not require LLVM or canonical IR. The next self-host gap is

to build the Hey IR producer executable itself with LLVM, then expand canonical

IR coverage toward modules/import identity, actors/jobs, source spans, effects,

and complete boxed typing.

v0.99.211a canonical-IR and dominance update

The core LLVM path now has a source-independent typed-IR consumer shared by native and Hey-written frontends. Wider LLVM coverage still uses direct source lowering until canonical IR gains modules/imports, complete boxed typing, actors/jobs, effects, and source spans.

The v0.99.210a exact closure failure in `tools/check_ergonomic_examples.hey` was invalid SSA: `%checked.addr` was allocated in one sibling branch and used in another. Reused lexical scalar slots are now predeclared in the entry block.

v0.99.210a boxed parameter and matrix update

LLVM now classifies `len(parameter)` as boxed use, fixing spec 816's array/string ABI confusion. The compiler matrix proves native LLVM application output and an LLVM-built self-host compiler executable separately. Self-hosted LLVM application output remains blocked on typed-IR emission.

v0.99.209a desugaring repair

Spec 793 exposed a front-end recursion before LLVM emission: qualified

`collections.*` calls were repeatedly receiver-desugared. Defined qualified

calls now take precedence, and LLVM type/boxed inference has a bounded fallback.

The focused suffix/collection programs build and match native output. This does

not change the stale full denominator: 570/646 detailed correctness, 58

unsupported, 16 run failures, and zero mismatches from v0.99.205a.

v0.99.208a focused unsupported closure

The latest complete denominator remains v0.99.205a: 646 programs, LLVM detailed correctness 570/646, 58 unsupported, zero build failures, 16 run failures, and zero output mismatches. `test/llvm-v205-unsupported.paths` preserves those exact 58 source paths.

After general runtime-facade delegation, boxed value propagation, parameter/return ABI separation, local dataflow fixes, dynamic-key inference, and scope-restoration fixes, a fresh focused compile of that exact inventory is 58/58 with zero unsupported or build-failed rows. This is a build/lowering closure, not a newly inferred full-report percentage. Some development tools require command-line arguments, and the generic stdlib bridge is still source-compiler-backed.

Next: run meaningful rows for output parity, classify argument-requiring tools explicitly, replace the transitional bridge with direct runtime ABIs, and close the known 16-row run-failure set before refreshing the complete denominator.

This slice focuses on self-host timeout removal and does not rerun the LLVM denominator. The next LLVM slice should close the largest shared family among those 58 unsupported rows.

LLVM parity gap dashboard

v0.99.137a makes the remaining LLVM gap explicit instead of treating every unsupported row as the same problem. The current uploaded baseline is clean but incomplete: native-C and self-hosted-C are correctness oracles, while LLVM still has unsupported rows and no known output-mismatch/build-failure class in the v0.99.135a report.

Command

./bin/subcmds/hey-llvm-gap-report
./bin/subcmds/hey-llvm-gap-report --tsv build/backend-world-full/backend-world.tsv

The report reads `build/backend-world-full/backend-world.tsv` plus row-local LLVM build logs under the same output directory. It emits JSON rows so follow-up tools can consume them without awk/perl/Python/Ruby one-offs.

Output contract

The first rows are:

  • `llvm_gap_source`: source TSV, base directory, existence, and byte count.
  • `llvm_gap_summary`: total unsupported/problem rows, output mismatches, build failures/timeouts, unknown reasons, self-hosted-on-LLVM rows, example/porcelain rows, expected-error rows, and weighted score.
  • `llvm_gap_bucket`: weighted scores for strings, data_model, functions, actors, jobs, io, control, modules, c10k, and other.
  • `llvm_gap_signal`: counted detailed signals such as `string_values_in_functions`, `string_trim_slice_split`, `data_model_facades`, `user_function_calls`, `actor_runtime`, and `jobs_runtime`.
  • `llvm_gap_reason_code`: stable reason-code counts.
  • `llvm_gap_top_row`: the top weighted rows, including source, role, reason code, bucket, signal, expected-error flag, and source reason.

Stable reason-code families

  • `llvm.string.*`: string predicates, trim/slice/split, find/len/concat, string literals, string expressions, and string function bodies.
  • `llvm.data_model.*`: object/array literals and facade calls such as get/put/has/keys/values/entries/json/to_json.
  • `llvm.call.unsupported`: user/module calls not yet expressed through the general call lowerer.
  • `llvm.actor.runtime` and `llvm.jobs.runtime`: actor/message/jobs runtime gaps.
  • `llvm.io.process_file` and `llvm.net.c10k`: file/process/socket/C10K runtime gaps.
  • `llvm.control.unsupported`: control lowering gaps.
  • `llvm.module.import`: module/import/classification gaps.
  • `llvm.other.unsupported`: deliberately rare fallback; top rows should not stay here.

Current interpretation

The latest latest uploaded pre-v0.99.146a promotion evidence showed LLVM failing by coverage, not wrong output: LLVM good=362/504, unsupported=130, output_mismatch=0, and build_failed=0. The weighted focus is dominated by strings, then data model and generic calls. The near-term closure order is therefore:

1. strings in functions/control/data-model contexts,

2. boxed/data-model expressions over arbitrary values,

3. generic user/module calls through canonical symbols and typed IR,

4. actors/jobs runtime internals after their dependencies are ready,

5. file/process/net/C10K rows,

6. compiler/tooling self-hosted-on-LLVM proof rows.

v0.99.137a closure slice

v0.99.137a adds the dashboard and closes the first string-control function row. LLVM string functions with control are no longer misclassified into the linear string emitter, and string locals can be updated from string expressions inside control-heavy LLVM functions. Spec 785 proves the path with a loop, branch, string slice, string concatenation, string local update, and string return.

Smoke target

make llvm-gap-report-smoke

The smoke target builds a tiny synthetic backend-world TSV/log tree and proves that the report emits a summary, a stable string reason code, and at least one top-row record without requiring a full promotion sweep.

v0.99.138a closure slice

v0.99.138a continues Phase 2 by moving another string island into general user-function lowering. Linear string-returning functions now lower text edge helpers through existing LLVM runtime ABI calls: replace/gsub/replace_all, strip-prefix/suffix, surround, repeat, pad/truncate, indent, and dedent-marker. Spec 786 proves these helpers compose across local variables, calls, and returned strings. This targets the `string_values_in_functions` and `string_trim_slice_split`/text-edge focus signals without adding row-specific benchmark shortcuts.

The same slice fixes the v0.99.137a promotion-report accounting false-negative for no-program/direct-run tool rows. If native-C ran successfully and recorded reference output, self-hosted `ok`/`reference` output now counts as parity, so tools such as `tools/hey_llvm_gap_report.hey` no longer fail self-hosted promotion merely because their backend-world row is classified as no-program metadata.
