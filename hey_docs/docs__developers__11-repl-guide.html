Hey REPL Guide

The persistent interactive REPL is currently implemented in C and shares the C runtime with compiled programs. Complete source fragments can already bypass that evaluator and use the product LLVM pipeline.

Start:

./bin/hey-repl

The REPL also persists history across sessions. Selection order and inline-comment behavior are documented in REPL history and callable values.

Interactive terminals support in-session command history and a small Vim-style

line editor. The prompt starts in insert mode. Use Up/Down to recall history,

Left/Right to move the cursor, or press Esc to enter Vim normal mode. In normal

mode, `h`/`l`, `0`, `^`, `$`, `w`, and `b` move; `x` deletes a character; `D` or

`d$` deletes to the end; `C` changes to the end; `r<char>` replaces one

character; and `i`, `a`, `I`, or `A` return to insert mode. Piped REPL specs and

pasteable scripts still use plain stdin behavior.

Compile a complete fragment through LLVM

Use the complete-fragment lane when session persistence is not required:

./bin/hey-repl --llvm-fragment path/to/fragment.hey arg1 arg2

This command assigns compile-request mode `repl-fragment`, lowers the source

through semantic HIR and ABI MIR, compiles it with the direct LLVM consumer, and

executes the cached native program. It does not enter the C expression

evaluator. Warm execution must report `cache=hit backend=llvm` and must not

start a compiler.

This is not yet the persistent LLVM REPL: bindings, imports, function

redefinition, transactional session generations, and restricted `eval` still

use the interactive session described below.

Pasting tutorial blocks

The REPL accepts complete `actor ... end`, `fn ... end`, `module ... end`, and `program ... end` blocks. It also combines multiline JSON object and array literals inside imported source and pasted blocks, including common actor/job forms such as `send queue, { ... }`.

The REPL supports the same `worker`, `workers(...)`, and worker-backed Jobs foundation as compiled programs. No import is required for the primitives. `import 'std:Workers'` adds `Workers.await_idle(...)`; `import 'std:Jobs'` loads additional Jobs helpers. Define the worker first, then create the pool or Job inside a `program ... end` block. `Jobs.result(Job, receipt, timeout_ms)` waits for a receipt and returns a `{status, ok, value/error}` result. Jobs may use fixed `count:` sizing or the complete adaptive `min_workers:`/`max_workers:`/`target_queue_latency_ms:` trio.

Imports

The REPL can import Hey source files, including stdlib modules with nested imports. Imported files define their functions, modules, and actors in the current REPL session; `program` and `spec` blocks in imported files are not run.

Reflection metadata in the REPL

After importing `std:reflection`, functions defined interactively retain normalized source text and the physical input line where the declaration began. Their location path is `<repl>`. Functions imported from a file retain that file path and declaration line. Callables without source metadata return `nil` safely.

Evaluate reflected source

import 'std:Eval'
let copy = eval(greet.source())
says copy('Jay')
says copy.source_location().path
says Eval.expression('20 + 22')

`eval(...)` is shorthand for `Eval.callable(...)`. It installs one restricted function or worker in the current session and returns the callable. `Eval.into_session(...)` also accepts modules and actors. Evaluated source is labeled `<eval>` line `1`, cannot import capabilities, and cannot use filesystem/process/network/native/package/environment/archive/direct-I/O builtins. `Eval.available?()` is true only in `hey-repl`; precompiled programs do not embed this evaluator. See Evaluate source in the REPL.

import 'std:Reflection'

fn greet(name)
  return "hello, #{name}"
end

says greet.source_location().path
says greet.source_location().line
says greet.source()

Reflection lookup is defensive: absent executable sidecars, absent function records, and missing module metadata do not emit internal JSON-access diagnostics.

import "stdlib/Json.hey"
import "stdlib/Jobs.hey"
import "stdlib/HeyC.hey"

Example:

import "stdlib/HeyC.hey"
let source = 'program\n  says 42\nend\n'
let emitted = HeyC.emit_text('inline', source)
says emitted.ok
quit

Copy/paste: process ring

actor done
  receive msg
  says msg
end

actor node(id, done_actor)
  receive next
  forever
    receive hops
    if hops == 0
      send done_actor, 'done at node ' + id
    else
      send next, hops - 1
    end
  end
end

program
  let done_actor = spawn done
  let one = spawn node(1, done_actor)
  let two = spawn node(2, done_actor)
  let three = spawn node(3, done_actor)

  send one, two
  send two, three
  send three, one

  send one, 5
  join done_actor

  stop one
  stop two
  stop three
end

quit

REPL and compiled-product smoke

Persistent transcript specs live in `specs/repl/*.in` and are run by:

make repl-specs

The REPL is an interactive language service. Application compilation uses the single product pipeline—LLVM-hosted Hey -> semantic HIR -> ABI MIR -> direct LLVM—and its behavior is covered by the language/compiler contracts.

Jobs/file/evented transcripts

The REPL suite includes persistent transcripts for jobs import diagnostics, file descriptor I/O, and evented-I/O facade execution:

specs/repl/201_named_jobs.in
specs/repl/202_adaptive_jobs.in
specs/repl/170_file_descriptor_io.in
specs/repl/180_evented_io_descriptor.in

They are intentionally small so REPL parity failures can be diagnosed one contract at a time.
