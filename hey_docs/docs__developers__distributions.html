Distributions and platform support

v0.99.259a three-root SDK closure

A target-contained SDK is staged from three explicit roots:

HEY_SDK_TOOLCHAIN_ROOT  -> libexec/hey/toolchain
HEY_SDK_SYSROOT_ROOT    -> libexec/hey/sysroot
HEY_SDK_NATIVE_ROOT     -> lib/hey/native

Build it with:

make dist-sdk-host-contained \
  HEY_SDK_TOOLCHAIN_ROOT=/path/to/clang-lld-resources \
  HEY_SDK_SYSROOT_ROOT=/path/to/target-sysroot \
  HEY_SDK_NATIVE_ROOT=/path/to/relocatable-native-libraries

Installed compilation uses the SDK-local compiler, sysroot, and native libraries. Linux applications receive an SDK path plus `$ORIGIN/../lib/hey/native`; Darwin applications receive an SDK path plus `@loader_path/../lib/hey/native`. A run bundle can therefore copy the application and native tree without shipping the compiler.

Two artifact kinds

Hey release tooling now creates two deliberately different artifacts.

Compiler SDK

The SDK contains the north-star compiler commands, private MIR consumer, standard library, documentation, examples, and a **precompiled target runtime object**:

bin/hey
bin/hey-selfc
bin/hey-package
libexec/hey/heyc-bootstrap
lib/hey/runtime/hey_runtime.o

Ordinary SDK application builds link the precompiled runtime object. Recipients no longer need to compile `runtime/hey_runtime.c` or install its C/OpenSSL development headers merely to build a Hey program.

The compiler searches for LLVM in this order:

1. `HEY_LLVM_CC` or `CLANG` when explicitly configured;

2. `libexec/hey/toolchain/bin/clang` inside the SDK;

3. `clang` on `PATH` as the development-checkout fallback.

A release builder may stage a relocatable LLVM tree with:

make dist-sdk-host-contained HEY_SDK_TOOLCHAIN_ROOT=/path/to/relocatable/toolchain

The supplied tree must expose `bin/clang`, `bin/ld.lld`, and `lib/clang`. SDK-local builds select LLD explicitly and discover compiler resource headers from the copied tree. Linux-x64 has been proven through the exact non-default release graph with compiler discovery removed from `PATH`, emitting HIR and compiling/running a real application from a relocated SDK. v0.99.259a adds explicit target-sysroot and native-library roots. Linux relocation and run-bundle closure are proven; Darwin uses the same layout and retains a required Apple Silicon host receipt.

Precompiled application bundle

A run bundle contains one already-compiled native application and no compiler:

make dist-run-host \
  RUN_APP=build/my-app \
  RUN_APP_NAME=my-app

The result contains `bin/my-app` plus a capability receipt. It is the smallest artifact for recipients who only need to run the application.

Relocation and compiler closure

`make dist-host` is the canonical SDK construction graph used by local releases and container vendoring. Vendoring does not maintain a second compiler bootstrap. The graph pins `hey-selfc`, `hey`, and `hey-package` to the bootstrap compiler inside their own release directory.

Installed tools must work outside the source checkout:

  • `bin/hey-selfc` discovers `libexec/hey/heyc-bootstrap` and the copied `stdlib/`;
  • `bin/hey` resolves the SDK root from its installed path and passes that root into cached applications;
  • delegated pure-Hey stdlib calls use temporary files under `HEY_STDLIB_DELEGATION_DIR`, `TMPDIR`, or `/tmp`, never a checkout-local `build/` directory.

The Linux release gate builds the SDK with a non-default build directory, changes to an unrelated working directory, unsets discovery overrides, sets `PATH=/nonexistent`, checks the staged toolchain receipt, emits semantic HIR, compiles and runs `examples/hello.hey`, and runs target JSON/version verification. This proves fixed-point release and compiler-toolchain closure, not target sysroot/native-library self-containment.

Shell-script policy

Released public commands are compiled Hey binaries. Distribution staging does not ship Bash, Ruby, Python, or Node launchers.

A source checkout may retain a tiny shell bootstrap shim when no installed `hey` exists yet. `bin/hey-zip-source` is such a bootstrap; its archive policy, validation, and prompt generation live in `tools/hey_zip_source.hey`.

Normalized SDK-tree copying

Compiler resource directories and platform SDKs may contain internal compatibility symlinks. They are not copied with the ordinary package/source-tree copier: `files.copy_tree` intentionally rejects every symlink.

Distribution staging and vendoring use `files.copy_sdk_tree` instead. It:

  • never follows a symlink while walking;
  • preserves relative links whose lexical targets remain inside the copied root;
  • rejects absolute links, root escapes, sockets, devices, and other special files;
  • preserves internal alias cycles without traversing them;
  • applies the same policy when staging the SDK and when publishing it into an application vendor directory.

This single policy replaces the previous mixture of `cp`, `tar`, and the symlink-rejecting package copier.
