Timer and counter semantic intrinsics

Status: roadmap/design note. This is not implemented yet.

Hey should avoid introducing numeric subtypes just to make common runtime values safer or faster. `Timer` and counter-style modules can be standard-library porcelain with compiler-recognized intrinsic contracts underneath.

The pattern is:

Public API: nice module abstraction
Compiler view: known primitive with invariants
Backend view: target-specific lowering

This keeps the language small while still giving the compiler enough facts to optimize and validate code.

Timer

A `Timer` module should expose monotonic time and elapsed-duration helpers:

let timer = Timer.start()
# work
let elapsed = timer.elapsed_ns()

Suggested public surface:

Timer.now_ns      -> monotonic instant
Timer.start       -> captured monotonic start instant
Timer.elapsed_ns  -> non-negative duration in nanoseconds
Timer.elapsed_s   -> floating reporting value derived from elapsed_ns

Compiler contract:

Timer.now_ns is effectful and nondeterministic.
Timer.now_ns must not be reordered across observable effects unless a future explicit rule allows it.
Timer.elapsed_ns returns signed Int64 with a proven non-negative invariant.
Timer.elapsed_s is derived from elapsed_ns and exists for reporting, not as the core duration value.

The important value category is not unsigned arithmetic. It is a signed integer duration with a non-negative proof:

DurationNs = signed Int64 where value >= 0 when produced by Timer elapsed/difference APIs

That gives the optimizer a useful fact without inheriting unsigned wraparound bugs. For example, a branch checking whether a trusted elapsed duration is negative can be eliminated.

Backend lowering should be implementation-specific but contract-compatible:

Native-C backend: clock_gettime(CLOCK_MONOTONIC, ...) or a Darwin monotonic primitive
LLVM backend: runtime intrinsic or LLVM-compatible host call
Self-hosted backend: emit the same runtime intrinsic call
REPL: host runtime monotonic clock

All backends must agree on the same observable contract even if the host primitive differs.

Counter families

A single vague `Counter` would hide too much. Split the semantics so the compiler/runtime can use stronger facts:

Counter              mutable non-negative count
MonotonicCounter     count that only increments
AtomicCounter        shared/thread-safe counter
ReductionCounter     worker-local counts merged at a structured join

Shared facts:

Counter.value >= 0
Counter.increment mutates state
Counter.increment_by(k) is a checked add when k >= 0
Public APIs cannot underflow
Overflow behavior must be explicit: trap, saturate, or return an error

Specialized facts:

MonotonicCounter never decreases.
AtomicCounter.increment must preserve explicit atomic ordering.
ReductionCounter is valid only across structured job/supervisor joins.
ReductionCounter may be privatized per worker and merged at the join point.

The biggest optimization is not choosing unsigned integers. It is recognizing reduction semantics. A hot parallel loop should not need every worker to perform expensive atomic increments when the work can be represented as worker-local counts plus a final sum.

Possible Hey shape:

let c = Counter.new()
c.increment()
c.increment_by(5)
let n = c.value()

Possible reduction shape:

let c = ReductionCounter.new()

jobs.perform(items) do |item|
  if item.valid?()
    c.increment()
  end
end

print(c.value())

Compiler/runtime lowering for the reduction case may become:

worker_1_count += 1
worker_2_count += 1
...
global_count = sum(worker_counts) at the structured join

Invariant vocabulary

The useful future compiler signal is a small invariant vocabulary, not a full public type system:

Int64
NonNegative<Int64>
Positive<Int64>
DurationNs
MonotonicInstantNs
CounterValue

These are compiler/backend facts. They do not require user-visible type declarations in the first implementation.

The facts enable optimizations and safety checks such as:

this value cannot be negative
this value only increases
this operation reads time and is not pure
this counter can be reduced across workers

Language and backend contracts

Document these as implementation-independent contracts before lowering them broadly:

Timer.now_ns:
  - returns signed Int64 monotonic timestamp
  - effectful and nondeterministic
  - not pure
  - lowered as a backend/runtime intrinsic

DurationNs:
  - signed Int64
  - non-negative when produced by Timer elapsed/difference APIs
  - convertible to Float/Double for reporting

Counter:
  - signed Int64 internally
  - invariant: value >= 0
  - underflow unavailable through public API
  - overflow behavior explicit

MonotonicCounter:
  - value never decreases
  - decrement unavailable
  - useful for metrics, ids, sequence numbers

AtomicCounter:
  - explicit atomic semantics
  - backend lowers to host atomics/runtime atomic helpers

ReductionCounter:
  - valid only across structured job/supervisor joins
  - backend/runtime may privatize and merge

Roadmap rule

Start as standard-library modules backed by runtime intrinsics. Add tiny implementation-independent specs first, then lower the shared intrinsic contracts across native-c, self-hosted, LLVM, and the REPL. Do not add unsigned numeric types or public type syntax merely to support these modules.
