v0.99.205a LLVM/tooling closure

LLVM `files.read_text` is now one ordinary string-expression capability rather than an unresolved imported function. That single lowering closes the uploaded report's complete 11-row build-failure cluster across vendor, source-archive, distribution, dependency-audit, and build-contract programs. The tiny self-host compiler service also handles every string-length alias, removing the report's only LLVM output mismatch. A 14-row focused backend-world gate is now 14/14 on both native-C and LLVM with no mismatches. C11 reserved identifiers are also escaped centrally across C-emitting backends. The architectural next step remains typed IR/LLVM object output from the Hey-owned compiler, not more emitted-C specialization.

v0.99.204a strategy update

Pure LLVM programs now direct-link without the monolithic runtime object, and the comparison harness includes an in-program integer recurrence rather than relying on process-wall smoke timing. The first maintained result is a substantial LLVM win (4 ms versus 177 ms at 1,000,000 iterations here), but promotion still requires representative runtime-bearing workloads and the remaining 55 unsupported rows. Self-host parsing is content-addressed and shared, but final self-host output remains C/cc.

v0.99.177a update: continues LLVM compatibility/performance by lowering ordinary C10K (`c10k_ping`/`net.c10k`), file_partitions, file-backed 1BRC station formatting, and boxed get-plus aggregation through normal runtime/data-model ABI paths. Specs/IR 822 and 823 pin the C10K/file-partition/file-backed-1BRC and 1BRC-style aggregate cases; `benchmarks/hey/one_brc_style_aggregate.hey` now builds/runs under LLVM without benchmark program-shape shortcuts. Latest pasted baseline before this slice was LLVM correctness=523/538 and good=507/538 with 15 unsupported rows; rerun promotion-report to measure the new denominator.

v0.99.177a strategy update: C10K and file-backed 1BRC are now ordinary LLVM runtime-surface compatibility work, not policy exceptions. `c10k_ping`, `net.c10k`, `file_partitions`, one_brc aggregate/materialized helpers, and file-backed station formatting lower through runtime ABI shims and general boxed/string/scalar lowering. This keeps the no-shortcut rule intact while removing arbitrary LLVM rejections.

v0.99.176a strategy update: correct the earlier overcorrection around 1BRC/C10K. LLVM must not use benchmark program-shape shortcuts or canned answers, but it also must not arbitrarily reject public runtime functions used by challenge programs. one_brc_* now lowers through ordinary runtime ABI shims, allowing the same generated 1BRC scale/full-proof sources to run under LLVM.

v0.99.133a LLVM parity update

The v0.99.132a promotion report was healthy for native-C/self-hosted but exposed one LLVM output mismatch at `specs/language/744_regex_replace_empty_match_contract.hey`. v0.99.133a fixes the runtime-shim semantics for zero-width `re.replace` and lowers the actor-backed remote package fetch row 770. The copy-paste-safe `./bin/subcmds/hey-llvm-priority-proof` now covers 20 cumulative rows.

Strategy note: runtime shims must mirror Hey stdlib semantics, not merely the nearest C/POSIX primitive. For regex replacement, POSIX zero-width matching differs from `stdlib/re.hey`; the LLVM shim now follows the Hey loop rule by consuming one character after an empty match and avoiding terminal empty replacement.

Next LLVM focus should be broad, weighted unsupported reduction from the latest promotion focus: strings first, data-model facade calls second, then user function calls, actor runtime, jobs runtime, file/process I/O, and c10k/net.

v0.98.44a LLVM runtime collection path note

The LLVM backend now handles the existing 1BRC runtime generation and summary specs that require runtime object key output, nested `result["results"][N]` reads, and nested nil checks. This is still runtime-backed subset growth, not full LLVM language parity.

Tiny jobs specs now define the next jobs lowering ladder: queue metadata, actor specs, duration/retry values, job ids, envelopes, worker refs, result delivery, and dead letters should be lowered as general contracts before treating the whole production jobs fixture as proof.

The fastest backend parity path is now a ladder of small specs: lower core data-model/output contracts, stdlib facade contracts, and jobs contracts independently before treating large fixtures as promotion evidence.

v0.99.46a actor/job ABI step

LLVM now lowers the first general runtime-facing actor slice without challenge-specific shortcuts. The supported subset is intentionally narrow but real: zero-argument one-shot actors whose receive body performs direct output/key-output and exits; `spawn`, `send`, `join`, `stop`, `monitor`, `monitor_info`, and `sleep`; and jobs runtime calls that can be expressed through those handles (`jobs.worker_ref`, `jobs.start_pool`, `jobs.perform`, `jobs.stop`, and `jobs.status`). This is the bridge from descriptor-only jobs lowering toward the actor/mailbox/result-waiter work needed by `jobs_production`.

Still unsupported by design: stateful actors, actors with constructor args, arbitrary control flow in actor bodies, challenge-specific C10K/1BRC helper shortcuts, and full result-waiter production jobs under LLVM.

v0.99.54a timing/sleep note

LLVM is timed separately via `./bin/test --backend-timing` / `--triple-timing`; it does not accelerate native-C or self-hosted rows. Sleep now accepts decimal millisecond values (`sleep 0.5`) across interpreter/native-C/self-hosted and LLVM literal lowering, backed by runtime `nanosleep` with OS-scheduler-bounded precision.

v0.99.142a strategy note

The latest blocker was not LLVM correctness; it was native-C failing to build a valid spec due bare `return` emission. Fixing all front doors together keeps native-C as a usable oracle while LLVM continues toward full parity.
