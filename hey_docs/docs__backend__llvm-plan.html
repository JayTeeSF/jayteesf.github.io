LLVM backend plan

LLVM is the next major backend track for Hey. The goal is portable optimizing binaries and a user distribution that can compile Hey programs without asking users to manage C or LLVM toolchains manually. Full-compliance sequencing, the general-lowering ladder, and the shared typed-IR pivot now live in the three-backend parity plan; this doc keeps the LLVM-specific research questions and slice history.

Current backend truth

• `native-c` is active, default, and fully available.

• `llvm` has typed IR/ABI handoff tooling and is selectable for a checked scalar/string/control-flow/stdout subset, plus string file I/O, string slice/compare, and literal for-loops. It still requires clang in maintainer/source checkouts until no-toolchain packaging is complete.

• `self-hosted-c` is now a selectable full-parity alternative after parser/build/spec/REPL parity and backend-world matched native-C; native-C remains default/oracle while LLVM/no-toolchain distribution continues.

• Growth path: no more per-line pattern lowering beyond the current subset; new coverage arrives by making `llvm` consume the typed IR (parity plan Track C) and climbing the spec-clustered ladder (Track D) — data model first, then text/file facades, modules on canonical symbols, actors, jobs, and evented I/O.

Why LLVM

LLVM is a mature optimizer and code generator. It provides tutorials for building a language frontend, generating LLVM IR, adding optimization/JIT support, and compiling object code. Rust's main codegen path lowers from MIR through an LLVM backend into machine code, which is a useful model for Hey's planned typed IR handoff. Go's standard compiler is a useful comparison point for an SSA-based native compiler pipeline, while gollvm/llvm-goc are useful examples for Go-family LLVM experiments rather than proof that Go itself requires LLVM.

Research questions before implementation-dependent slices

1. Should Hey first emit textual `.ll`, LLVM bitcode/object files, or call LLVM APIs directly?

2. Should the first backend shell out to `llc`/`clang` in maintainer builds, or link LLVM libraries into a Hey compiler binary?

3. What is the smallest typed IR that both native-C and LLVM can consume without duplicating frontend logic?

4. How should Hey values cross the runtime ABI boundary: boxed `HeyValue`, unboxed scalars, tagged unions, or specialized layouts?

5. How will actors, mailboxes, and scheduler/runtime calls be represented in IR?

6. How will garbage/lifetime rules work for strings, arrays, maps, actors, and external resources?

7. How will the distribution package target triples, runtime objects, linker support, and platform-specific libraries?

8. How do we preserve source-location diagnostics and debuggability while optimizing?

9. Which benchmark proves promotion: 1BRC, ring, C10K, compile time, binary size, startup time, or a blended gate?

Alpha slices

• `v0.98.24a`: define the typed IR and runtime ABI boundary. **Done:** `heyc ir` and `--emit-ir` expose the shared handoff.

• `v0.98.25a`: emit textual LLVM IR for scalar arithmetic and simple strings in maintainer builds. **Done:** `heyc llvm-ir` and `--emit-llvm-ir`.

• `v0.98.26a`: build one standalone executable through LLVM tooling in maintainer builds. **Done:** `heyc llvm-smoke` emits `.ll` and shells to `clang` when available.

• `v0.98.27a`: compare LLVM vs native-C on startup, binary size, compile time, and scalar-smoke run time. **Done:** `bin/perf llvm-compare`.

• `v0.98.28a`: lower integer comparisons and `while`/`until` hot loops in LLVM smoke IR. **Done:** `bin/perf llvm-compare` reports `hot_loop_ready=true` only when the hot-loop fixture matches Native-C output.

• `v0.98.43a-v0.98.49a`: harden object emission, target triples, runtime linkage, platform packaging, and no-toolchain distribution.

Non-goals for the first LLVM slice

• Do not replace native-C immediately.

• Do not promise full self-hosted parity.

• Do not expand `--backend llvm` beyond the checked subset until each added construct has parity fixtures and honest limitations.

• Do not make end users install LLVM manually as the intended product experience.
