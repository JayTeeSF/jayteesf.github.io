Hey developer documentationAudience-specific Hey documentation

Static browser playground and learning portal

v0.99.276a publishes the Tour of Hey as a static HTML/CSS/JavaScript application. It is generated as part of the ordinary documentation site, so it can be committed directly to `jayteesf.github.io` and served by GitHub Pages without an API server.

Publish with the documentation site

./bin/hey doc-site \
  --out "$HOME/dev/jayteesf.github.io/hey_docs" \
  --kind all \
  --theme water \
  --jobs 11

The command writes the interactive site under:

$HOME/dev/jayteesf.github.io/hey_docs/learn/

The generated pages use relative asset paths and remain valid below `https://www.jayteesf.com/hey_docs/`.

Run without a server

Open the generated `learn/index.html`, or publish the documentation tree normally. The application needs only:

  • `learn/index.html`;
  • `learn/assets/style.css`;
  • `learn/assets/app.js`;
  • `learn/assets/hey-browser.js`;
  • `learn/catalog.json`;
  • the service worker and web manifest.

There is no `/api/run` endpoint and no `./bin/hey learn serve` command.

Browser backend

The browser backend is a constrained Hey-to-JavaScript transpiler and runtime. Execution occurs in a Web Worker:

Hey editor source
  -> browser-safe parser/transpiler
  -> generated JavaScript
  -> isolated Web Worker runtime
  -> bounded output returned to the page

It supports the complete maintained eight-lesson Tour surface:

  • immutable arrays and objects, including spread updates;
  • functions and first-class callable values;
  • isolated actors with private state, `send`, and `join`;
  • stateless workers and bounded pools;
  • Jobs, receipts, and results;
  • bounded streams with ordered collection;
  • virtual in-memory files, handles, and Result values;
  • route values and pure web dispatch.

This is deliberately not the native compiler pretending to be available on a static host. Browser support is an explicit secondary output backend with parity fixtures against the ordinary Tour programs. Native Hey remains the performance and deployment north star.

Compile one source file to JavaScript

Maintainers can inspect the generated module directly:

./bin/heyc browser-js examples/tour_01_values.hey \
  -o build/tour-values.mjs

The output exports `run(runtime)` and uses the same browser runtime shipped by the documentation site.

Safety boundary

The browser runtime has no host filesystem, process, socket, package, environment, or native-library access. The Tour's file examples use an in-memory virtual filesystem. Execution has a source limit, output limit, and Web Worker deadline; a timed-out worker is terminated.

This protects the page from accidental infinite work. It is not a claim that arbitrary untrusted JavaScript can be made harmless inside the same origin, so the backend exposes only generated code and its closed runtime API.

Verification

./bin/hey learn verify
node test/browser_learning_runtime_test.js .

test/regression_learning_portal_product_lane.sh
test/regression_learning_portal.sh

./bin/heyc specs/language/1020_browser_playground_profile.hey --test
./bin/heyc specs/language/1021_integrated_learning_portal.hey --test
./bin/heyc specs/compiler/997_browser_learning_portal_product_contract.hey --test

The runtime test executes all eight canonical Tour source files and compares exact stdout with the native Tour receipts. The documentation regression generates a complete static site and rejects server endpoints, absolute asset paths, or omitted browser assets.
