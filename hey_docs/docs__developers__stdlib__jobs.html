Jobs

Jobs are Hey's normal interface for managed background and parallel work. A Job owns a bounded pool of one stateless worker declaration and adds receipts, result records, acknowledgements, retries, cancellation, dead letters, idempotency, optional durable replay, metrics, and lifecycle.

Start with Workers and Jobs, step by step for complete annotated programs.

`jobs.define`

let ImportUsers = jobs.define(
  'ImportUsers',
  ImportUser,
  min_workers: 2,
  max_workers: 12,
  target_queue_latency_ms: 25,
  capacity: 256,
  retries: 2,
  backoff_ms: 100,
  max_backoff_ms: 5000,
  result_capacity: 1024,
  dead_letter_capacity: 128,
  idempotency_capacity: 4096,
  journal_path: 'var/jobs/import-users.jsonl',
  journal_compact_after: 4096
)

Arguments and options:

  • `'ImportUsers'`: human-readable operational name used by status, diagnostics, and journal filtering;
  • `ImportUser`: the actual worker value, not a string used for dispatch;
  • `count`: a fixed worker count; use this alone for deterministic fixed-size Jobs;
  • `min_workers`: adaptive pool floor and initial worker count;
  • `max_workers`: adaptive pool ceiling;
  • `target_queue_latency_ms`: oldest queued-work latency that triggers bounded scale-up;
  • `capacity`: bounded messages waiting to start;
  • `retries`: attempts allowed after the first failed attempt;
  • `backoff_ms`: delay before the first retry;
  • `max_backoff_ms`: cap for exponential retry delay;
  • `result_capacity`: retained receipt/result records;
  • `dead_letter_capacity`: retained terminal failures;
  • `idempotency_capacity`: remembered message idempotency keys;
  • `journal_path`: optional append-only durable journal; parent directory must exist;
  • `journal_compact_after`: event count that triggers atomic snapshot compaction.

Choose either `count:` or the complete adaptive trio `min_workers:`, `max_workers:`, and `target_queue_latency_ms:`. Mixing the fixed and adaptive forms, or supplying only part of the adaptive trio, is rejected. Adaptive workers are lightweight pooled actors on the existing managed scheduler; scaling does not create one operating-system thread per worker. Scale-up and retirement are rate-limited, and sustained idleness returns the pool to `min_workers`.

Without `journal_path`, the Job is process-local. With it, accepted work and terminal outcomes are synchronized before the runtime reports their state transition.

Thin `job Name ... end` declarations

`job Name ... end` is syntax only. It lowers before semantic HIR to the same `jobs.define(...)` call documented above:

worker ImportUser(message)
  return import_user(message)
end

job ImportUsers
  worker ImportUser
  count: 4
  capacity: 256
  result_capacity: 1024
end

program
  let receipt = ImportUsers.enqueue({user_id: 42})
  says jobs.result(ImportUsers, receipt, 5000).value
  let stopped = ImportUsers.stop()
end

The declaration must appear before `program`, contains exactly one `worker WorkerName` line, and accepts the same `key: value` options as `jobs.define(...)`. It does not create a second Job implementation or a new compiler representation. Explicit `jobs.define(...)` remains available when a Job must be constructed conditionally or inside a function. Thin declarations and explicit definitions produce equivalent semantic HIR and ABI MIR.

Keyed partitioning

Use `partition_by:` when messages for one business key must not execute concurrently:

fn ByAccount(message)
  return message.account_id
end

worker UpdateAccount(message)
  return apply_update(message)
end

let Updates = jobs.define(
  'AccountUpdates',
  UpdateAccount,
  count: 8,
  capacity: 256,
  partition_by: ByAccount,
  partitions: 32
)

`partition_by` must be an unbound ordinary function with one parameter. `partitions` must be a positive integer and is required whenever `partition_by` is present. The runtime canonicalizes the returned key, hashes it into one of the fixed shards, and prevents two handlers in that shard from executing at the same time. Other shards continue in parallel on the same bounded worker pool.

The fixed shard count bounds memory and shutdown work. It also means hash collisions deliberately serialize unrelated keys that land in the same shard. Increase `partitions:` when status shows a hot shard; do not assume one shard per distinct key. Partitioning guarantees non-overlap, not completion order or strict FIFO lock acquisition. Use explicit sequence values and stream `.resequence` when output order matters.

Receipts include `partition`. Job status adds `partitioned`, `partition_by`, `partition_count`, `partition_enqueued`, `partition_waits`, and `partition_wait_max_ns`. Retries keep the receipt's original partition.

Partitioning works with fixed `count:` pools and with adaptive `min_workers`/`max_workers` pools. It does not create one operating-system thread, actor, or queue per key.

Enqueue receipts

let receipt = ImportUsers.enqueue({user_id: 42})

Receipt fields:

  • `id`: monotonically increasing identity within this Job;
  • `job`: Job name;
  • `accepted`: whether bounded capacity admitted the message;
  • `status`: `queued` or `rejected` at submission time;
  • `duplicate`: whether an idempotency key reused an earlier receipt;
  • `retries`: configured retry limit;
  • `partition`: fixed shard index when the Job uses `partition_by`.

Always check `accepted` when overload must be handled immediately.

Worker acknowledgements

Returning an ordinary value acknowledges success:

worker ResizeImage(image)
  return resize(image)
end

`result.ok(value)` acknowledges success explicitly. `result.error(message)` reports a retryable failure:

import 'std:result'

worker ResizeImage(image)
  if image.valid == false
    return result.error('invalid image')
  end

  return result.ok(resize(image))
end

After retries are exhausted, the receipt becomes failed and its original payload enters the bounded dead-letter collection.

`jobs.result`

let outcome = jobs.result(ImportUsers, receipt, 5000)

The third argument is the maximum number of milliseconds the caller waits. Outcome fields:

  • `status`: `completed`, `failed`, `cancelled`, `rejected`, `timeout`, or `unknown`;
  • `ok`: success flag;
  • `value`: worker return value on success;
  • `error`: failure/cancellation reason;
  • `attempts`: handler attempts made.

A caller timeout does not currently terminate an arbitrary native handler. Ownership-safe compiler checkpoints remain a later reliability milestone.

Retries and delayed backoff

A failed attempt retries until `retries` is exhausted. Delays double from `backoff_ms` and stop growing at `max_backoff_ms`.

Delayed attempts live in one runtime timer queue; they do not occupy worker instances while waiting.

Cancellation

let cancellation = jobs.cancel(ImportUsers, receipt)

The returned object contains `cancelled` and `status`. Queued and delayed work can be cancelled before execution. A handler already executing may finish before observing the request.

Dead letters

for letter in jobs.dead_letters(ImportUsers)
  warn letter.id
  warn letter.attempts
  warn letter.error
  warn letter.payload
end

The collection is bounded by `dead_letter_capacity`.

Idempotency

let receipt = ImportUsers.enqueue({
  idempotency_key: 'user-42',
  user_id: 42
})

A duplicate key reuses the original receipt/result while the key remains inside `idempotency_capacity`. This prevents duplicate worker execution within retained state and is the recommended defense for durable at-least-once replay.

Durable replay and compaction

With `journal_path`, the runtime writes canonical JSON Lines records:

1. an `enqueue` record is synchronized before scheduling accepted work;

2. a `completed`, `failed`, or `cancelled` record is synchronized at terminal completion;

3. startup restores completed results and idempotency keys;

4. enqueue records without terminal records are scheduled again.

The fourth rule is at-least-once delivery. A crash may happen after a side effect and before the terminal journal write, so handlers should make side effects idempotent.

`journal_compact_after` atomically rewrites retained current state after the configured event count, preventing indefinite journal growth.

Deterministic reliability tests

jobs.test_clock_set(0)
jobs.test_clock_advance(100)

These test helpers control Jobs retry deadlines and adaptive scaling decisions. Advancing the clock wakes delayed retries and lets specs age queued or idle work without arbitrary wall-clock sleeps. They do not alter file, network, process, or general time APIs.

Status

let status = ImportUsers.status()

Job fields include:

  • `enqueued`, `completed`, `failed`, `retried`, `cancelled`, `rejected`;
  • `retry_delay_total_ms`;
  • `idempotent_reuses`;
  • `pending_results`, `result_capacity`, `dead_letters`, `dead_letter_capacity`;
  • `durable`, `journal_events`, `journal_errors`, `replayed`;
  • `partitioned`, `partition_by`, `partition_count`, per-shard `partition_enqueued`, `partition_waits`, and `partition_wait_max_ns`;
  • `pool`, containing the worker-pool status fields.

Adaptive pool status includes `adaptive`, `current_workers`, `desired_workers`, `min_workers`, `max_workers`, `peak_workers`, `oldest_queue_wait_ns`, `target_queue_latency_ms`, `scale_up_events`, `scale_down_events`, `workers_started`, `workers_retired`, and `last_scale_reason`. Fixed `count:` Jobs report `adaptive: false` and keep their configured worker count.

Stop

ImportUsers.stop()

Stopping rejects new work, cancels queued and delayed attempts, and allows active worker turns to finish.
