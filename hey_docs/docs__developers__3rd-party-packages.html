Third-party Hey packages

Hey packages are ordinary source repositories with immutable versioned releases.

Package APIs, compatibility matrices, and release notes belong in each package

repository. Hey defines the generic consumer trust boundary: graph resolution,

locks, integrity, installation, audit, import verification, and registered

commands.

Declare project dependencies

A consuming project commits `hey-package.json` and `hey.lock.json`, but not the

installed `.hey/` tree:

{
  "name": "quiz_app",
  "version": "0.1.0",
  "registry": "https://example.invalid/hey-registry-v1.json",
  "dependencies": {
    "hey_web": "^1.2.0",
    "hey_config": "~> 0.4.0"
  }
}

A dependency may also pin a local/remote immutable release source explicitly:

{
  "dependencies": {
    "local_example": {
      "version": "1.0.0",
      "release": "vendor/local_example-1.0.0.release.json",
      "release_sha256": "...",
      "archive": "vendor/local_example-1.0.0.zip"
    }
  }
}

Resolve, restore, verify, and audit

# Resolve the complete graph and atomically update the lock/install tree.
hey package ensure --project . --update-lock

# Recreate the installation exactly from the existing lock. Registry metadata is
# not consulted and lock bytes cannot change.
hey package ensure --project . --locked

# Offline complete-tree verification.
hey package verify --project .

# Reverify installed bytes and reacquire registry/release/archive evidence.
hey package audit --project .

The resolver is deterministic highest-compatible with backtracking. It rejects

cycles and impossible version intersections and does not publish a partial graph.

Requirements support exact/comparator ranges, comma/whitespace intersections,

`||` alternatives, caret, tilde, and wildcards.

The lock binds project identity, root declarations, every graph edge, exact

versions, release/archive/manifest/content hashes, per-file hashes, archive

sources, module aliases, and registered package commands. An update rejects a

new payload that reuses an already locked name/version with different content.

Import locked modules

import 'pkg:hey_web@1.2.3/main'

A `pkg:` import is accepted only when the installed package, manifest alias,

lock entry, and source bytes agree. Package manifest module keys are stable

public aliases; their archive paths are private implementation details.

Use package-provided commands

A package may declare ordinary Hey command source files in its manifest. The

project lock derives a collision-checked registry:

hey package commands --project .
hey package command-path serve --project .
hey serve --port 8080

Built-in commands cannot be shadowed, two packages cannot claim the same command,

and command source paths must be safe declared package files. Hey verifies the

complete installed tree before every lookup/dispatch.

Registry index

Hey consumes a registry-neutral JSON index. Each package/version entry binds an

immutable release JSON and archive plus their expected hashes. Local paths and

bounded HTTP/HTTPS sources use the same contract. Hosting layout, credentials,

publication policy, and Git workflow remain external.

See Package plumbing and porcelain

for the full index and lock contracts.

Package contents

A package release declares identity, version, public module aliases,

dependencies, commands, exact files, and file hashes in `hey-package.json`.

Native packages may additionally include `hey.native.json`, package-owned C

headers/sources, and build recipes. Native manifests describe stable ABI types,

ownership, nullability, blocking behavior, thread safety, libraries, and the

descriptor symbol; they must not expose `HeyValue` or raw host pointers.

Author and publish immutable assets

Generic authoring begins with:

hey package init my_package --path "$HOME/dev/my_package"
cd "$HOME/dev/my_package"
bin/check

Release construction and publication are delegated to independently versioned

`hey_packager` tooling. Publish immutable assets such as:

NAME-VERSION.zip
NAME-VERSION.release.json

The release metadata binds archive, manifest, canonical content, and file hashes.

Moving branch archives and mutable `latest` URLs are not verified releases.

Required documentation sets, website paths, source/LLM handoffs, Git commits,

tags, credentials, and registry publication policy do not belong in Hey core.

Native artifact boundary

Verified package installation proves source/release bytes, not that a target

native library was built. Native output belongs in an application/deployment

build directory. Package recipes must record targets/toolchains, vendored hashes

or system requirements, flags/libraries/frameworks/licenses, generated native

manifest lock data, artifact hashes, and compile/load/run receipts.

Native mobile and television packages

UI and operating-system services remain package-shaped:

  • target-neutral lifecycle/navigation/action/accessibility contracts;
  • iOS/tvOS UIKit/TVUIKit, Keychain, entitlement, focus, and signing adapters;
  • Android/Google TV Activity/View/Compose, Keystore, focus/remote, Gradle/JNI,

and manifest adapters;

  • application-owned screen vocabulary, room/session protocol, QR presentation,

scoring, credentials, privacy/store metadata, and release policy.

Hey owns target triples, object emission, stable ABI/callback plumbing, opaque

handles, general network/image facilities, and small build/install/receipt

porcelain. A package bug should be reduced to a package-neutral compiler/runtime

fixture before changing core.

.hey/
**/.hey/

Commit manifests and locks. A project that deliberately vendors immutable

payloads should use a separately documented location such as `vendor/packages/`

rather than weakening the default cache rule.
