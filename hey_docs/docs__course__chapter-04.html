Chapter 4 — Values and Types

A **type** classifies values and determines what operations make sense

on them. Hey's type story has two layers: a small set of *scalar*

types that map directly onto machine representations, and a universal

*boxed* layer for everything structured or dynamic. Understanding this

split explains half of the compiler's back end.

Scalars

A **scalar** is a value the machine can hold in a single register — no

internal structure to point into. Hey's scalars:

TypeValuesMachine shape
`Int`whole numbers64-bit integer (the compiler calls this `i64`)
`Float`fractional numbers64-bit floating point (`f64`)
`Bool``true` / `false`one bit, stored as an integer
`Str`textpointer to immutable string data

(Strings are "scalar-shaped" for the compiler — one pointer — even

though the text itself lives in memory.)

Sidebar: why you will rarely see a `Float`

Hey has `Float`, yet idiomatic Hey code — including the compiler

itself and the flagship data-crunching examples — is overwhelmingly

integer-typed. That is a deliberate consequence of Chapter 1's

philosophy. **Floating-point** numbers approximate fractions in a

fixed number of bits, which makes their arithmetic *rounded* and

their printing genuinely subtle — the same computation can format

differently across platforms and libraries. A language whose test

culture is "compare every output byte-for-byte" (Chapter 11) and

whose bootstrap proof is "byte-identical, bit-for-bit" (Chapter 12)

prizes arithmetic that is *exact*.

So for decimal-looking data, idiomatic Hey uses **fixed-point

arithmetic**: parse `"1.5"` with the builtin `parse_fixed`, which

returns the *scaled integer* 15 ("how many tenths?") of type `Int`;

compute everything in exact integer math; scale back only when

formatting the result. The repository's billion-row-challenge

examples compute `min/mean/max` temperatures this way and print

`3.0/4.0/5.0` without a single float in the arithmetic. `Float`

remains available — the ABI reserves `f64` for it — for domains that

truly need it; the design simply refuses to make approximation the

default path.

Boxed values

Anything with structure lives in the **boxed** world. A boxed value is

allocated in memory with a small header (the "box") that records what

kind of value it is; the program passes around a pointer to the box.

Boxing is what lets one variable hold *any* shape of data, at the cost

of a pointer indirection.

Hey's boxed data comes in two literal shapes:

  • **Arrays** — ordered collections: `[1, 2, 3]`, indexed as `arr[0]`,

measured with `len(arr)`, extended with `push(arr, x)` (which, per

immutability, returns a *new* array).

  • **Objects** — named-field collections (also called *records*):

`{name: 'Ada', age: 36}`, read as `obj.name` or `get(obj, 'name')`,

extended with `put(obj, key, value)`.

Arrays and objects nest freely, which makes boxed values a natural

match for JSON-style data. There are also `bytes` values (raw binary

data, written `b"..."`) and `nil` (the deliberate "nothing here"

value).

Type inference and annotations

Hey is **statically typed**: the compiler determines a type for every

binding *before the program runs*, and rejects programs that misuse

types. But you rarely write types, because of **type inference** — the

compiler deduces types from evidence:

let n = parse_int('8080')   # n is Int: parse_int always returns Int
let s = 'port: ' + n        # s is Str: + with a string makes a string
let cfg = {port: n}         # cfg is boxed: object literals are boxed

The compiler knows the **signature** of every built-in function — its

parameter types and return type — from a signature table inside the

semantic-analysis stage (Chapter 7), so results like `parse_int(...)`

get precise scalar types automatically.

When you want to be explicit — for documentation, or at a module's

public boundary — annotations use `:` after the name:

fn greet(name: Str) -> Str
  return 'hello, ' + name
end

let port: Int = parse_int('8080')

`-> Str` declares the return type. Function signatures written this

way appear throughout Chapters 6–10, because the compiler's own

modules annotate their public functions.

Why the scalar/boxed split matters

The split is invisible in day-to-day Hey code but decisive inside the

compiler:

  • Scalars compile to *direct machine arithmetic* — an `Int` add is one

processor instruction.

  • Boxed values compile to *runtime calls* — creating an object, or

reading `arr[i]`, calls a helper in the runtime library

(Chapter 10) that understands the box format.

So the more precisely the compiler can infer a scalar type, the faster

the code it can emit. That is why the built-in signature table exists:

a value the type system can prove is an `Int` never pays the boxed

toll. You will see this split again as a visible seam in the MIR

(Chapter 8), where every binding is marked `i64`, `str`, `bool`, or

`boxed`.

Discussion questions

1. Dynamically typed languages (Python, JavaScript) effectively box

*everything*. What do they gain? Measure the cost: how many machine

steps might `a + b` take if both operands must be unboxed first?

2. Hey infers `let n = parse_int(s)` as `Int` because the builtin's

signature is known. What could the compiler do if `parse_int` might

fail on bad input? (Look back at "Errors and results" in

Chapter 3.)

3. Immutable arrays mean `push` returns a new array. Propose an

implementation trick that avoids copying the whole array every

time, and one program pattern where naive copying would hurt.
