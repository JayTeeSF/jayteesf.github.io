Hey Language Reference

This is a compact current-surface reference. The tutorial should remain the first read for style and motivation.

Top-level forms

import 'path/to/file.hey'

module Name
  fn helper(arg)
    return arg
  end

  actor worker
    receive msg
    says msg
  end
end

fn function_name(arg1, arg2)
  return arg1
end

actor name(optional, params)
  # statements
end

spec 'name'
  expect stdout '...'
end

program
  # statements
end

`program` is the executable entry block. `spec` blocks make a file testable with `heyc <file> --test`.

Scope and imports

Current Hey scope is explicit and conservative:

  • `let` introduces a binding in the current statement/function-style scope.
  • `set` rebinds an existing local binding.
  • Module functions/actors are called with their module prefix, such as `Greeter.hello(user)`.
  • `import 'stdlib:web'` loads a bundled stdlib module without exposing its checkout path.
  • `import './file.hey'` loads an app-local module relative to the importing file.
  • `import 'pkg:cards@1.2.3/main'` loads an installed package module from `${HEY_PACKAGE_PATH:-.hey/packages}`.
  • Imports do not inject every imported name unqualified into every scope.
  • There is no user-facing mutable global variable scope and no implicit root/main namespace automatically imported into modules.
  • A future language slice should add `const`/immutable module exports and import aliases/from-imports deliberately, with canonical-symbol collision diagnostics. See `docs/modules-and-packages.md` for the canonical import path contract.

print expr

write expr

  statements

  statements
end

  statements
end

  statements
end

  statements
end

send actor_expr, message_expr
join actor_expr
stop actor_expr
sleep milliseconds_expr
fail message_expr
return expr
return

`write expr` writes to stdout without appending a newline. `write target, value` writes to a file target. `write target, value, mode` accepts explicit file modes where supported by the runtime. `log target, value` appends a line.

Core maps, arrays, and JSON codecs

let doc = {name: 'Hey', visits: 2}
says doc['name']
set doc = {...doc, visits: doc['visits'] + 1}
says to_json(doc)

let xs = [1, 2.5, true, nil, 'A']
says to_json([...xs, {name: 'Hey'}])

Arrays, floats, object/map literals, and JSON codec helpers are supported in the current runtime. Primitive helpers such as `put` remain available, but ordinary public examples should prefer spread literals where supported.

I/O helpers and collection helpers

Current public I/O statements include `says`, `print`, `warn`, `write`, `log`, and `read`.

Common core value helpers include `get`, `at`, `put`, `push`, `len`, `length`, `has`, `del`, `delete`, `keys`, `values`, `entries`, `items`, `pairs`, `merge`, `type`, `shape`, and `shape_id`.

JSON codec compatibility helpers include `json`, `to_json`, and `json_text`. The standard-library facade exposes `json.decode` and `json.encode` via `stdlib/json.hey`.

Truthiness

`false` and `nil` are falsey. Everything else is truthy. Boolean expressions can compose predicates with `or` and `and`; all three compiled backends and the interpreter now share that surface.

Typing

Hey source has no required or visible static types. Compiler-learned type facts may later live in `.hey/` sidecar files.

Plumbing and porcelain rule

Language/runtime primitives should stay small: values, functions, modules, imports, actors, mailboxes, supervision, links/monitors, I/O targets, scheduler hooks, shape metadata, and allocation/performance introspection.

Higher-level workflows such as JSON codecs, job queues, stream processors, HTTP, and 1BRC helpers belong in the standard library or packages unless missing runtime plumbing blocks them.

Runtime error example

program
  says 'oops' - 1
  fail 'boom'
end

Strings

Triple-quoted strings use `"""` delimiters for multi-line text. They preserve newlines and support interpolation. See `docs/multiline-strings.md`.
