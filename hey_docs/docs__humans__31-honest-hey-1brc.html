Honest ordinary-Hey 1BRC

The maintained One Billion Row Challenge programs are application-shaped

compiler and runtime probes. They are not permission to add challenge-specific

syntax, filename branches, canned answers, hidden native entry points, or public

shared mutable state.

Integrity rule

Every implementation follows the ordinary product pipeline:

Hey source
  -> LLVM-hosted Hey frontend
  -> semantic HIR
  -> ABI MIR
  -> direct LLVM
  -> native executable

The private native bootstrap may seed a clean build and consume validated MIR,

but it is not an alternative application backend. Performance improvements must

benefit normal values, bytes, functions, loops, actors, workers, allocation, or

lowering.

Maintained applications

examples/one_brc_single.hey
examples/one_brc_case.hey
examples/one_brc_scale.hey
examples/one_brc_actors.hey

They share `examples/lib/one_brc.hey` and must retain exact output parity.

Current ordinary algorithm

Single-core:

1. map the file read-only;

2. optionally locate an explicitly requested row boundary;

3. walk the selected byte range with bounded delimiter searches;

4. decode each station slice and parse its fixed-point temperature;

5. update an ordinary immutable object keyed by station name, where each value

is an immutable `{minimum, maximum, sum, count}` record;

6. sort station names and render with the general byte-buffer API.

Actors:

1. map once and split the byte range at newline boundaries;

2. send each worker one immutable mapping plus one independent range;

3. let each worker build one ordinary immutable statistics object;

4. send one partial value per worker to the collector;

5. merge partial records with reusable Hey functions;

6. format and verify the same result as single-core/reference.

There is no row-by-row actor mailbox path and no benchmark-owned runtime ABI.

The runtime provides only general mapping, byte-view, actor, object, and buffer

services.

Generalized compiler facts

This application exposed four compiler issues that are now maintained as a

standalone fixture rather than benchmark exceptions:

  • `div(a, b)` is a semantic integer result and lowers to MIR `i64`;
  • `json(text)` is boxed parsed data rather than a string-shaped return;
  • string concatenation accepts ordinary integer, boolean, and boxed operands

through the general boxed emitter;

  • nested properties bind to their actual receiver, so `entry[1].type` cannot be

captured by an unrelated lexical field named `type`.

Both independent Hey compiler hosts must emit identical HIR and MIR for the

fixture, and direct-MIR LLVM execution must match native-C output.

Measurement boundaries

`--rows N` validates the completed row count. `--limit-rows N` is the explicit

prefix path and reports its serial boundary-discovery cost. Build time is never

included in runtime measurements.

The scale harness executes exactly one workload per case. It records rows,

worker count, runtime, stations, parity, status, speedup, and efficiency. Exact

full-file runs do not perform a prefix scan.

./bin/perf 1brc scale \
  --data ../1brc/measurements.txt \
  --rows 1000000000 \
  --cores "$(sysctl -n hw.logicalcpu)" \
  --timeout-seconds 300 \
  --out "$HOME/hey-perf-results/one-brc-scale"

`scale.tsv.best.tsv` records the measured best worker count. `1brc auto`

calibrates candidates on an explicit prefix before the full run; it does not

pretend one worker count is universally optimal.

Bounded algorithm and worker sweep

The maintained sweep follows the reusable ideas behind fast C, Rust, and

multiprocess implementations without importing challenge-specific runtime

operations: read-only mapping, disjoint newline-aligned partitions, bounded

byte scans, fixed-point integer parsing, worker-local aggregation, and one final

merge. It compares three lanes:

  • `c-reference`, the independent optimized C correctness and speed control;
  • `hey-single`, one direct-LLVM process with chunk-size variants;
  • `hey-actors`, direct-LLVM actors with worker counts from one through the CPU

cap and the same chunk-size variants.

The C lane is mandatory as the per-prefix oracle. Every successful Hey result

must have the same SHA-256. One-time builds and each case have independent hard

timeouts. The measured sweep stops launching cases after the total case budget,

projects one billion rows from the two largest successful sizes per candidate,

and uses the slower projection plus a safety factor.

./bin/perf 1brc sweep \
  --data "$HOME/dev/1brc/measurements.txt" \
  --cores 12 \
  --workers 1-12 \
  --algorithms c-reference,hey-single,hey-actors \
  --row-ladder 1000,10000,100000,1000000,10000000 \
  --chunk-bytes-list 1048576,8388608,33554432 \
  --build-timeout-seconds 300 \
  --case-timeout-seconds 60 \
  --max-total-seconds 300 \
  --max-projected-seconds 300 \
  --target-seconds 5 \
  --projection-safety-factor 1.25 \
  --out build/one-brc-sweep

`winner.tsv` records the exact Hey version, input sample and size, machine

identity, algorithm, worker count, chunk size, binary hash, measured rows, and

conservative projection. `authorized=true` means only that the projection is at

most five minutes. `target_met=true` is the separate five-second qualification.

The full command performs no tuning and will not start unless the winner receipt,

machine, data, binary identity, five-minute safety bound, and requested target

all match:

./bin/perf 1brc full \
  --data "$HOME/dev/1brc/measurements.txt" \
  --winner build/one-brc-sweep/winner.tsv \
  --rows 1000000000 \
  --timeout-seconds 300 \
  --max-projected-seconds 300 \
  --require-target-seconds 5 \
  --out build/one-brc-full

A winner projected above five seconds is deliberately refused by that command.

The five-second requirement is therefore an executable qualification gate, not

a performance claim made from smaller or different hardware.

Equivalent C control

A maintained standalone C control uses the same high-level work: bounded

separator search, fixed-point parsing, station aggregation, sorting, and output.

It does not include or call the Hey runtime.

./bin/perf 1brc compare \
  --data ../1brc/measurements.txt \
  --rows 1000000000 \
  --warmup 1 \
  --repeat 3 \
  --timeout-seconds 300 \
  --out "$HOME/hey-perf-results/one-brc-compare"

The workflow builds first, runs precompiled programs, requires byte-identical

results, records every repetition, writes machine metadata, and reports the

best Hey/C ratio. Historic measurements are diagnostic only; rerun on the target

machine before making a current performance claim.

Why elite challenge C may remain faster

Equivalent C controls generic language/runtime overhead. Elite entries use

different algorithms and representations: SIMD delimiter and number parsing,

denser specialized buckets, reduced hashing/comparison, cache-aware

partitioning, target-specific instructions, and tuned worker placement.

Closing that gap should use reusable optimizations such as compact immutable map

storage, better string-key hashing, allocation reduction, unboxed record lanes,

vectorizable byte scans, and measured actor scheduling. It should not recreate a

public mutable collection used by one benchmark.
