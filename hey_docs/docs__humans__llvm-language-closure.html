Direct LLVM language-closure plan

Hey's product promise is that ordinary Hey source reaches native code through:

Hey source -> semantic HIR -> ABI-explicit MIR -> direct LLVM -> native executable

Stage-0 native-C exists only to bootstrap and recover the compiler. It is not a

second product backend. `v0.99.359a` removed Stage-0 compatibility execution

from `hey SCRIPT`: ordinary source execution now publishes only LLVM cache

entries and fails closed on an unsupported construct. `v0.99.360a` then closed

the maintained package-index application itself through that LLVM-only path.

The broader all-repository closure and promotion gates below remain distinct

from this maintained 66-program receipt.

Closed in v0.99.360a: package-index application

The package-index generator now compiles and executes through semantic HIR,

ABI MIR, direct LLVM, and the content-addressed shebang cache. The maintained

outside-in regression proves:

  • namespaced calls are not confused with stream receiver sugar;
  • parameter names are resolved as symbols, not substring matches inside other

call names;

  • dynamic collection parameters remain boxed instead of defaulting to integers;
  • String facts flow through paths, comparisons, helper returns, and

`Text.trim(result.stdout)`;

  • boxed object properties such as `Result.ok` remain on the object path;
  • dynamic filesystem iteration, `Files.join`, and atomic publication lower

directly;

  • program `exit` reaches an `i32` cleanup epilogue while actor exit semantics are

unchanged;

  • the first run records an LLVM cache miss/publication, the second records an

LLVM cache hit, and both produce the same five package-document links;

  • the maintained strict-LLVM build inventory passes 66/66.

This is not a claim that every source file in the repository is already direct-

LLVM clean. Release-C/A/B, the full backend world, and promotion remain the

broader evidence levels.

Closed in v0.99.348a: immutable sorting ergonomics

The collection-sorting slice now reaches direct LLVM without stdlib interpreter delegation:

  • `items.sort` and `items.sort(order: 'descending')`;
  • stable `items.sort_by(key_function)` with each key evaluated exactly once;
  • descending key order without reversing equal-key stability;
  • `items.natural_sort` and `items.natural_sort_desc`;
  • generated-program dispatch of one-argument key callables;
  • language, product, HIR, and MIR regressions.

This closes one collection/callable intersection, not the whole language. The

package-index gaps described below were subsequently closed in `v0.99.360a`;

the section is retained as the design record that led to the fix.

Why the package-index script exposed the gap

The static package-index generator is ordinary scripting code, but it combines

several surfaces that were not yet closed together in the direct MIR consumer:

  • boxed locals and parameters;
  • `for` iteration over dynamic collections;
  • string literals, character comparisons, and string-returning helpers;
  • object/property values from filesystem and process results;
  • array spreads and collection accumulation;
  • imported `stdlib:Streams` functions, including functions unused by the root

program;

  • worker-backed ordered streams;
  • filesystem traversal and atomic publication.

At `v0.99.347a`, the HIR producer still had heuristic inference errors: a

parameter could be mistaken for a callable because its name appeared inside

another function name, and string/collection APIs were incomplete in the known

return-shape table. `v0.99.360a` replaced those specific guesses with

identifier-boundary call detection, arity-aware stream sugar, conservative

boxed defaults, and call-position type facts. The historical diagnosis remains

useful; the maintained package-index regression now proves the repaired path.

Closure rule

A construct is complete only when all of these are true:

1. the parser and semantic analyzer accept it;

2. semantic HIR contains explicit, correct types and ownership shape;

3. ABI MIR represents it without source-level inference;

4. both MIR consumers accept the same validated MIR;

5. native-C and direct LLVM output match the embedded behavioral oracle;

6. a micro-spec, HIR golden, MIR golden, and outside-in product regression cover

it;

7. the all-repository LLVM world reports no fallback or unsupported reason for

that construct.

Fixing a special application in the LLVM emitter without closing those gates is

not completion.

Ordered implementation campaign

1. Reachability before lowering

Imported modules currently make every imported function eligible for emission.

A root that imports `stdlib:Streams` can therefore fail because an unused helper

contains a construct the LLVM consumer cannot lower. Build a call graph from

canonical HIR symbols and emit only reachable functions, module initializers,

actor/worker entrypoints, reflected symbols, and explicitly exported functions.

Required gates:

  • an import containing an unreachable unsupported function does not poison a

valid application;

  • reflection/export roots remain reachable;
  • native-C and LLVM consume byte-identical pruned MIR;
  • a golden records the reachable-symbol manifest.

2. Typed HIR without name-shape guessing

Replace remaining body-text heuristics with types produced from parsed forms and

symbol references. Every parameter/local must have one stable kind: scalar,

string, bytes, callable, actor, or boxed value. Calls must use resolved canonical

symbols rather than substring searches.

Required micro-specs include:

  • parameter names that are substrings of other call names;
  • strings passed through nested helper calls;
  • collection parameters iterated by `for`;
  • object/property access and dynamic indexing;
  • unary `!`/`not` around receiver-method sugar;
  • branch-local type joins and nullable values.

3. Boxed local and parameter ABI

Teach the LLVM MIR consumer to allocate, retain, release, assign, branch-merge,

and return boxed locals and parameters. Remove every `__hir_local boxed`

unsupported result.

Required gates:

  • let/set of arrays, objects, Results, and nil;
  • boxed branch dominance and loop-carried values;
  • boxed parameter/return calls across modules;
  • ownership counters remain balanced under error and early-return paths.

4. Dynamic control flow

Complete `for` over arrays, objects/entries, streams, and generic iterables;

complete `while`, `until`, `break`, `continue`, and returns nested in those

blocks. MIR must expose iterator initialization, next, done, and value ownership.

Required gates:

  • empty/single/multiple iteration;
  • loop-carried boxed accumulation;
  • nested loops and early exits;
  • ordered stream collection remains deterministic.

5. String completeness

Close string literals and locals in every expression position, character/string

comparisons, concatenation, interpolation, heredocs, slicing, replacement,

case conversion, indentation, and path helpers. Remove all generic

`expression \`"..."\`` and string-body diagnostics.

Required gates:

  • Unicode and byte-boundary behavior match the oracle;
  • string comparisons in boolean expressions;
  • string-returning user functions and module calls;
  • heredoc interpolation and escaping;
  • no application-specific helper is added to the runtime ABI.

6. Collections and object data model

Complete array/object literals, spreads, dynamic keys, properties, `has?`,

`get`, `put`, `push`, deletion, keys/values/entries, sorting, and generic length.

The MIR representation must make scalar-versus-boxed arguments explicit.

Required gates:

  • immutable update semantics;
  • nested arrays/objects;
  • comparator callables;
  • Result and metadata objects;
  • collection operations inside loops and worker functions.

7. Filesystem, process, and CLI

Close all current stdlib facades used by normal scripts: argument parsing,

recursive entries/files, metadata, reads/writes, atomic publication, process

capture, and process replacement. `Process.exec_env_argv` must remain direct in

both compiler lanes and preserve argv, environment, stdio, and exit behavior.

8. Workers, Jobs, streams, actors

Lower worker declarations/callables, bounded pools, ordered resequencing,

partitioned work, Jobs, actor state/receive/send, supervision, cancellation, and

shutdown through explicit MIR/runtime ABI contracts.

Required gates:

  • workers 1/2/N;
  • ordering and backpressure;
  • exception/error propagation;
  • deterministic shutdown with no leaked threads or handles;
  • scheduler profile and contention receipts.

9. Remaining language inventory

Close modules, package aliases, reflection, native FFI, bytes/maps/buffers,

network/TLS/HTTP, regex, eval, mobile target intrinsics, and platform shims. Each

bucket must move from gap-report reason codes to direct consumer parity.

Executable campaign gates

Run the narrow gates while developing:

./bin/test --compiler-contract
./bin/test --hir-mir
./bin/test --mir-consumers
./bin/test --conformance
./bin/test --backend-world-hang-probe

Run the complete inventory with bounded parallelism:

HEY_BACKEND_WORLD_JOBS=8 ./bin/test --backend-world-full
./bin/hey tools/hey_llvm_gap_report.hey build/backend-world-full/results.tsv

Promotion requires:

time HEY_BACKEND_WORLD_JOBS=8 ./bin/test --promotion-report

`v0.99.359a` completed the source-runner fallback deletion checklist:

1. Stage-0 application compilation was removed from `tools/hey_run.hey`;

2. the shebang regression requires `backend.txt = llvm`;

3. fallback receipts are no longer accepted;

4. Stage-0 remains only in clean bootstrap/fixed-point compiler construction.

`v0.99.360a` additionally proves the package-index generator through default

LLVM. The remaining broader promotion conditions are intentionally separate:

  • imports must not create reachable unsupported-function failures;
  • the full all-repository gap report must have zero unsupported direct-LLVM

rows;

  • Release-C/A/B and promotion-report must pass on maintained hosts.
