Great Command-Line Scripting with Hey

A practical guide to structuring, rendering, parallelizing, compiling, and benchmarking a real Hey command-line program. The running example generates a static package index for a directory of versioned Hey packages.

1. The shape of a good Hey script

A maintainable Hey command-line program normally has five layers:

  1. A small launcher that selects the intended Hey installation.
  2. Explicit standard-library imports.
  3. An UpperCamelCase module that owns the program's behavior.
  4. A small public entry function such as PackagesIndex.run(args).
  5. A tiny program block that passes cli.args() into the module.

The launcher keeps the selected Hey checkout explicit:

#!/usr/bin/env sh
set -eu

SCRIPT_DIR="$(CDPATH= cd -- "$(dirname -- "$0")" && pwd)"
HEY_ROOT="${HEY_ROOT:-$HOME/dev/hey-lang-bootstrap-plan}"
HEY_SOURCE="$SCRIPT_DIR/generate-packages-index-main.hey"

export HEY_ROOT
export HEY_STDLIB_PATH="${HEY_STDLIB_PATH:-$HEY_ROOT/stdlib}"

exec "$HEY_ROOT/bin/heyc" "$HEY_SOURCE" --run -- "$@"

The actual Hey source stays namespaced:

import 'stdlib:cli'
import 'stdlib:files'
import 'stdlib:streams'
import 'stdlib:text'

module PackagesIndex
  fn run(args)
    # command behavior
  end
end

program
  PackagesIndex.run(cli.args())
end

This provides the organizational benefit of a Ruby service class without constructing an object whose only purpose is namespacing.

2. Command-line arguments and help

cli.args() returns application arguments. cli.positionals(args) extracts positional values, while cli.flag?, cli.flag_int, and cli.missing_value? support readable option handling. 

let args = cli.args()

if cli.flag?(args, 'help') or cli.flag?(args, 'h')
  PackagesIndex.usage()
  return
end

if cli.missing_value?(args, 'workers')
  fail '--workers requires an integer value'
end

let paths = cli.positionals(args)
let workers = cli.flag_int(args, 'workers', cpu_count())

Use a raw squiggly heredoc for help text:

private fn usage_text()
  return <<~'USAGE'
    Usage: ./generate-packages-index.hey [OPTIONS] [PACKAGES_DIR]

    Options:
      --workers N   Number of parallel workers
      -h, --help    Show help
  USAGE
end

3. Filesystem discovery

The stdlib:files module owns traversal, metadata, path manipulation, and publication.

let packages = files.entries('./packages')
let all_files = files.files('./packages/hey_web', '')
let markdown = files.files('./packages/hey_web', '.md')
  • files.entries(path) returns immediate children.
  • files.files(root, suffix) recursively returns matching regular files.
  • An empty suffix returns every regular file below the root.
  • files.directory?, files.file?, and files.metadata provide filesystem checks.
  • files.write_atomic(path, content) publishes through a sibling temporary file and rename.

For this index generator, one recursive scan per package is preferable to rescanning the same tree for every extension:

let package_files = files.files(package_dir, '')
let docs_dirs = PackagesIndex.docs_directories(package_dir, package_files)
let linkable_files = PackagesIndex.linkable_files(package_files)

4. Collection pipelines

After importing stdlib:collections, receiver-oriented collection helpers can be chained:

let sorted = paths
  .uniq
  .sort(PackagesIndex.compare_paths)

The complete program uses natural numeric ordering so 0.2.10 sorts after 0.2.2, rather than relying on plain lexical ordering.

5. Rendering with heredocs

Heredocs are the clearest default for complete HTML sections:

return <<~HTML
  <section class="package">
    <h2>#{PackagesIndex.html_escape(package_name)}</h2>
  #{text.indent(docs_html, '  ')}
  #{text.indent(files_html, '  ')}
  </section>
HTML

Use text.join and text.indent to compose rendered fragments:

let links_html = text.indent(text.join(links, "\n"), '  ')

return <<~HTML
  <ul>
  #{links_html}
  </ul>
HTML

6. Escape at the boundary

Structured path data should remain ordinary strings until rendering. Escape it only when it crosses into HTML text or a URL attribute. 

private fn html_escape(value)
  return value
    .replace_all('&', '&amp;')
    .replace_all('<', '&lt;')
    .replace_all('>', '&gt;')
    .replace_all('"', '&quot;')
    .replace_all("'", '&#39;')
end

7. Parallel work without manual threads

One package is a natural independent work unit. Package trees can therefore be scanned and rendered with a bounded parallel stream: 

let sections = tasks
  .map(PackagesIndex.render_package, workers: workers)
  .collect()

Parallel stream stages preserve input order by default, so work may finish in any internal order while the generated page remains deterministic.

Parallelism is not automatically faster.

This example contains only seven small package trees. Creating workers, scheduling jobs, communicating results, and preserving order costs more than the filesystem work saved by parallel execution. The fastest measured configuration uses one worker.

The program therefore keeps an explicit sequential fast path:

if workers == 1
  for task in tasks
    set sections = [...sections, PackagesIndex.render_package(task)]
  end
else
  set sections = tasks
    .map(PackagesIndex.render_package, workers: workers)
    .collect()
end

More workers may become useful for hundreds of packages, much larger trees, Markdown rendering, archive inspection, hashing, link validation, or network-bound work. Worker count should be measured rather than assumed.

8. Atomic output

Do not truncate the live index before its replacement is complete:

files.write_atomic(output_path, html)

Readers see either the previous complete index or the newly generated complete index, rather than a partially written page.

9. A minimal build-up version

A deliberately smaller version teaches the basic structure before the complete program adds validation, escaping, sorting, and parallelism:

import 'stdlib:cli'
import 'stdlib:files'
import 'stdlib:text'

module SimplePackagesIndex
  fn run(args)
    let paths = cli.positionals(args)
    let root = './packages'

    if paths.length == 1
      set root = paths[0]
    end

    let links = []
    for path in files.files(root, '.md')
      let relative = text.strip_prefix(path, root + '/')
      set links = [
        ...links,
        %qq(<li><a href="#{relative}">#{relative}</a></li>)
      ]
    end

    let html = <<~HTML
      <!doctype html>
      <html lang="en">
      <body>
        <h1>Packages</h1>
        <ul>
      #{text.indent(text.join(links, "\n"), '    ')}
        </ul>
      </body>
      </html>
    HTML

    files.write_atomic(files.join(root, 'index.html'), html)
  end
end

program
  SimplePackagesIndex.run(cli.args())
end

The complete linked implementation adds:

  • argument validation and --help;
  • Markdown and HTML links;
  • versioned docs/ links, including empty directories;
  • HTML and URL escaping;
  • natural numeric sorting;
  • optional package-level parallelism;
  • a sequential fast path;
  • deterministic output;
  • atomic publication;
  • a generated UTC timestamp.

10. Performance: interpreted source, compiled Hey, and Ruby

Execution mode matters. Running the source through heyc --run includes Hey parsing and interpreter startup on every invocation. For a meaningful runtime comparison, the Hey implementation was compiled once with the native-C backend and the resulting executable was benchmarked directly. 

export HEY_ROOT="$HOME/dev/hey-lang-bootstrap-plan"
export HEY_STDLIB_PATH="$HEY_ROOT/stdlib"

mkdir -p ./build

"$HEY_ROOT/bin/heyc" build \
  --backend native-c \
  ./generate-packages-index-main.hey \
  -o ./build/generate-packages-index

The warmed benchmark used three warm-up runs followed by thirty measured runs:

hyperfine \
  --warmup 3 \
  --runs 30 \
  './build/generate-packages-index --workers 1 ./packages' \
  './build/generate-packages-index --workers 2 ./packages' \
  './build/generate-packages-index --workers 7 ./packages' \
  './generate-packages-index.rb ./packages'
Program Mean Standard deviation Range Relative to Ruby
Compiled Hey, 1 worker 13.7 ms 1.0 ms 12.4–17.7 ms 8.20× faster
Compiled Hey, 2 workers 30.1 ms 0.8 ms 27.4–32.0 ms 3.72× faster
Compiled Hey, 7 workers 46.2 ms 5.4 ms 42.1–67.3 ms 2.43× faster
Ruby 112.1 ms 8.6 ms 100.6–138.6 ms Baseline

Redirecting normal output to /dev/null produced essentially the same result: compiled single-worker Hey averaged 13.6 ms, Ruby averaged 112.2 ms, and Hey was 8.26 times faster. Terminal output was therefore not responsible for the difference.

Ruby also benefits from warm filesystem and operating-system caches: its first hand-timed invocation was slower than subsequent runs. The benchmark deliberately warms every command before measurement, so the comparison is between steady-state executions rather than one cold program and one warm program.

What the numbers show
  • Compilation is the dominant performance improvement for this small program.
  • Compiled single-worker Hey is approximately 8.2 times faster than the Ruby implementation in the warmed benchmark.
  • Two workers are approximately 2.21 times slower than one worker.
  • Seven workers are approximately 3.38 times slower than one worker.
  • The best worker count depends on the size and cost of the workload, not merely on the number of available CPU cores.

These results should not be generalized to every Hey or Ruby program. This is a small, filesystem-oriented workload with only seven package directories. It does, however, demonstrate two practical rules: benchmark the compiled artifact when evaluating compiled-language performance, and measure parallelism instead of assuming that more workers will help.

11. Verification workflow

Run the source through the launcher:

chmod +x \
  ./generate-packages-index.hey \
  ./generate-packages-index-main.hey

./generate-packages-index.hey --help
./generate-packages-index.hey ./packages

Compile once for runtime benchmarking:

export HEY_ROOT="$HOME/dev/hey-lang-bootstrap-plan"
export HEY_STDLIB_PATH="$HEY_ROOT/stdlib"

"$HEY_ROOT/bin/heyc" build \
  --backend native-c \
  ./generate-packages-index-main.hey \
  -o ./build/generate-packages-index

Check the generated page:

test -s ./packages/index.html
grep -n 'docs/' ./packages/index.html
grep -nE '\.(md|html)</a>' ./packages/index.html

The complete implementation linked at the top of this article supports normal documentation trees, nested documentation, multiple versions, natural sorting such as 0.2.2 versus 0.2.10, and empty versioned docs/ directories.
