Distributions and platform support

Product requirement: one target archive, no recipient toolchain

Hey should be installed the way users experience Go or Rust: download one archive for the exact operating system and architecture, extract it into one versioned directory, and put its public commands on `PATH`.

That does **not** require the archive to contain literally one executable. The compiler still needs its standard library, target runtime object, linker, target system interface, and package metadata. Combining all of those into one enormous executable would make updates, signing, diagnostics, and licensing harder without improving the user experience.

The release goal is therefore:

hey-VERSION-linux-x64.tar.gz
hey-VERSION-darwin-arm64.tar.gz
hey-VERSION-windows-x64.zip

Each target archive must:

  • run `hey`, `heyc`, and package commands without an installed C/C++ compiler;
  • contain the target LLVM compiler/linker pieces actually used by Hey;
  • contain a precompiled Hey runtime and every redistributable native dependency;
  • compile and run a real application after the archive is moved away from its build path;
  • install under `/opt/hey/VERSION` or `~/.local/hey/VERSION` on Unix, and `%LOCALAPPDATA%\Programs\Hey\VERSION` on Windows;
  • expose small links or launchers in `/usr/local/bin`, `~/.local/bin`, or the Windows user `PATH`.

Visual Studio Build Tools, Xcode, Homebrew LLVM, GCC, and Clang are release-builder dependencies only. They must not be ordinary recipient prerequisites once a target archive is promoted.

Choose the correct release command

Preflight a product release before starting the expensive closure gate:

./bin/hey target doctor --target darwin-arm64
./bin/hey target doctor --target linux-x64

`./bin/hey target release` creates an actual self-contained compiler archive

for `darwin-arm64` or `linux-x64`. The temporary Windows command is deliberately

different:

./bin/hey windows runtime-test-bundle --out hey-windows-native-test.zip

That ZIP collects native Win32 runtime evidence and temporarily uses Build

Tools. It is not a compiler distribution. There is no supported

`target release --target windows-x64` until prebuilt Hey/LLVM/linker/runtime

closure exists. See the complete command-line guide.

Linux and Darwin target archive pack, verify, and release proof

Target archive v2 wraps a Linux x64 or Darwin arm64 contained SDK in one recipient download:

./bin/hey target pack \
  --sdk dist/hey-VERSION-linux-x64 \
  --target linux-x64 \
  --out "$HOME/Downloads/hey-VERSION-linux-x64.tar.gz"

./bin/hey target verify \
  --archive "$HOME/Downloads/hey-VERSION-linux-x64.tar.gz" \
  --target linux-x64

The same lifecycle applies to Apple Silicon:

./bin/hey target pack \
  --sdk dist/hey-VERSION-darwin-arm64 \
  --target darwin-arm64 \
  --out "$HOME/Downloads/hey-VERSION-darwin-arm64.tar.gz"

./bin/hey target verify \
  --archive "$HOME/Downloads/hey-VERSION-darwin-arm64.tar.gz" \
  --target darwin-arm64

Payload verification uses `sha256sum` where available and macOS

`shasum -a 256` otherwise. Packaging preserves relative SDK symlinks and

executable modes and supports GNU or BSD tar. Cross-host verification may prove

layout and checksums, but reports native execution as skipped rather than

promoting the target.

A matching native host may build the exact release artifact with:

./bin/hey target release \
  --target linux-x64 \
  --out "$HOME/Downloads/hey-VERSION-linux-x64.tar.gz"

Use `darwin-arm64` on Apple Silicon. The release command runs the strict SDK

closure gate, consumes that exact exported SDK, packs it, and verifies it. It

refuses host/target mismatches. Signing/notarization and publication remain

required after this local release receipt.

Use `--extract-to DIR` when the already-verified payload should be retained for

installation or a target-host receipt. Verification still expands and checks

the archive once; it simply does not delete that checked tree afterward.

The packer refuses SDKs that do not report `self_contained_sdk=true`, stages

the complete tree with SDK-safe symlink handling, and emits:

  • one deterministic `tar.gz` with a single versioned top-level directory;
  • `manifest.json`, `VERSION.txt`, and `TARGET.txt`;
  • `SHA256SUMS` covering regular payload files;
  • a publication `.sha256` sidecar for the complete archive;
  • `install.sh` and `uninstall.sh`.

After extracting the archive, a normal user install is:

./install.sh --user

This installs under `~/.local/hey/VERSION` and links the public commands in

`~/.local/bin`. A system installation uses:

sudo ./install.sh --system

This installs under `/opt/hey/VERSION` and links commands in `/usr/local/bin`.

Custom test or managed prefixes use `--prefix DIR --bin-dir DIR`. The installer

verifies `SHA256SUMS` before publication. The uninstaller removes only symbolic

links that still point to its own installed version, then removes that version

root.

v0.99.282a binds this mechanism to the maintained real Linux closure gate. The

gate exports the exact SDK only after relocated compiler, sysroot, native

library, and compiler-free run-bundle execution pass. The archive is then

verified, installed into an unrelated prefix, and used with `PATH=/nonexistent`,

`LD_LIBRARY_PATH=/nonexistent`, and checkout discovery removed to compile and

run the OpenSSL fixture. `ldd` must resolve `libssl` and `libcrypto` from the

installed archive. This is the first executable one-download compiler archive

receipt; public signing/publication automation remains separate work.

Three-root SDK closure

A target-contained SDK is staged from three explicit roots:

HEY_SDK_TOOLCHAIN_ROOT  -> libexec/hey/toolchain
HEY_SDK_SYSROOT_ROOT    -> libexec/hey/sysroot
HEY_SDK_NATIVE_ROOT     -> lib/hey/native

Build it with:

make dist-sdk-host-contained \
  HEY_SDK_TOOLCHAIN_ROOT=/path/to/clang-lld-resources \
  HEY_SDK_SYSROOT_ROOT=/path/to/target-sysroot \
  HEY_SDK_NATIVE_ROOT=/path/to/relocatable-native-libraries

Installed compilation uses the SDK-local compiler, linker, sysroot, runtime object, and native libraries. Linux applications receive `$ORIGIN/../lib/hey/native`; Darwin applications receive `@loader_path/../lib/hey/native`.

Darwin contained builds now require `bin/ld64.lld` in the staged toolchain and pass its exact path to relocated Clang through `--ld-path=...`. This prevents relocated Clang from falling back to a missing `ld` on `PATH`.

The staged toolchain also merges the runtime library trees adjacent to both Clang and `ld64.lld`. Homebrew can install `llvm` and `lld` as separate formulae; the linker retains `@loader_path/../lib` references such as `liblldELF.dylib`, so copying only the LLVM prefix is not a self-contained linker closure.

Release provenance and publication truth

Target archive v3 includes `RELEASE-PROVENANCE.json`, `THIRD-PARTY-INVENTORY.json`, and `SHA256SUMS`. The checksum file is the canonical file inventory. Provenance records the compiler pipeline, target, deterministic archive metadata, and whether native receipt, signing, license inventory, and SBOM evidence are complete.

A packed or natively verified archive is not automatically publishable. Current archives deliberately contain:

publication_ready=false
license_inventory_complete=false
sbom_complete=false

Publication tooling may change those fields only after exact third-party versions/licenses, a complete SBOM, the native target receipt, source provenance, and platform signatures/notarization are attached.

Two artifact kinds

Compiler SDK

The SDK contains the north-star compiler commands, private MIR consumer, standard library, documentation, examples, and a precompiled target runtime object:

bin/hey
bin/hey-selfc
bin/hey-package
libexec/hey/heyc-bootstrap
libexec/hey/toolchain/bin/clang
libexec/hey/toolchain/bin/ld.lld or ld64.lld
libexec/hey/sysroot
lib/hey/runtime/hey_runtime.o
lib/hey/native
stdlib

Ordinary application builds link the precompiled runtime object. Recipients do not compile `runtime/hey_runtime.c` and do not install C/OpenSSL development headers merely to use Hey.

The compiler searches for LLVM in this order:

1. `HEY_LLVM_CC` or `CLANG` when explicitly configured for development;

2. `libexec/hey/toolchain/bin/clang` inside the installed SDK;

3. `clang` on `PATH` only as a source-checkout fallback.

A contained Darwin compiler also resolves `libexec/hey/toolchain/bin/ld64.lld` and passes it explicitly. A contained Linux compiler selects the bundled LLD driver.

Precompiled application bundle

A run bundle contains one already-compiled native application and no compiler:

make dist-run-host \
  RUN_APP=build/my-app \
  RUN_APP_NAME=my-app

The result contains `bin/my-app` plus the native libraries that application needs. It is the smallest artifact for recipients who only run an application.

Relocation and compiler closure

`make dist-host` is the canonical SDK construction graph used by local releases and container vendoring. Vendoring does not maintain a second compiler bootstrap. The graph pins `hey-selfc`, `hey`, and `hey-package` to the bootstrap compiler inside their own release directory.

Installed tools must work outside the source checkout:

  • `bin/hey-selfc` discovers `libexec/hey/heyc-bootstrap` and the copied `stdlib/`;
  • `bin/hey` resolves the SDK root from its installed path and passes that root into cached applications;
  • delegated pure-Hey stdlib calls use temporary files, never checkout-local `build/` state;
  • target linking uses only the staged compiler, linker, sysroot, runtime object, and native roots.

Endpoint security and release signing

Unsigned, newly generated Windows executables may be blocked before they start.

The promotion harness records that as endpoint-security/application-control

evidence and never asks the user to disable protection. Public Windows archives

must be Authenticode-signed and timestamped; public macOS archives must be

properly code-signed and notarized. Artifact signing is a publication gate, not

a compiler semantic.

Structural verification versus native promotion

Compiler matrices must remain runnable on ordinary development hosts. Contract

979 therefore invokes the Darwin closure gate with

`HEY_TARGET_RECEIPT_MODE=optional`: when `ld64.lld`, the macOS SDK, or OpenSSL

is unavailable, the contract records an explicit skip and continues checking

the source/layout invariants.

Promotion remains stricter. Running

`test/regression_darwin_sdk_target_closure.sh` directly uses required mode,

fails on any missing prerequisite, and must compile and execute the relocated

application on Apple Silicon. Optional mode can never set a platform promoted.

Windows command-line use

Windows users routinely use command-line development tools through Windows Terminal, PowerShell, Command Prompt, IDE terminals, and editor tasks. Hey does not need a GUI merely to run programs.

The intended product commands are:

hey .\hello.hey
heyc build .\hello.hey -o .\hello.exe
.\hello.exe

A `.hey` file can also be associated with `hey.exe` later, but file association is convenience rather than the language execution model. Editor integration and a desktop IDE are separate products; they are not prerequisites for a working Windows compiler.

Shell-script policy

Released public commands are compiled Hey binaries. Distribution staging does not ship Bash, Ruby, Python, or Node launchers.

A source checkout may retain tiny bootstrap scripts before an installed `hey` exists. Target promotion harnesses may also contain platform scripts, but those scripts must not be confused with the final product archive.

Normalized SDK-tree copying

Compiler resource directories and platform SDKs may contain internal compatibility symlinks. Distribution staging and vendoring use `files.copy_sdk_tree`, which:

  • never follows a symlink while walking;
  • preserves relative links whose lexical targets remain inside the copied root;
  • rejects absolute links, root escapes, sockets, devices, and other special files;
  • preserves internal alias cycles without traversing them;
  • applies the same policy during staging and final vendor publication.
