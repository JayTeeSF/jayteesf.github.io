Hey Language Guide

This guide shows copyable examples first.

See Hey reserved words and backend-safe names for the canonical distinction between Hey syntax words, expression literals/operators, and C11 names that remain legal in Hey.

Hello

program
  says 'Hello, Hey'
end

Values

let answer = 40 + 2
says answer
says answer == 42
says 'Count: ' + answer
says true
says nil

Strings

Double-quoted strings support escapes (`\"`, `\\`, `\n`) and `#{...}` interpolation. Single-quoted strings are raw, Ruby-style: backslashes and `#{...}` are literal text, and only `\'` (a quote) and `\\` (one backslash) are recognized escapes. Prefer single quotes for ordinary non-interpolated text, keys, paths, and fixture strings; use double quotes when you need interpolation or escape sequences such as `\n`. This keeps source cheaper for the front door to normalize and teaches the compiler that no interpolation scan is needed.

let name = 'Ada'
says "hello #{name}"
says 'raw #{name} and c:\temp stay literal'
says 'it\'s easy to quote "both" kinds'

Core object/map values

For fixed object fields, prefer JavaScript-style dot access; see Object field access for dynamic keys and built-in-name collisions.

Hey uses immutable core map/array values for structured data. JSON is a standard-library codec boundary over those values.

let user = {name:'Ada', visits:2, ok:true}
says user.name
set user = {...user, visits: user.visits + 1}
says user.to_json

Valid identifier keys do not need quotes. Invalid keys still use strings:

let doc = {name:'Hey', 'data-id':123}

Core arrays and JSON parsing

let xs = [1, 2.5, true, nil, 'A']
set xs = [...xs, {name:'Hey'}]
says xs[0]
says xs.length
says xs.size
says xs.to_json
says json('42')

Hey source object and array literals are expression-capable core values, so they can also hold runtime values such as actor handles. Use `stdlib:json` when crossing an actual JSON boundary; inside Hey, prefer dot access for fixed object fields, brackets for indexes/dynamic or colliding keys, spread updates, and value properties.

Functions

Function names may use a trailing `?` predicate convention or `!` effectful-operation convention. The suffix is part of the name and adds no automatic behavior. See Callable name suffixes.

The compiler selects efficient internal representations from ordinary source usage. Those decisions remain compiler-owned and do not change how functions are written.

fn greet(name)
  return 'Hello, ' + name
end

program
  says greet('Hey person')
end

A function name without parentheses is a callable value:

fn double(value)
  return value * 2
end

fn apply(value, callable)
  return callable(value)
end

program
  let pipeline = [double]
  says apply(21, pipeline[0])
end

Module functions, bound callables, actor-definition values, and arrays of callables follow the same rule. See Functions and callable values.

Modules

Modules package related functions and actors. They are the preferred current alternative to classes for organizing behavior around JSON/hash data.

module Users
  fn name(user)
    return user.name
  end

  actor echo(prefix)
    receive msg
    says prefix + msg
  end
end

program
  let user = {name:'Ada'}
  says Users.name(user)

  let p = spawn Users.echo('p')
  send p, 'ong'
  stop p
end

Core arrays can contain actor handles:

let p = spawn Users.echo('p')
let workers = [p]
send workers[0], 'ing'
stop p

Stateless workers

Workers are interchangeable handlers for independent immutable messages:

worker Normalize(message)
  return text.trim(message)
end

program
  let normalizers = workers(Normalize, count: 4, capacity: 256)
  normalizers.enqueue('  Ada  ')
  normalizers.stop()
end

A worker has one message parameter, no persistent state, and no mutable local bindings. Multiple instances drain one bounded hidden mailbox in parallel. Use actors for identity/state and Jobs for reliability/scaling policy.

Actors

Actor names support the same trailing `?` and `!` callable suffixes, including module-qualified spawn expressions.

actor echo
  receive msg
  says msg
end

program
  let p = spawn echo
  send p, 'ping'
  stop p
end

Actor state

actor counter(label)
  state count = 0
  receive msg

  if msg == 'inc'
    set count = count + 1
    says label + ': ' + count
  else
    if msg == 'stop'
      exit
    end
  end
end

Actor data rule

Actors do not share mutable objects.
Messages are copied, immutable, uniquely moved, or explicitly share-safe.

Current runtime behavior: values sent to actors are cloned into FIFO mailboxes.

JSON helpers

program
  let doc = {name:'Ada', visits:2, ok:true}
  says doc.has?('name')
  says doc.keys.to_json
  set doc = {...doc, city:'SF', visits:3}
  set doc = doc.del('ok')
  says doc.to_json
end

Run:

./bin/hey examples/json_helpers.hey --test

Multi-line JSON literals

program
  let doc = {
    name:'Ada',
    tags:[
      'hey',
      'json',
    ],
    nested:{ok:true,},
  }
  says doc.to_json
end

Run:

./bin/hey examples/json_multiline.hey --test

Supervisors

actor once
  receive msg
  says 'handled ' + msg
  exit
end

program
  let sup = supervise once max 3
  send sup, 'one'
  sleep 20
  send sup, 'two'
  sleep 20
  stop sup
end

Run:

./bin/hey examples/supervisor_restart.hey --test

I/O statements

says 'line'      # stdout plus newline
print 'partial'  # stdout without newline
warn 'careful'   # stderr plus newline
write 'value'    # stdout without newline
write 'tmp/out.txt', 'contents'
log 'tmp/app.log', 'started'
let text = read('tmp/out.txt')

`write` to a file path overwrites. `log` appends one line. Future file/socket/string-buffer handles should plug into the same target-oriented forms.

Collection snapshots

let doc = {name: 'Ada', visits: 1}
says doc.keys.to_json
says doc.values.to_json
says doc.entries.to_json

`keys`, `values`, and `entries` return immutable snapshot arrays. `put`, `delete`, `merge`, and `push` return new values and do not mutate existing values.
