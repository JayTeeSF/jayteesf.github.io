Documentation organization

Hey keeps user documentation and maintainer records in the same repository, but they serve different audiences.

Public documentation

Public pages explain how to write and run Hey programs. They use current syntax, complete examples, normal prose, and links to deeper guides. They do not begin with release notes, backend scorecards, or lists of recently completed implementation slices.

The public HTML site includes:

  • the root README and documentation index;
  • quick-start, tutorial, language, REPL, actor, module, visibility, and compiler guides;
  • standard-library guides;
  • stable architecture and language-influence explanations.

Maintainer records

Maintainer records describe work rather than teach the language. They include:

  • version plans, current handoffs, todo lists, done lists, and open questions;
  • release notes and defect ledgers;
  • backend parity plans, gap reports, and performance evidence;
  • historical feature-slice documents and implementation RFCs.

These files remain valuable source material for contributors and release automation, but the documentation generator excludes them from the public site. A public guide may link to a maintainer record only when the reader explicitly needs implementation or contribution context.

Writing rules

When updating a public page:

1. Describe the current behavior, not the sequence of versions that introduced it.

2. Prefer current readable syntax such as `value[key]`, array/object spreads, and scheme imports.

3. Explain a concept completely enough that the reader does not need release notes to understand it.

4. Link to one authoritative deeper guide instead of repeating partial explanations in several files.

5. Move historical notes and unfinished work to the appropriate roadmap, todo, handoff, or release-notes file.
