Package plumbing and porcelain

Decision

Hey core is registry-neutral but owns the complete consumer trust boundary.

Applications should not reimplement dependency graphs, locks, checksums, or

installation under project-local wrapper scripts.

Public project workflow

# Resolve the complete declared graph and publish a new lock atomically.
./bin/hey package ensure --project . --update-lock

# Restore exactly from the existing lock. Do not consult registry metadata or
# change lock bytes.
./bin/hey package ensure --project . --locked

# Offline: verify project identity, graph closure, installation shape, metadata,
# and every installed file byte.
./bin/hey package verify --project .

# Online/local source audit: verify the installed tree, then recheck registry,
# release metadata, archive, manifest, content, and same-version immutability.
./bin/hey package audit --project .

Package-provided commands are also lock-bound:

./bin/hey package commands --project .
./bin/hey package command-path COMMAND --project .
./bin/hey COMMAND [args...]

Top-level dispatch searches from the current directory to the project root,

verifies the complete installed tree, rejects built-in/cross-package collisions,

and executes only a command source path declared by the exact locked package

manifest.

No package-authoring language syntax is justified. Packages remain ordinary

manifests, modules, declared files, immutable release payloads, and imports.

Registry index v1

A project may declare a registry source in `hey-package.json` or pass

`--registry INDEX`. The index is data, not executable policy:

{
  "registry_version": 1,
  "packages": {
    "hey_example": {
      "1.2.3": {
        "release": "hey_example/1.2.3/hey_example-1.2.3.release.json",
        "release_sha256": "...",
        "archive": "hey_example/1.2.3/hey_example-1.2.3.zip",
        "archive_sha256": "...",
        "manifest_sha256": "...",
        "content_sha256": "..."
      }
    }
  }
}

Local paths and bounded HTTP/HTTPS sources use the same verification contract.

The resolver supports exact/comparator intersections, alternatives, caret,

tilde, and wildcard requirements. It tries versions highest-first but snapshots

and backtracks when a later transitive/root requirement makes an earlier choice

invalid. Cycles and impossible intersections fail without publishing a lock or

partial package tree.

Lock v2

`hey.lock.json` binds:

  • project name/version identity;
  • exact root dependency declarations;
  • resolver and lock format versions;
  • each package version, requirement, direct/transitive status, dependencies,

release source/hash, archive source/hash, manifest hash, canonical content

hash, module map, command map, and per-file hashes;

  • graph package and edge counts;
  • the derived package-command registry.

An update compares same-name/same-version identity against the previous lock.

Different release/archive/manifest/content identity is rejected as content

replacement even when the new payload is internally self-consistent.

`--locked` uses only lock-bound archive sources and hashes. Registry and release

metadata can be unavailable or changed without affecting an exact restore.

Atomicity and verification

The resolver stages the complete graph beside the project installation. Every

archive passes safe path/link checks and all integrity layers before publication.

The package tree and lock are swapped as one transaction with rollback. A

conflict, cycle, command collision, checksum mismatch, unsafe archive, write

failure, or interrupted transaction cannot publish a partial new graph.

Offline verification rejects:

  • changed project identity or root declarations;
  • invalid/missing graph edges;
  • extra package names or versions;
  • missing or extra files/directories;
  • symlinks and special files;
  • changed installation metadata, modules, commands, or file declarations;
  • any file hash mismatch.

`audit` performs all offline checks first, then reacquires and verifies the

registry entry, release JSON, archive bytes, manifest/content identity, and

same-version immutability evidence.

Generic supporting facilities

`stdlib:Schema` provides immutable data-first descriptors for objects, arrays,

maps, strings, integers, numbers, booleans, nil, literals, enums, and unions.

Validation returns structured error paths such as `$.dependencies.web.version`.

It is reusable by packages and applications and does not embed registry policy.

Filesystem publication, bounded downloads, safe ZIP extraction, canonical JSON,

SHA-256, structured argv/environment execution, EHY/text templates, and target

matrix JSON remain general facilities with independent contracts.

Authoring and publication boundary

`hey package init` may create a generic package repository. The independently

versioned `hey_packager` owns required public documentation, source handoffs,

release ZIP/JSON generation, website layout, Git commit/tag behavior, credentials,

and publication. The language repository must not encode a particular hosting

path or publisher policy.
