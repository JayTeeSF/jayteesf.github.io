Chapter 9 — The Back End: Emitting LLVM

The back end turns MIR into something a machine can run. Hey does not

generate machine instructions itself; it generates **LLVM IR** and

lets the LLVM toolchain do the last mile. This chapter explains that

choice, teaches you to read a little LLVM IR, and maps the emitter —

the largest module in the compiler.

LLVM in three paragraphs

**LLVM** is an industrial compiler infrastructure used by C/C++

(clang), Rust, Swift, and many others. Its heart is a typed,

target-independent IR with a textual form (`.ll` files). Feed LLVM a

`.ll` file and it applies decades of optimization work and emits

native code for essentially any mainstream processor.

LLVM IR looks like assembly for an idealized machine. Values are

named `%something`; each instruction produces at most one new value;

types (`i64`, `i32`, `i1`, `ptr`) are written everywhere; code is

organized into labeled basic blocks (Chapter 8) inside functions

(`define ... { ... }`); global constants are named `@something`.

For Hey, targeting LLVM completes a clean division of labor: every

*decision* about what a Hey program means is made in Hey source

(Chapters 6–8 plus this emitter), while raw code-generation muscle —

register allocation, instruction selection, optimization — is

delegated to a battle-tested external tool through a documented,

textual interface. The same boundary discipline as MIR, one level

down.

Reading the output

For hello-world's MIR (`say str "Hello, world"`), the emitted `.ll`

has this shape (sketch, abbreviated):

@.str.0 = private unnamed_addr constant [13 x i8] c"Hello, world\00"

define i32 @main() {
entry:
  %t0 = call i32 @puts(ptr @.str.0)
  ret i32 0
}
  • String literals are **interned** — stored once as numbered global

constants (`@.str.0`) and referenced by pointer.

  • Temporaries are numbered `%t0, %t1, ...` in emission order.
  • A `say` of a plain string becomes a call to `puts`, the C library's

print-line routine. No Hey runtime needed for this program.

Now a real excerpt from a compiled actor's message dispatch — this is

Chapter 5's `msg.kind == 'scan'` comparison, exactly as emitted:

  %t3 = call ptr @hey_llvm_value_to_raw_string_cstr(ptr %t2)
  %t4 = call i32 @strcmp(ptr %t3, ptr @.str.1)
  %t5 = icmp eq i32 %t4, 0
  br i1 %t5, label %if.then0, label %if.else1

Read it: extract the C-style string from a boxed value (a runtime

call — the `@hey_llvm_*` prefix marks the runtime's emitter-facing

helpers), compare with the interned literal, turn the comparison into

a one-bit value (`icmp`), branch. Structured `if`/`while` from the

source has become labeled blocks and branches — this is what

"lowering control flow" ultimately means.

Two mechanics worth naming:

  • **The prelude.** Every `.ll` file begins with declarations of the

external functions it may call (the C library and the runtime's

helper surface), emitted by `hey_llvm_prelude_declares()`.

  • **The runtime-required flag.** The emitter tracks whether the

program actually touched boxed values, actors, or other runtime

services. Purely scalar programs (like hello) link into tiny

executables with no runtime initialization at all; the moment a

runtime feature appears, the emitter flips a header flag and emits

the initialize/shutdown plumbing around `main`.

Where it lives: `stdlib/HeyLlvm.hey`

The compiler's largest module (~10,400 lines) with its smallest

public API:

module HeyLlvm
  fn emit_text(mir_text: str) -> boxed
  fn emit_file_to_path(mir_path: str, ll_path: str) -> bool
end

Internally it is organized as one MIR walker plus a family of *typed

expression evaluators* — each takes an expression and the current

context and returns the instructions, new temporaries, and result

name it produced:

FunctionEmits
`hey_llvm_emit_text_value(mir_text: str) -> boxed` (l. 6848)whole-module driver: walks MIR records, dispatches every statement
`hey_llvm_int_expr_value(expr, locals, fns, string_start, t_start)` (l. 5360)integer arithmetic
`hey_llvm_str_expr_value(...)` (l. 4536)string expressions and concatenation
`hey_llvm_bool_expr_value(...)` (l. 4109), `hey_llvm_cmp_expr_value(...)` (l. 3556)conditions and comparisons
`hey_llvm_call_expr_value(...)` (l. 4773)user-function calls (mangled `@hey_fn_<name>`)
`hey_llvm_boxed_expr_value(...)` (l. 972)array/object literals, element access — the boxed world's runtime calls
`hey_llvm_interpolated_fold_value(...)` (l. 3248)`#{...}` string interpolation, as a left-to-right concatenation fold
`hey_llvm_actor_raw_emit_value(...)` (l. 6484)actor receive-body statements
`hey_llvm_callable_trampoline_value(...)` (l. 6191)functions-as-values dispatch
`hey_llvm_prelude_declares() -> str` (l. 6176)the declaration prelude

The threading convention visible in those signatures — every

evaluator receives `(locals, fns, string_start, t_start)` and returns

its consumed counters — is how the emitter guarantees *deterministic*

numbering of temporaries and interned strings: same MIR in, same

bytes out, every time. Determinism is not cosmetic; Chapter 11's

exact-output specs and Chapter 12's byte-identical fixed point both

stand on it.

Discussion questions

1. Interned strings are deduplicated per module. What program pattern

benefits most? Sketch the emitter bookkeeping needed to intern

"per use" instead, and what it would cost.

2. The runtime-required flag makes hello-world runtime-free. Why is

it worth engineering effort to keep trivial programs trivial?

Consider startup time, binary size, and teaching.

3. Deterministic output means the emitter can never consult a hash

table with unstable ordering, the clock, or randomness. Find one

place in *any* compiler you know where nondeterminism leaks into

output, and propose the fix.
