Cross-compilation and v0.99.219a 1BRC handoff

Status: **evidence and implementation plan; no cross-toolchain fix in this

handoff**.

v0.99.219a 1BRC evidence

The 100M real-file prefix selected 12 workers:

runtime_ms=2053
rows_per_second=48,709,206.04
projected_runtime_1b_seconds=20.53
confidence=high
compiler_mode=selfhost-c->llvm-mir

The actual 1B-row run with those 12 workers reported:

measured_rows=1,000,000,000
stations=41,343
runtime_ns=43,242,597,000
runtime_ms=43,242
rows_per_second=23,125,345
estimated_seconds_for_1b=43
sub_60_seconds=true
sub_5_seconds=false
rows_match=true
status=ok
compiler_mode=selfhost-c->llvm-mir

The real proof is therefore **43.242 seconds**, not the 20.53-second 100M

projection. Throughput fell by about 52.5% between the 100M prefix and the full

file. The worker-selection ramp remains useful, but the current `high` label is

too optimistic for predicting full-file throughput.

A previously supplied legacy direct-source LLVM run completed 1B rows in about

35.505 seconds with 8 workers. That older result is faster by roughly 7.7

seconds, but it is not an apples-to-apples product-lane comparison: the lane,

worker count, cache state, and run conditions differ. The next proof must run

both maintained product lanes in one controlled session and should test at least

8, 10, and 12 workers on the full file.

Why `full-proof.tsv` disappeared

The command did create:

build/1brc-selfhost-c-1b/full-proof.tsv

The subsequent Release-A started at 23:23:01 and begins with `make clean`, which

removes `build/`. The attempt to read the file happened at 23:23:31, after the

cleanup. This is an artifact-retention problem, not evidence that the 1BRC

command failed to write its summary.

`build/` is intentionally ephemeral. Preserve benchmark evidence outside it:

RESULT_ROOT="$HOME/hey-perf-results/$(date +%Y%m%d-%H%M%S)"

./bin/perf 1brc-full \
  --data ../1brc/measurements.txt \
  --rows 1000000000 \
  --workers '8 10 12' \
  --implementations 'selfhost-c->llvm-mir selfhost-llvm->llvm-mir' \
  --no-progress \
  --out "$RESULT_ROOT/1brc-full" \
  --tsv

cat "$RESULT_ROOT/1brc-full/full-proof.tsv"

Do not run a `make clean`, Release-A, or another command that removes the chosen

output directory until the result has been copied or committed to a dedicated

performance-results location.

Current Linux vendoring defect

Darwin vendoring is currently healthy, but the real Linux-x64 Docker command is not:

Fatal error: can't create build/vendor-linux-x64/release-Linux-x86_64/heyc.o: No such file or directory

The structural nested-directory spec passes, which means the plan/contract exists, but the actual container build reaches a path that was not created before parallel compilation. Treat the real command as the authority. The next implementation slice must reproduce the end-to-end container path, create the target directory in the exact build context, and verify the resulting Linux compiler and stdlib inside Linux before publication.

This documentation slice intentionally does not change vendoring code.

Why Linux vendoring is slow today

The macOS host produces Mach-O/arm64 objects and links against macOS system

libraries. A Linux-x64 distribution needs ELF/x86-64 objects, a Linux ABI, Linux

startup objects, a Linux libc/pthread set, target headers, and a compatible

linker. LLVM IR by itself does not supply those operating-system pieces.

The current Docker path is slow because it repeatedly creates a Linux build

environment, installs/locates tools, compiles the large C runtime and bootstrap

compiler for Linux, builds the Hey-owned public binaries, verifies them under

Linux, and copies the staged distribution back to macOS.

Can LLVM replace Docker?

Yes for the hot build path, provided Hey ships or locates a complete target

toolchain. Clang supports cross compilation through a target triple and

`--sysroot`; LLD can link ELF on macOS. The missing ingredient is not LLVM code

generation but the Linux sysroot/runtime contract.

Target direction:

Hey HIR/MIR
    -> LLVM IR with x86_64-linux target triple/data layout
    -> clang --target=x86_64-unknown-linux-musl --sysroot=<linux-sysroot>
    -> target runtime object compiled from runtime/hey_runtime.c
    -> ld.lld + musl CRT/libc/pthread
    -> Linux ELF executable

Musl is the preferred first target because a controlled static or mostly-static

sysroot is easier to package reproducibly than binding to an arbitrary host

glibc version. Docker/Podman should remain an execution-verification boundary,

not the normal compiler.

Cross-target implementation queue

1. Introduce target descriptors containing triple, CPU/features, pointer mode,

object format, linker flavor, libc, and sysroot identity.

2. Emit the correct LLVM target triple and data layout.

3. Build/cache `hey_runtime.o` per target/profile/source hash.

4. Package or fetch a checksum-pinned Linux-x64 musl sysroot and CRT objects.

5. Invoke target Clang/LLD directly from macOS.

6. Stage the distribution without entering a container.

7. Verify the produced ELF in a minimal container or VM.

8. Compare bytes/checksums against a native Linux build.

9. Keep Docker fallback until the direct cross-target path passes Release-B and

vendoring tests.

10. Add timings for sysroot acquisition, runtime compile, compiler build,

public binaries, verification, and publication so the remaining cost is

visible.
