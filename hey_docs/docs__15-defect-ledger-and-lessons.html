v0.99.225a lessons: byte-range ownership and hidden quadratic collection reads

The first ordinary-Hey 1BRC actor version still let a coordinator parse and

package work. The correct architecture is one newline-aligned mapped byte range

per worker, one private mutable table per worker, and one compact table returned

for merge. Actors express ownership and completion; they must not create one

mailbox interaction per input record.

A 50,000-station stress case then remained above five minutes even though the

scan-only loop completed quickly. The actual defect was general LLVM collection

lowering: every `array.length` and `array[index]` deep-cloned the complete

immutable array. A normal output loop was therefore O(n^2). LLVM now borrows the

collection and clones only the selected element. Preserve this distinction:

immutable language semantics do not require deep-copying the container for a

read.

Output assembly had another avoidable immutable-growth pattern. Use the general

ByteBuffer for repeated append, then freeze once. Do not solve parser/output

benchmarks with challenge-specific runtime functions.

Byte-table slot selection also exposed weak low-bit distribution when FNV hashes

were masked by a power-of-two capacity. Apply a final avalanche before masking;

measure collision/probe behavior for future table APIs.

v0.99.225a lesson: an elapsed-time threshold is not a timeout

The original honest 1BRC scale harness measured a case and only afterward checked whether `runtime_ms >= 300000`. A hung or extremely slow case could therefore run forever and exhaust the machine. The fix isolates every case in a child process group and uses the general `process.run_argv_timeout` primitive to terminate it at the deadline.

Repeat: safety limits around untrusted or performance-exploratory work must be enforced outside the work being limited. Preserve partial results before launching the next case.

Avoid: cooperative checks inside a hot loop as the only protection against deadlock, runaway allocation, blocked I/O, or a wedged actor join.

v0.99.221a parallel runtime-object publication

A clean extracted `make -j4` exposed two native application builds compiling the shared cached runtime object directly to the same final pathname. One linker could observe the other compiler's truncated/partial object and report hundreds of undefined runtime symbols. Runtime objects now compile to per-process temporary files and publish with atomic rename. Preserve `test/regression_runtime_object_cache_parallel_publish.sh`; do not serialize all application builds merely to hide this cache-publication defect.

v0.99.221a lesson: a benchmark builtin is not language performance

The previous 1BRC path moved parsing, hashing, aggregation, and threading into C and let LLVM optimize only a wrapper. That is invalid as proof that Hey is fast. Challenge algorithms must remain ordinary Hey source; native support may expose only general reusable primitives. Performance failures are requirements for better language/runtime/compiler facilities, not permission to hide the workload below the language boundary.

v0.99.220a handoff lessons

  • A benchmark result written under `build/` is disposable. Release-A begins with

`make clean`, so preserve expensive evidence under a dedicated results root

before starting release gates.

  • A 100M real-file prefix is useful for worker selection but did not predict the

1B result: 20.53 seconds projected versus 43.242 seconds actual. Treat the

full-file proof as authoritative and rerun adjacent worker counts at 1B.

  • Compiler-host choice and application runtime are separate dimensions. The two

product hosts should emit identical MIR; compare their compile/MIR timings

separately from the runtime of the resulting application.

  • Configuration fields are not implementation. `max_workers`, `scale_on`, and

network backend descriptors do not prove dynamic autoscaling, queue-latency

sampling, or distributed execution.

  • The desired Jobs abstraction is a named `perform` unit of work enqueued by

`perform_later/in/at`; actor envelopes, result acknowledgements, schedulers,

and worker pools are implementation details for the common case.

  • LLVM can emit a foreign target object, but cross-platform linking still needs

the target ABI, sysroot, CRT, libc/pthread, runtime object, and linker. Move

Docker from compiler to final execution verifier only after those artifacts

are explicit and checksum-pinned.

  • Formatter-driven syntactic sugar must be AST-based and semantics-preserving.

Never rewrite object keys, raw strings, comments, or precedence with regexes.

v0.99.215a lessons: corpus closure is the compiler test

  • Treat C-hosted and LLVM-hosted Hey frontends as independent producers. Equal output alone is insufficient; require byte-identical MIR so frontend drift is visible before backend execution.
  • Application HIR must compile local modules but represent stdlib imports as stable capabilities. Recursively embedding stdlib implementation bodies made compiler programs pathological and coupled sidecars to installed library source.
  • Corpus closure exposed general defects that small fixtures missed: optional actor metadata, module symbol identity, parameter inference, actor-state dominance, boxed built-ins, bytes/property typing, mixed concatenation, null/reset representation, and runtime-capability mapping. Fix semantic families centrally; never add example-name branches.
  • A single 88-unit cold run can exceed environment caps. Keep deterministic offsets/chunks and reusable compiler artifacts so closure is resumable without weakening the inventory.
  • Public scripting requires the executable path, cache identity, sidecar validation, compile-on-miss, and argument forwarding to work together. A shebang smoke must execute the staged distribution, not only a source-checkout wrapper.

v0.99.212a defect lessons

  • **Imported compiler code treated as an opaque stdlib facade:** LLVM originally refused to emit `stdlib:hey_ir`. Compiler implementation libraries that form the frontend must be directly emitted; ordinary stdlib calls remain ABI boundaries.
  • **Heuristic ABI inference cannot scale to compiler code:** recursive emitters and boxed syntax trees exposed pointer-as-integer corruption. Optional explicit signatures now carry authoritative boundary facts.
  • **String-return values passed to `put` were lowered as integers:** LLVM now selects the string map-update ABI for string expressions, not only string literals.
  • **Conformance aggregation must isolate implementations:** a single long-lived runner mixed compiler lifetimes and made a slow aggregate look stalled. The launcher now starts one process per implementation while the manifest stays backend-opaque.

v0.99.211a defect lessons

  • **Conformance leakage:** implementation/backend assertions had accumulated under `specs/language`. New shared conformance lives in a backend-opaque manifest; compiler and IR structure checks live separately.
  • **Spec metadata contamination:** the self-host parser skipped a `spec` block structurally but merged its `expect` statements into the executable unsupported inventory. Skipped metadata must not affect executable parsing.
  • **LLVM dominance:** allocating a repeated lexical local at its first textual `let` can place the slot in one sibling basic block while another sibling stores through it. Reused scalar slots are now predeclared in the function entry block.
  • **Fake shared backend:** a frontend-to-LLVM claim is invalid if the consumer reopens `.hey` source. The compiler contract deletes source before consuming either producer's typed IR.
  • **Specification boundary:** canonical typed IR is optional compiler interoperability, not a requirement for Java, Rust, or other conforming Hey implementations.
  • **Missing outer terminators:** the self-host parser could let a nested `end` close an inner branch and then accept EOF as the end of the surrounding program/function/actor/module. Every block-owning form must require an explicit terminating `end`; the opaque negative conformance row pins rejection without prescribing diagnostics.
  • **Version-only compiler caches:** a parser fix within v0.99.211a rebuilt neither the reusable self-host compiler nor its parser cache, so conformance executed stale behavior. Reusable compiler executables now fingerprint compiler implementation sources, and parser-cache directories include the exact emitter identity.

v0.99.210a defect lessons

Backend names must describe independent dimensions

The old backend-timing wrapper parsed a marker stream by the wrong shape and called a three-way view “triple timing,” obscuring whether LLVM described the compiler executable or the generated app. The maintained matrix now records compiler implementation, compiler backend, and output backend independently.

Generic `len(parameter)` is boxed until proven otherwise

LLVM treated `len(xs)` inside a user function as string length because the classifier constructed a `needle_len` probe but omitted it from the boxed-use decision. Array arguments then reached `strlen`. The probe now participates in classification; spec 816 and Release-C execute the LLVM result.

Documentation examples are executable contracts

Two package ergonomics examples used `.main`, which is not a Hey value literal, and caused the only two native-suite failures in JayTeeSF's v0.99.209a run. They now use `['main']`, and the corresponding package docs match.

v0.99.209a defect lessons

Receiver sugar must not consume qualified module calls

The v0.99.208a method recognizer treated `collections.uniq(names)` as though

`collections` were a runtime receiver. Recursive type inference repeatedly

rewrote the same call with another first argument, causing spec 793 to grow to

about 1.99 GB RSS in eight seconds under the parallel IR-golden run.

Non-recurrence rules:

  • resolve an actually defined qualified callable before attempting receiver

sugar;

  • scan self-host expressions only for top-level method markers;
  • keep direct module calls in the regression fixture beside receiver calls;
  • bound recursive inference/desugaring so a future cycle cannot exhaust the

host;

  • diagnose the active parallel job, not merely the last completed line printed

by another worker.

Do not nest heavyweight compiler parallelism

Release-A placed the large typed-IR build matrix beside an internally parallel

IR-golden matrix inside another four-job batch. Even after the recursion bug was

fixed, that multiplied peak memory and CPU contention on developer laptops.

Release-A now sequences the typed-IR and IR-golden gates, then caps the remaining

LLVM batch at two jobs.

v0.99.208a defect lessons

  • Typed-IR parameters must be classified independently from return types. The old shortcut turned numeric counts into strings and boxed objects into scalars once mixed ABI inference expanded.
  • Diagnostic compiler analyses need the same cycle protection as executable lowering. Memoized string-function states and bounded boxed-parameter recursion prevent large import graphs from hanging or overflowing.
  • Control-flow syntax must be recognized before an implicit-return fallback; otherwise `if`/`while` headers become fake returns in HIR and hide actual ABI errors.
  • Dot syntax must be parsed after known module/function/member forms but before generic expression fallback; built-in property/method spellings retain precedence and bracket access remains the collision escape hatch.
  • Parameter ABI and return ABI are independent facts. Treating a boxed-input boolean helper as boxed-returning caused invalid call signatures across unrelated tooling rows.
  • LLVM emitter scope pointers must be restored on every return path. An early-return path left `current_locals` pointing at cleared storage and produced nondeterministic later-function classification.
  • Removing an `unsupported` guard can expose real dataflow defects. The 58-row closure required shared local joins, dynamic key classification, repeated-parentheses handling, and ignored-result call lowering—not 58 row names.

v0.99.206a lessons

  • Do not call C11 words Hey reserved words. They are backend-reserved spellings and remain legal Hey identifiers; generated C alone is renamed.
  • Parser-cache hits do not solve emission-time quadratic work. Measure parse, reachability, environment construction, and per-form emission independently.
  • Imported functions and actors are not roots merely because they exist. Follow references, but retain conservative all-actor behavior for dynamic actor APIs.
  • A self-host timeout should be fixed by shared capability lowering or algorithmic work, never by increasing the promotion timeout.

v0.99.204a defect lessons

  • **String literal property ordering:** self-hosted C emission treated `"Brainscape".length` as a generic C string literal before property sugar ran, producing invalid C member syntax. Property/access sugar must be recognized before terminal literal fallbacks. Spec 903 and the 12-row dual suite pin this.
  • **Opaque-pointer toolchain drift:** generated LLVM IR used modern `ptr` syntax, but the stable Linux container's older Clang required explicit opaque-pointer mode. Generated-IR syntax and target linker invocation are one compatibility contract; spec 904 pins the switch.
  • **Unconditional runtime linkage:** the LLVM emitter generated a web handler and runtime initialization for every program, forcing the monolithic runtime object into pure scalar binaries. Runtime dependencies are now derived from emitted call sites; pure paths direct-link, while runtime paths remain explicit. Spec 904 and `llvm-compare-smoke` pin both classifications.

1. Interpreter global-environment race (fixed v0.99.4a)

  • **Symptom:** on 1-CPU linux-x64, jobs-pool specs printed garbage identifier errors ("undefined name: /", "do not understand: O") and crashed with SIGSEGV; on an 8-core M3 the same programs passed, so the bug was invisible on the development machine.
  • **Root cause:** `env_find` returned interior `ReplVar*` pointers into `global_env`, a realloc-growing array. The main thread grew and mutated it while actor threads read through the global-fallback lookup. A realloc moved the array under readers; value replacement released `HeyValue`s a reader was cloning.
  • **Why it existed:** the interpreter began single-threaded (REPL) and actors were added later; the shared-fallback lookup was never revisited for concurrency, and the fast machine's scheduling hid the window.
  • **Fix:** `repl_global_env_mutex` plus clone-under-lock accessors `env_lookup_clone`/`env_has`; `env_set` and `env_mark_done_if_var` lock when touching `global_env`.
  • **Guards:** `make sanitize-smoke` (ThreadSanitizer strict — any warning fails); specs `320`/`607`/`608` as concurrency canaries; the linux-x64 1-CPU environment as recurring evidence.
  • **Rule:** no interior pointer into shared mutable storage may escape the lock that protects it. Look up and clone under the lock; return values, not pointers. Any state reachable from more than one thread needs a written owner or a lock, even in "interpreter-only" code.

2. Teardown use-after-free of actor definitions (fixed v0.99.4a)

  • **Symptom:** intermittent SIGSEGV at or after program completion with correct stdout already flushed.
  • **Root cause:** `heyc` and REPL teardown freed actor definitions and the global environment before straggler actor threads were joined; late actors parsed freed body lines.
  • **Why it existed:** `hey_runtime_shutdown` did join all actors, but it ran after the interpreter-state frees; the ordering dependency between the interpreter layer and the runtime layer was implicit.
  • **Fix:** `hey_runtime_drain_actors()` (stop joins every thread) called before any interpreter-state free in all six teardown paths.
  • **Guards:** sanitize-smoke; repeated-run canaries (the crash trio passes 45/45 on 1 CPU).
  • **Rule:** join every thread before freeing anything it can read. Teardown order is an API contract; make it explicit in a function name, not a comment.

3. "Benign" shared counter race (fixed v0.99.4a)

  • **Symptom:** ThreadSanitizer warnings on `repl_line_no++` that drowned out the real races above.
  • **Fix:** `_Atomic int`.
  • **Rule:** there are no acceptable "benign" data races. They are undefined behavior in C11 and they bury real races in sanitizer output. Fix them immediately so TSan stays a zero-warning gate.

4. Self-hosted emitter leaks unknown calls as raw C identifiers (found v0.99.3a, `at`/`json_text` fixed v0.99.5a)

9. Native-C typed-scalar fast path printed booleans as raw 0/1 (fixed v0.99.8a)

  • **Symptom:** `examples/typed_scalar_fast_path.hey` and `specs/language/210_typed_scalar_fast_path.hey` (spec: `expect stdout "true\n1"`) compiled and ran clean, but a compiled build printed `1\n1` instead of `true\n1`. The backend-world-full gate reported this as a **self-hosted** `output_mismatch`, because self-hosted actually emitted the semantically correct `true\n1` and got diffed against native-c's own wrong capture, which the harness trusts unconditionally as the reference.
  • **Root cause:** the optimized typed-IR/unboxed-scalar native backend (`heyc_fast_emit_lines` in `runtime/heyc.c`, the `HEY_FAST_INT_BACKEND` path used when a program is fully scalar-eligible) represents every local as plain C `long` with no boolean tag, and `says`/`print`/`warn` always formatted with `%ld`. `./bin/heyc <file> --run` (interpreter path) and `--test` (spec-check path) both go through the boxed `HeyValue`-tagged interpreter and rendered correctly, so the bug was invisible outside a compiled build — exactly the kind of split the world-gate harness exists to catch, except the harness treats native-c's compiled-build output as ground truth without checking it against the spec's own `expect stdout`.

  • **Guards:** `./bin/test --full` (285/285), `make typed-ir-smoke`, `make native-build-smoke` all stay green; both affected fixtures now match their `expect stdout`.
  • **Rule:** any backend that introduces its own unboxed/typed representation for values the boxed interpreter tags (bool, in this case) must reproduce the boxed interpreter's *observable* formatting, not just its arithmetic. When a bug is backend-specific and invisible through the interpreter path, do not assume the compiled native-c oracle is correct just because it is the oracle — see item 12's open question below.

10. Self-hosted emitter collapsed multi-arg `spawn`/`supervise` into one malformed literal (fixed v0.99.8a)

  • **Symptom:** `benchmarks/hey/c10k_actor_ping.hey` (`spawn connection(1, sink)`) failed self-hosted `build_failed` with `clang: too many arguments to function 'hey_int'`. Generated C read `hey_actor_spawn_values("connection", hey_actor_connection, 1, (HeyValue[]){hey_int(1, sink)})` — arg count hardcoded to `1`, and the two actor args `1, sink` were passed as one string straight into `hey_int(...)`.
  • **Root cause chain:** (a) `hey_c_runtime_actor_arg_from_spawn_value` in `stdlib/hey_c.hey` returns the *entire* text between the spawn/supervise parens as a single string, never splitting on top-level commas, while the caller hardcoded the runtime arg count to `1` and wrapped that whole string in one `hey_c_runtime_expr_value` call; (b) `hey_c_runtime_int_literal_value` only checked that the *first* character of an expression was a digit, so the malformed multi-arg text `"1, sink"` silently passed as a valid integer literal and got wrapped as `hey_int("1, sink")` instead of failing loudly or erroring at the split step. The same two-part bug existed independently in the `spawn` and `supervise` call sites (near-duplicated code), and single-arg actors never exercised it, which is why it shipped unnoticed.
  • **Fix:** added `hey_c_runtime_actor_args_code_value`, which uses the existing generic top-level-comma splitter (`hey_c_runtime_object_entries_value`) to convert each actor argument independently and join with `, `; both `spawn` and `supervise` now compute the real arg count from the split list instead of hardcoding `1`. Also tightened `hey_c_runtime_int_literal_value` to require every character be a digit (not just the first), closing the general "text that starts with a digit passes as an int literal" hole for any other call site.
  • **Guards:** `./bin/test --full` (285/285); manual byte-for-byte native-c vs self-hosted diff on `benchmarks/hey/c10k_actor_ping.hey` (`c10k done clients: 5` on both).
  • **Rule:** when a helper returns "the raw text of N comma-separated things," never wrap it whole in a single-value converter — split first, convert each part, and derive counts from the split, never a literal. Literal-detection helpers must validate the whole token, not a prefix; a prefix check that returns true on partially-matching garbage is a silent-failure hole matching defect class 4.

11. LLVM never consulted its own `bool_locals` tracking when printing a bare variable (fixed v0.99.8a)

  • **Symptom:** the same two `typed_scalar_fast_path` fixtures from item 9 also mismatched under LLVM: `says ok` (where `let ok = total > 0`) printed `0`/`1` instead of `true`/`false`. Also: if a backend already tracks a fact (`bool_locals`), audit every place that fact should be consulted, not just the site where it was first needed.

12. Self-hosted module-qualified function calls: two compounding bugs (both fixed v0.99.9a)

  • **Symptom:** the dominant self-hosted `build_failed` cluster (~35 of 43 rows in the v0.99.7a M3 full gate, e.g. `tools/hey_ast_manifest.hey`) failed with `error: use of undeclared identifier 'hey_ast'` — a call like `hey_ast.manifest("stdlib")` got emitted as raw, uncompiled C text.
  • **Root cause, part A (fixed v0.99.8a):** `stdlib/hey_parser.hey` already namespaces functions declared inside `module X ... end` blocks with a dotted name (`X.fn`), and the call-site router `hey_c_runtime_function_call_value` in `stdlib/hey_c.hey` already mangles a qualified call through `hey_c_c_symbol_value` (`X.fn` -> `X__fn`) before checking `env["fn:" + symbol]`. But the functions doing the *registering* — `hey_c_runtime_function_env_value` (builds that `env`), `hey_c_runtime_function_decls_value`/`hey_c_runtime_function_defs_value` (emit the C declaration/definition), and the equivalent actor pair `hey_c_runtime_actor_decls_value`/`hey_c_runtime_actor_defs_value` — all used the *raw, unmangled* dotted name as the key/C-identifier. The registered key (`fn:X.fn`) could never match the looked-up key (`fn:X__fn`), so the call router always reported "not a known function" and fell through to raw-identifier emission (defect class 4), even for functions defined in the *same file*. Fixed by routing all five call sites through `hey_c_c_symbol_value`, matching what the caller already expected. Verified with a dedicated tiny fixture (`stdlib/hey_spec_module_helper.hey` + `specs/language/629_self_hosted_stdlib_module_call_routing.hey`) that now passes self-hosted end-to-end, plus `./bin/test --full` (286/286, no regression — non-namespaced top-level functions are unaffected since `hey_c_c_symbol_value` is a no-op on names without a `.`).
  • **Root cause, part B (fixed v0.99.9a):** even with part A fixed, most real-world failures (`tools/hey_ast_manifest.hey` importing `stdlib/hey_ast.hey`, etc.) still failed, because `stdlib/hey_parser.hey`'s `hey_parser_external_runtime_import_value`/`hey_parser_stdlib_import_supported_value` treat *any* `import` path containing `/stdlib/` as an opaque "external runtime" import to be silently skipped (not parsed, not merged into `forms`) unless it exactly matches a 4-file whitelist (`json.hey`, `hey_syntax.hey`, `markdown.hey`, `hey_parser.hey`). Every other `/stdlib/*.hey` file never got its function bodies merged, so the called function was never emitted as C regardless of part A's fix. Widening the whitelist naively would have traded the "undeclared identifier" failure class for a "redefinition" one: the parse walkers had no import de-duplication, so a file reachable via two chains (e.g. `hey_syntax.hey` imported both directly and via `hey_parser.hey`) was fully re-parsed and re-appended each time — the confirmed cause of the two pre-existing `redefinition` `build_failed` rows (`specs/language/595`, `568`).
  • **Fix (v0.99.9a):** (1) added `hey_parser_normalize_path_value` (collapses `seg/../` steps so one file reached through different chains maps to one canonical key) plus a `visited` map threaded through `hey_parser_parse_forms_value`/`hey_parser_parse_module_forms_value`/`hey_parser_parse_text_value`; an already-visited import merges nothing, and marking a path visited *before* recursing also closes a latent infinite-recursion on import cycles. External/whitelist checks still run on the raw resolved path, so follow/skip semantics are otherwise byte-identical. (2) With dedup in place, widened the stdlib whitelist to the pure-Hey modules the failing rows import — `fs.hey`, `hey_parser_expanded.hey`, `hey_semantics.hey`, `hey_ast.hey`, `hey_bootstrap.hey`, plus the two spec fixtures — each verified per-row before inclusion. `process.hey`, `time.hey`, `jobs.hey`, `net`-family, and `hey_c.hey`/`hey_dual*`/`hey_compiler_strategy.hey` remain excluded: the first group is runtime-routed porcelain (following them would emit unrouted `command`/`now_ns` calls into currently-green rows), and the second group is the emitter-emitting-itself step that needs its own slice.
  • **Guards:** micro-specs `629` (module-call routing) and `630` (diamond-import dedup, fails as duplicate C definitions without the fix); `./bin/test --full` 287/287; rows verified self-hosted end-to-end with byte-identical native diffs: specs `395`/`420`/`460`/`470`/`480`/`547`/`568`/`589`/`595`/`596`, `tools/hey_ast_manifest`/`hey_semantics_check`/`hey_parser_expanded_check`/`hey_source_hotspots` (build), `examples/hey_semantics_tooling`/`hey_expanded_parser`; dual-timing and selfhost-coverage unchanged.
  • **Rule:** when a call-routing mechanism has a registration side and a lookup side, a single shared naming transform (here `hey_c_c_symbol_value`) must be applied identically on both sides — grep every call site that builds a `"fn:" + name`-shaped key before assuming a routing gap is a missing feature rather than a mismatched key. Never widen an import/inclusion allowlist without first confirming (or adding) de-duplication for whatever graph it walks; diamond imports are the default in any real module graph, not an edge case. And widen allowlists one entry at a time with the previously-green importers re-verified per entry.

13. Interpreter C-stack overflow on deep Hey recursion (stdlib hot paths converted v0.99.9a; general guard still open)

  • **Symptom:** `./bin/heyc tools/hey_ast_manifest.hey --run` — a fixture that passes in some shells — died with SIGSEGV under the default 8 MB stack, and the identical binary passed with `ulimit -s 65520`. Reproduced at pristine v0.99.8a HEAD, so it predates any current-slice change; it flaps with environment because the stdlib tree sits exactly at the recursion cliff.
  • **Root cause:** the interpreter consumes C stack for every Hey call frame and performs no tail-call elimination and no depth guard. `hey_syntax_scan_lines_value` recursed once per source line (`stdlib/hey_c.hey` alone is ~2,300 lines) and `hey_ast_count_forms_value`/`hey_ast_forms_for_report_value` recursed once per collected form repo-wide, so `hey_ast.manifest("stdlib")` stacked thousands of frames and crossed 8 MB.
  • **Fix (v0.99.9a, partial):** converted those three hot walkers to `while` loops (signatures unchanged); `tools/hey_ast_manifest.hey` now passes under the default stack in both the interpreter and the self-hosted build. The general class — interpreter SIGSEGV instead of a clean "recursion too deep" error — is still open below.
  • **Rule:** stdlib code that walks every line or every form of arbitrarily many files must be iterative, not recursive, until the interpreter eliminates tail calls or guards depth. When a crash reproduces in some shells and not others, check `ulimit -s` before bisecting code.

14. Self-hosted emitter text-scanning defects exposed by the self-emission probe (fixed v0.99.12a; self-emission itself still open)

  • **Symptom:** whitelisting `stdlib/hey_c.hey` so the emitter could compile *itself* (the last big self-hosted `build_failed` tier) first crashed with `slice expects non-negative integer start and count`, then produced C that clang rejected in several distinct ways.
  • **Root causes, each a general emitter bug reachable from ordinary user code (micro-spec `631` isolates the first two):** (a) `hey_c_runtime_access_from_base_value` matched the *first* `]` after an index `[`, so a key containing a nested index or bracket (`fenv["fn:" + form["name"]]`) was truncated mid-string; (b) every string-key fast path (`get`/`has?`/`put`/`del`, method and bare forms) treated any key *starting* with a quote as one whole C string literal, so a concat key (`"fn:" + name`) was pasted raw into C; (c) `hey_c_runtime_find_top_level_op_scan_value` did not skip backslash escapes inside string literals, so `\"` inside a literal toggled the in-string flag and operator splits landed inside literals; (d) `stdlib/hey_c.hey` itself contained eight malformed legacy string literals (`" printf("%s\\n", "` with bare interior quotes) that the interpreter's loose expression parser tolerated but any faithful re-emission cannot — the single-quote raw-string direction now in docs/08-open-questions.md exists to prevent this class at the language level.
  • **Fixes:** balanced-bracket scanner (`hey_c_runtime_matching_bracket_scan_value`) for index extents; shared `hey_c_runtime_key_literal_value` guard (literal fast path only when the key is one whole literal) applied at all eleven key sites; backslash skip in the top-level-op scanner; the eight legacy literals corrected to properly escaped quotes.
  • **Guards:** micro-spec `631` (concatenated and nested-index string keys, native + self-hosted); full suite 288/288; previously-green self-hosted rows re-verified byte-identical.
  • **Still open (deliberately deferred):** actual self-emission. With the fixes above the emitter processes all ~2,300 lines of its own source without crashing, but the generated C still has at least three defect classes (trailing index after a call — `command(x)["ok"]`; nested-function emission when a body mis-terminates; `let` with unsupported RHS emitting a comment while later uses still reference the name), and each tier row costs ~4 minutes of emission, which would add roughly an hour to the 13-minute full gate. The `hey_c.hey`/`hey_dual_suite.hey`/`hey_compiler_strategy.hey` whitelist entries are therefore *not* shipped; `process.hey`/`time.hey` (tiny, verified) are. Re-attempt self-emission only with per-class fixes plus an emission-performance pass.
  • **Rule:** text scanners in the emitter must all share the same three invariants — balanced delimiters, string-literal awareness, and escape awareness; a scanner missing any one of them is a latent wrong-output bug for exactly the source most worth compiling (the compiler itself). And before whitelisting a large module for import-following, measure the emission cost: a correctness win that multiplies the gate's wall time is not shippable as-is.

15. Single-quoted raw strings landed as a front-door normalization (v0.99.13a)

  • **Direction:** JayTeeSF answered this in docs/08-open-questions.md: Ruby-style `'...'` raw literals (no interpolation; only `\'` and `\\` escape) to eliminate the escaped-quote defect classes (items 12/14) at the language level.
  • **Design:** implemented as a line-ingestion rewrite, not a new literal kind threaded through every backend. `hey_normalize_single_quotes` in `runtime/hey_repl.c` (shared by `heyc` via the existing `#include "hey_repl.c"`) rewrites `'...'` to an equivalent double-quoted literal (`"`→`\"`, `\`→`\\`, `#{`→`\#{`) before comment stripping in `heyc_split_lines` and in both REPL loops; `hey_parser_normalize_line_value` in `stdlib/hey_parser.hey` mirrors it at all five line/continuation intake points of the self-hosted pipeline. Every downstream consumer — interpreter, native-c emitter, fast path, typed IR, LLVM, spec harness, self-hosted emitter — therefore sees exactly one string syntax, byte-identically on both front doors. Apostrophes inside double-quoted strings are untouched; an unterminated `'` (a comment apostrophe) leaves the rest of the line unchanged, and whole-line/trailing comment stripping runs after normalization so garbled comment text can never reach the parser.
  • **Guards:** micro-spec `633` (raw semantics: embedded `"`, literal `#{...}`, literal backslashes, `\'`, apostrophe-in-double-quotes, raw/cooked concat) passing interpreted, compiled native-c, and self-hosted with byte-identical output, plus a REPL check; full suite 290/290 with the normalizer active over the whole repository; dual-timing, selfhost-coverage, repl-specs/matrix, sanitize-smoke green.
  • **Found while testing, later fixed:** the LLVM subset's string-interpolation scanner did not honor the `\#{` literal marker; v0.99.17a fixed it with an escape-aware interpolation marker scanner and spec `638_llvm_escaped_interpolation_marker.hey`.
  • **Rule:** when adding surface syntax whose meaning is expressible in the existing core syntax, normalize at the front door instead of teaching every backend a new case — one rewrite point keeps N backends in byte-for-byte agreement by construction. The two Hey emitters only compile the normalized form, so writing `'...'` in `stdlib/` sources is safe everywhere the whitelisted modules are followed.

17. LLVM text facades must share literal-marker and bool-rendering invariants (v0.99.17a)

  • **Symptom:** after single-quoted raw strings normalized `#{...}` to the literal marker `\#{...}`, LLVM still scanned for interpolation with a plain substring search. `says 'literal #{name}'` could therefore interpolate `name` under LLVM while native-c and self-hosted printed the literal marker text. That made `backend_summary self-hosted good=294/326` look like a coverage regression even though the new timeout guardrail was intended for runaway app execution.

  • **Fix:** split the harness settings into `--run-timeout-seconds` (default 10) and `--build-timeout-seconds` (default 0/off). The old `--timeout-seconds` flag remains an alias for the run timeout only, and parallel `--jobs` workers pass both settings explicitly.
  • **Guard:** default backend-world runs no longer classify slow-but-valid builds as `build_timeout`; build timeouts are opt-in for focused investigations.
  • **Rule:** build and run phases need separate timeout policies. A default app-run timeout may protect a long world gate, but it must not distort backend coverage by timing out expensive compiler/emitter work unless the caller explicitly asks for that experiment.

21. LLVM method-sugar facades must desugar before type/boxed/bool classification (v0.99.21a)

  • **Symptom:** LLVM's data-model coverage had direct lowering for core calls such as `get(obj, key)` and `has(obj, key)`, but method-sugar forms such as `obj.get('name')`, `obj.has?('visits')`, `obj.keys()`, and `obj.to_json()` remained unsupported or risked predicate output as raw `1`/`0` because the LLVM classifiers saw the pre-desugared expression.
  • **Root cause:** method sugar was handled in some expression paths but not at every classification boundary. The boxed-expression, string-expression, bool-expression, and generic expression emitters could therefore disagree about the same source expression.
  • **Fix:** LLVM now desugars method calls before boxed/string/bool/expression classification and lowers data-model method facades through the same runtime shims as their function-call forms. The slice added `has`/`has?` index support, `keys`, `values`, `entries`/`items`/`pairs`, method `to_json`/`json_text`, and typed-IR op contracts/goldens for the new shapes.
  • **Guard:** `specs/language/643_llvm_json_method_facades.hey` plus targeted backend-world over specs 190/260/551/643 reports native-c, self-hosted, and LLVM all positive_ok=4/4; `make ir-golden-smoke` pins the IR handoff.
  • **Rule:** any syntactic sugar that can change apparent return type must be normalized before every backend classification pass, not only immediately before final code emission. Predicate facades must prove boolean rendering as well as truth-value lowering.

22. LLVM file/process/text facades must separate string paths from boxed targets (v0.99.22a)

  • **Symptom:** LLVM could lower plain string file-I/O but still rejected target/mode forms such as `write file(path, mode), value`, boxed reads like `read(file(path, mode))`, and stdlib facades such as `read_lines`, `command`, `text.split`, `text.parse_int`, and `text.parse_fixed`.
  • **Root cause:** LLVM classification treated file/process/text parse facades as generic unsupported expressions instead of op-shaped runtime shim calls with explicit string-vs-boxed source handling.
  • **Fix:** added general runtime shims for boxed file values, target/mode writes, read-lines, command, split, and parse facades; LLVM now routes string path sources and boxed target/source values to different explicit shims rather than guessing late in emission.
  • **Guard:** spec `644_llvm_file_target_command_facades.hey`, targeted backend-world over specs 300/561/563/588/644, `llvm-declare-smoke`, and `ir-golden-smoke`.
  • **Rule:** file/process facades must classify their source/target shape before emission. A path string, boxed `file(...)`, and process result object are distinct lowering shapes even when they share public porcelain.

23. LLVM exact-call and facade classifiers must agree before local kind assignment (v0.99.24a)

  • **Symptom:** lowering new LLVM allocation/type/shape/json-codec facades exposed two classification gaps: zero-arg facades such as `allocations()` were not recognized by the exact-call scanner, while expressions such as `get(stats, 'json_gets') + get(stats, 'json_get_key_fast')` could be mistaken for a single boxed `get(...)` call. A separate prefixed boolean expression, `says 'shape changed? ' + (shape_id(a) != shape_id(b))`, printed `1`/`0` instead of `true`/`false`.
  • **Root cause:** LLVM had several overlapping call classifiers. Some accepted prefix-only matches and some rejected zero-argument calls, so boxed/string/bool/scalar assignment could disagree about the same expression before emission. Boolean detection also did not strip safe outer parentheses before checking comparisons.
  • **Fix:** tightened `heyc_llvm_line_call_name` to require a balanced full-expression call while accepting zero arguments, made boxed-call candidacy use that exact classifier, and taught boolean classification to trim outer parentheses. Allocation/type/shape/json-codec facades now route through op-shaped runtime shims.
  • **Guard:** specs `645_llvm_allocation_type_facades.hey` and `646_llvm_json_codec_shape_facades.hey`; targeted backend-world over specs 250/280/290/365/645/646 plus examples allocation/shapes/collection reports LLVM good=9/9, and targeted backend-world for new specs 645/646 reports native-c, self-hosted, and LLVM all positive_ok=2/2.
  • **Rule:** before assigning a local or RHS to boxed/string/bool/scalar lowering, LLVM must use the same exact-call classifier at every boundary. Prefix matches are not calls; parenthesized predicates must be recognized before stdout lowering.

24. LLVM data-model update facades must cover both boxed values and scalar/string/bool leaves (v0.99.24a)

  • **Symptom:** LLVM could lower basic object `put` and method sugar, but still rejected existing repository rows that used `push(xs, value)`, `merge(obj, update)`, `del(obj, key)`, array-index `put(xs, i, value)`, or variable-carrying spread literals such as `[...base, ...more]`. A separate row compared an integer local to a string literal and tried to emit the string literal as a numeric expression.
  • **Root cause:** the data-model line emitter handled only the early Rung 1 constant/update subset. Collection update facades needed explicit boxed/scalar/string/bool leaf routing, and mixed scalar/string equality needed to be classified before generic numeric comparison lowering.
  • **Fix:** added LLVM runtime shims and lowering for `push`, `merge`, `del`/`delete`, array-index `put`, boxed-value `put`, and variable-carrying array/object spread literals. Equality now treats one-string/one-non-string `==` as false and `!=` as true, matching Hey value semantics without coercion.
  • **Guard:** specs `647_llvm_json_update_facades.hey` and `648_llvm_mixed_equality.hey`; targeted backend-world over specs 010/076/090/180/635/647/648 reports LLVM good=7/7, and the new specs pass native-c, self-hosted, and LLVM.
  • **Rule:** data-model update facades must route leaf values by shape before emission. Do not assume update values are integers merely because an expression is not a string literal; boxed values, strings, booleans, and scalars need distinct shim calls.

25. LLVM module-qualified function calls require matching definitions (v0.99.28a)

  • **Symptom:** `specs/language/612_module_qualified_function_symbol_parity.hey` and `613_second_module_function_symbol_parity.hey` failed only under LLVM with undefined symbols such as `@hey_fn_MathBox__inc`.
  • **Root cause:** LLVM call emission already mangled module-qualified function calls into the same `Module__fn` symbol shape as native-c, but the LLVM definition pass skipped every function whose Hey name contained `.`, a guard originally added to avoid poisoning programs with imported stdlib facade wrapper bodies.
  • **Fix:** LLVM now emits module-qualified function definitions. Unsupported function bodies still fail loudly through the existing LLVM unsupported path; the fix only makes call/definition symbol handling symmetric.
  • **Guard:** existing specs 612 and 613 remain the regression guard and now pass native-c, self-hosted, and LLVM under targeted backend-world. No existing specs were removed.
  • **Rule:** do not fix undefined-symbol rows by skipping call sites or removing specs. Function symbol policy must be symmetric: if a backend can emit a call to a symbol, it must either emit the matching definition or reject the source before producing IR.

26. Specs are the regression floor, not cleanup targets (v0.99.28a)

  • **Rule:** existing specs must not be removed to hide regressions or make a slice look green. A spec may be deleted only when it is explicitly obsolete and replaced by stricter/equivalent coverage, and the final answer must call out the removal and replacement.

27. LLVM variable-carrying literals must fall back to op-shaped lowering (v0.99.28a)

  • **Symptom:** LLVM could lower constant JSON literals by folding through the interpreter, but rejected existing rows with variable-carrying literals such as `[1, value]` or `{ok: ok}` even when each leaf value had a supported scalar/string/bool/boxed shape.
  • **Root cause:** failed constant folding was treated as unsupported instead of falling back to structural literal emission with per-leaf shape routing. Float and bool locals also were not consistently promoted into boxed runtime values before `to_json`.
  • **Fix:** array/object literal lowering now falls back to op-shaped boxed construction for supported variable leaves, with explicit float, bool, scalar, string, and boxed shim paths.
  • **Guard:** existing float/boxed-output specs 569/598/601/603 plus new spec `649_llvm_variable_boxed_literals.hey` and its `.hir` golden; targeted native-c+LLVM backend-world reports the touched rows green.
  • **Rule:** failed interpreter constant-folding is not itself an unsupported-language proof. If every leaf has a supported runtime shape, LLVM must lower the literal structurally or reject the specific unsupported leaf loudly.

28. Directly-lowered stdlib facade imports need skip rules plus call-site guards (v0.99.28a)

  • **Symptom:** LLVM could lower bare file/process/text primitives, but importing `stdlib/process.hey`, `stdlib/files.hey`, `stdlib/fs.hey`, or `stdlib/text.hey` still risked poisoning a directly-supported program because LLVM tried to compile the facade implementation bodies.
  • **Root cause:** imported facade modules mix tiny public wrappers with slower helper functions that are not themselves the intended lowering target. Compiling every imported function body conflates "module imported" with "all module internals are LLVM-supported".
  • **Fix:** LLVM now skips directly-lowered stdlib facade definitions for process/files/fs/text prefixes and lowers the supported public calls explicitly at the call site through runtime shims.
  • **Guard:** specs `650_llvm_stdlib_process_files_facades.hey` and `651_llvm_text_trim_facade.hey`, plus existing rows 300/589/594; `.hir` goldens pin the current handoff surface.
  • **Rule:** skipping an imported facade definition is safe only when every supported public call has an explicit call-site lowering. Skips must not hide unsupported user-defined functions or unsupported calls inside the program block.

29. String-returning function definitions need pointer signatures, not scalar defaults (v0.99.28a)

  • **Symptom:** module-qualified function definitions that returned one of their string parameters could be emitted with scalar `i64` parameter/return assumptions under LLVM, even though the call site passed string pointers.
  • **Root cause:** LLVM's function-definition path inferred string-ness less completely than the call/emission paths and defaulted function parameters to scalar values.
  • **Fix:** LLVM now recognizes the supported string-return helper shape and emits `ptr` signatures/returns for functions that return a string parameter directly.
  • **Guard:** existing module-symbol/text helper rows, including spec 618 in the targeted native-c+LLVM backend-world run.
  • **Rule:** function-definition type inference must match call-site type inference. Do not lower a call to `ptr` while emitting an `i64` definition for the same symbol.

31. The self-hosted emitter tier is not a spec-cleanup target (v0.99.29a)

32. REPL imports must execute real source imports, not acknowledgements (v0.99.31a)

Defect: the REPL accepted `import "stdlib/json.hey"` as a hardcoded acknowledgement and rejected every other source import. That meant valid compiled Hey syntax such as `import "stdlib/hey_c.hey"` followed by `hey_c.emit_text(...)` failed interactively with `unknown function: hey_c.emit_text`.

Fix: REPL import now reads source files recursively, de-duplicates nested imports, defines imported functions/modules/actors, skips comments in executed bodies, and normalizes multiline object/array literals in imported function bodies. Existing jobs import coverage was strengthened and a new `hey_c.emit_text` REPL transcript pins the direct case.

33. LLVM string locals in boxed arrays are not integers (v0.99.31a)

Defect: LLVM lowered array literals containing a string local, for example `[value]`, through the integer push path. The emitted app printed a pointer address JSON array instead of `["A"]`, causing the unicode rows 570 and 619 to mismatch native-c.

Fix: boxed array lowering now recognizes string locals and calls the string push shim. Spec `652_llvm_string_local_array_value.hey` pins the small root case across native-c, self-hosted, and LLVM.

Rule: when a lowering path has separate scalar and boxed/string shims, local type tracking must be consulted before falling back to numeric emission.

34. Compile-time emitter data must stay direct, not hidden behind native-c (v0.99.32a)

Defect: self-hosted rows that called `hey_c.emit_text(...)` or `hey_c.emit_file(...)` with literal source data failed by leaving the module-qualified call in generated C. A tempting workaround would be to bridge to native-c or whitelist the entire large emitter module prematurely.

Fix: the self-hosted runtime emitter now recognizes compile-time string arguments, invokes the Hey-written parser/emitter at self-hosted compile time, and embeds the resulting `{ok, c, error}` object as JSON data in generated C. Simple string constants are carried through `let` and invalidated on `set` so the optimization remains deterministic and local.

Rule: closing a compiler-tooling row by embedding compile-time data is allowed only when the data is produced by the Hey-written compiler path and the generated C contains no native-oracle bridge marker. Do not confuse this narrow constant-folded subset with full imported-module self-hosting.

35. Runtime type errors must not become C operator accidents (v0.99.32a)

Defect: self-hosted direct scalar lowering could compile subtraction expressions such as `'oops' - 1` into C-level arithmetic instead of routing through the boxed runtime and raising Hey's `- expects two integers` error. The row then exited successfully or failed for the wrong reason, breaking expected-error parity.

Fix: self-hosted runtime-body selection now treats subtraction expressions as boxed-runtime-needed, preserving native-c's runtime error behavior for the intentional negative fixtures.

Rule: direct scalar shortcuts are valid only when they preserve Hey's type checks and errors. If an operator has runtime validation, the self-hosted emitter must either prove the operand types or route through the boxed runtime helper.

36. Bridge-marker detection must not match data literals (v0.99.32a)

Defect: backend-world searched generated self-hosted C for the bridge marker as an unanchored substring. A program that intentionally searched for the marker text as ordinary data could therefore be misclassified as `bridge_forbidden` even though no bridge was generated.

Fix: the harness now matches only the exact generated bridge marker comment line.

Rule: harness sentinels must be structurally matched. Never classify generated code by an unanchored substring that user/source data can contain legitimately.

37. Imported programs must not become the self-hosted entry point (v0.99.43a)

Native-C compiled the entry file's program, but the self-hosted runtime emitter selected the first parsed `program` form, so the imported tool wrapper ran instead of the spec's program. The v0.99.42a promotion report surfaced this as the only self-hosted `output_mismatch` row.

Fix: `hey_c_runtime_program_value_with_env_value` now scans all parsed forms and emits the last/entry `program` body, matching the existing simple emitter behavior and native-C's import semantics. Imported files still contribute functions, modules, and actors; their executable wrapper programs do not become the app entry.

Guard: existing spec `656_selfhost_build_time_focus_tool.hey` now byte-matches native-C under self-hosted, and new tiny spec `658_selfhost_imported_program_entry.hey` imports `support/imported_program_side_effect.hey` to pin that imported program bodies are skipped.

Rule: import following must preserve the entry file's program identity. Any compiler path that flattens imported forms needs an explicit entry-program selection rule; do not pick the first `program` encountered after import expansion.

38. Build-time focus must not read stale smoke staging data (v0.99.44a)

That is useful only while an explicit TSV is being copied for a single invocation. 39. String-returning function classification must account for local string temporaries (v0.99.44a)

Defect: LLVM string-function classification looked only at the first executable expression and understood literals, parameters, and direct concatenation. A function that built a local string first and returned it could be treated as scalar or unsupported even though every operation was already in the string subset. The typed IR had the same symptom for user string-function calls: it emitted `i64`/`boxed` where the real contract was `str`.

Fix: string-function analysis now tracks local string temporaries in order, typed IR records string params/returns and call results as `str`, and LLVM string-function emission allocates/stores/loads local string pointers before returning. Specs 655 and 660 plus golden IR pin the contract.

Rule: type/lowering classification must be flow-aware for local facts that were just established in the same function. Do not infer string function support from a single unparsed body line when the language already has `let`/`set` steps.

40. Imported jobs facade bodies must not poison directly lowered LLVM calls (v0.99.45a)

Defect: importing `stdlib/jobs.hey` could force LLVM to inspect helper/facade function bodies that were not part of the selected entry program's supported lowering path. A tiny descriptor program that only needed `jobs.queue(...)` or `jobs.actor_spec(...)` could therefore become unsupported because unused helper bodies referenced deeper actor/runtime constructs.

Fix: LLVM now skips imported `jobs.`/`jobs_` facade/helper definitions when supported call sites lower directly through runtime-backed shims. The supported public descriptor calls still emit explicit LLVM calls, and unsupported jobs execution calls remain unsupported instead of silently bridging to native behavior.

Guard: existing descriptor specs 552-557, 577, 578, 586, and 605 run under LLVM in targeted proof; new spec 661 pins descriptor construction through the public jobs facade.

Rule: facade modules with direct LLVM shims should not emit unused helper bodies that are outside the supported subset. Lower the supported public call sites explicitly; unsupported call sites must fail loudly and honestly.

41. Nested boxed descriptor reads need one general chain lowering, not output-specific shortcuts (v0.99.45a)

Defect: jobs descriptors are intentionally nested (`queue['scheduler']['strategy']`, backend configuration, envelopes with payloads). One-level key shortcuts make deeper descriptor inspection unsupported, and adding per-source or per-output rewrites would repeat the old benchmark-shaped lowering mistake.

Fix: LLVM now lowers general boxed key/index chains through `HeyValue*` runtime shims, so nested descriptor reads work for jobs data and for ordinary boxed objects/arrays without a jobs-specific pattern.

Guard: spec 662 pins nested queue scheduler and payload reads across native-C, self-hosted, and LLVM.

Rule: when a runtime descriptor API grows nested shape, add general boxed chain lowering plus tiny specs. Do not add source-specific stdout rewrites or benchmark-specific descriptor shortcuts.

42. Challenge proof commands must not be shortcut-lowered (v0.99.46a)

45. String-returning function detection must include helper-call bodies (v0.99.49a)

Imported helpers such as `text.trim` and `files.dirname` were therefore emitted as calls to undefined `@hey_fn_text__trim` / `@hey_fn_files__dirname` instead of the existing runtime-backed shims, producing LLVM `build_failed` rows in backend-world.

Rule: when a public facade already has a direct LLVM shim, that shim must have precedence over generic imported-function lowering. Generic function calls are the fallback, not the first choice.

48. Pooled actors must return mailbox nodes to the shared slab (v0.99.52a)

Defect: the pthread actor receive path recycled consumed `HeyMessage` nodes, but the pooled worker loop used `free(node)` after each message and on stop messages. 58. Self-hosted parser/emitter bracket logic must distinguish literals from top-level indexing (v0.99.113a)

Defect: a package example with nested multiline calls like `packages.registry([ packages.manifest(..., ['main']) ])` failed only under self-hosted-C. 60. Self-hosted compiler handoff should wait for generated C, not throwaway cleanup (v0.99.113a)

Defect: the `hey-selfc` wrapper ran the generated Hey-written emitter app through a pipe and waited for the process to exit. For large imported programs, the app could successfully write the generated C and print the success marker, then spend enough time in throwaway emitter cleanup to make the wrapper look like a build failure.

Fix: `hey-selfc` now runs the emitter app in the background, waits until both the success marker and generated C file are present, then terminates the throwaway emitter process and proceeds to compile the generated C. Rule: handoff wrappers should wait on the durable artifact they need, not on cleanup of a temporary compiler process. Keep a follow-up open for reducing large-import emitter cleanup cost inside the runtime/self-hosted compiler itself.
