Documentation site generator

`stdlib:docs` and `./bin/hey-doc-site` build a static HTML site from Hey's curated public documentation.

./bin/hey-doc-site \
  --source . \
  --out build/hey-docs \
  --theme bootstrap \
  --jobs 8

What the site includes

The generator uses an explicit public-document catalog. It includes the README, learning guides, language references, REPL and actor guides, module and visibility documentation, and standard-library guides.

It deliberately excludes work-tracking and historical material, including:

  • todo, done, in-progress, handoff, and open-question files;
  • release notes and version plans;
  • backend parity, LLVM gap, and performance evidence;
  • defect ledgers, feature-slice records, and implementation RFCs.

This boundary prevents release telemetry and terse implementation notes from becoming part of the user-facing site merely because one public page links to them.

Options

--source PATH      repository or distribution root
--out PATH         destination directory
--title TITLE      site title
--theme THEME      bootstrap or plain
--kind KIND        user, developer, agent, or all
--jobs COUNT       concurrent page-render jobs
--no-cdn           omit CDN CSS and JavaScript
--help             show command help
--version          print the Hey version

Publishing the project site

./bin/hey-doc-site \
  --source . \
  --out "$HOME/dev/jayteesf.github.io/hey_docs" \
  --theme bootstrap \
  --jobs 8

Markdown links are rewritten only when the target is in the curated public catalog. Links to excluded maintainer files are rendered as plain text rather than pulling those files into the site.

Documentation kinds

`stdlib:docs` owns both API metadata and HTML site generation. Documentation is classified into three public kinds:

  • `user`: tutorials, guides, language reference, and stdlib usage.
  • `developer`: implementation architecture, backend design, and contributor material.
  • `agent`: handoff, roadmap, todo, defect, and continuation context intended for coding agents.

Render one kind or all kinds together:

./bin/hey-doc-site --source . --out build/user-docs --kind user
./bin/hey-doc-site --source . --out build/developer-docs --kind developer-docs
./bin/hey-doc-site --source . --out build/agent-docs --kind agent-docs
./bin/hey-doc-site --source . --out build/all-docs --kind all

Programmatic discovery uses the same classification:

import 'stdlib:docs'

says docs.kinds()
says docs.classify('docs/03-current-handoff.md')
says docs.pages_for('.', 'agent')

`tools/hey_docs.hey` remains an implementation command on top of `stdlib:docs`; `hey_docs` is not a second stdlib module.

Aliases ending in `-docs` normalize to the same kinds; they do not create additional catalogs or modules.
