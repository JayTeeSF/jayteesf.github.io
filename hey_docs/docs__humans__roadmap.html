Roadmap after v0.99.363a ordinary-value closure

Product direction

Hey source -> LLVM-hosted Hey -> semantic HIR -> ABI MIR -> direct LLVM
one self-contained product archive per OS/architecture
immutable values + isolated actors + stateless workers + Jobs + bounded streams
registry-neutral package graphs + immutable locks + byte verification
small CLI porcelain + independently versioned packages and applications

General-purpose channels and shared mutable state are not public language

primitives. Native resources remain opaque handles. Optional frameworks,

database integrations, UI hosts, and application policy belong in packages.

Completed implementation through v0.99.363a

  • Native desktop/mobile structural targets, target archive provenance, Windows

runtime evidence, source-handoff safety, learning/docs surfaces, reflection,

restricted eval, actors/workers/Jobs/streams, and the fixed-point HIR/MIR/direct

LLVM compiler remain intact from earlier slices.

  • v0.99.360a compiled the package-index application through the ordinary

LLVM-only source cache and expanded strict closure to 66/66.

  • v0.99.361a added ordinary recursive scalar calls while preserving proven

`tail.self`/LLVM `musttail`; strict closure is 67/67.

  • v0.99.362a added complete package graph resolution, deterministic

highest-compatible backtracking, lock v2, exact offline restore, complete

installed-tree verification, online audit, command registration, reusable

schema validation, and observable isolated source handoffs.

  • v0.99.363a rewrote all maintained 1BRC applications with mapped bytes,

ordinary immutable objects, functions, loops, and actor-owned partials;

removed the specialized mutable byte-keyed table surface from every compiler

and runtime lane; and closed generalized division, parsed-JSON, mixed-value

stringification, and nested-receiver facts through HIR, MIR, and direct LLVM.

Immediate alpha slices

1. **v0.99.364a — CamelCase modules.** Rename every public stdlib module and all

first-party qualified references in one breaking alpha migration. Mechanically

reject lowercase public qualifiers from maintained source rather than keeping

compatibility aliases.

2. **v0.99.365a — Time/Duration coherence.** Store Duration in nanoseconds and

align explicit constructors/converters with clocks, timeout, scheduler,

process, stream, and timer APIs across all compiler lanes.

3. **v0.99.366a — independent package migrations.** Apply package, namespace,

and time contracts to every supplied `hey_*` repository. Begin with the seven

application-ecosystem packages only when their immutable source releases,

manifests, and callable documentation are available; do not infer APIs from

package names or public-page summaries.

4. **v0.99.367a — tvOS and party quiz.** Reuse the native iOS object/application

path for tvOS. Add honest structural/native receipts and build a package-owned

living-room quiz vertical slice.

5. **v0.99.368a — Google TV/Android TV.** Extend the Android NDK/JNI path with

Android TV launcher, focus/remote behavior, emulator/device receipts, and the

same package-owned quiz protocol. Cast remains an optional later package or

application, not Android TV native evidence.

6. **v0.99.369a — receipts and safe parallelism.** Finish target-host receipts,

retain timed continue-all release verification, and parallelize only gates

with isolated build/cache/output paths.

Detailed acceptance criteria are in

`docs/agents/version-slices-v0.99.360a-v0.99.369a.md`.

Television application boundary

TV application
  |-- native TV question/display screen
  |-- QR code containing the session URL
  |-- package-owned HTTP/WebSocket or polling service
  |-- phone web application
  |-- session/question/answer persistence
  `-- scoring and lobby policy

The ownership split is deliberate:

  • **Hey core:** target triples, object emission, runtime ABI, lifecycle callbacks,

SDK inspection, capability discovery, and honest structural/native receipts.

  • **Packages:** native TV UI/lifecycle adapters, QR rendering, web-service

composition, configuration, persistence, and UI contracts.

  • **Application repository:** questions, lobby behavior, scoring, visual design,

deployment, credentials, and release cadence.

The seven named application packages are plausible owners, but implementation

must inspect their actual release manifests and documentation before binding the

demo to `hey_app_ecosystem`, `hey_app`, `hey_project`, `hey_config`, `hey_cli`,

`hey_data`, or `hey_ui_contract`.

Package ownership

Hey core owns project declarations, complete graph resolution, registry-neutral

indexes, release/archive/manifest/content verification, immutable locks, exact

restore, complete installed-tree verification, online audit, safe

filesystem/archive/process APIs, generic templates and schema validation,

collision-checked package command registration, and target/capability discovery.

Independent package tooling owns authoring workflow, documentation policy,

release generation, website/Git publication, credentials, and registry hosting

policy. Applications own dependency choice and product behavior.

Release verification

Fast iteration uses:

HEY_BUILD_JOBS=12 HEY_COMPILER_CONTRACT_JOBS=12 ./bin/hey verify --focused

The maintainer continue-all pass uses:

HEY_BUILD_JOBS=12 HEY_COMPILER_CONTRACT_JOBS=12 HEY_RELEASE_JOBS=12 ./VERIFY-HEY-RELEASE.sh

Optional documentation publication, application vendoring, and promotion remain

explicit. Every failure stays visible and logged; the verifier attempts all

scheduled gates and exits nonzero after the aggregate summary.

Beta gate

Beta still requires multiple signed promoted desktop archives, stable package

contracts and migration policy, blocking performance budgets over ordinary Hey

code, reproducible CI/SBOM/license evidence, truthful mobile/television receipts,

and at least two maintained applications.
