1BRC scale-confidence matrix

Hey has three different concepts that are easy to mix together:

1. **Native-C runtime proof**: the current real 1BRC path. This is the only path

that has proven the 1B-row file on JayTeeSF's M3.

2. **Self-hosted front door**: the self-hosted compiler entrypoint. The tracked

dual-suite is direct (`bridged_count=0`), but full parser/build/spec/REPL

parity still blocks promotion.

3. **LLVM backend**: selectable for a small scalar/string/control-flow/stdout

subset. It intentionally does not lower 1BRC-specific helper calls as

benchmark proof shortcuts.

Use `./bin/perf 1brc-scale` to run increasing 1BRC row counts across the

available implementation/front-end paths:

./bin/perf 1brc-scale \
  --rows "1000 10000 100000 1000000" \
  --workers "1 2 4 8" \
  --implementations "native-c self-hosted llvm" \
  --out build/1brc-scale

cat build/1brc-scale/scale.tsv

For a higher-confidence ramp, use larger rows:

./bin/perf 1brc-scale \
  --rows "1000000 10000000 100000000" \
  --workers "1 2 4 8" \
  --implementations "native-c self-hosted llvm" \
  --out build/m3-1brc-scale

Confidence labels:

| Rows | Confidence | Meaning |

|---:|---|---|

| 1k, 10k, 100k | warmup | Good for correctness and smoke testing only. |

| 1M | low | First useful throughput signal, still noisy. |

| 10M | medium | Useful trend signal, still a 100x projection to 1B. |

| 100M | high | First strong projection point, only 10x from 1B. |

| 1B | actual | The real proof. |

The matrix writes `scale.tsv` with these important columns:

| Column | Meaning |

|---|---|

| `implementation` | `native-c`, `self-hosted`, or `llvm`. |

| `compiler_mode` | For self-hosted, `direct` or `native_oracle_bridge`. |

| `runtime_ms` | Measured inside the compiled Hey app around `one_brc_summary(...)`. |

| `build_real_seconds` | Build/front-end wall time; informational only. |

| `process_run_real_seconds` | Shell process wall time; informational only. |

| `projected_runtime_1b_seconds` | Linear runtime-only projection from the internal measurement. |

| `status` | `ok`, `build_failed`, `run_failed`, or `unsupported`. |

Current expected backend behavior

`native-c` should run all 1BRC scale rows. That is the production proof path.

`self-hosted` should run the same rows through `bin/subcmds/hey-selfc`. The generated

1BRC scale source should now show `compiler_mode=direct` when the Hey-written

runtime-backed emitter lowers `one_brc_summary(...)`, key/index access, `div(...)`,

equality, and output concatenation directly. A `native_oracle_bridge` row means that

source still escaped through the Native-C C-emission oracle.

`llvm` is expected to report `status=unsupported` or `status=build_failed` for generated 1BRC scale sources today. That is correct and intentional: LLVM must not silently fall back to Native-C or recognize benchmark helper names as a shortcut. LLVM can become useful for 1BRC only after it lowers the general runtime calls, file/string parsing, data model, loops/functions, and concurrency primitives needed by ordinary Hey benchmark code.

Real full-file proof

The full proof remains:

./bin/perf 1brc-full \
  --data ../1brc/measurements.txt \
  --rows 1000000000 \
  --workers "8" \
  --no-progress \
  --out build/m3-1brc-proof-no-progress

This command does not generate, count, validate, or scan the huge file with shell

helpers. Runtime is measured inside the native Hey app.

Real-data prefix ramp:

./bin/perf 1brc-scale \
  --data ../1brc/measurements.txt \
  --rows "1000 10000 100000 1000000 10000000" \
  --workers "1 2 4 8" \
  --implementations "native-c llvm" \
  --out build/m3-1brc-scale-native-vs-llvm

When `--data` is supplied, the harness creates prefix samples with compiled Hey runtime plumbing. LLVM rows still report honest unsupported while the generated scale program uses `one_brc_summary(...)`; LLVM must eventually pass this through ordinary file/data-model/control-flow lowering rather than a benchmark-specific shortcut.
