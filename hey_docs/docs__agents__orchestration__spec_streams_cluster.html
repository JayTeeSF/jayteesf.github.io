Streams/net/http/io cluster — implementation spec

Emitter file: `~/dev/claude_01/stdlib/HeyLlvm.hey` (5410 lines). All 10 MIR files read in full from

`scratchpad/item5/survey/<name>.mir`. Every construct below was verified absent by grep (no call

site, only sometimes a `declare`).

Key dispatcher anchors (closest handlers for almost everything in this cluster):

  • `hey_llvm_boxed_expr_value` (L373-900) — RHS of `local X:boxed = <expr>`. All new facade-call

branches (Streams__*, Http.*, Net.*, Web.*, Jobs.define, Files.open, method chains) get a

sibling `if hey_llvm_starts_with?(clean, '...')` branch here, following the existing pattern

used for `.stop()` (L850), `.status().` -style method calls, and single-dot `name.field`

(L874-883, i.e. absolute L373+501..510).

  • `hey_llvm_str_part_or_access_value` (L1362-1511) — resolves one `+`-joined string part; same

single-dot-only limitation as the boxed dispatcher (L1498-1510).

  • `hey_llvm_int_expr_value` (L1882-1978) — int-returning builtins/facades; `Text.parse_int(arr[i])`

is the existing model (L1884-1901).

  • Statement dispatcher inside `hey_llvm_emit_text_value` (`local `/`store ` branch L4339-4560+,

`say boxed` L4301-4337, `write ` L3870-3935, `sleep i64` L3824-3837 — this generic statement

block already applies inside plain (non-restricted) `fn` bodies, so int-typed fn bodies like

`SlowFirst` reuse it verbatim).

No example in this cluster has any `Streams__`, `Http.`, `Net.`, `Web.`, or `Jobs.` handling in the

emitter today (`grep -n '"Streams__\|"Http\.\|"Net\.\|"Web\.\|"Jobs\.'` → zero hits). Several

runtime entry points are already **declared** in the prelude but have **zero call sites** anywhere

in the emitter — i.e. the C runtime side is ready, only the Hey-side lowering is missing:

`hey_llvm_streams_{from,unordered,resequence,collect,map,filter,through,each}_value`,

`hey_llvm_join_value_string`, `hey_llvm_text_value_string`, `hey_llvm_c10k_ping_value`,

`hey_llvm_file_open_value`, `hey_llvm_stream_{read_line,write_all,close}_value`,

`hey_llvm_result_{ok,error,value_value,is_ok_i64}`, `hey_llvm_jobs_define_worker_value`,

`hey_llvm_parse_fixed_{i64,value_i64}`, `hey_llvm_find_sequence_value_i64`,

`hey_llvm_web_dispatch_with_handler_value`, `hey_llvm_web_serve_with_handler_value`,

`hey_llvm_callable_named_value`/`callable_{kind,arity,name_cstr,bound_count}_i64`.

`Http.parse_request`, `Http.header`, `Http.route_match`, `Http.response`, `Http.response_text`,

`Web.routes`, `Web.get`, `Web.text`, `Net.event_loop` have **no declare at all** — the wire shape

is unknown and must come from a native pin.

---

1. stream_resequencing (HAS native pin: `p18_stream_resequencing.ll`)

MIR (`survey/stream_resequencing.mir`), in order:

1. `fn SlowFirst(value:i64) -> i64` with body `sleep i64 (5 - value) * 20; ret i64 value` —

**already supported**: `sleep i64 <chain>` (L3824) + `hey_llvm_int_chain_mod_value` (L1808)

matches `(5 - value) * 20` exactly (paren-group times int literal). No work needed here in

isolation, but this fn is also referenced *by name as a value* below, which is unsupported.

2. `Streams__unordered([1, 2, 3, 4])` — new handler in `hey_llvm_boxed_expr_value`. Native shape

(p18 L667): `call ptr @hey_llvm_streams_unordered_value(ptr %array)` where `%array` is the

already-built json array temp. Declare exists, no call site.

3. `SlowFirst` used as a bare callable value (2nd arg of `Streams__map`) — **new handler needed,

highest leverage of the whole cluster.** Native shape (p18 L668, L581-627):

  • `%tN = call ptr @hey_llvm_callable_named_value(ptr @.str.<"SlowFirst">, i64 <arity>)`
  • a synthesized dispatch trampoline `define ptr @hey_program_call_callable_<arity>(ptr %callable, ptr %arg0[, ...argN]) {...}` emitted once per distinct arity actually used, with one `callable.names.N` string-compare arm per distinct named fn/worker referenced at that arity (checks `callable_kind_i64`==1, `callable_arity_i64`==N, `strcmp` against the fn's name, `callable_bound_count_i64`==0, converts args via `hey_llvm_value_to_i64`/raw ptr per param kind, calls `@hey_fn_<Name>`, wraps result via `hey_llvm_int_value`/`hey_llvm_nil_value` on failure paths).
  • `hey_program_spawn_actor_<arity>` boilerplate is *also* emitted alongside (p18 L629-651) even

though this program spawns no actors — appears to be paired scaffolding per arity, not

conditional on actual actor use. Confirm via native pins whether it's unconditional or only

appears when `spawn` co-occurs with callables (only one data point so far).

4. `Streams__map(<stream>, SlowFirst, 4)` — new handler. Native shape (p18 L669):

`call ptr @hey_llvm_streams_map_value(ptr %stream, ptr %callable, i64 4, ptr @hey_program_call_callable_1)`.

The literal `4` (3rd MIR arg) passes straight through as `i64`.

5. `Streams__collect(<stream>)` — new handler, trivial: `call ptr @hey_llvm_streams_collect_value(ptr %stream)`.

6. `join(completion_order, ",")` inside `say str` — new handler in `hey_llvm_str_expr_value` /

`hey_llvm_str_part_or_access_value`. Native shape (p18 L673): `call ptr @hey_llvm_join_value_string(ptr %array, ptr @.str.<",">)`, result fed straight to `puts`.

7. `Streams__resequence(Streams__map(...))` — new handler, trivial: `call ptr @hey_llvm_streams_resequence_value(ptr %stream)`. Composes with #3/#4 (nested call parsing — `hey_llvm_boxed_expr_value` must recurse into a facade-call argument that is itself a facade call; currently only array/object literals recurse (L433-446, L543-556), calls don't).

2. tour_06_streams (needs native pin)

MIR: `fn double(i64)->i64`, `fn greater_than_four(i64)->bool`, then

`Streams__collect(Streams__filter(Streams__map([1,2,3,4], double, 2), greater_than_four, 2))`,

`say str join(values, ",")`.

  • Same array/callable/map/collect/join constructs as #1, reusing the same new handlers.
  • **New variant**: callable trampoline for `i64 -> bool` return (`greater_than_four`) feeding

`Streams__filter` (declare `hey_llvm_streams_filter_value(ptr, ptr, i64, ptr)`, same 4-arg shape

as map). No native pin exists for a `bool`-returning callable — the SlowFirst-only pin only

proves the `i64->i64` variant's result-wrap (`hey_llvm_int_value`); a bool-returning callable

presumably wraps via a bool/int value constructor instead — **needs native pin to confirm.**

  • Nested `Streams__filter(Streams__map(...))` — same nested-facade-call recursion gap as #1.

3. parallel_streams (needs native pin)

MIR: `fn double`, `fn over_twenty(i64)->bool`, `fn observe(boxed)->boxed`,

`worker AddTen(value:boxed)->boxed { ret boxed value + 10 }`, then in main:

`Jobs.define("ExampleStreamAdditions", AddTen, count: 2, capacity: 4)`,

`Streams__through(Streams__filter(Streams__map([5,10,20,30], double, 2), over_twenty, 2), Additions)`,

`say str join(values, ",")`, `Streams__each(values, observe, 2)`, `Additions.stop()`.

  • `worker AddTen(...)` header — **already supported** generically (L3446-3447 mirrors `fn `).
  • `Jobs.define("name", AddTen, count: 2, capacity: 4)` — **new handler**, sibling to the existing

`workers(Fn, count:, capacity:)` sugar branch (L4526-4550). That branch is the closest template:

parse elements via `hey_llvm_split_elements_value`, validate `count:`/`capacity:` labels, look up

`AddTen` in `workers` registry, `put(locals, name, bid + ':boxed')`. Difference here: an explicit

string-literal name arg precedes the fn, and the runtime call is far more complex —

`hey_llvm_jobs_define_worker_value(ptr,ptr,ptr,i64×11,ptr,i64,ptr,ptr,i64)` (19 params, declared,

0 call sites) vs. the simple 4-arg `hey_llvm_workers_create_value` used by the `workers(...)`

sugar. **Needs native pin** — argument order/meaning (retry/backoff/result_capacity/etc. flags)

cannot be inferred from the declare signature alone.

  • `Streams__through(<stream>, Additions)` — new handler, 2-arg, no callable/trampoline (2nd arg is

a boxed local, loaded via `.obj` like any boxed operand): `call ptr @hey_llvm_streams_through_value(ptr %stream, ptr %pool)`.

  • `Streams__each(values, observe, 2)` — new handler; same map/filter shape

(`hey_llvm_streams_each_value(ptr, ptr, i64, ptr)`) but callable trampoline variant is

`boxed -> boxed` (no `value_to_i64`/`int_value` wrap — likely a straight pass-through call to

`@hey_fn_observe(ptr)`). **Needs native pin** to confirm (no boxed-typed callable pin exists yet).

  • `say boxed Streams__each(...)` — the each call's return value said directly; depends on

`hey_llvm_boxed_expr_value` support existing before `say boxed` can reuse it (L4326 already falls

back to `hey_llvm_boxed_expr_value` for non-trivial payloads, so this composes for free once #4 exists).

  • `Additions.stop()` — **already supported** (L850-856, `hey_llvm_workers_stop_value`) once

`Additions` is registered as a `:boxed` local by the new `Jobs.define` handler above.

4. public_evented_io (needs native pin)

MIR: `Net.event_loop({backend: "poll"})`, `Net.c10k(5)`, then `say boxed loop.kind` /

`.backend` / `result.kind` / `.completed` / `.ok`.

  • `Net.event_loop({backend: "poll"})` — **new handler, no declare exists at all.** Return value

needs `.kind`/`.backend` fields — could be a plain JSON object built at compile time (no runtime

call) if `Net.event_loop` is actually a pure/static descriptor constructor, or a real runtime

call with a name not yet in the prelude. **Needs native pin** — cannot tell without it.

  • `Net.c10k(5)` — new handler. Likely maps to the already-declared `hey_llvm_c10k_ping_value(i64)`

(same runtime primitive as example #5's bare `c10k_ping(5)`) — but confirm arg/prefix handling

via native pin since the same underlying builtin is spelled two different ways across examples.

  • `say boxed loop.kind` etc. — **already supported**, single-dot boxed field access

(`hey_llvm_boxed_expr_value` L874-883). No work needed once `loop`/`result` are bound as `:boxed`.

5. evented_io_c10k (needs native pin)

MIR: `local b1 result:boxed = c10k_ping(5)`, then 5× `say boxed result.<field>`.

  • `c10k_ping(5)` (bare call, no facade prefix) — new handler in `hey_llvm_boxed_expr_value`,

mapping to declared-but-uncalled `hey_llvm_c10k_ping_value(i64)`: `call ptr @hey_llvm_c10k_ping_value(i64 5)`.

Simplest example in the cluster — good first native pin to cut (single new call, no callables,

no chains). Confirm the literal `5` passes straight through as `i64` (likely, matches `Streams__map`'s literal-passthrough pattern).

  • `say boxed result.kind/.backend/.clients/.completed/.ok` — **already supported** (single-dot).

6. http_request_response (needs native pin)

MIR: `Http.parse_request(raw)`, `req.method + " " + req.path + " from " + Http.header(req, "Host")`,

`Http.route_match("/posts/:id", req.path)`, `"post id: " + hit.params.id`,

`Http.response(200, {"content-type": "text/html"}, "<h1>hey</h1>")`,

`find(Http.response_text(resp), "HTTP/1.1 200 OK") == 0`.

  • `Http.parse_request(raw)` — **new handler, no declare exists.** Needs native pin for the runtime

symbol name and whether `raw` (a `str` local, itself built by `+`-concatenation of `\r\n`

literals) needs conversion before the call.

  • `req.method + " " + req.path + " from " + Http.header(req, "Host")` — string left-fold

(already-supported mechanism) over parts, but two of the parts are new: single-dot boxed field

read as string is already fine (`req.method`, `req.path`); `Http.header(req, "Host")` is a new

2-arg call needing a new branch in `hey_llvm_str_part_or_access_value` (no declare found —

needs native pin).

  • `Http.route_match("/posts/:id", req.path)` — new handler, no declare, needs native pin.
  • `hit.params.id` — **chained two-level boxed field access, new handler needed.** The existing

single-dot branches (`hey_llvm_boxed_expr_value` L874-883, `hey_llvm_str_part_or_access_value`

L1498-1510) both explicitly reject any 2nd dot (`hey_llvm_c_name_value(dot_key) == dot_key` fails

once `dot_key` contains a `.`). Generalize both branches to loop over `.`-separated segments,

emitting one `hey_llvm_json_get_key_value` per segment before the final

`hey_llvm_value_to_raw_string_cstr` (str context) — moderate leverage, also needed by

tour_08_web's `request.params.name`.

  • `Http.response(200, {...}, "<h1>hey</h1>")` — new handler, mixed int/object/string literal args,

no declare, needs native pin.

  • `find(Http.response_text(resp), "HTTP/1.1 200 OK") == 0` — two new pieces: `Http.response_text(resp)`

(new handler, no declare, needs native pin) and bare `find(str, str)` as a program-level builtin

(new handler in `hey_llvm_int_expr_value`, likely `hey_llvm_find_sequence_value_i64` with a

0/len(haystack) window — needs native pin to fix the exact 4-arg call convention). This `find(...)`

builtin is also needed by #2's `Text.find` (see below) — shared leverage.

7. tour_08_web (needs native pin)

MIR: `fn hello(request:boxed)->boxed { ret boxed Web.text("Hello, " + request.params.name) }`,

`Web.routes([Web.get("/hello/:name", hello)])`, `Web.dispatch(routes, {method:..., path:..., headers:{}, body:""})`,

`say boxed response.status` / `.body`.

  • `request.params.name` — same chained-dot generalization as #6 (`hit.params.id`), string context.
  • `Web.text("Hello, " + request.params.name)` — new handler, `ret boxed <call>` path (see

`hey_llvm_boxed_expr_value` reuse via the `ret boxed` statement branch, L4757+/4897+/4921+

region for `unsupported return expression`) — no declare exists, needs native pin.

  • `hello` referenced by name inside `Web.get("/hello/:name", hello)` — same callable-value

machinery as #1/#2/#3 (`hey_llvm_callable_named_value`), `boxed -> boxed` arity-1 variant like

`observe`. `Web.get(path, fn)` itself — new handler building a route descriptor object

(probably `{path: ..., handler: <callable>}` via existing json-object-literal machinery once the

callable value exists), no declare, needs native pin.

  • `Web.routes([...])` — new handler wrapping an array of route descriptors, no declare found

(only `web_dispatch_with_handler_value`/`web_serve_with_handler_value` are declared) — needs

native pin to see whether `Web.routes` is a real runtime call or purely array-literal passthrough.

  • `Web.dispatch(routes, {method:.., path:.., headers:{}, body:""})` — new handler. Declared

`hey_llvm_web_dispatch_with_handler_value(ptr, ptr, ptr)` strongly suggests a 3rd `ptr` trampoline

arg reusing the same callable-dispatch infra as `Streams__map`/`Streams__filter` (dispatching to

whichever route's `hello`-like handler matched) — needs native pin to confirm.

  • `response.status`, `.body` — already-supported single-dot access, free once `response` is bound.

8. stateful_file_io (needs native pin) / 9. tour_07_io_and_results (needs native pin)

Near-identical MIR shape (tour_07 is the smaller/simpler sibling — do it first):

`Files.open(path, "write").value`, `writer.write_all(content)`, `writer.close()`,

`Files.open(path, "read").value`, `reader.read_line(N)[.value.data]`, `text(<bytes>, "utf8")`,

`reader.close().ok`.

  • `Files.open(path, "write")` — new handler, maps to declared-but-uncalled

`hey_llvm_file_open_value(ptr, ptr)`. Returns a Result; `.value` unwrap is the next problem.

  • `.value` on a Result — **new handler, zero call sites anywhere for any of

`hey_llvm_result_{ok,error,value_value,is_ok_i64,value_or_value}`.** Ambiguous whether `.value`

should route through the generic single-dot json-get path (treating Result as a plain object with

a "value" key) or through the dedicated `hey_llvm_result_value_value(ptr)` call — **needs native

pin**, this is the riskiest guess in the whole cluster.

  • `writer.write_all(content)` — new handler, `content` is a `str` local; maps to declared-but-uncalled

`hey_llvm_stream_write_all_value(ptr, ptr)`. Needs native pin for whether `content` (str) is

passed as a raw cstr or first wrapped via `hey_llvm_string_value`.

  • `writer.close()` — new handler, `hey_llvm_stream_close_value(ptr)` (declared, uncalled).
  • `reader.read_line(64)` / `reader.read_line(128)` — new handler with an int-literal arg, maps to

declared-but-uncalled `hey_llvm_stream_read_line_value(ptr, ptr-or-i64?)` — signature is

`(ptr, ptr)` per the declare, so the int arg `64` likely gets boxed via `hey_llvm_int_value`

first — **needs native pin** to confirm.

  • `.value.data` chain (stateful_file_io only; tour_07 just uses `line.value` directly without

`.data`) — same chained-dot generalization as #6/#7, compounded with the Result-unwrap ambiguity

above. tour_07's simpler `line.value` (single dot, but on a Result not a plain object) is a good

first target to nail the Result-unwrap shape before tackling stateful_file_io's 2-level chain.

  • `text(<bytes>, "utf8")` builtin — new handler in the str dispatcher, maps to declared-but-uncalled

`hey_llvm_text_value_string(ptr, ptr)`. Shared with nothing else in this cluster but cheap once

the arg (a chained field-access result) is available.

  • `reader.close().ok` — method-call-then-field-access on the *result* of a call, not a local —

a new shape entirely (existing dot-access branches all key off `hey_llvm_local_typed_value`,

i.e. a bound local name, not an arbitrary sub-expression). Needs either a general

"evaluate boxed expr then apply field access to the resulting ptr" combinator, or a dedicated

`.close().<field>` special case mirroring the existing `.status().<field>` branch (L1483-1497 /

its boxed-context twin) — **needs native pin** to see which shape native actually emits.

10. streaming_file_and_parse_primitives (needs native pin)

MIR: `write "build/v046_lines.txt", content, "w"`, `Files.lines(path)`, `say i64 len(rows)`,

`say boxed rows[0/1]`, `Text.split("station;12.3", ";")`, `say i64 len(parts)`,

`Text.parse_int(rows[2])`, `Text.parse_fixed(parts[1], 1)`, `Text.parse_fixed("-1.5", 1)`,

`Text.find("abcdef", "bc")`.

  • `write "build/v046_lines.txt", content, "w"` — **closest existing handler, smallest gap in the

whole cluster.** The 3-literal-arg `write "path","content","mode"` branch already exists

(L3889-3921, → `hey_llvm_write_path_mode_string`) but hard-requires all three args to be

single-quoted literals (L3894-3899 checks `hey_llvm_literal_render_value` on each). Extend that

branch: when the middle arg is a bound `str` local instead of a literal, load it

(`load ptr, ptr %<name>.addr`) and pass the loaded ptr instead of a `@.str.N` global. No new

runtime declare needed (`hey_llvm_write_path_mode_string` presumably already takes cstr ptrs).

  • `Files.lines(path)` — new handler, `path` is a `str` local (not literal, unlike the existing

`read("literal")` special case at L1405-1412). Likely maps to declared-but-uncalled

`hey_llvm_read_lines_path_value(ptr)` or `hey_llvm_read_lines_value(ptr)` — needs native pin to

pick the right one and its arg convention (path string vs already-read content).

  • `len(rows)` / `len(parts)` on a boxed array — check emitter support for `len(<boxed local>)`

separately from `len(<str local>)` (`.length` is already handled for `str`, L1414-1422); `len(...)`

as a bare call on boxed is a distinct token shape — likely trivial (`hey_llvm_json_length_i64`-style

call) but unverified; grep found no existing `len(` builtin call-site either — **new handler**.

  • `rows[0]`, `rows[1]` — boxed array index read in `say boxed` context; the boxed-expr dispatcher

has a bracket-index branch (`hey_llvm_boxed_expr_value` absolute ~L887-889, `brack > 0` check)

— likely **already supported**, low risk, worth trying without a native pin first.

  • `Text.split("station;12.3", ";")` — new handler; maps to declared `hey_llvm_split_value(ptr, ptr)`.

Both args here are literals, should be a straightforward wire-up, low risk without pin.
