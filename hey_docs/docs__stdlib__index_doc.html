Standard Library Direction

The standard library is Hey-lang porcelain built on stable language/runtime plumbing.

Core rule:

Syntax is scarce.
Plumbing is stable.
Porcelain belongs in libraries.

The standard library should provide useful workflows without preventing third-party alternatives.

Current starter modules

stdlib/json.hey
stdlib/jobs.hey
stdlib/files.hey
stdlib/text.hey
stdlib/time.hey
stdlib/static_files.hey
stdlib/web.hey
stdlib/vendor.hey
stdlib/hey_expressions.hey

`stdlib/json.hey` exposes:

json.decode(text)
json.encode(value)

These delegate to compatibility runtime helpers today. They establish the boundary: JSON is a codec over core immutable values, not the language's permanent data model.

`stdlib/jobs.hey` exposes the starter jobs/queues facade over actor plumbing. See jobs.

`stdlib/files.hey` exposes `files.lines(source)` and `files.read_text(source)`.

`stdlib/text.hey` exposes `text.split`, `text.find`, `text.index_of`, `text.parse_int`, and `text.parse_fixed`.

`stdlib/static_files.hey` exposes conservative path-policy helpers for serving files under one configured root. See static files.

`stdlib/web.hey` exposes the official request/router/response web porcelain. See web.

`stdlib/vendor.hey` exposes the Hey distribution vendoring workflow behind `./bin/hey-vendor`. See vendor.

`stdlib/hey_expressions.hey` exposes friendly first-run/easter-egg/tagline helpers plus a maintained reference catalogue for `./bin/hey expressions`.

Future stdlib porcelain

json / jsonlines codecs
jobs and queue helpers
collection helpers
stream processors
file/socket/process helpers
secure static file serving
workflow chains/rings
Timer and counter modules

None of these should require new syntax unless a missing plumbing primitive blocks library implementation.

Timer and counters

Timer and counter modules should be stdlib porcelain over compiler-recognized runtime intrinsics, not new public numeric types. See Timer and counter semantic intrinsics.
