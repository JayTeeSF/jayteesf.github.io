Tooling-facade cluster spec (10 examples)

All 10 examples have no native pin (`p*_<name>.ll`) in scratchpad. All need:

./bin/heyc build-mir --backend llvm <scratchpad>/survey/<name>.mir -o <scratchpad>/pin_<name>

for: hey_ast_manifest, hey_bootstrap_gate, hey_parser_tooling, hey_semantics_tooling,

hey_syntax_tooling, hey_tiny_c_emit, ehy_templating, markdown_tooling, allocation_stats,

actor_send_move_and_mailbox_slab.

Line refs are to `/Users/jthomas/dev/claude_01/stdlib/HeyLlvm.hey` as of this survey.

Shared infrastructure gaps (fix once, unblocks many examples)

G1. Module-facade call shape: `Module.method(args...) -> boxed`

No dispatch anywhere recognizes a dotted-module call (`HeyAst.manifest(...)`,

`HeyBootstrap.gate(...)`, `HeyParser.parse_text(...)`, `HeySyntax.line_kind(...)`,

`HeySyntax.check_file(...)`, `HeyC.emit_text(...)`, `Ehy.render_file_with(...)`,

`Ehy.result_with(...)`, `Markdown.check(...)`). `hey_llvm_call_expr_value` (line 1613)

only resolves calls to program-defined `fns` (checked via `has?(fns, name)`); it never

sees a `.` in the head. `hey_llvm_boxed_expr_value` (line 373) has no case for a dotted

head either — falls through to its final `return {ok:false...}` at line 897.

  • **Precedent to copy**: `Text.parse_int(...)` (line 1884), `Time.now_ns()` /

`Time.elapsed_ns(...)` / `Time.ns_to_ms(...)` (lines 1902-1924), `Cli.args()` (line

787) are each matched by exact/prefix string match inside `hey_llvm_int_expr_value`

or `hey_llvm_boxed_expr_value` and lowered to ONE declared `hey_llvm_*` runtime call.

Each new facade needs the same treatment: a `hey_llvm_starts_with?(clean,

'HeyAst.manifest(')` (etc.) case in `hey_llvm_boxed_expr_value`, emitting a call to a

new `declare ptr @hey_llvm_hey_ast_manifest_value(ptr)`-style runtime symbol.

  • **First step for every one of these 8 examples**: get a native pin to learn (a) the

exact runtime symbol name the C emitter calls, (b) argument marshalling (str literal

vs boxed object arg), (c) whether the object-literal arg (`{name: "Ada"}` in

`Ehy.render_file_with`) is built via the existing `hey_llvm_boxed_expr_value` object

path (line 480) and passed as a second `ptr` arg, or via a different flattening.

  • Extend point: `hey_llvm_boxed_expr_value` line ~373-390 (add cases before the

array/object-literal branches) or a small new dispatch fn called from there.

G2. `and` / `or` boolean combinators — entirely absent

`grep -n "' or '"` / `"' and '"` over HeyLlvm.hey returns nothing. `hey_llvm_cmp_expr_value`

(line 1178), which is the ONLY function backing `if bool`, `while bool`, `until bool`

(dispatched at line 3731), `ret bool` (line 4919), and `say/print bool`, splits on a

single top-level comparison operator and has no combinator handling at all. This is a

**very** high-leverage gap: `hey_semantics_tooling.mir` alone uses `or`/`and` chains in

dozens of `if bool` conditions and its very first failure is:

ret bool (value >= "a" and value <= "z") or (value >= "A" and value <= "Z") or (value >= "0" and value <= "9") or value == "_" or value == "?" or value == "."

(hey_semantics_tooling.mir:14, blocker per hey_semantics_tooling.llerr).

  • Extend point: new function `hey_llvm_bool_expr_value` that top-level-splits `cond` on

` or ` then ` and ` (paren/quote-depth aware, same depth-tracking style already used

at line 1229-1255), recursing into `hey_llvm_cmp_expr_value` per atomic term (stripping

balanced outer parens), short-circuit via `select`/`br` chains or eager `and`/`or` i1

ops (native pin needed to confirm which — likely eager `or i1` / `and i1` chains given

the existing style of this emitter, not branchy short-circuit).

  • Callers to rewire: line 3731 (`if/while/until bool`), line 4919 (`ret bool`), and the

`say bool` path (search `'say bool '` handling — same family as `say str`/`say boxed`

at line 3758/4301).

  • **Recommended order**: land G2 before touching hey_semantics_tooling in depth — most

of that file's `if bool X or Y` and `ret bool ... and/or ...` lines depend on it.

G3. String-to-string (non-literal) comparison: `strLocal == strLocal`

`hey_llvm_cmp_expr_value`'s `seq_at` special case (line 1213) only fires when the RHS is

a **quoted literal** ending in `"`. Two `str` locals compared (`ch == quote`, seen

repeatedly in hey_semantics_tooling.mir, e.g. line 44/95/159/219) fall through to the

generic op-scan branch (line 1298-1311), which calls `hey_llvm_int_expr_value` on both

sides — that rejects `str`-kind locals (only `i64`/`bfor` survive

`hey_llvm_int_operand_value`, line 1055-1075). Needs a new case in

`hey_llvm_cmp_expr_value`: when both sides resolve to `str`-kind locals, load both as

`ptr`, `call i32 @strcmp`, `icmp eq/ne i32 ..., 0` (mirrors the literal-RHS path at

line 1221-1223 but with both operands loaded rather than one interned global).

G4. Boxed value / boxed dot-field used as an int operand in a generic comparison

`hey_llvm_boxed_int_term_value` (line 1083) already implements exactly this coercion

(bare boxed local → `hey_llvm_value_to_i64`; `boxedLocal.field` →

`hey_llvm_json_get_key_i64`) but it is only ever called from the `ret` handler (line

4742) and one `!x.enqueue(...)` special case (line 1265) — **not** from

`hey_llvm_cmp_expr_value`'s generic op branch or from `hey_llvm_int_operand_value`.

Blocks: `allocation_stats.mir` (`say bool gets >= 1` / `puts >= 1`, where `gets`/`puts`

are `:boxed` locals) and `actor_send_move_and_mailbox_slab.mir` (`stats.send_moves >=

2`, `stats.message_allocs >= 1`, `stats.message_returns >= 2`, direct boxed dot-field

compared to int literal).

  • Fix: in `hey_llvm_int_operand_value` (line 1055) or in `hey_llvm_cmp_expr_value`'s

generic-op lhs/rhs evaluation (line 1300/1304), fall back to

`hey_llvm_boxed_int_term_value` when the plain int path fails and the token names a

`boxed`-kind local or `name.field` where `name` is boxed.

G5. Chained/nested dot access (`a.b.c`, `a.b[0]`, `a.b[0].c`)

Every dot-access special case (`hey_llvm_boxed_expr_value` line 874 bare dot, line 886

bracket-index, `hey_llvm_cmp_expr_value` line 1180 `.length`,

`hey_llvm_str_part_or_access_value` line 1424 `.has?(`) requires the target **before**

the first `.`/`[` to be a *bare* local name (`hey_llvm_local_typed_value(locals,

slice(clean, 0, dot))`). None support a chain where the first segment is itself a dot

or bracket access:

  • `hey_ast_manifest.mir:15-16`: `manifest.totals.has?("program")`,

`manifest.forms.length > 0`

  • `hey_parser_tooling.mir:15,18-19`: `tree.forms[0]`, `form.body[0].kind`,

`form.body[1].kind`

  • `markdown_tooling.mir:17`: `control boxed for b2 issue in result.issues` (for-loop

target must also be bare per line 3674)

Fix: generalize by resolving the head as a temp `%tN` pointer produced by recursively

evaluating everything up to the last `.`/`[` segment through `hey_llvm_boxed_expr_value`

itself (i.e. make the head-lookup try `hey_llvm_boxed_expr_value` on the prefix before

falling back to `hey_llvm_local_typed_value`), rather than requiring a bare local.

G6. `print boxed` / `warn boxed` unimplemented (runtime symbols already declared)

  • `print boxed <expr>` is **unconditionally rejected** at line 4302-4303 (`if

hey_llvm_starts_with?(line, 'print boxed ') return {ok:false, error:'unsupported MIR

operation...'}` — checked and rejected before even looking at payload). Blocks

`ehy_templating.mir:14,16` (`print boxed file_view`, `print boxed

Ehy.result_with(...)`).

  • `warn boxed <expr>` has no case at all (only `warn str`, line 3839). Blocks

`markdown_tooling.mir:18` (`warn boxed issue`, where `issue` is a `bfor` loop var from

`control boxed for`).

  • Both runtime symbols already exist: `declare void @hey_llvm_print_value(ptr)` (line

2448) and `declare void @hey_llvm_warn_value(ptr)` (line 2449) — currently unused.

Fix is pure MIR-dispatch: copy the `say boxed` block (line 4301-4337) verbatim for

`print boxed`/`warn boxed`, swapping the runtime call to `hey_llvm_print_value` /

`hey_llvm_warn_value` and removing the `print boxed` early-reject at 4302-4303.

G7. `reset_allocations()` / `allocations()` bare builtin calls

Runtime symbols already declared and unused: `declare ptr

@hey_llvm_allocations_value()` and `declare void @hey_llvm_allocations_reset()` (lines

2402-2403). No MIR dispatch calls them. Blocks `allocation_stats.mir:13,17` and

`actor_send_move_and_mailbox_slab.mir:19,26` (`local bN ignored:boxed =

reset_allocations()`, `local bN stats:boxed = allocations()`).

  • Extend point: `hey_llvm_boxed_expr_value` (line 373) — add `if clean ==

'reset_allocations()'` → emit `call void @hey_llvm_allocations_reset()` then produce

a nil/ignorable boxed result (native pin needed: does it emit `hey_llvm_nil_value()`

or reuse a null ptr constant for the `ignored` binding?); `if clean ==

'allocations()'` → `%t = call ptr @hey_llvm_allocations_value()`.

G8. `find(strExpr, "literal") -> i64`

`declare i64 @hey_llvm_string_find(ptr, ptr)` exists (line 2026) and is unused. No case

in `hey_llvm_int_expr_value`/`hey_llvm_operand_or_call_value` recognizes bare

`find(a, b)`. Blocks `hey_tiny_c_emit.mir:16-17`: `find(emitted.c, "long answer = 42")

>= 0` (first arg is a boxed dot-field, needs G5 or a direct str-cstr-of-boxed-field

case; second arg is a literal → intern).

  • Extend point: `hey_llvm_operand_or_call_value` (line 1724), add a `find(` case

alongside the existing `len(`/`div(` cases (line 1736/1752), lowering arg0 through

`hey_llvm_str_part_or_access_value` (handles the boxed-field → raw-cstr case already,

line 1498-1510) and arg1 through literal interning, then `call i64

@hey_llvm_string_find(ptr, ptr)`.

G9. `Files.exists?`, `Files.join`, `Files.dirname` (secondary, deep in hey_semantics_tooling.mir)

Not supported anywhere (grep for `'Files.'` outside the import line returns nothing).

Used repeatedly past line 497 of hey_semantics_tooling.mir

(`hey_semantics_package_import_path_value`, `hey_semantics_import_path_value`,

`hey_semantics_check_imports_value`). Not reachable until G2/G3 above are fixed (file

fails at line 14 today), but flag for the same implementer since it's the same file.

Per-example construct list (file order) and blockers

hey_ast_manifest.mir (17 lines)

1. `HeyAst.manifest("examples")` boxed facade call — **G1**. Native pin required.

2. `manifest.totals.has?("program")` chained dot+method — **G5** (extends

`hey_llvm_str_part_or_access_value` `.has?(` case, line 1424, or a boxed-bool

equivalent reached via `say bool`).

3. `manifest.forms.length > 0` chained dot+`.length` compared to int — **G5** (extends

`hey_llvm_cmp_expr_value`'s `.length` case, line 1180, which today requires a bare

local before `.length`).

Order: G1 → G5.

hey_bootstrap_gate.mir (18 lines)

1. `HeyBootstrap.gate("examples")` boxed facade call — **G1**. Native pin required.

2. `say boxed gate.kind` / `.full_self_hosted` / `.full_self_host_target` /

`.native_oracle_required` — all single-level dot access, **already supported**

(`hey_llvm_boxed_expr_value` line 874-885, reached via the `say boxed` fallback at

line 4326). No further work once G1 lands.

hey_parser_tooling.mir (20 lines)

1. `local b1 source:str = "..."` — plain literal, already supported.

2. `HeyParser.parse_text("example", source)` boxed facade call, 2 args (literal + str

local) — **G1**. Native pin required.

3. `tree.forms[0]` — dot then bracket-index chain — **G5**.

4. `say boxed form.kind` — bare dot, already supported once `form` local exists.

5. `say i64 len(form.body)` — `len()` of a dot-accessed boxed field; `len(` case at

line 1752 requires a bare local (`hey_llvm_local_typed_value(locals, target_name)`)

— needs **G5** generalization too.

6. `form.body[0].kind` / `form.body[1].kind` — triple chain (dot, bracket, dot) —

**G5**.

Order: G1 → G5.

hey_syntax_tooling.mir (16 lines)

1. `say boxed HeySyntax.line_kind("program")` — facade call **inline inside `say

boxed`**, not via a `local` first — **G1**; once `hey_llvm_boxed_expr_value` gets the

facade-call case, this is reached automatically through the existing `say boxed`

fallback (line 4326), no separate `say`-side change needed.

2. `HeySyntax.check_file("examples/hello.hey")` boxed facade call — **G1**.

3. `say boxed report.ok` — bare dot, already supported.

Order: G1 only (this is the smallest example in the cluster — good first target once a

native pin exists).

hey_tiny_c_emit.mir (18 lines)

1. `HeyC.emit_text("example", source)` boxed facade call, 2 str args — **G1**.

2. `say boxed emitted.ok` — bare dot, already supported.

3. `find(emitted.c, "long answer = 42") >= 0` — **G8** + boxed-dot-field-as-str-arg

(`emitted.c`, handled by existing `hey_llvm_str_part_or_access_value` dot case, line

1498-1510, IF that function is what G8's arg0 evaluation reuses).

4. `find(emitted.c, "const char *name = \"Ada\"") >= 0` — same as above, note the

literal contains an escaped `"` — check `hey_llvm_literal_render_value` handles

embedded escaped quotes inside a `find(...)` literal argument correctly (it already

handles this generally for say/print literals; just confirm the arg-splitter used by

G8, likely `hey_llvm_split_elements_value`, doesn't mis-split on the embedded `"`).

Order: G1 → G8.

ehy_templating.mir (17 lines)

1. `Ehy.render_file_with("examples/templates/hello.ehy", {name: "Ada"})` boxed facade

call, args = (str literal, object literal) — **G1**. Native pin required; the

object-literal arg likely reuses the existing `{...}` construction path

(`hey_llvm_boxed_expr_value` line 480) then passes the resulting `ptr` as arg 2.

2. `print boxed file_view` — **G6**.

3. `local b2 inline_view:str = "..."` — plain multi-line string literal, already

supported (check embedded `\n` handling in `hey_llvm_literal_render_value` — likely

fine, same as other multi-line literals elsewhere in the suite).

4. `Ehy.result_with(inline_view, {name: "Ada", items: ["a", "b"]})` boxed facade call,

args = (str local, object literal with a nested array value) — **G1** (nested-array

object value already supported per line 543-555).

5. `print boxed Ehy.result_with(...)` — combines **G1** + **G6** (print-boxed

dispatch must route an unresolved-local payload through `hey_llvm_boxed_expr_value`,

same as `say boxed` does at line 4326).

Order: G6 → G1 (G6 is pure plumbing and cheap; G1 needs a native pin either way).

markdown_tooling.mir (22 lines)

1. `Markdown.check("docs")` boxed facade call — **G1**.

2. `if bool result.ok` — bare dot on a boxed local used as a bool condition. Check

`hey_llvm_cmp_expr_value`'s op_at<0 fallback (line 1288-1296): it calls

`hey_llvm_int_expr_value(clean,...)` then `icmp ne ..., 0` — for `result.ok` (dot

access, no int literal), this needs either **G4**-style boxed-dot int coercion or a

dedicated boxed-dot-as-bool case (load ptr, `hey_llvm_json_get_key_value`, then

`hey_llvm_value_to_i64` + `icmp ne 0`, OR a `hey_llvm_value_is_truthy_i64`-style

runtime call — native pin will show which).

3. `control boxed for b2 issue in result.issues` — for-loop target is a dot-access

chain, not bare — **G5** extension to the for-loop dispatch at line 3605-3708

(`target_entry = hey_llvm_local_typed_value(locals, for_parts[3])`, line 3674).

4. `warn boxed issue` — **G6**.

5. `fail str "docs broken"` — nested three levels deep (inside `if`'s `else`, inside

`control...for`'s body). The `fail str` case itself (line 3810) has no nesting

restriction — this should already work once everything above unblocks reaching it;

flag as risk only, not a known gap.

Order: G1 → (bool-dot coercion) → G5 (for-target) → G6 (warn boxed).

allocation_stats.mir (24 lines)

1. `reset_allocations()` — **G7**.

2. `{name:"Ada"}` object literal — already supported.

3. `{...doc, visits: 1}` single-spread + field — already supported (line 484-514).

4. `next.visits` bare dot — already supported.

5. `allocations()` — **G7**.

6. `say str type(stats)` — `type(` case already exists in

`hey_llvm_str_part_or_access_value` (line 1441-1481) for a bare boxed local; should

work once `stats` exists via G7.

7. `say bool stats.has?("xmalloc_calls")` — bare dot + `.has?(` in a **bool** context;

check `hey_llvm_cmp_expr_value` has no `.has?(` case at all today (only

`hey_llvm_str_part_or_access_value` does, for str contexts) — likely needs a new

`.has?(` branch added to `hey_llvm_cmp_expr_value` mirroring line 1424-1440's

`phas_` logic but returning i1 directly instead of a selected string.

8. `stats.json_gets + stats.json_get_key_fast` — **new**: top-level `boxed = boxedField

+ boxedField` addition (no existing top-level case; only the nested-in-object-literal

`vplus` pattern at line 557-573 exists). Needs a new case in

`hey_llvm_boxed_expr_value`: split on top-level ` + `, resolve each side via the G5

dot-chain resolver, `call ptr @hey_llvm_add_value(ptr, ptr)`.

9. `stats.json_puts + stats.json_put_key_fast` — same as #8.

10. `say bool gets >= 1` / `say bool puts >= 1` — boxed local vs int literal — **G4**.

Order: G7 → (new `.has?` bool case) → (new top-level boxed `+`) → G4.

actor_send_move_and_mailbox_slab.mir (30 lines)

1. `actor sink()` / `actor.receive msg` / `actor.exit` — **already supported**

(actor-fn scaffolding, receive at line 3060-3064, exit at line 3139/2974).

2. `spawn sink` (x2) — **already supported** (line 4422-4523).

3. `actor.send a, {kind: "first"}` / `actor.send b, {kind: "second"}` — **already

supported** (line 4160-4290, inline object-literal payload path).

4. `actor.join a` / `actor.join b` — **already supported** (line 4160-4229).

5. `reset_allocations()` — **G7** (same as allocation_stats.mir).

6. `allocations()` — **G7**.

7. `stats.send_moves >= 2`, `stats.message_allocs >= 1`, `stats.message_returns >= 2` —

bare-dot boxed field vs int literal, all three — **G4**.

Order: G7 → G4. This example is otherwise fully covered by existing actor

infrastructure — **second-smallest lift in the cluster after hey_syntax_tooling** once

G7/G4 land (G4 is shared with allocation_stats, so land allocation_stats first).

hey_semantics_tooling.mir (580 lines — largest, self-hosted lexer/scanner)

This is a from-source function-body implementation (11 `fn` definitions +

`HeySemanticBuiltins__validate_tree/_source`, `HeySemantics__analyze_text/_file`,

`entry main`), **not** a thin facade-call wrapper like the other 9. First (and

currently only reached) blocker:

  • Line 14: `ret bool (value >= "a" and value <= "z") or (value >= "A" and value <= "Z")

or (value >= "0" and value <= "9") or value == "_" or value == "?" or value == "."`

— **G2** (and/or combinators), compounded by string `>=`/`<=` comparisons against

literals (need to confirm `hey_llvm_cmp_expr_value`'s generic op-word path supports

`str >= "a"`/`str <= "z"` — currently `hey_llvm_cmp_word` (line 1107) maps the

operator fine, but the operand evaluation goes through `hey_llvm_int_expr_value`,

which is int-only; string relational comparison via `strcmp`-sign-check is a **new**

primitive, not yet present anywhere).
