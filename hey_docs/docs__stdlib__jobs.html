Jobs stdlib

`stdlib/jobs.hey` is the standard-library facade for asynchronous work. Jobs are ordinary actors underneath, but the goal is that programmers do not have to think about actors, supervisors, monitors, links, mailboxes, or restart plumbing for the common case. They should think: queue, worker, job, result.

That matters for background work and for tiny async operations. A short operation can still be modeled as a job when the developer wants queueing, supervision, retries, result lookup, cancellation, or durability.

Smallest useful async shape

Use defaults and let the jobs runtime own the supervised worker:

import 'stdlib:jobs'

actor import_worker
  receive job
  let completed = jobs.complete(job, {message: job['payload']['name'] + ' imported'})
end

actor import_sink
  receive msg
  says msg['result']['message']
  exit
end

program
  let runtime = jobs.start_default('import_worker')
  let receipt = jobs.perform_with_id(runtime, 'import-Ada', {name: 'Ada'})
  let sink = spawn import_sink
  let requested = jobs.result_to(runtime, receipt['id'], sink)
  join sink
  let stopped = jobs.stop(runtime)
end

Defaults: one worker, a single scheduler, local actor backend, no retry delay, status/result/dead-letter APIs, cancellation requests, and metrics. This example is intentionally REPL-pasteable.

Worker contract

Workers receive a `job_envelope` and explicitly acknowledge success or failure:

jobs.complete(job, result)
jobs.fail(job, reason)

The envelope includes `job_id`, `client_job_id`, `queue`, `attempt`, `retries`, `payload`, `runtime_pid`, and `worker_pid`.

Results and status

Hey actors are message-first, so query APIs send to a sink actor:

jobs.result_to(runtime, receipt['id'], sink)
jobs.dead_letters_to(runtime, sink)
jobs.status_to(runtime, sink)

`jobs.result_to` is await-style: if the result or dead-letter already exists, it sends it to the sink immediately; otherwise the queue stores the sink and replies when the job completes or dies. Correctness tests should synchronize with a sink + `join`, not sleep for an assumed number of milliseconds.

Retries and duration units

For human-scale jobs, milliseconds are fine:

let queue = jobs.queue({name: 'imports', retries: 2, backoff_ms: 10})

For low-level async work, prefer explicit duration values:

let queue = jobs.queue({
  name: 'imports/fast',
  retries: 2,
  backoff_delay: jobs.micros(500)
})

Available helpers:

jobs.nanos(500)
jobs.micros(250)
jobs.millis(10)
jobs.seconds(2)
jobs.duration(250, 'us')

Nanoseconds are smaller than microseconds. Duration values preserve `value_ns` and expose `effective_ms`; the current delayed-job timer actor is millisecond-resolution, so positive sub-millisecond scheduled delays are rounded up to one effective millisecond and do not run early. For ordinary job completion, prefer result sinks over timer sleeps so async code can complete as soon as the actor work is done.

Durable local disk backend

For durable local queues, use the write-ahead log backend:

let queue = jobs.queue({
  name: 'durable/imports',
  backend: jobs.disk('build/jobs/imports.wal'),
  retries: 2,
  backoff_delay: jobs.micros(500)
})

let runtime = jobs.start(queue, 'import_worker')
let receipt = jobs.perform_with_id(runtime, 'import-Ada', {name: 'Ada'})

The disk backend records enqueue, retry, complete, fail, cancel, and dead-letter events as JSON lines. Recover the log with:

let recovery = jobs.recover(queue)
says recovery['count']

`jobs.network(endpoint)` exists as the network backend descriptor shape. The transport adapter itself is a future runtime/backend slice.

Detailed production configuration

let queue = jobs.queue({
  name: 'production/imports',
  min_workers: 2,
  max_workers: 2,
  scheduler: 'round_robin',
  retries: 3,
  retry: jobs.exponential_backoff(jobs.micros(500), jobs.millis(20)),
  backend: jobs.disk('build/jobs/production-imports.wal'),
  dead_letter: 'production/imports/dead'
})

let runtime = jobs.start(queue, 'import_worker')

Caller-owned worker pools

Owned pools are preferred, but caller-owned pools remain supported:

let w1 = supervise import_worker max queue['max_restarts']
let runtime = jobs.start_pool(queue, [w1])

Compatibility helpers

These older helpers remain for simple examples and backward compatibility:

jobs.perform_later(queue, worker, payload)
jobs.perform_at(queue, worker, at, payload)

New production code should perform against a runtime, not directly against a raw worker pid.
