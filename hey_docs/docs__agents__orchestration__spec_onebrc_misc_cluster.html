Spec: one_brc + misc cluster (item 5)

Group: one_brc_single, one_brc_actors, one_brc_case, one_brc_scale, function_actor,

json_actor, jsonl_sort_mailinator_users, bytes_codecs, bytes_strings,

tour_02_functions, self_host_compile_pilot, self_hosted_supported_subset,

ergonomics_package_stream_app, ergonomics_porcelain_app. All fail-closed, all in

survey/results.tsv. HeyLlvm.hey = `~/dev/claude_01/stdlib/HeyLlvm.hey` (5410 lines).

HEADLINE FINDING: the generic stdlib-facade dispatcher

`p16_one_brc_single.ll` (the only native pin in this group) proves native does NOT

hand-roll a C runtime call per stdlib facade. Any `Module.function(args...)` call

that isn't one of the emitter's hard-coded intrinsics (`Time.now_ns()`,

`Text.parse_int`, `json()`, `monitor*`, `actor_info`, `file()`, `div()`, `len()`...)

compiles through one generic runtime entry point, already declared in the prelude

but never invoked by the walker:

declare ptr @hey_llvm_stdlib_json_call(ptr module_name, ptr fn_name, ptr args_array, i64 argc)

Confirmed shape (from `Files.write_text(target_path, result.to_json)` at

p16_one_brc_single.ll:855-878, and `Time.elapsed_ns`/`Time.ns_to_ms` used as nested

OBJECT-FIELD VALUES at .ll:1864-1875, 2169 area):

  • Two globals per call site, per-use interned: `@.str.N = "<Module>.hey"`,

`@.str.N+1 = "<Module>.function"` (fully-qualified dotted name).

  • Args array: `alloca [<argc> x ptr]`, each slot `getelementptr` + `store`, filled

left to right in source order with EVERY arg boxed first:

  • `str` local → `load .addr` then `call ptr @hey_llvm_string_value(ptr %cstr)`.
  • `i64` value → `call ptr @hey_llvm_int_value(i64 %v)`.
  • a field-access pseudo-property `.to_json` on a boxed local → `load .obj`,

`call ptr @hey_llvm_json_stringify_cstr(ptr %obj)`, then wrap that cstr with

`hey_llvm_string_value` too (so `x.to_json` evaluates as a `str`-boxed value).

  • (not observed here, but consistent with existing array/object-literal codegen)

boxed locals pass their loaded `.obj` ptr straight through, no rewrap.

  • Call: `%tN = call ptr @hey_llvm_stdlib_json_call(ptr @.str.M, ptr @.str.M+1, ptr %args, i64 argc)`.

Result is a boxed value usable like any other (get_key/put/len/etc.).

  • IMPORTANT: this is context-sensitive, not universal. `Time.elapsed_ns`/

`Time.ns_to_ms` used as a **plain int-expr** (`local x:i64 = Time.ns_to_ms(...)`,

or inside the paren-chain evaluator) still compile to the existing direct

`now_ns_i64`/`sdiv` short-circuit HeyLlvm.hey already has (int_expr_value,

line ~1902-1930) — confirmed unchanged in `p13_llvm_integer_hot_loop_timed.ll`

(no stdlib_json_call there). The generic dispatcher fires only where the walker's

boxed-value assembly code (object/array literal field values, statement-level

calls whose result is discarded, `ret boxed <call>`) doesn't already have a

specific intrinsic for the callee.

**Recommended new handler**: `hey_llvm_facade_call_value(expr, locals, fns, ...)`,

tried as a fallback inside `hey_llvm_boxed_expr_value` (~line 373) after the

existing intrinsics (`json(`, `monitor*`, `actor_info(`, `file(`) and before

failing closed; also called from the statement-level dispatcher (~line 3810 area,

alongside `fail str`) for void-context calls like `Files.write_text(...)` used bare

(not assigned), and from `ret str`/`ret boxed`/`ret i64` expression handling for

`ret expr` cases when the RHS is a bare `Module.function(...)` call (jsonl,

jobs_*, tour_05, tour_08 all hit this). Recognize the callee as

`<Capitalized ident>.<lower_snake ident>(...)`, where `<Capitalized ident>` is

NOT already one of the special-cased intrinsic prefixes.

**Leverage**: this single handler is very likely what turns the ~30 other

`unsupported boxed expression: local bN x:boxed = Module.fn(...)` fail-closed

entries across the WHOLE 100-example survey (ehy_templating, ergonomics_*,

evented_io_c10k, hey_ast_manifest, hey_bootstrap_gate, hey_parser_tooling,

hey_tiny_c_emit, http_request_response, jobs_*, markdown_tooling,

public_evented_io, self_host_compile_pilot, self_hosted_supported_subset,

shapes_data_model, stream_resequencing) from "first blocker is a facade call"

into "first blocker is something else, if anything." It won't single-handedly

turn any of them in-class-identical (each still needs its own args, nested-call,

array-literal-of-calls, etc. support), but it collapses the survey's dominant

blocker category into one code path. Implement and re-survey before doing

per-example follow-up work.

Per-example construct inventory (file order)

one_brc_single.mir (383 lines) — representative of the whole one_brc family

Shared free-fn helpers (`one_brc_min_value` … `one_brc_boundaries_value`,

lines 11-223) are almost entirely ALREADY-SUPPORTED shapes (if/ret, while,

nested if/else, arithmetic) EXCEPT the following, each a **new handler** in

`hey_llvm_int_expr_value`/`hey_llvm_operand_or_call_value`/

`hey_llvm_boxed_expr_value`/`hey_llvm_str_expr_value` unless noted:

1. `if bool cursor < finish and byte_at(data, cursor) == 45` (line 48) —

compound `and` condition where the second operand is a builtin call, not a

plain comparison. Closest: `hey_llvm_cmp_expr_value` (~1178); needs `and`-chain

support extended to allow a call-based sub-condition.

2. `byte_at(data, cursor)` (48, 54) — bare builtin, runtime symbol

`hey_llvm_byte_at_value_i64(ptr, i64)` already declared (prelude ~2562), zero

emitter call sites. New handler in `hey_llvm_operand_or_call_value` (~1724) or

`hey_llvm_int_expr_value`.

3. `store value:i64 = value * 10 + byte - 48` (line 56) — 3-term unparenthesized

arithmetic chain (`*` then `+` then `-`). `hey_llvm_int_expr_value` (~1882)

only splits on the FIRST depth-0 binary op and hands the remainder to

`hey_llvm_operand_or_call_value`, which does not itself split further — so

chains of >1 operator (outside the specific paren+mod-tail shape in

`hey_llvm_int_chain_mod_value`, ~1808) are unsupported. Needs a general

left-to-right multi-op fold in `hey_llvm_int_expr_value` (native precedence

here is pure left-to-right, matching the existing paren-chain evaluator's

fold order — likely reuse/generalize `hey_llvm_int_chain_mod_value`).

4. `ret boxed {minimum: value, maximum: value, sum: value, count: 1}` (65) —

plain object literal, should already work via `hey_llvm_boxed_expr_value`'s

`{`-literal branch (~481) if `ret boxed <object-literal>` return-position

routes there; verify return-expression dispatch calls boxed_expr_value for

`{`-prefixed rets (check ret-expr router, likely fine already).

5. `!stats.has?(station)` (72) and `stats.put(station, incoming)` (73) —

**dynamic (non-literal) key** has?/put. Current handlers

(`.has?("` at ~1424, `.put("` at ~804) require a literal-quoted key

substring immediately after the method name and fail to match a bare

identifier key at all. Runtime already supports it —

`hey_llvm_json_has_key_i64(ptr,ptr)` / `hey_llvm_json_put_key_value(ptr,ptr,ptr)`

accept any boxed key ptr (confirmed used with a `%tN` register key in

p16_one_brc_single.ll:1265,1274,1312, i.e. `one_brc_merge_record_value`).

New handler: variable-key put/has? branch alongside the literal-key ones.

6. `stats[station]` (75) — dynamic-key index read via `[...]` on a boxed local

where the index is a **string local**, not an int literal/local. Check

`hey_llvm_str_part_or_access_value`/`hey_llvm_call_expr_value`'s `[` handling

(~613-660, ~880) — those paths key off `json_get_index_value` (i64 index) or

`json_get_key_value` with a literal string; a variable-string index needs the

`json_get_key_value(ptr, ptr <loaded str local>)` variant (no global needed).

7. `stats.put(station, { minimum: one_brc_min_value(...), ... })` (76) — put with

a dynamic key AND an object-literal value containing nested fn calls; compose

of #5 plus existing nested-call-in-object-literal support (already used

elsewhere, e.g. `mean: call one_brc_mean_tenth_value(...)`-style values are

supported per the array/object literal element grammar — verify nested

`one_brc_min_value(current.minimum, incoming.minimum)` bare-call (no `call`

keyword) as an object-literal field value; MIR only prefixes `call` at

statement/local-assignment position, nested expression calls appear bare

(`one_brc_min_value(x, y)`) — confirm `hey_llvm_split_elements_value`/object

literal value-parsing invokes `hey_llvm_call_expr_value`-equivalent for bare

calls, not just the `call `-prefixed form.

8. `control boxed for b36 station in source.keys` (one_brc_merge_stats_value,

MIR ~117) — for-in target is `<local>.keys`, a dotted property, not a bare

local. The for-in handler (~3605-3690) does

`hey_llvm_local_typed_value(locals, for_parts[3])` on the raw 4th token,

which fails for anything containing `.`. New handler: detect a `.keys` suffix

on the for-in target, emit the existing `hey_llvm_json_keys_value(ptr)` call

first, iterate that result.

9. `"" + station` (118, and repeatedly: 127, 131, 152 `"" + stations[index]`) —

str-coercion idiom (empty-string-literal concat forces a boxed value to str).

Check `hey_llvm_str_expr_value`/`hey_llvm_str_part_value` for a `'' + <boxed

local-or-index-expr>` pattern; likely NOT supported (str concat normally

expects a str operand, not a raw boxed-index-expr coercion) — new handler,

probably `hey_llvm_value_to_raw_string_cstr` wrap similar to string comparisons

already used elsewhere (slice 55 notes `to_RAW + strcmp` for string compares).

10. `[...next, station]` (133, 136, 139, 219, 222) — spread-then-append array

literal (`[...list, item]`), single spread only. Should already work — the

array-literal `...` handling (~392-434) supports a leading spread plus

trailing elements; verify it accepts a boxed-local spread + single scalar

tail (this shape already appears in json_spread_literals per slice 52's

notes, so likely ALREADY SUPPORTED — flag as verify-only, not build).

11. `byte_buffer(1048576)` (149), `byte_buffer_append(output, ...)` (160-168),

`byte_buffer_bytes(output)` (171) — bare builtins, runtime symbols already

declared (`hey_llvm_byte_buffer_new_value(i64)`,

`hey_llvm_byte_buffer_append_value_i64/cstr_i64(ptr,ptr)`,

`hey_llvm_byte_buffer_bytes_value(ptr)`), zero emitter call sites. New

handlers, likely in `hey_llvm_boxed_expr_value` (buffer is boxed) and a new

"int-returning builtin call used as a statement, result discarded" path for

the `local bNN len:i64 = byte_buffer_append(...)` idiom (result bound but

body only cares about side effect).

12. `format_fixed(minimum, 1)` (164, 166, 168) — explicit 2-arg builtin call

form, DIFFERENT from the existing automatic "format-then-concat" heuristic

(`hey_llvm_format_fixed_string` is already called automatically when

str-concatenating with an int, e.g. ~1009, 1351, 1420) — but there is no

handler for `format_fixed(x, precision)` as an explicit call expression fed

into another builtin (`byte_buffer_append`). New handler needed; likely

trivial (reuse `hey_llvm_format_fixed_string(i64, i64)`, already declared).

13. `text(byte_buffer_bytes(output), "utf-8")` (171) and

`text(station_bytes, "utf-8")` (103) — bare 2-arg `text(bytes, "encoding")`

builtin, runtime symbol `hey_llvm_text_value_string(ptr, ptr)` declared,

zero call sites. New handler.

14. `{ rows: rows, stations: stats.keys.length, text: ... }` (177) — object

literal field value is a chained property access `X.keys.length` (two dots).

`.keys` alone is supported (`hey_llvm_json_keys_value`, used at ~704, 1474)

but chaining `.length` onto that result needs a new compound-access path;

the existing `.length` support (`hey_llvm_json_len_i64`) expects a single

local/field, not `<local>.keys.length`.

15. `map_readonly(path)` (185) — bare builtin, `hey_llvm_map_readonly_value(ptr)`

declared (line 2223), zero call sites. New handler (boxed-returning).

16. `mapped.ok` (187), `mapped.value` (190) — plain field access on a boxed

local from a builtin's result; should already work via the standard

`.field` get_key path once #15 lands (verify only).

17. `find_nth_byte(data, 10, max_rows, 0)` (194) — bare builtin, 4-arg,

`hey_llvm_find_nth_byte_value_i64(ptr,i64,i64,i64)` declared, zero call

sites. New handler.

18. `find_byte_between(data, 59, cursor, finish)` (94, 98, 212) — bare builtin,

4-arg, runtime symbol `hey_llvm_find_byte_value_i64(ptr,i64,i64,i64)`

declared (line 2283), zero call sites, signature likely

`(data, byte, start, finish)`. New handler.

19. `slice(data, cursor, separator - cursor)` (102) and

`slice(data, 1, 2)` (bytes_codecs) — bare 3-arg `slice(bytes, start, len)`

builtin returning `bytes`/boxed. No runtime symbol found under this name in

current declares — check native pin for the actual symbol (likely

`hey_llvm_bytes_slice_value` or similar, NOT yet declared — may need a new

`declare` line too, unlike the byte_at/map_readonly/etc. family which are

pre-declared). Cross-check against a NATIVE PIN once made (see below).

20. `actor one_brc_range_worker(collector:actor)` body (263-272) — see

"actor-body dispatch" section below; this is the dominant blocker

(`if bool msg.kind == "scan"` is literally the survey.tsv first-blocker

line for all 4 one_brc examples).

21. `actor one_brc_collector(...)` body (273-292) — same section; additionally

exercises `Files.write_text(target_path, result.to_json)` as a bare

(unassigned) actor-body statement — this is the exact construct the

headline finding decodes.

22. `fail str "1BRC row count mismatch: expected " + expected_rows + ", got " + result.rows`

(run_one_brc_single, MIR ~340) — `fail str` handler (~3810) only accepts a

single-quoted literal (`hey_llvm_single_quoted_literal?`); concatenated

payload unsupported. New handler: route `fail_payload` through

`hey_llvm_str_expr_value` instead of requiring a literal, when it isn't one.

23. `cpu_count()` (one_brc_actors MIR only, not one_brc_single) — bare 0-arg

builtin, `hey_llvm_cpu_count_i64()` declared, zero call sites. New handler,

same style as `Time.now_ns()` direct-dispatch (~1902).

one_brc_actors.mir — delta vs one_brc_single

Identical helper/actor bodies; extra constructs in `run_one_brc_actors`/`main`:

  • `cpu_count()` (item 23 above).
  • `Files.remove_tree(state_path)` (MIR ~344) — bare-string-arg builtin/facade;

runtime symbol `hey_llvm_path_remove_tree_value(ptr)` already declared (2546),

zero call sites — this one has a DIRECT declared symbol (unlike write_text

which routes through the generic dispatcher above), so check the native

pattern carefully once a pin exists: it may go direct, not through

`stdlib_json_call`, precisely because it's declared as a first-class runtime

fn already. Needs its own NATIVE PIN to confirm (see below — no pin covers

this call shape).

  • `output_path + ".json"` / `workers + "\t" + ...` string concat chains with a

leading str local + literal + more locals — should already work via

`hey_llvm_str_expr_value`'s multi-part concat (used pervasively elsewhere);

verify-only.

one_brc_case.mir — delta vs one_brc_single/actors

  • `if bool workers > 1` / `else` branch selecting between calling

`OneBrcActors__run` vs `OneBrc__single` and storing into the same `result`

boxed local from both arms — should already work (if/else with `store` in

both arms, existing pattern), verify-only.

  • `metadata_path + ".workers.json"` — same concat pattern, verify-only.
  • `{ rows: result.rows, workers: workers, runtime_ms: runtime_ms, stations: result.stations, result_path: result_path }`

object literal mixing field-access values and plain locals — should already

work (standard object-literal value grammar), verify-only.

  • `metadata.to_json` (MIR ~140-141) — same `.to_json` pseudo-field as

one_brc_single item 21/headline finding.

  • Everything else is a subset of one_brc_single/actors; no new constructs beyond

the shared list above.

one_brc_scale.mir — delta vs one_brc_actors (largest incremental example)

New constructs beyond the shared one_brc list:

24. `Process.run_argv_timeout([ case_app, path, "" + workers, ... ], stop_ms)`

(MIR ~340) — facade call whose FIRST arg is an array literal of mixed

str-coerced (`"" + workers`) and plain boxed locals, second arg a boxed

int. Exercises headline-finding dispatcher PLUS array-literal-of-coerced-

values as a facade arg. `outcome.timed_out` / `outcome.ok` / `outcome.stderr`

field reads on the result — should be fine once the call itself resolves to

a boxed value.

25. `Files.exists?(metadata_path)` / `!Files.exists?(...) or !Files.exists?(...)`

(MIR ~343) — facade call returning presumably a bool/boxed-bool, combined

with `or`. Need to confirm bool-coercion of a facade-call result in a

condition (`if bool !A or !B` where A, B are calls) — condition parsing

(`hey_llvm_cmp_expr_value`) doesn't currently split on `or` combined with

negated call operands; likely a new compound-condition handler.

26. `Text.trim(Files.read_text(metadata_path))` — nested facade calls, one

facade call's boxed/str result fed as the sole arg to another facade call.

27. `while bool row_target <= largest_rows and !stopped` /

`while bool workers <= cores and !stopped` (MIR ~365, ~372) — compound

`and` condition mixing a comparison with a bare bool-local negation. Check

`while bool` condition parsing for `A and !B` shape (mirrors #1's `and`

extension need).

28. `div(baseline_runtime_ms * 1000, result.runtime_ms)` and

`div(speedup_x1000, workers)` — should be fine per existing `div()` +

single-op-arg support (verify-only, already covered above).

29. `row_target + "\t" + workers + "\t" + result.runtime_ms + "\t" + ...` long

tab-joined concat chain (7+ parts, mixing field access and plain locals) —

should already work via the general str-concat machinery (many parts is

already exercised elsewhere per slice notes), verify-only.

30. `Files.append_text(output_path, row)` — facade call, statement position,

args (str local, str local) — exercises headline finding again, simplest

possible 2-str-arg shape (closest to the exact write_text pattern already

decoded from the pin).

31. `store row_target:boxed = row_target * 10` — boxed-typed local receiving an

arithmetic result (mixing i64 arithmetic into a `:boxed` store) — check

whether `store <name>:boxed = <int-expr>` auto-boxes via `hey_llvm_int_value`

the way object-literal fields do; likely a NEW handler (existing `store

<name>:boxed = <recv>` etc. are all boxed-to-boxed moves, not int-to-boxed

stores).

32. `if bool status == "ok" and (best_runtime_ms == 0 or result.runtime_ms < best_runtime_ms)`

(MIR ~438) — three-way compound condition with a parenthesized `or` nested

inside an `and`. New handler in `hey_llvm_cmp_expr_value`; likely the

single hardest condition-parsing case in the whole cluster.

33. Recursion-shaped `one_brc_scale_case`/`one_brc_scale_run` calling each other

— standard fn-to-fn calls, should be fine once each individual construct

above resolves.

function_actor.mir (38 lines)

  • `fn label_count(label:boxed, count:boxed) -> str; ret str label + ": " + count`

— str fn with two boxed params concatenated with a literal; should already

work (str-return fn support landed slice 43, boxed params supported

elsewhere) — verify-only, first-blocker is downstream in the actor.

  • `actor counter(label:boxed)` state `count:i64 = 0`, receive `msg` (plain str

payload, not object) — this is the SIMPLE end of the actor-body spectrum:

`if bool msg == "inc"` (existing `ifmsg:` token, ~3307, SUPPORTED),

`store count:i64 = count + 1` (existing `incby:` token, SUPPORTED),

**`say str call label_count(label, count)`** (line ~24) — NOT SUPPORTED: the

actor stmt dispatcher's `say str ` handling only recognizes literal-or-concat

("saymix") shapes (~3140-3225), no `say str call <fn>(<args>)` token. This is

exactly the survey's reported first blocker. New token, e.g.

`saystrcall:<fnname>:<arg-piece>|<arg-piece>` reusing the same

Actor-body generalized dispatch (the other big architectural item)

Current actor-body codegen (HeyLlvm.hey ~2700-3400, statement loop starting

~3060) is a flat, per-line STRING-PATTERN matcher: every `actor.receive` body

line is tested against ~20 hardcoded shapes (`ifmsg:`, `ifmsgint:`, `sayrecv`,

`saystate:`, `sayparam:`, `sayrecvfield:`, `sayrecvidx:`, `sayrecvlen`,

`saymix:`, `exitmsg`, `sendparamrecv:`, `ifstatenil:`, `setstaterecv:`,

`sendstateint:`, `sendstatesub:`, `sleeplit:`, `sleeprecv`, `ifrecvcmp:`,

`incby:`, `increcv:`, `ifcmp:`) and the FIRST line that matches none of them

fails closed with "unsupported actor operation". There is NO token for:

  • `local <bid> x:<type> = <any-general-expr>` inside an actor body (no actor

body in this group can bind a fresh local at all beyond the fixed

`msg`/state/param names) — blocks json_actor, all 4 one_brc variants.

  • `<recv>.<field>` used as a CONDITION operand beyond the exact `msg == "lit"`/

`msg == <int>`/`msg > <int>`/`msg < <int>` shapes — `if bool msg.kind ==

"scan"` (dotted field access on the LHS) is a different shape than the

existing `ifmsg:`/`ifmsgint:` (which compare `msg` itself, not `msg.field`).

This is literally the survey's reported blocker for one_brc_single/actors/

case/scale.

  • `<recv>.<field>` or `<local>.<field>` used as a CALL ARGUMENT inside an actor

body (`OneBrc__scan_range(msg.data, msg.start, msg.finish)`,

`OneBrc__merge_stats(stats, msg.stats)`).

  • `actor.send <target>, <object-literal>` (only scalar payloads are tokenized

today: `sendparamrecv`, `sendstateint`, `sendstatesub`).

  • `Files.write_text(...)` / any facade call as a bare actor-body statement.
  • boxed STATE mutation via a general fn call (`store stats:boxed = call

OneBrc__merge_stats(stats, msg.stats)`) — only `setstaterecv` (state = recv)

and the int `incby`/`increcv` forms exist; no `store <state>:boxed = call

<fn>(<state-or-msg-args>)`.

  • `"" + output_path` str-coercion of an actor PARAM inside an actor body

(`local b107 target_path:str = "" + output_path`).

**Recommended approach** (matches the in-progress.md slice-46 note, "needs a

generalized actor-body dispatch rather than more tokens... the per-actor

stash from slice 45 is the right substrate"): instead of adding more per-shape

tokens, treat `msg`/actor states/actor params as ordinary named locals seeded

into the SAME `locals` map used by fn bodies (`msg` → `:boxed`, states/params

per their declared types), and re-route actor-body statement lines through the

EXISTING fn-body statement dispatcher (the big `if hey_llvm_starts_with?(line,

'local ')`/`'store '`/`'if bool '`/`'actor.send '`/etc. chain used for

ordinary fn/entry bodies, ~3810 onward and ~4340 onward) rather than the

separate token-string mini-language. This reuses ALL of the general-purpose

machinery (call expressions, object literals, facade dispatcher, concat,

field access) for free inside actors, at the cost of reconciling the

per-actor-stash/grouped-emission ordering from slice 45 (actor fns are all

emitted before ALL step fns, with a shared running temp/label/global counter)

with the fn-body dispatcher's own counter bookkeeping. This is the single

largest re-architecture in the whole cluster; sequence it AFTER the headline

facade dispatcher (so the reused fn-body statement chain already knows how to

emit `Files.write_text(...)`, object-literal sends, etc.) and validate first on

function_actor (scalar msg, `say str call fn(...)` gap) and json_actor

(object-literal spread of `msg`) as smaller fixed points before attempting the

full one_brc actor bodies.

Cluster summary / recommended order

Highest leverage first:

1. **Generic facade-call dispatcher** (`hey_llvm_stdlib_json_call`, headline

finding above). Unlocks the first-blocker on self_host_compile_pilot,

self_hosted_supported_subset immediately (1-arg literal calls, simplest

possible case) and removes the FIRST blocker (though not all blockers) from

jsonl_sort_mailinator_users, ergonomics_package_stream_app,

ergonomics_porcelain_app, plus ~20 more examples outside this group per

survey/results.tsv. Do this first; it's also a prerequisite for the actor

body work (Files.write_text inside one_brc_collector) and for `.to_json`.

2. **Multi-op int arithmetic chains** (`value * 10 + byte - 48`) and **bare

builtin call family** (byte_at, map_readonly, find_byte_between,

find_nth_byte, byte_buffer*, format_fixed(x,p), text(bytes,enc), cpu_count,

dynamic-key put/has?/index) — all mechanical, all runtime symbols already

declared except `slice()`. This is the bulk of one_brc's free-fn helper

layer and has no actor complexity; do this before the actor-body rewrite so

one_brc's actor bodies can call already-working helper fns.

3. **Generalized actor-body dispatch** (reuse fn-body statement chain for

actor bodies). Validate incrementally: function_actor (say-str-call token

or full reuse) → json_actor (object-literal local + spread of msg +

object-literal send) → one_brc_range_worker/collector (field-access

conditions or a generalized `msg.field` condition parser, call args from

fields, boxed state mutation via call, Files.write_text statement).

4. **`:callable` type + callable-dispatch subsystem** (tour_02_functions) —

isolated, self-contained, no actor/facade dependency; can be built any time

in parallel with the above by a different track.

5. **`:bytes` type + byte-literal/hex/base64 builtin family**

(bytes_codecs, bytes_strings) — isolated, no actor/facade dependency, but

BLOCKED on a native pin (runtime ABI for `b"..."` literals and the

hex/bin/oct/base64 family is not inferable from any pin currently in the

scratchpad).

6. **Compound boolean conditions** (`A and <call>==lit`, `A and !B`,

`A and (B or C)`) — needed by one_brc_scale specifically; smallish grammar

extension to `hey_llvm_cmp_expr_value`, sequence after step 2.

7. one_brc_scale's `Process.run_argv_timeout`/`Files.append_text`/

`Files.exists?`/`Text.trim` are all facade calls (step 1) once declared/

dispatched generically — no separate work beyond confirming their exact

native call shape via a pin (Files.remove_tree in particular may go DIRECT

rather than through the generic dispatcher, since it already has its own

declared runtime symbol — needs its own pin to disambiguate).
