Visibility and private imports

Hey defaults to explicit-but-public module functions: when a module is imported, its public module functions are callable with their module prefix.

Use `private fn` for helpers that belong to the current source file or module implementation:

module Cards
  private fn normalize_title(title)
    return trim(title)
  end

  fn create(title)
    return {title: Cards.normalize_title(title)}
  end
end

`Cards.create(...)` can call `Cards.normalize_title(...)`, but importers cannot call `Cards.normalize_title(...)` directly. To importers, private functions behave as if they do not exist.

Top-level private helpers are also allowed:

private fn slugify(name)
  return trim(name)
end

Those helpers are callable from the same source file and hidden from other files.

Private imports

Use `private import` when a file needs an implementation dependency without re-exporting that dependency to its own importers:

private import './slug_helpers.hey'

module Decks
  fn slug(name)
    return SlugHelpers.slugify(name)
  end
end

A file that imports `Decks` can call `Decks.slug(...)`, but it cannot call `SlugHelpers.slugify(...)` through `Decks`' transitive dependency.

Ramifications

• `private fn` is about API visibility, not security sandboxing.

• Private functions are still compiled as part of the program so public functions can call them.

• Private imports keep dependency details out of downstream module APIs.

• Imported modules keep their namespaces. Private imports do not create unqualified names.

• Hey has no module-level mutable variable exports today, so private import currently applies to imported functions/actors/modules rather than variables.

• Future export lists and import aliases should compose with these rules rather than replacing them.
