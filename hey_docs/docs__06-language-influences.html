Language Influences

Ruby

Keep the joy: low ceremony, great strings, good names, and expressive code.

C

Use C as the first native backend and runtime ABI. Hide manual string/memory pain from normal Hey code.

JavaScript object practice

Hey JSON/hash notation intentionally borrows the practical object-literal shape common in JavaScript style guides: use literal objects, prefer readable properties, and only quote invalid identifier keys. Hey does **not** copy JavaScript's prototype model.

let user = {name:'Ada', visits:2, 'data-id':123}

Erlang/OTP

Actors, mailboxes, fail-fast thinking, and supervisors are core influences.

Pony

Data-race avoidance matters. Hey should avoid shared mutable state across actors by default.

Go

Hey should borrow operational practicality: simple binaries, good tooling, and useful runtime diagnostics.

Lisp

AST clarity and eventual code transformation remain important, but macros should wait until the language core is stable.

Explicit non-influence: user-facing async/await

Hey should not copy async/await as its ordinary concurrency model. Hey code should remain actor/message-passing-first: `spawn`, `send`, actor mailboxes, `receive`, `join`, `stop`, and supervisors.

The implementation may still use asynchronous I/O, fibers, continuations, worker threads, or a VM internally. These are runtime mechanisms, not user-facing concepts.

Hey should achieve multicore parallelism by scheduling many isolated actors across a pool of OS threads. Actors should not share mutable state by default. Values sent between actors should be copied, immutable, uniquely moved, or explicitly share-safe.

The goal is Erlang-style resilience, Pony-style data-race avoidance, Go-style runtime/tooling practicality, and Ruby-like source elegance.

My strongest guidance: keep actors; do not add async/await; build a real runtime; start with run-to-completion actor handlers; use OS threads internally; forbid shared mutable state across actors by default.
