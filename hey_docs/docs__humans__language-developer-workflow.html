Hey language developer workflow

This is the copy/paste workflow for humans maintaining the Hey language. It uses

`$HOME/Downloads` for received archives, `$HOME/dev` for working trees and extracted projects, and `$HOME/Desktop` for generated outbound artifacts. Apply and commit

each alpha source archive separately so failures remain attributable to one

version slice.

The preferred final-release path is the repository-root processor returned with

a source handoff. Source acceptance is test-authoritative: no SHA-256 comparison

or byte-for-byte ZIP identity is required. During focused iteration, complete

replacement files or executable overlay scripts may be applied directly and

reviewed with `git diff`; then run the focused tests. For a full ZIP, the

processor extracts its installer, stages the complete tree safely, runs the

selected ordered gates under `time`, commits only after success, and emits the

next archive and continuation prompt:

cd "$HOME/dev/hey-lang-bootstrap-plan"

HEY_HANDOFF_JOBS=12 \
HEY_VERIFY_MODE=full \
HEY_BUILD_JOBS=12 \
HEY_COMPILER_CONTRACT_JOBS=12 \
HEY_RELEASE_JOBS=12 \
HEY_RUN_PROMOTION=1 \
./PROCESS-HEY-SOURCE-HANDOFF.sh \
  "$HOME/Downloads/hey-lang-bootstrap-plan-vVERSION.zip"

Full and report handoffs require promotion as the final commit-blocking gate.

The processor still runs it after the continue-all verification receipt, even

when earlier gates failed, and refuses to commit or fast-forward on any failure.

The manual workflow below remains the transparent fallback and documents each

individual step.

Apply one source ZIP as one commit

The ZIP is a transport container, not the source identity. It may be regenerated

or recompressed without updating a checksum. `unzip -tq`, exact central-directory names, the complete canonical stdlib

module ledger, collision-free case-folded paths, staged roots/sentinels, mode

checks, file-inventory equality, the repository diff, and executable tests are

the source-handoff acceptance evidence. This does not relax checksums for package

locks, published packages, native targets, or release artifacts.

Assume the downloaded archive is in `$HOME/Downloads` and the existing checkout

is `$HOME/dev/hey-lang-bootstrap-plan`:

cd "$HOME/dev/hey-lang-bootstrap-plan"

if HEY_SOURCE_ARCHIVE="$HOME/Downloads/hey-lang-bootstrap-plan-vVERSION.zip" sh -eu <<'HEY_SOURCE_APPLY'
archive="$HEY_SOURCE_ARCHIVE"
test -f "$archive"
unzip -tq "$archive"

installer="$(mktemp)"
cleanup() { rm -f "$installer"; }
trap cleanup EXIT HUP INT TERM

unzip -p "$archive" bin/hey-apply-source-zip > "$installer"
chmod +x "$installer"
"$installer" "$archive" --repo "$PWD"
HEY_SOURCE_APPLY
then
  git status --short
  git add -A
  ./bin/hey-normalize-canonical-module-index .
  git commit -m "Release vVERSION"
else
  echo "Source archive apply failed; nothing was committed." >&2
fi

Repeat that complete block for each version in ascending order. Do not apply

several archives and combine them into one commit.

On case-insensitive macOS filesystems, filesystem spelling and Git index spelling

must be verified separately. `git add -A` alone may fail to stage a case-only

rename. The normalization helper explicitly removes the retired lowercase index

entries and re-adds the canonical CamelCase entries with case folding disabled.

Sub-minute architecture gate

Use this before any fixed-point build when changing compiler ownership, native

boundaries, release frontend selection, or the REPL migration plan:

HEY_BUILD_JOBS=12 ./bin/hey verify architecture

This builds only `build/heyc`, audits every native source, runs the LLVM-first

language/compiler contracts, executes the LLVM-hosted release-frontend fixture,

and checks the verification manifest. It deliberately does not build

`hey-selfc`, generate a source ZIP, run SDK matrices, or attempt promotion.

Fast current-slice debugging

Do not rerun the broad platform or release matrix while diagnosing one active

release-candidate failure. Rebuild incrementally and run only the newest gates:

cd "$HOME/dev/hey-lang-bootstrap-plan"
HEY_BUILD_JOBS=12 ./bin/hey verify current-slice

This tier runs the canonical filename audit, inherited-child process regression,

its compiler contract, the source-handoff fail-fast language and shell regressions,

the generated-child Windows component/receipt/bundle checks, and the verification

manifest. It intentionally skips source-ZIP construction,

SDK matrices, documentation generation, and release tiers. Run the broader

focused/release tiers only after this command is green.

Malformed ZIP tests must mutate archive entries directly; never attempt to create

case-colliding aliases in one host directory, because default macOS filesystems

cannot represent that fixture. Run the behavioral shell regression once so its

diagnostics remain direct; the language contract should inspect source shape and

verify gate registration without launching the expensive regression again.

For malformed negative fixtures, invoke the existing Hey-owned archive checker

through `heyc --run`; do not compile another full runtime unless native emission

is the behavior under test.

Guarded focused verification

Run this after each applied slice. The child shell stops on the first failure

without changing the options of the interactive terminal:

cd "$HOME/dev/hey-lang-bootstrap-plan"

if sh -eu <<'HEY_FOCUSED_VERIFY'
make clean
make -j12

./bin/hey --version
./bin/hey verify --focused
./bin/test --hir-mir
./bin/test --conformance
./bin/heyc tools/check_verification_manifest.hey --run

make docs-smoke
make docs-layout-smoke
make source-zip-smoke

test -z "$(git status --short)"
HEY_FOCUSED_VERIFY
then
  echo "Focused verification passed."
else
  echo "Focused verification failed; later gates were not run." >&2
fi

Run the longer release blocks separately. Twelve workers is the maintained default on a 12-core development machine.

Lower `HEY_RELEASE_JOBS` explicitly on memory-constrained hosts; individual

release groups still cap lanes that cannot safely share outputs:

cd "$HOME/dev/hey-lang-bootstrap-plan"
time env HEY_RELEASE_JOBS=12 ./bin/test --release-c

cd "$HOME/dev/hey-lang-bootstrap-plan"
time env HEY_RELEASE_JOBS=12 ./bin/test --release-a

cd "$HOME/dev/hey-lang-bootstrap-plan"
time env HEY_RELEASE_JOBS=12 ./bin/test --release-b

When a complete defect inventory is more useful than fail-fast behavior, run the

maintained aggregate verifier. It streams every gate, prints a large failure

block, continues with all remaining gates, retains one log per gate, and exits

nonzero after the final summary:

cd "$HOME/dev/hey-lang-bootstrap-plan"

HEY_BUILD_JOBS=12 \
HEY_COMPILER_CONTRACT_JOBS=12 \
HEY_RELEASE_JOBS=12 \
./VERIFY-HEY-RELEASE.sh

Documentation publication and application vendoring remain optional. Promotion

is enabled by default and is always the final gate:

HEY_VERIFY_PUBLISH_DOCS=1 \
HEY_VERIFY_VENDOR=1 \
HEY_VENDOR_JOBS=12 \
HEY_BACKEND_WORLD_JOBS=12 \
./VERIFY-HEY-RELEASE.sh

The source-handoff processor can use the same continue-all catalog inside its

disposable worktree with `HEY_VERIFY_MODE=report`; it still refuses to advance

the real checkout when any gate fails.

Run promotion last and alone when not using the aggregate script:

cd "$HOME/dev/hey-lang-bootstrap-plan"
time env HEY_BACKEND_WORLD_JOBS=12 ./bin/test --promotion-report

Generate the Windows test bundle locally

The source handoff does not need to include a generated Windows bundle. Build it

from the exact committed source slice:

cd "$HOME/dev/hey-lang-bootstrap-plan"

./bin/hey windows runtime-test-bundle \
  --out "$HOME/Desktop/hey-windows-native-test-v$(./bin/hey --version).zip"

unzip -tq "$HOME/Desktop/hey-windows-native-test-v$(./bin/hey --version).zip"
shasum -a 256 "$HOME/Desktop/hey-windows-native-test-v$(./bin/hey --version).zip"

Copy that ZIP to Windows, use **Extract All**, open the extracted folder, and

double-click `START-HERE.cmd`. Do not open PowerShell manually, do not run as

Administrator, and do not disable CrowdStrike. Return the generated

`hey-windows-results.zip`.

Vendor Hey into the maintained application

Darwin arm64:

cd "$HOME/dev/hey-lang-bootstrap-plan"
export HEY_OPENSSL_PREFIX="$(brew --prefix openssl@3)"

./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target darwin-arm64 \
  --macos-trust local

Linux x64 through Docker:

cd "$HOME/dev/hey-lang-bootstrap-plan"

time ./bin/hey vendor \
  --repo "$HOME/dev/elders_prayer_app" \
  --target linux-x64 \
  --container docker \
  --jobs 12

The Linux command requires a running Docker-compatible daemon.

Publish the complete documentation site

The recommended presentation is Water.css with the sidebar layout: readable,

spacious, and still structurally owned by Hey's local HTML/CSS.

cd "$HOME/dev/hey-lang-bootstrap-plan"

rm -rf "$HOME/dev/jayteesf.github.io/hey_docs"

./bin/hey doc-site \
  --out "$HOME/dev/jayteesf.github.io/hey_docs" \
  --kind all \
  --theme water \
  --layout sidebar \
  --jobs 11

Use `--theme reference` when a completely local/offline style is required.

Generate the native iOS structural SDK

cd "$HOME/dev/hey-lang-bootstrap-plan"

./bin/hey mobile ios doctor

rm -rf "$HOME/dev/hey-ios-sdk"
./bin/hey mobile ios sdk --out "$HOME/dev/hey-ios-sdk"

cat "$HOME/dev/hey-ios-sdk/STATUS.md"
file "$HOME/dev/hey-ios-sdk"/objects/*.o

For the previously downloaded v0.99.319a SDK archive:

cd "$HOME/dev"
rm -rf "$HOME/dev/hey-ios-sdk-v0.99.319a"
tar -xzf "$HOME/Downloads/hey-ios-sdk-v0.99.319a.tar.gz" -C "$HOME/dev"
cd "$HOME/dev/hey-ios-sdk-v0.99.319a"
cat STATUS.md
file objects/*.o

Mach-O object emission is structural evidence. Simulator linking/execution,

physical-device installation, signing, and IPA generation remain separate gates.

Build and launch the direct-native iOS application

On macOS with Xcode installed:

cd "$HOME/dev/hey-lang-bootstrap-plan"

./bin/hey mobile ios native-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/dev/hey-ios-native-app" \
  --bundle-id com.hey-lang.native-demo \
  --display-name 'Hey Native' \
  --launch

cat "$HOME/dev/hey-ios-native-app/STATUS.md"
cat "$HOME/dev/hey-ios-native-app/receipt.json"

Build, sign, and launch on a physical iPhone/iPad

Inspect the command before supplying signing material:

cd "$HOME/dev/hey-lang-bootstrap-plan"
./bin/hey mobile ios device-app --help
xcrun devicectl list devices

Then set the Apple Development identity, team, and matching provisioning profile

shown by the help output and run:

./bin/hey mobile ios device-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/Desktop/hey-ios-device-app" \
  --bundle-id com.hey-lang.native-demo \
  --display-name 'Hey Native' \
  --device DEVICE_ID

This proves development-signed device installation and launch. IPA, TestFlight,

App Store validation, and App Review remain separate application release gates.

For the VSS application:

cd "$HOME/dev/vss"
HEY_ROOT="$HOME/dev/hey-lang-bootstrap-plan" ./bin/vss ios device --help

Generate the native Android structural SDK

cd "$HOME/dev/hey-lang-bootstrap-plan"

./bin/hey mobile android doctor

rm -rf "$HOME/dev/hey-android-sdk"

./bin/hey mobile android sdk \
  --out "$HOME/dev/hey-android-sdk"

file "$HOME/dev/hey-android-sdk"/objects/*.o
cat "$HOME/dev/hey-android-sdk/manifest.json"

This emits real Android ELF objects. It does not yet claim a JNI shared library,

APK/AAB, emulator/device execution, signing, or Play publication.

Generate the Android Studio application

cd "$HOME/dev/hey-lang-bootstrap-plan"
rm -rf "$HOME/dev/hey-android-app"

./bin/hey mobile android app \
  --out "$HOME/dev/hey-android-app" \
  --plan-only

Open `$HOME/dev/hey-android-app` in Android Studio and run the `app`

configuration. On a shell with Gradle, Android SDK/NDK, and adb configured:

./bin/hey mobile android app \
  --out "$HOME/dev/hey-android-app" \
  --build \
  --launch

Local agent branch workflow

Generate a repository-local agent prompt without creating a source archive:

./bin/hey-zip-source --agent

The agent records the starting state, works on a non-`trunk` branch, implements ordered version slices, adds small focused verification commands, and commits each completed slice only after its maintained focused/current-slice gates pass. Broad/full verification runs after minor failures are resolved. The maintainer reviews and merges the resulting branch.

See `docs/agents/local-agent-workflow.md` for the complete contract.
