Queued runtime/Jobs defects (war-room report)

14. GO-PARITY WEB RUNTIME PROGRAM (maintainer bar 2026-07-24: "as

fast as Go for web requests"; NOT covered by R8 — hey_runtime.c

is kept code). Baseline, same machine/payload/oha flags: Go

net/http no-keep-alive 45,309 req/s (~22us service); Hey

stdlib:Web workers:1 5,977 req/s (~167us) — ~8x per-request CPU.

The user's `sample` profile NAMES the costs (not I/O: write 65 +

accept 20 + close 36 + poll 117 samples):

(a) ~1,700 samples strcmp/hey_shape_intern + ~380 mutex — every

request/response JSON object RE-INTERNS its shape by string

compare under a GLOBAL LOCK. DESIGN PRINCIPLE (maintainer

2026-07-24: "we have an immutable language — why do we have a

lock?"): interned shapes are immutable-once-created and

append-only, exactly the structure that earns LOCK-FREE

READS — atomic publication, readers never synchronize, only

the rare first-intern coordinates. Rule going forward: in

this runtime, HOT PATHS HOLD NO LOCKS; immutability is the

enabling invariant, use it. Plus a GLOBAL-LOCK AUDIT of the

runtime under the same `sample` methodology (allocation,

actor registry, anything else mutex-shaped on request paths).

This lock is also the prime suspect for the INVERSE worker

scaling (workers:1=5977 -> workers:8=1589) — expect that to

flip positive with the fix.

(b) ~420 samples memmove/strlen/strdup — string copy churn in the

request/response path; also why writes cost ~0.2ms/KB where

Go writes 35KB in ~10us. Fix: reference-passing buffers,

single-pass serialization, writev batching.

(c) keep-alive per the earlier design (idle-timeout request loop

replacing hardcoded connection: close at hey_runtime.c:3897).

(d) SHAPE LIFECYCLE (maintainer requirement 2026-07-24: immortal

unbounded structures are unacceptable for long-running

processes). Facts: per-request VALUES are refcounted and

freed continuously (immutability makes refcounting COMPLETE —

acyclic by construction); SHAPES were always immortal (the

pre-Stage-1 mutex list also never freed them) — safe only

under the bounded-layouts assumption, which DATA-DRIVEN KEYS

break (the Ruby-symbols leak precedent). Fix: shapes become

reference-counted from the values that carry them, reclaimed

via epoch-based deferral compatible with the lock-free

readers (per-worker epoch counters — uncontended cacheline

stores, hot path stays lock-free); table slots tombstone under

the writer lock and re-publish. ACCEPTANCE: a leak test

driving N requests with data-driven object keys asserts

bounded RSS and bounded table occupancy; throughput numbers

may not regress vs Stage 1. Until it lands, document

data-driven keys as a long-running-process anti-pattern.

Staging: (a) first (biggest + fixes scaling), then (b), then (c),

then (d) before the program is declared complete for production

servers (60K bar covers speed; (d) covers longevity).

ACCEPTANCE BAR (maintainer, restated 2026-07-24: "as fast as Go,

60K+"): the PROGRAM is done only at >=60,000 req/s with

closes to 60k+. Every stage's numbers printed via oha + the

prior stage's numbers.

STAGE 4 PLAN (decided 2026-07-24, after the Go-architecture

question): Go needs no kernel load-balancing — it runs ONE

shared listener + a kqueue/epoll netpoller + goroutines

multiplexed across threads; work is distributed in USERSPACE

after accept, connections never pin a thread. Mirror that:

(4a) event loop per worker — kqueue/epoll readiness loop over

many connections per worker, listener as a readiness source;

kills the workers>=connections cap, platform-independent

(substrate exists: Net.event_loop facades, c10k smoke — BUT

Net.backend() reports readiness=poll today; kqueue/epoll is part

of 4a's DEFINITION OF DONE per the 2026-07-24 research note,

poll(2) rescans O(fds) per wakeup and becomes the next ceiling).

(4b, the bigger lever) route the 60K bar through the COMPILED

lane by landing delegation-tax item A (precompiled stdlib

archive, direct calls, dead-strip): native handlers cost ~us,

and the fixed socket layer becomes the whole story; also moves

the user's 4.2-4.5ms delegation plateau and "batch two".

EXPLICITLY NOT the plan: deep interpreter optimization

(hey_repl.c is R6-retiring; even a perfect interpreter cannot

reach Go's ~16us/request service). Order: 4b is already the

queue's PRIMARY item; 4a is independent runtime work. Stage (d)

still gates program-done. MEASUREMENT RULES (lessons):

assert the response body before measuring (a stale listener on

the same port serves the wrong bytes — wildcard *:PORT

coexists with a specific bind under SO_REUSEADDR, see defect
