Concurrency, actors, supervisors, and Jobs

Start with functions. Use an actor when one long-lived owner should process messages sequentially. Use a supervisor when that owner must be restarted after failure. Use Jobs when callers need a queue, worker pool, retries, results, cancellation, durability, metrics, or latency goals.

Hello from an actor

actor greeter
  receive name
  says "Hello, #{name}!"
end

program
  let worker = spawn greeter
  send worker, 'Ada'
  stop worker
end

Each actor processes one mailbox message at a time. Different actors may run in parallel on the runtime worker pool.

A supervised worker

actor calculator
  receive number
  says number * number
end

program
  let supervised = supervise_named(calculator, 3)
  send supervised, 12
  stop supervised
end

The supervisor owns restart policy. Links and monitors are lower-level tools for code that needs explicit failure relationships or notifications.

A scalable Jobs queue

import 'stdlib:jobs'

actor factorial_worker
  receive envelope
  let n = envelope['payload']
  let answer = 1
  let i = 2
  while i <= n
    set answer = answer * i
    set i = i + 1
  end
  jobs.complete(envelope, answer)
end

program
  let queue = jobs.queue({
    name: 'math/factorial',
    scheduler: 'round_robin',
    min_workers: 2,
    max_workers: 8,
    scale_on: {latency_ms: 20},
    retries: 2
  })

  let runtime = jobs.start(queue, {actor: factorial_worker})
  let receipt = jobs.perform(runtime, 10)
  says jobs.result_to(runtime, receipt, 1000)
  jobs.stop(runtime)
end

The queue configuration states the operational intent: start with two workers, permit up to eight, and treat 20 ms queue latency as the scaling target. The runtime may add workers when backlog age exceeds the target and retire idle workers without dropping below `min_workers`.

Runtime model

Hey actors are multiplexed over an internal native worker pool. A runnable actor can be resumed by another worker; it is not permanently tied to one operating-system thread. Mailbox processing is cooperative at message boundaries, and blocking I/O belongs behind runtime-aware socket/process/file operations so a blocked operation does not require every other actor to stop.

The implementation still needs promotion evidence for full preemption, dynamic Jobs autoscaling, remote queues, and cluster membership. Documentation must distinguish the configuration contract from features not yet proven by runtime specs. The roadmap targets one Jobs vocabulary for local pools, durable queues, and eventually clustered nodes, while preserving explicit limits and observable backpressure.
