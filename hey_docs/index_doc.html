Hey

Hey is an alpha compiled language for readable programs, native executables, immutable data, built-in specs, and actor-based concurrency that can use modern multicore systems without forcing application code to manage threads directly.

The command-line runner is `hey`, the compiler is `heyc`, the interactive shell is `hey-repl`, and source files use the `.hey` extension.

Learn Hey

Start with the path that matches what you are trying to do:

  • Quick start — build Hey and run a first program.
  • Tutorial — learn the language by building progressively larger examples.
  • Language guide — focused explanations of everyday syntax and behavior.
  • Language reference — concise reference for supported forms and built-ins.
  • REPL guide and REPL recipes — explore Hey interactively.
  • Actors and mailboxes — concurrency, supervision, and message passing.
  • Modules, packages, and visibility — local, standard-library, and third-party imports, including `private import` and `private fn`.
  • Standard library — web, jobs, files, JSON, packages, documentation, and distribution tooling.

The documentation index groups the complete public documentation by topic.

A first program

spec 'hello'
  expect stdout "Hello, Hey!\n"
end

program
  let name = 'Hey'
  says "Hello, #{name}!"
end

Run and test it:

./bin/heyc hello.hey --run
./bin/heyc hello.hey --test

Build the source checkout

make clean
make
./bin/heyc --version
./bin/heyc --backends

Run the maintained verification suites:

./bin/test --release-c
./bin/test --release-a
./bin/test --release-b

The full backend promotion report is intentionally separate because it is expensive:

time HEY_BACKEND_WORLD_JOBS=8 ./bin/test --promotion-report

Backend status

`native-c` is the default behavioral oracle. `self-hosted-c` is selectable and must preserve the same language behavior. `llvm` is selectable and is progressing toward full compatibility before it can be evaluated as the primary backend.

Backend implementation plans, release history, defect records, and version-by-version work tracking are maintainer material. They are kept in the repository but are deliberately excluded from the public HTML documentation site.
