Publishing a Hey package

Repository layout

my_package/
  hey-package.json
  README.md
  src/
    main.hey
  specs/
    main_spec.hey

A minimal manifest:

{
  "manifest_version": 1,
  "name": "my_package",
  "version": "0.1.0",
  "description": "A useful Hey package",
  "modules": {
    "main": "src/main.hey"
  },
  "files": [
    "README.md",
    "src/main.hey"
  ],
  "dependencies": {}
}

Every importable module must be declared under `modules`, and every released file must be declared under `files`. Published versions are immutable: changing released source requires a new version.

Verify and pack

hey package verify .
hey package pack . --out dist

The maintained command body is Hey; release distributions compile it with LLVM for the target environment. Packing uses `stdlib:archive` deterministic ZIP bytes and emits canonical JSON release metadata with archive, manifest, and content SHA-256 values. Run it from a clean checkout. Commit the manifest and lockfile, but normally ignore `dist/`.

Publish with GitHub CLI

git tag -a v0.1.0 -m "hey package v0.1.0"
git push origin main --tags

gh release create v0.1.0 \
  dist/my_package-0.1.0.zip \
  dist/my_package-0.1.0.release.json \
  --title "my_package 0.1.0" \
  --notes "Initial immutable Hey package release."

Consumers should use the release asset URL and hashes from the release JSON, never a branch-generated GitHub source archive.

Local development

Develop against a sibling checkout, then pack and verify exactly what will ship. A package manager may copy a verified local source into `.hey/packages`; the compiler must not silently import editable sibling files during locked builds.

Installing and locking a release

hey package install \
  --release dist/my_package-0.1.0.release.json \
  --root .hey/packages \
  --lock hey.lock.json

Installation verifies the archive, exact manifest bytes, and canonical declared content before an atomic rename into the package cache. The lockfile is also replaced atomically. Compilers and the REPL verify the imported module against `.hey-package-install.json` and require the package to be pinned in `hey.lock.json`.

Use `hey package verify-install PATH` to audit an installed package. `HEY_PACKAGE_VERIFY=off` is an explicit development-only escape hatch and should not be used in builds or releases.

Vendoring a Hey distribution

hey package vendor-dist --source dist/hey-VERSION-TARGET --dest vendor/hey/TARGET

The destination is assembled in a sibling staging directory and moved into place atomically after its required binaries are validated.
