Agent handoff documentationAudience-specific Hey documentation

Read this first before coding.

Starting point

  • Hey is a private alpha language. Analyze the complete uploaded repository before changing code.
  • The product frontend is one fixed-point path: LLVM-hosted Hey frontend -> semantic HIR -> ABI MIR. A private native `heyc-bootstrap` still seeds clean builds and performs validated MIR-to-LLVM emission/object/link work; it is an internal dependency, not an application backend.
  • The compiler pipeline is Hey source -> semantic HIR -> ABI MIR -> direct LLVM. HIR precedes MIR.
  • The concurrency north star is immutable values, isolated stateful actors, stateless workers, runtime-managed Jobs, and worker-backed streams. Do not add general-purpose user-facing channels or shared mutable state.
  • Compiler representation and ABI decisions are inferred from ordinary source and remain compiler-owned.
  • `runtime/hey_runtime.c` is the minimal target runtime/ABI, not the compiler architecture. Keep parsing, semantics, HIR/MIR policy, tooling policy, and application logic in Hey. Keep runtime services general and application-independent.
  • The legacy runner backend is removed. Do not restore it or document it as active.
  • Do not add benchmark program-shape shortcuts, canned challenge answers, filename branches, or challenge-specific runtime functions.

Working rules

  • Fix the compiler/runtime; never delete or weaken a valid spec to hide a regression.
  • Add focused mini-specs for hard failures before broad implementation.
  • Preserve the fixed-point LLVM-hosted frontend while shrinking the private native bootstrap/MIR consumer; retire it only after Hey-owned MIR-to-object/link and seed-free release bootstrap are proven.
  • Keep live OS resources as opaque handles, never JSON descriptors.
  • Keep timeouts enforceable, process capture dual-pipe, and blocking readiness honestly labeled.
  • Do not document function-name strings as first-class callable values.
  • Keep `stdlib:eval` an explicit restricted REPL/tooling capability; do not embed a hidden compiler, downloaded native-code path, or second parser in ordinary applications.
  • For native mobile and television work, read `docs/humans/native-mobile-plumbing-porcelain.md`; keep extraction package-first, prefer small target-neutral tooling, and preserve the **no new mobile syntax** rule unless an irreducible compiler/runtime requirement is proven.
  • Update all affected developer tutorials, stdlib guides, language references, human architecture/roadmap/changelog records, and agent todo/in-progress/done/lessons in the same slice.
  • Keep `docs/agents/todo.md` synchronized with `docs/humans/roadmap.md`.
  • Keep `docs/agents/done.md` synchronized with `docs/humans/CHANGELOG.md`.
  • Update the versioned EHY prompt when workflow, truth, required evidence, or source-document paths change.
  • Preserve executable modes in source and distribution archives.
  • In every progress update and final handoff, state the major north-star goals, the current milestone, and concrete version-slice progress including what is working and what remains broken.
  • Include a brief design reflection on whether the evidence suggests a different direction, approach, or idea should be considered; do not continue a weak approach merely because it is already underway.

Handoff process

1. Make substantive implementation progress.

2. Prefer the returned source-handoff processor with an explicit 12-worker

budget; it must verify in a disposable worktree and preserve ordered

shared-output gates.

3. Run focused specs and regressions while iterating.

4. Update all relevant documentation and tracking records.

5. Run docs and source-archive smoke gates.

6. Run Release-C, Release-A, and Release-B as appropriate.

7. Run promotion-report last, only when requested or when final promotion evidence is required.

8. Generate the clean source archive with `./bin/hey-zip-source --downloads`.

9. Present commands in this order: downloads; unzip/add/commit; final verification; promotion-report alone last.

10. State explicitly whether any `rm` commands are needed for renamed/deleted files.

Mandatory LLM source-ZIP reminder

Every `./bin/hey-zip-source` result must print copy/paste commands that run `bin/hey-apply-source-zip` from that exact candidate archive. Never substitute an unzip-only handoff, a static removal list, or deletions derived from an unreadable/empty manifest.

Mandatory zip handoff commands

Every final response that provides a downloadable source zip must include copy/paste-ready installation commands. Never provide a zip link by itself. The commands must appear in this order:

1. Extract `bin/hey-apply-source-zip` from the candidate archive and let it validate integrity, fully stage the tree, verify required roots/sentinels/modes/file count, and only then derive tracked deletions. Never prune from an unreadable or incomplete manifest and never maintain a static deletion list.

2. Validate the archive, then unzip it over the existing repository root with `unzip -o`.

3. Run `git status --short`, then `git add .`, then commit the slice.

4. Run final verification beginning with `make clean` and `make`.

5. Put the promotion-report command last and alone when promotion evidence is requested.

Use a concrete block shaped like this, adjusted for the actual archive name, version, deleted paths, and commit message:

cd "$HOME/dev/hey-lang-bootstrap-plan"

if sh -eu <<'HEY_SOURCE_APPLY'
archive="$HOME/Downloads/hey-lang-bootstrap-plan-vVERSION-DESCRIPTION.zip"
test -f "$archive"
unzip -tq "$archive"

installer="$(mktemp)"
cleanup() { rm -f "$installer"; }
trap cleanup EXIT HUP INT TERM

unzip -p "$archive" bin/hey-apply-source-zip > "$installer"
chmod +x "$installer"
"$installer" "$archive" --repo "$PWD"
HEY_SOURCE_APPLY
then
  git status --short
  git add .
  git commit -m "Release vVERSION: DESCRIPTION"
else
  echo "Source archive apply failed; nothing was committed." >&2
fi

Strict shell options must stay inside the child `sh` process. Never emit a bare

`set -e` or `set -eu` into an interactive terminal: those options persist after

the pasted block and can make a later expected test failure terminate the

user's entire shell session.

Do not use `git diff --stat` or `git diff --cached --stat` in copy/paste command blocks. Use `git status --short`.
