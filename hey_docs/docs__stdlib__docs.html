Docs stdlib (`stdlib:docs`)

`stdlib:docs` builds a static HTML site from the user-facing Hey documentation.

The command wrapper is:

./bin/hey-doc-site --out build/site

For GitHub Pages:

./bin/hey-doc-site --source . --out "$HOME/dev/jayteesf.github.io/hey_docs" --title "Hey language"

Defaults

The generator focuses on docs useful to application authors:

  • root `README.md`
  • `docs/README.md`
  • tutorial and quick-start docs
  • language guide/reference docs
  • REPL docs
  • `docs/modules-and-packages.md`
  • `docs/visibility-and-private-imports.md`
  • all `docs/stdlib/*.md`

It excludes LLM/release/backend handoff docs by default, including current handoff, planned versions, todo-by-version, open questions, backend parity gap, measurement, defect ledger, release notes, and similar maintainer-only context.

Use `--include-internal` when you intentionally want everything under `docs/`.

Options

./bin/hey-doc-site --help

Supported options:

  • `--source PATH` repo or dist root, default `.`
  • `--out PATH` destination directory
  • `--title TITLE` site title
  • `--theme bootstrap|plain`, default `bootstrap`
  • `--no-cdn` avoid CDN CSS/JS
  • `--include-internal` include handoff/backend/release docs
  • `--jobs COUNT` render pages through a Jobs queue; default `4`
  • `--version`

`bootstrap` uses the Bootstrap CDN. `plain` emits a small inline stylesheet for offline/local use.

Async page rendering

`hey-doc-site` uses `stdlib:jobs` internally. `docs.generate(...)` plans the page list, starts a small `docs/site` queue, enqueues one page-render job per Markdown document, waits for the receipts, and then writes the site index. This keeps the command behavior deterministic while making page rendering an ordinary Jobs workload.

Use a larger page-render pool for bigger docs sets:

./bin/hey-doc-site --source . --out build/site --theme bootstrap --jobs 8

The generated pages also rewrite Markdown document links such as `Tutorial` into the generated HTML filenames, so links keep working when the site is copied to GitHub Pages.

Markdown features

The renderer intentionally supports the subset needed for Hey docs today:

  • headings
  • paragraphs
  • fenced code blocks
  • unordered lists
  • Markdown links to generated docs pages
  • simple pipe tables

Example table:

| Call | Meaning |
|---|---|
| `web.get` | route descriptor |

That renders as an HTML `<table>` with `<th>` and `<td>` cells. More advanced Markdown features should be added only when the docs need them and should be pinned with a language spec.
