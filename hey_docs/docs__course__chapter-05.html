Chapter 5 — The Concurrency Model

**Concurrency** means structuring a program as multiple activities

that make progress independently — handling many web requests,

crunching a dataset on all processor cores. It is also where most

languages hide their sharpest knives: two activities touching the same

memory at the same time (a **data race**) produce bugs that appear and

vanish with timing.

Hey's answer is architectural, not advisory: *make data races

inexpressible.* The model rests on the immutability you met in

Chapter 1 plus four building blocks.

Actors: the only home for changing state

An **actor** is an isolated unit that owns private state and a

**mailbox** — a queue of incoming messages. Nothing outside the actor

can touch its state; the *only* way in is to send it a message, and

the actor processes messages one at a time. One-at-a-time processing

means the actor's state never faces two writers at once — the data

race is gone by construction, not by programmer discipline.

Here is the complete actor example from the repository's `examples/`

directory (`examples/actor_echo.hey` — the `spec` block at the top is

its built-in test, a form Chapter 11 explains):

spec 'actor echo'
  expect stdout "ping\n"
end

actor echo
  receive msg
  says msg
end

program
  let p = spawn echo
  send p, 'ping'
  stop p
end

`actor echo ... end` defines the actor; `receive msg` names the

incoming message, and the body below it runs once per delivered

message. The program creates a running instance with `spawn`, sends it

a value with `send p, 'ping'`, and `stop` ends it after the mailbox

drains. Sends are **asynchronous**: the sender does not stop and wait

for the receiver to process the message. A program can also wait for

an actor to finish — called **joining**.

Richer actors take parameters and carry private state bindings that

their receive body may `set` — the counter pattern (accumulate on each

message, report on request) is a stateful actor whose messages are

objects like `{kind: 'add', amount: 2}`, inspected as `msg.kind`.

Because messages are ordinary Hey values — and all values are

immutable — sending can never create a shared mutable path between

sender and receiver, whatever the actor's internal state does.

Workers: pure parallel computation

An **actor** is for *state*; a **worker** is for *work*. Workers are

stateless functions the runtime may run in parallel on many processor

cores. Because they hold no state between invocations and receive only

immutable inputs, any number can run simultaneously with no

coordination at all. Workers are how a Hey program saturates a

multi-core machine on data-parallel jobs.

Jobs and the scheduler

You do not create operating-system threads in Hey. The runtime

(Chapter 10) maintains a **worker pool** — a set of OS threads, sized

to the machine — and schedules **jobs** (runnable pieces: an actor

with a non-empty mailbox, a worker invocation) onto that pool. Idle

threads *steal* queued jobs from busy ones, keeping cores evenly

loaded without any program-side tuning. This is the same

"many-lightweight-tasks on few-threads" architecture as Go's

goroutines or Erlang's processes, specialized by the guarantee that

tasks never share mutable memory.

Streams: pipelines of data in flight

A **stream** connects stages of a data pipeline: a source produces

values, worker-backed stages transform them (`map`, `filter`), and the

results are collected — in arrival order or original order, your

choice. Streams give the everyday "process this collection in

parallel" pattern a name, so programs do not hand-build actor networks

for it. From `examples/tour_06_streams.hey`:

let values = [1, 2, 3, 4]
  .map(double, workers: 2)
  .filter(greater_than_four, workers: 2)
  .collect()

Each `.map`/`.filter` stage names the function to apply and how many

workers run it; `.collect()` gathers the results. Behind the fluent

syntax these are ordinary worker pools fed in sequence.

What is deliberately missing

Hey has **no shared mutable memory** and **no general-purpose

channels** (the free-floating pipes some languages provide, which

reintroduce shared-state reasoning through the back door: who reads?

who closes it? in what order?). Every concurrency need routes through

values + actors + workers + jobs + streams. That short list is a

design bet: fewer primitives, each with a strong guarantee, beat a

large toolbox where safe use is the programmer's problem.

Discussion questions

1. Immutability makes message *sending* safe. Explain precisely why:

what goes wrong if a language lets you send a pointer to a mutable

structure?

2. An actor processes messages one at a time — a potential

bottleneck. How would you restructure a hot counter actor into many

actors plus a combining step? (This is the map-reduce idea.)

3. Work stealing balances load automatically. Invent a workload where

stealing behaves badly, and suggest a refinement.
