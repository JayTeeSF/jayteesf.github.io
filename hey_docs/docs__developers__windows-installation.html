Install and verify Hey on Windows x64

> **Current scope:** this ZIP is a native runtime promotion test, not the final Hey compiler installer. It temporarily discovers or installs Microsoft C++ Build Tools so Windows itself compiles the receipt. The promoted product archive must instead contain `hey.exe`, its LLVM/linker/runtime closure, and require no Visual Studio installation.

Hey's native Windows promotion test is distributed as a small ZIP. You do not need Git, Make, a Unix shell, a complete Hey source checkout, a specific ZIP filename, or a specific extracted-folder name on the Windows machine. Extract it anywhere and double-click `START-HERE.cmd`; do not type PowerShell commands.

Administrator rights are not required to execute the test. If Microsoft C++ Build Tools are absent, Windows may show a UAC prompt only for that prerequisite installation. Running the test as Administrator does not bypass CrowdStrike, Defender, AppLocker, or WDAC.

When the test completes or is blocked, it automatically creates `hey-windows-results.zip` in the extracted folder. Return that one ZIP; it contains the log and every evidence file that exists.

Endpoint-security and application-control failures

A successful compile followed by `Program ... failed to run: Access is denied`

means Windows refused to launch the generated executable. On managed machines,

CrowdStrike, Defender, AppLocker, WDAC, or another endpoint policy may make that

decision before the program enters `main`.

The current bundle therefore:

1. compiles a minimal kernel32-only launch probe;

2. stages it under `%LOCALAPPDATA%\Programs\Hey\PromotionTest\VERSION`

instead of executing from Downloads;

3. records SHA-256, Authenticode status, and Zone.Identifier presence;

4. runs the launch probe before the complete runtime receipt;

5. writes `windows-native-execution-blocked.txt` when launch is denied;

6. keeps `windows_promoted=false` unless native execution actually succeeds.

Do **not** disable CrowdStrike, Defender, or application-control policy. Send

back these files when present:

windows-test.log
windows-native-compilation-evidence.txt
windows-native-execution-blocked.txt

If the minimal probe is also blocked, use `./bin/hey windows azure-create` for

a clean temporary host or obtain an enterprise-approved signing certificate or

policy exception.

Optional enterprise Authenticode signing

The promotion harness can sign generated test executables when the current

Windows account has access to an approved code-signing certificate:

$env:HEY_WINDOWS_CERTIFICATE_THUMBPRINT = "CERTIFICATE_THUMBPRINT"
$env:HEY_WINDOWS_TIMESTAMP_URL = "https://approved-rfc3161-timestamp-service"
.\START-HERE.cmd

An explicit `HEY_WINDOWS_SIGNTOOL` path is optional. Signing uses SHA-256,

RFC3161 timestamping, and `signtool verify /pa`. A self-signed certificate or a

signature that is merely present is not a production trust decision.

Fastest route: temporary Microsoft Azure Windows VM

This route creates a short-lived Windows Server VM hosted by Microsoft, restricts RDP and SSH to your current public IP, copies the test ZIP, and opens an RDP file on macOS.

Requirements

  • an Azure account with an active subscription;
  • Homebrew on the Mac;
  • Microsoft Windows App or another RDP client on the Mac.

From the Hey repository, run:

./bin/hey windows azure-create

The helper performs these steps visibly:

1. installs Azure CLI with Homebrew when it is absent;

2. opens `az login` when authentication is required;

3. builds and validates `hey-windows-native-test.zip`;

4. creates a Windows Server 2022 x64 VM;

5. restricts RDP and SSH to the Mac's current public IP;

6. installs Microsoft OpenSSH Server through Azure Run Command;

7. copies the ZIP into the Windows user's home directory;

8. writes and opens `~/Downloads/hey-windows-test.rdp`.

Azure asks for the Windows password while creating the VM. Enter the same password when `scp` asks for it and when RDP connects.

Inside the Windows desktop:

1. open **File Explorer**;

2. open `C:\Users\heytester` unless `HEY_AZURE_ADMIN` selected another user;

3. right-click `hey-windows-native-test.zip` and choose **Extract All**;

4. open the extracted folder;

5. double-click **`START-HERE.cmd`**;

6. approve a UAC prompt only if Windows needs to install Microsoft C++ Build Tools.

Check the remote VM at any time with:

./bin/hey windows azure-status

After saving the receipt and log, delete the VM, disks, public IP, and network resources immediately:

./bin/hey windows azure-delete

The VM and public IP can incur Azure charges while they exist. Deleting the complete resource group is the supported cleanup operation.

Local or existing Windows machine

1. Build the Windows test ZIP on macOS or Linux

From the Hey repository:

./bin/hey windows runtime-test-bundle \
  --out "$HOME/Desktop/hey-windows-native-test-v$(./bin/hey --version).zip"

The command packages only the Windows runtime sources, the native receipt, the visible-progress test scripts, and the one-click Windows launcher.

2. Copy the ZIP to Windows

The easiest choices are a shared folder, Remote Desktop file copy, USB storage, or cloud storage.

A shared folder, Remote Desktop file copy, USB storage, or cloud storage avoids any Windows command-line setup. Advanced users may also use an already-configured SSH service.

3. Run the test on Windows

1. Open **File Explorer**.

2. Find `hey-windows-native-test.zip`.

3. Right-click it and choose **Extract All**.

4. Open the extracted folder.

5. Double-click **`START-HERE.cmd`**.

6. Approve a UAC prompt only if Windows needs to install Microsoft C++ Build Tools.

The `START-HERE.cmd` launcher derives its root from its own `%~dp0` location, calls `RUN-WINDOWS-TEST.cmd` in the same extracted folder, and resolves `bin\hey-windows-install-and-test.ps1` from that root. Neither the ZIP nor the folder needs a version suffix or any other specific name. The v0.99.277a launcher incorrectly moved to the parent directory, which produced the exact `...\Downloads\bin\hey-windows-install-and-test.ps1 does not exist` failure. v0.99.278a and later fix that path and fails early with an extraction-specific message if the ZIP contents are incomplete.

The launcher installs Microsoft C++ Build Tools and the Windows SDK through WinGet when they are absent. It initializes the x64 developer environment, compiles the maintained Win32 runtime receipt, and executes it.

Every stage is printed before it runs. Compiler output and receipt execution output stream to the terminal instead of being held until completion. The same lines are appended to:

windows-test.log

The CMD window pauses on success or failure. A failed compiler, executable, prerequisite installer, or PowerShell script therefore leaves the last completed stage visible and preserves the log.

The first run may install a large Microsoft toolchain. If the installer requests a restart, restart Windows and double-click `START-HERE.cmd` again.

The Build Tools payload is roughly a full native C/C++ toolchain and can take a long time even after the download meter reaches 100%, because hundreds of packages are verified and installed. The returned screenshots showing `STAGE install-build-tools`, the Visual Studio Installer progress window, and no `HEY WINDOWS TEST FAILED` line indicate that the corrected launcher is now on the intended path. Do not start a second installer concurrently. Let the first run finish; if Windows requests a reboot, reboot once and double-click `START-HERE.cmd` again.

This installation is temporary promotion scaffolding. It will be removed from the final Windows language archive.

Current component-matrix run

The latest returned v0.99.286a evidence is more specific than the first access-denied run:

  • the minimal kernel32-only launch probe executes successfully from the Hey user-install area;
  • the richer aggregate runtime receipt compiles and links;
  • endpoint policy denies that aggregate before `main`.

v0.99.291a runs a control image plus every independently compiled receipt, and continues the matrix after a policy block:

launch probe -> control -> platform -> process -> IOCP -> HTTP -> managed -> aggregate

The control uses the same component-receipt scaffold but no Hey runtime API. If it is blocked after the launch probe passes, the evidence points to policy shared by the component-image form rather than a specific runtime API. If control passes, the first later block isolates the runtime boundary.

Return `windows-native-component-evidence.txt`, `windows-endpoint-policy-evidence.txt`, and the log. Each attempt records a distinct SHA-256, Authenticode state, Zone.Identifier state, and available Code Integrity, AppLocker, Defender, and CrowdStrike/Falcon Application events. The aggregate omits the separately tested unused managed object and does not link `winhttp.lib`; WinHTTP is loaded dynamically. Keep CrowdStrike and other endpoint protection enabled.

Returned `kernel32.lib` / C1083 failure

The first complete Build Tools run reached `cl.exe` but the verifier omitted the MSVC `/link` argument boundary. The compiler consequently treated `kernel32.lib`, `ws2_32.lib`, and `winhttp.lib` as C source files and reported C1083. v0.99.280a placed `/link` before the required system libraries. Regenerate the bundle and rerun it; the existing Build Tools installation should be discovered and reused.

Double-clicking `START-HERE.cmd` is the only supported user action. The CMD launcher may invoke Windows PowerShell internally for environment discovery, prerequisite installation, stage logging, and receipt creation; the user does not open or type commands into PowerShell.

Successful receipt

A successful run prints:

windows native runtime receipt ok platform=true process_child=true files=true mappings=true threads=true iocp=true winhttp=true system_trust=true
windows native promotion evidence ok compiler=... executable=...
HEY WINDOWS TEST PASSED

Send back this one file:

hey-windows-results.zip

What the receipt proves

The executable test covers the native Windows x64 boundaries for:

  • UTF-16 paths, executable discovery, and clocks;
  • threads, events, waits, and handle closure;
  • generated child-process launch, wait, and exit-status preservation without `cmd.exe`, PowerShell, or PATH lookup;
  • file reads, writes, seeks, and mappings;
  • Winsock startup and IOCP completion;
  • WinHTTP loading, bounded timeout configuration, and Windows system certificate trust.

Cross-compiled COFF objects are not enough to promote Windows. Promotion requires this compiled-and-executed target-host receipt.

How Windows users will use the language

A GUI is not required. Windows developers commonly run compilers from Windows Terminal, PowerShell, Command Prompt, Visual Studio Code, or another IDE terminal. The intended installed commands are:

hey .\hello.hey
heyc build .\hello.hey -o .\hello.exe
.\hello.exe

The final Windows archive should install under `%LOCALAPPDATA%\Programs\Hey\VERSION`, add its `bin` directory to the user `PATH`, and include a precompiled runtime plus contained LLVM/LLD components. Optional `.hey` file association, editor extensions, and a desktop IDE can be added later without changing the command-line language model.

Generate the test ZIP from the source checkout

Language maintainers should generate the target-host bundle locally rather than

receiving a pre-generated ZIP:

cd "$HOME/dev/hey-lang-bootstrap-plan"
./bin/hey windows runtime-test-bundle \
  --out "$HOME/Desktop/hey-windows-native-test-v$(./bin/hey --version).zip"
