Self-hosted emitter tier analysis

This note captures the current self-hosted blocker cluster so it does not get

rediscovered or papered over by deleting specs.

Current status

The self-hosted backend is direct for the tracked compatibility suite

(`supported=12 unsupported=0`, `bridged_count=0`). v0.99.32a reduces the

full-repository **emitter-self-hosting tier** by constant-folding compile-time

`hey_c.emit_text(...)` / `hey_c.emit_file(...)` calls when their arguments are

known strings, and by embedding current coverage/decision summaries as data.

The remaining tier is still real backend/compiler work. The unresolved rows

import or depend on one of:

  • `stdlib/hey_c.hey`
  • `stdlib/hey_dual_suite.hey`
  • `stdlib/hey_compiler_strategy.hey`

The historical symptom in generated self-hosted C was a module-qualified call

left in source form, for example:

HeyValue emitted = hey_value_clone(hey_c.emit_text("inline", source));

That is invalid C because `hey_c` is not a C object. The correct lowered call

shape, once the imported module is compiled into the self-hosted output, is a

canonical symbol such as:

hey_fn_hey_c__emit_text(...)

Is `hey_c.emit_text(...)` bad syntax?

No. `hey_c.emit_text(...)` is valid Hey module-qualified function syntax. It is

supported by native-c today, and the repository intentionally contains concrete

examples/specs that exercise it:

  • `specs/language/440_hey_tiny_c_emitter.hey`
  • `specs/language/450_self_host_compile_pilot.hey`
  • `examples/hey_tiny_c_emit.hey`
  • `examples/self_host_compile_pilot.hey`
  • `tools/hey_tiny_c_emit.hey`
  • `tools/hey_self_compile_pilot.hey`

These specs are not obsolete. They are the regression floor for the language's

module function contract and for the self-hosting compiler path.

Why native-c works while self-hosted fails

Native-c follows imports and emits the imported Hey module functions into the C

program, so `hey_c.emit_text(...)` is compiled as a real function call.

The self-hosted path deliberately still treats `hey_c.hey` and friends as an

unclosed tier. Simply whitelisting `hey_c.hey` makes the self-hosted compiler try

to parse and emit its own large emitter. That exposes both a performance problem

and remaining self-emission correctness risks.

Measured locally on v0.99.29a, a naive `hey_c.hey` whitelist attempt did not

finish the targeted `specs/language/440_hey_tiny_c_emitter.hey` self-hosted row

within the slice budget. The earlier v0.99.12a ledger also recorded that every

`hey_c`-importing row could cost minutes when the emitter compiles itself. The

parallel backend-world gate makes the project loop faster, but it does not make

that tier shippable by itself.

Why not route to native-c or remove specs?

Do not route these rows through native-c and do not remove the specs. That would

hide the exact parity gap this tier exists to close.

Allowed directions:

1. Make the self-hosted compiler able to compile the imported module directly.

2. Make the self-hosted compiler emit the shared typed IR, then let the existing

backend consume that IR.

3. Ledger a row as an intentional M4 exclusion only if the exact source row,

reason, and replacement milestone are named.

Current partial closure

v0.99.31a fixed a related but separate front-door gap: the C REPL can now recursively load real stdlib source imports, so `import "stdlib/hey_c.hey"` followed by `hey_c.emit_text('inline', source)` works interactively. That proves the module syntax and imported definitions are valid at the REPL front door.

v0.99.32a closes the narrow compile-time `emit_text`/`emit_file` subset without a native bridge: if the source/path arguments are compile-time strings, the self-hosted emitter parses and emits at compile time using the same Hey-written emitter and embeds the resulting `{ok, c, error}` data into generated C. This is intentionally not the same as compiling the full imported `hey_c.hey` module into every output. The dynamic `hey_dual_suite.run()` timing harness and complete module-function emission remain M4/IR-emission work unless they can be closed directly and cheaply.

Work breakdown

E1: make problem rows first-class evidence

`bin/subcmds/hey-backend-world` now emits `problem_row=...` lines for every row

whose TSV status is `output_mismatch`, `build_failed`, `build_timeout`, or

`run_failed`. That means the final full-gate command can stay short while still

capturing source paths for regressions.

E2: isolate the self-hosted output mismatch

The summary line alone is not enough. The next full gate should show a

`problem_row` line like:

problem_row=N source=... backend=self-hosted build=ok run=ok output=output_mismatch ...

Once the source is known, compare:

cat build/backend-world-full/outputs/<safe>.native.out
cat build/backend-world-full/outputs/<safe>.self.out
cat build/backend-world-full/outputs/<safe>.self.diff

Treat the row as a backend bug unless native-c is proven to violate the spec's

`expect stdout` contract.

E3: reduce the `hey_c.emit_text` tier to smaller specs

Before whitelisting `hey_c.hey`, add/keep micro-specs for the exact language

contracts used by the emitter:

  • module-qualified function calls compile to canonical symbols
  • imported module functions are available to program rows
  • call-result index access remains supported
  • nested module functions preserve names
  • string-builder accumulator loops do not clone on every append
  • unsupported RHS in a `let` fails loudly instead of leaking names

E4: choose the structural close

Preferred M4 close: self-hosted emits the same typed IR consumed by LLVM/native

backend infrastructure. That avoids making the Hey-written C emitter compile its

own C emitter as the long-term architecture.

Short-term close is allowed only if the measured per-row self-hosted build time

is acceptable and the generated C stays direct (no native-oracle bridge marker,

no hidden fallback).
