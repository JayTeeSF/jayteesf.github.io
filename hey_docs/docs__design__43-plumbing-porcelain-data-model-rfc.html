RFC v0.43: Plumbing, Porcelain, Data Model, and Concurrency Boundaries

Status: accepted as the Hey-lang architecture standard in v0.44.0. See also `docs/architecture-standard.md`.

Problem

Hey-lang has been drifting toward syntax for every good idea: actors, supervisors, jobs, queues, JSON helpers, async I/O, benchmark workflows, and more. That is a mistake. Syntax is expensive. Runtime/library structure is cheap and composable.

Git is the model: keep a small set of stable plumbing operations, then build friendly porcelain on top. Git's user-facing commands are porcelain; its low-level commands expose inner machinery and can be chained or scripted. Hey-lang should do the same.

Rule

Do not add syntax for a feature if the feature can be expressed as a library over existing language plumbing.

Syntax is reserved for:

  • module and function definition;
  • local binding and assignment;
  • control flow;
  • literals for the core value model;
  • function call and method-style call sugar;
  • low-level concurrency primitives only where the runtime needs special scheduling or memory semantics.

Everything else starts as a library.

Plumbing

The language/runtime plumbing should be:

values
functions
modules
imports
records/maps/arrays/strings/bytes/numbers/bools/nil
shape metadata for records/maps/modules where useful
actors
mailboxes
send/receive
links/monitors
supervisors
I/O targets
scheduler hooks
allocation/benchmark introspection

Porcelain

Porcelain belongs in the standard library or packages:

json.encode / json.decode
job queues
perform_later / perform_at
workflow chains/rings
retry policies
queue scaling policies
stream processors
1BRC helpers
HTTP server/client
process pipes

Porcelain can be excellent and batteries-included. It should not block users from building a different porcelain on the same plumbing.

Jobs and queues

Jobs are not new syntax for v1.

A job system should be a library over modules, functions, actors, supervisors, and serializable payload values. A Sidekiq-like package should be possible without fighting the language.

Recommended v1 library shape:

import jobs

module ImportCards
  fn perform(payload)
    let deck_id = payload['deck_id']
    return {ok: true, deck_id: deck_id}
  end
end

program
  let queue = jobs.queue({name: 'local/imports', min_workers: 1, max_workers: 16, scale_on: {latency_ms: 500}, retries: 5})
  let job_id = jobs.perform_later(queue, ImportCards.perform, {deck_id: 123})
  says job_id
end

`job_id` should be a library value, probably wrapping queue namespace plus actor/supervisor identity. It should not require special language syntax.

Persistence

Do not implement JSONL/local database persistence as the first queue backend.

In-memory supervised queues are the first target because they preserve async/runtime semantics without pretending distributed synchronization is solved. Persistent distributed queues require explicit semantics for ownership, leases, duplicate execution, replay, ordering, failover, and clock/partition behavior.

A JSONL queue is allowed later as a package/backend, but not as the default semantics until the distributed story is clear.

JSON and data model

Hey-lang should not make a particular JSON standard the deep language core.

Instead:

  • the language core owns maps/arrays/strings/bytes/numbers/bools/nil;
  • JSON is a standard-library codec over those values;
  • today’s JSON literal syntax is kept as convenient map/array literal syntax;
  • `to_json`, `json.encode`, and `json.decode` are library/runtime services;
  • future JSON variants or typed JSON standards should be implemented as codecs/packages, not as language-breaking syntax changes.

This preserves convenience while keeping the language from being trapped by future JSON evolution.

Shapes

The optimizer should track shapes for map/record-like values.

A shape is runtime/compiler metadata describing a stable layout or key-set, for example:

{name, visits}
{event, data}
{ok, deck_id, next}

Shapes should accelerate common operations:

  • string-key get/put/has;
  • object construction;
  • JSON encode/decode;
  • message payload dispatch;
  • typed job payload validation;
  • actor/job mailbox routing.

Shapes are plumbing. JSON codecs, job payload schemas, and user record abstractions are porcelain built on shapes.

Decision checklist for new features

Before adding syntax, answer:

1. Can this be implemented as a normal function/module over existing plumbing?

2. Does this require runtime scheduling, memory, or parser support that libraries cannot express?

3. Would third-party libraries be prevented from implementing alternatives?

4. Can the feature be made excellent as standard-library porcelain first?

5. Would adding syntax make future standards or alternate implementations harder?

If answers 1, 3, or 4 are yes, keep it out of syntax.

Implementation standard

Starting in v0.44.0, roadmap items must identify whether they are language plumbing or library porcelain. JSON codecs and job/queue systems are library porcelain unless a missing plumbing primitive blocks them.
