Hey developer documentationAudience-specific Hey documentation

Hey tutorial

Hey is an alpha compiled language for small, readable programs that still want native builds, actor-style concurrency, immutable data, built-in specs, and a path toward self-hosted tooling.

Hey's quickest story is web-shaped: describe a route, handle a request with normal functions, return immutable response values, and move expensive work to Jobs when needed. The official web path is `stdlib:web`.

This tutorial teaches the application-facing surface. Lower-level runtime, backend, and release-tracking details belong in the human maintainer records.

For deeper application work, continue with stateful files and mappings, processes and networking, and functions and callable values.

How to read examples

Each larger example starts with a small contract block before the code:

  • **What it does** explains the user-facing behavior.
  • **Important calls** lists the main stdlib calls and what shape they return.
  • **Expected output** shows what you should see.

Hey has a first alpha signature surface for cataloged symbols: `./bin/hey doc SYMBOL` and `./bin/heyc --describe SYMBOL`. Tutorial examples still document argument shapes inline while docs, CLI describe, and compiler contracts converge on one metadata source.

Before choosing names, read Hey reserved words and backend-safe names. Hey syntax words and backend-only C11 keywords are deliberately different categories.

Hello

**What it does:** prints one line and also acts as its own test.

Create `hello.hey`:

spec 'hello'
  expect stdout "Hello, Hey!\n"
end

program
  says 'Hello, Hey!'
end

Run it:

./bin/heyc hello.hey --run
./bin/heyc hello.hey --test

Expected output from `--run`:

Hello, Hey!

`program` is the entry point. `spec` is an executable contract for the file.

A tiny web app over a real socket

Create `hello_web.hey`:

import 'stdlib:web'

module GreetingController
  fn show(request)
    return web.json({message: 'Hello from Hey!', path: request.path})
  end
end

program
  let routes = web.routes([
    web.get('/', GreetingController.show)
  ])

  web.serve(routes, {
    host: '127.0.0.1',
    port: 3072,
    workers: 0
  })
end

Start it in one terminal:

./bin/heyc hello_web.hey --run

Call the live TCP server from another terminal:

curl http://127.0.0.1:3072/

Expected response:

{"message":"Hello from Hey!","path":"/"}

`workers: 0` asks the runtime to use its online-CPU default. Keep socket workers focused on request parsing and responses; enqueue slow or retryable work through Jobs. `web.service(routes, options)` returns a composable service descriptor. Pure `web.dispatch(...)` remains useful for fast controller specs, but both are test/composition boundaries—not substitutes for the live `web.serve(...)` server demonstrated above.

See the web stdlib guide and the `hey_web` package entry for reusable server/client commands.

Strings

Use single quotes for ordinary strings. Use double quotes when you need interpolation or escape sequences.

program
  let name = 'Ada'
  says 'plain text'
  says "hello #{name}"
end

Expected output:

plain text
hello Ada

Values, bindings, and interpolation

**Important calls and operators:**

FormMeaning
`let name = value`creates a binding in the current scope
`set name = value`rebinds an existing local name
`#{expr}`interpolates an expression into a double-quoted string
`nil`empty/no-value sentinel
`==`equality comparison
program
  let name = 'Ada'
  let answer = 40 + 2

  says "Hello, #{name}!"
  says "answer: #{answer}"
  says answer == 42
  says nil
end

Expected output:

Hello, Ada!
answer: 42
true
nil

Hey source currently avoids visible type ceremony. The compiler and runtime still collect type and shape facts for optimization and backend lowering.

There is no dedicated `const` form yet. Treat `let` bindings as normal scoped bindings today; any future `const` syntax must include parser, REPL, HIR/MIR/LLVM, and diagnostic contracts.

Immutable arrays, maps, and collection helpers

For the complete lookup and collision rules, see Object field access.

Hey arrays and maps are immutable values. Updating a collection returns a new value.

**Important calls and operators:**

FormMeaning
`[...xs, value]`returns a new array with `value` appended
`{...obj, key: value}`returns a new map with updated fields
`xs[index]`reads an array item
`obj.field`reads a field from a map
`value.to_json`renders a Hey value as JSON text
program
  let names = ['Ada', 'Grace']
  let more = [...names, 'Katherine']
  says more.to_json
  says more[1]

  let user = {name: 'Ada', visits: 1}
  let next = {...user, visits: user.visits + 1, city: 'SF'}
  says next.to_json

  # Original values are unchanged.
  says names.to_json
  says user.to_json
end

Expected output:

["Ada","Grace","Katherine"]
Grace
{"name":"Ada","visits":2,"city":"SF"}
["Ada","Grace"]
{"name":"Ada","visits":1}

For higher-level collection algorithms, import `stdlib:collections`. Receiver-oriented functions then read naturally as `names.uniq`, `names.includes?('Ada')`, `items.chunk(20)`, or `records.group_by_key('team')`. Constructors without a receiver, such as `collections.range(2, 5)`, remain module calls. See Collection method syntax.

JSON is a boundary

Hey object and array literals are core values. JSON is a codec over those values.

**Important calls:**

CallArgumentsReturns
`json.decode(text)`JSON textHey object/array/string/number/bool/nil value
`json.encode(value)`Hey valueJSON text
`keys(map)`maparray of string keys
import 'stdlib:json'

program
  let user = json.decode("{\"name\":\"Ada\",\"visits\":1}")
  let next = {...user, visits: user.visits + 1}

  says json.encode(next)
  says next.keys.to_json
end

Expected output:

{"name":"Ada","visits":2}
["name","visits"]

This keeps the language data model independent from any one wire format while preserving convenient JSON-shaped literals.

Functions

Function names may end in `?` for predicates or `!` for operations whose effects

deserve attention. These are naming conventions, not hidden behavior; see

Callable name suffixes.

**Important forms:**

FormMeaning
`fn name(args...) ... end`defines a function
`return value`returns immediately
final expressioncan be the function result in the supported subset
fn greet(name)
  return "Hello, #{name}!"
end

fn badge(name, visits)
  "#{name}: #{visits}"
end

program
  says greet('Ada')
  says badge('Grace', 2)
end

Expected output:

Hello, Ada!
Grace: 2

A bare `return` exits the current function/program with no explicit value, which behaves as `nil` in value contexts.

Functions and actor definitions are first-class values. They can be stored in arrays and objects, sent through actor messages and Jobs payloads, invoked or spawned dynamically, and partially applied with `bind`. See Functions and callable values. Do not use function-name strings as a substitute for callable values.

Scope, modules, and imports

Modules group related functions and actors. Imports link definitions from another source file into the entry program.

Current scope rules are intentionally conservative:

  • `let` bindings created inside `program`, `fn`, `actor`, loops, or branches are local to that block/function-style scope.
  • Module members live in their module namespace and should be called with the module prefix, such as `Greeter.hello(user)`.
  • Imports make another file's definitions available, but imported module members are still module-qualified.
  • There is no user-facing global variable scope today. Top-level source should contain imports, modules, actors, functions, specs, and one `program` entry point, not mutable global state.
  • There is no implicit main/root namespace automatically injected into every module. Shared state should be passed as values or returned from functions.

`lib/greeter.hey`:

module Greeter
  fn hello(user)
    return 'Hello, ' + user.name
  end
end

`main.hey`:

import 'lib/greeter.hey'

program
  let user = {name: 'Ada'}
  says Greeter.hello(user)
end

Expected output:

Hello, Ada

Modules plus immutable maps are the current organization path. Hey does not need classes to start building useful programs.

Python-like import conveniences such as `from Greeter import hello`, import aliases, or explicit re-export lists are useful future ergonomics, but they should be added deliberately with collision diagnostics and canonical-symbol support. Until that slice lands, examples should prefer explicit module prefixes because they make scope and ownership obvious.

Control flow and loops

Use `let` for a new binding and `set` to rebind a local name. `if`/`else`, `while`, `until`, and `for ... in` are the everyday control-flow forms.

**Important forms:**

FormMeaning
`if expr ... else ... end`branch on truthiness
`while expr ... end`repeat while a condition remains true
`until expr ... end`repeat until a condition becomes true
`for name in collection ... end`iterate over an array-like value
`fail message`stop with a runtime error
`return value`exit the current function/program
program
  let total = 0

  while total < 2
    set total = total + 1
  end

  until total == 3
    set total = total + 1
  end

  for name in ['Ada', 'Grace']
    says 'hi ' + name
  end

  if total == 3
    says 'done'
  else
    fail 'unexpected total'
  end
end

Expected output:

hi Ada
hi Grace
done

For hot loops, keep scalar counters and strings in simple locals. That gives HIR/MIR analysis and LLVM code generation the best chance to stay on unboxed typed lanes instead of falling back to boxed dynamic values.

I/O

The complete guides cover the common I/O model, stateful files and mappings, and processes, pipes, TCP, UDP, readiness, and cancellation.

Hey has small public I/O statements today.

**Important calls/statements:**

FormMeaning
`says value`stdout plus newline
`print value`stdout without newline
`warn value`stderr plus newline
`write value`stdout without newline
`write path, text, mode`writes text to a file
`log path, text`appends text to a log file
`read(path)`reads a file as text
program
  says 'line'
  print 'partial'
  write ' value'
  says ''
  warn 'careful'

  write 'build/tutorial.txt', "first\n", 'w'
  log 'build/tutorial.log', 'started'

  let text = read('build/tutorial.txt')
  says text
end

The architecture goal is target-oriented I/O: files, sockets, buffers, and event loops plug into stable plumbing while higher-level convenience belongs in libraries.

Actors

Start with the progressive concurrency and Jobs guide.

Actor names may also end in `?` or `!`, including module-qualified forms such as

`spawn Workers.writer!`.

Actors own state and communicate with messages.

**Important forms:**

FormMeaning
`actor name(args...) ... end`defines an actor
`spawn actor(args...)`starts an actor and returns a pid/reference
`send pid, message`sends a message
`receive msg`binds the next received message
`join pid`waits for actor termination
`exit`exits the current actor
actor counter(label)
  state count = 0
  receive msg

  if msg == 'inc'
    set count = count + 1
    says label + ': ' + count
  else
    if msg == 'stop'
      exit
    end
  end
end

program
  let visits = spawn counter('visits')
  send visits, 'inc'
  send visits, 'inc'
  send visits, 'stop'
  join visits
end

Expected output:

visits: 1
visits: 2

Actors do not share mutable objects. Current runtime behavior clones values into FIFO mailboxes. The language direction is copied, immutable, uniquely moved, or explicitly share-safe messages.

Actors can be spawned directly or selected through first-class actor-definition values. Actor handles, definitions, callables, and arrays of callables may all cross mailboxes. See Actors and mailboxes and Functions and callable values.

Supervisors and jobs

The concurrency and Jobs guide builds from hello-world actors to supervised workers and scalable queues with latency goals.

The next step after plain actors is deciding how much infrastructure you want to own.

Use supervision when you want direct control over one actor's lifecycle. Use `stdlib:jobs` when you want queue-shaped background work with worker ownership, result delivery, retry policy, and shutdown handled by library porcelain.

Supervision: small runtime plumbing

actor once
  receive msg
  says 'handled ' + msg
  exit
end

program
  let sup = supervise once max 3
  send sup, 'one'
  sleep 0.5
  send sup, 'two'
  sleep 0.5
  stop sup
end

Expected output:

handled one
handled two

Jobs: stdlib porcelain over actors

**What it does:** starts a default job runtime, enqueues one import job, waits up to 1000ms for the result, prints the result, then stops the runtime.

**Important calls:**

CallArgumentsReturns
`jobs.start_default(worker_name)`actor name as textruntime object/pid bundle
`jobs.perform_with_id(runtime, client_id, payload)`runtime, string id, object payloadreceipt object with an id
`jobs.complete(job, value)`job message, result valuecompletion acknowledgement
`jobs.result_to(runtime, receipt, timeout_ms)`runtime, receipt object, timeout integerresult object `{ok, value, ...}`
`jobs.stop(runtime)`runtimeshutdown result

Pasteable REPL/file example:

import 'stdlib:jobs'

actor import_worker
  receive job
  jobs.complete(job, {message: job.payload.name + ' imported'})
end

program
  let runtime = jobs.start_default('import_worker')
  let receipt = jobs.perform_with_id(runtime, 'import-Ada', {name: 'Ada'})
  let result = jobs.result_to(runtime, receipt, 1000)
  says result.ok
  says result.value.message
  let stopped = jobs.stop(runtime)
end

Expected output:

true
Ada imported

The synchronous `result_to(runtime, receipt, timeout_ms)` form is tutorial porcelain. The actor-routing form is still available for programs that want actor-style result delivery: `jobs.result_to(runtime, receipt.id, sink)` sends the completed result to `sink`, and `join sink` is the synchronization point.

This is the plumbing/porcelain rule in practice: actors and supervision stay small and stable; queue policy, retries, durability, and result APIs live in the library.

Result values and expected failures

Use `fail` for bugs and actor crashes. Use Result values for expected failures.

**Important calls:**

CallMeaning
`result.ok(value)`success result
`result.error(message)`failure result
`result.is_ok(value)`true when success
`result.is_error(value)`true when failure
`result.value_or(value, default)`unwraps success or returns default
`result.message(value)`error message
import 'stdlib:result'

program
  let cfg = result.error('missing config')
  let ready = result.ensure(1 + 1 == 2, 'math broke')

  says result.value_or(cfg, 'default')
  says result.is_ok(ready)

  if result.is_error(cfg)
    warn result.message(cfg)
  end
end

Expected stdout:

default
true

Expected stderr:

missing config

CLI and safe process execution

For scripts, use `stdlib:cli` instead of hand-parsing flags.

import 'stdlib:cli'

program
  let args = ['serve', '--port=3072', '--verbose', '--no-cache', 'public']
  let parsed = cli.parse(args)

  says cli.command(args, 'help')
  says cli.parsed_int(parsed, 'port', 3072)
  says cli.parsed_bool(parsed, 'verbose', false)
  says cli.parsed_bool(parsed, 'cache', true)
  says parsed.positionals.to_json
end

Expected output:

serve
3072
true
false
["serve","public"]

When executing tools with user-provided values, prefer argv-shaped process execution over shell-shaped strings.

import 'stdlib:process'
import 'stdlib:text'

program
  let ran = process.run_argv(['/bin/echo', 'hello world'])
  says ran.ok
  says text.chomp(ran.stdout)
end

Expected output:

true
hello world

Text, regex-lite, collections, and debug helpers

The stdlib contains small batteries for common scripting work. Prefer these helpers over hand-rolled loops in user-facing examples.

import 'stdlib:collections'
import 'stdlib:text'
import 'stdlib:debug'
import 'stdlib:re'

program
  let names = ['Ada', 'Grace', 'Ada']
  let records = [{name: 'Ada', team: 'compiler'}, {name: 'Grace', team: 'compiler'}]

  says names.uniq.to_json
  says collections.range(2, 5).to_json
  says names.first(nil)
  says names.includes?('Grace')
  says records.group_by_key('team').to_json
  says text.replace('hello world', 'world', 'Hey')
  says text.strip_suffix('archive.zip', '.zip')
  says re.matches('^h.llo$', 'hello')
  says debug.typed_label('names', names.uniq)
end

Expected output:

["Ada","Grace"]
[2,3,4]
Ada
true
{"compiler":[{"name":"Ada","team":"compiler"},{"name":"Grace","team":"compiler"}]}
hello Hey
archive
true
names: array

`stdlib:re` is regex-lite, not full Perl/Ruby/Python regex parity. It supports anchors, wildcard dot, simple quantifiers, scan, replace, and selected character-class/alternation behavior. Captures/backreferences/lookaround are not the portable baseline yet.

Packages, validation, debug events, and lower-level service descriptors

This section demonstrates stdlib porcelain: readable APIs that create structured values. Normal web apps should start with `stdlib:web`; this section shows lower-level package/debug/net descriptors.

**What it does:** builds a package manifest, resolves a dependency, creates a validation result, formats a debug event, and describes an HTTP service.

**Important calls:**

CallArgumentsReturns
`packages.manifest(name, version, modules)`string, string, array of module namesmanifest object
`packages.source(name, modules)`string, arraysource object
`packages.with_source(manifest, source)`manifest, sourcemanifest with source metadata
`packages.registry(manifests)`array of manifestsregistry object
`packages.dependency(name, version_req)`string, version requirementdependency object
`packages.resolve(registry, dependency)`registry, dependencyresolution object with a `code` field
`errors.validation(field, code)`field name, error codevalidation error object
`errors.validation_result(errors)`array of validation errorsresult object with an `error` field
`debug.event(name, data)`event name, object metadatastructured debug event
`debug.log_line(event)`debug eventtext line
`net.listener(host, port, options)`host, port, object optionslistener descriptor
`net.route(method, path, handler)`method, path, handler nameroute descriptor
`net.http_service(listener, routes)`listener descriptor, array of routesservice descriptor
`net.service_summary(service)`service descriptorobject with summary fields such as `routes`
import 'stdlib:packages'
import 'stdlib:debug'
import 'stdlib:errors'
import 'stdlib:net'

program
  let manifest = packages.with_source(
    packages.manifest('cards', '1.0.0', .cards),
    packages.source('stdlib', .cards)
  )
  says packages.import_path(manifest, 'cards')

  let registry = packages.registry([manifest])
  let resolved = packages.resolve(registry, packages.dependency('cards', '=1.0.0'))
  says resolved.code

  let validation = errors.validation_result([errors.validation('name', 'required')])
  says validation.error

  let event = debug.event('compile', {files: 3})
  says debug.log_line(event)

  let listener = net.listener('127.0.0.1', 3072, {concurrency: 8})
  let service = net.http_service(listener, [net.route('GET', '/', 'home')])
  says net.service_summary(service).routes
end

Read this as a contract for application ergonomics. The package, validation, debug, and service values are ordinary Hey objects. The friendly API is the porcelain; lower-level details such as lockfile cache layout, diagnostic sinks, socket accept loops, TLS policy, and supervised listener workers are plumbing.

This example matters for LLVM because it forces the backend to support realistic language shapes: imports, module-qualified calls, nested calls, arrays, object literals, string fields, boxed values, indexing, and structured return values.

Production networking needs one additional step beyond descriptors. A clearer future shape would separate description from execution:

let service = net.http_service(listener, routes)
let server = net.serve(service)

Today, treat `net.http_service(...)` as “describe this service,” not “start serving internet traffic.”

Static files for service apps

The static-files layer is path-policy porcelain, not a complete HTTP server.

import 'stdlib:static_files'

program
  write 'build/public-ok.txt', 'ok', 'w'

  let good = static_files.read('build', '/public-ok.txt?cache=1')
  says good.status
  says good.body

  let blocked = static_files.resolve('build', '../secret.txt')
  says blocked.status
  says blocked.reason
end

Expected output:

ok
ok
error
unsafe-path

The current default rejects `..`, dotfiles, backslashes, colons, and percent-encoded input before reading. Future HTTP middleware still needs URL decoding, realpath/symlink confinement, MIME/cache/range/compression/sendfile, and actor-supervised socket integration.

EHY templates

`stdlib:ehy` is Hey's ERB-style template layer. Templates are plain text plus three tags:

TagMeaning
`<% code %>`statement/code block
`<%= expr %>`interpolate expression
`<%# comment %>`comment, dropped entirely
`-%>`trim the newline after the tag
import 'stdlib:ehy'

program
  let t = 'Hello, <%= name %>!' + "\n" + '<% for n in [1, 2] -%>' + "\n" + 'n=<%= n %>' + "\n" + '<% end -%>' + "\n"
  print ehy.result_with(t, {name: 'Ada'})
end

Expected output:

Hello, Ada!
n=1
n=2

`ehy.result_with(template, bindings)` turns each binding into a `let`, compiles the template to a standalone Hey program, runs it, and returns stdout.

Lower-level HTTP requests, responses, and routes

`stdlib:http` owns the reusable HTTP protocol and client foundation: request/response values, parser, serializer, URL parsing, bounded HTTP/HTTPS text requests, redirects, and tiny route matching. Plain HTTP uses `stdlib:net`; HTTPS uses certificate-validating `stdlib:tls`.

import 'stdlib:http'

program
  let raw = 'GET /posts/42?draft=1 HTTP/1.1' + "\r\n" + 'Host: example.test' + "\r\n" + "\r\n"
  let req = http.parse_request(raw)

  says req.method
  says req.path
  says req.params.draft
  says http.header(req, 'Host')

  let hit = http.route_match('/posts/:id', req.path)
  says hit.params.id

  let resp = http.response(200, {'content-type': 'text/plain'}, 'hello')
  print http.response_text(resp)
end

Expected first lines:

GET
/posts/42
1
example.test
42
HTTP/1.1 200 OK

Requests are immutable objects with fields such as `method`, `target`, `path`, `query`, `params`, `version`, `headers`, and `body`. Header names normalize to lowercase on parse. `http.response_text` emits CRLF line endings and an automatic `content-length`.

Client example:

let fetched = http.get('https://example.com/health', {timeout_ms: 2000, max_response_bytes: 65536, max_redirects: 3})
if fetched.ok
  says fetched.value.body
end

`https://` validates both certificate trust and the requested hostname and never silently becomes plain TCP. High-level response bodies remain buffered text; compressed/binary archive downloads still need the streaming byte-preserving download API described by `http.transport_capabilities()`.

Tutorial maintenance rules

A tutorial example is not complete unless it says what it does, explains the non-obvious calls, and shows expected output.

New examples should prefer stdlib porcelain over raw primitives unless the tutorial is teaching the primitive itself. Raw socket/file/process/actor/runtime examples belong in reference docs and primitive-focused specs.

When adding or changing a REPL-pasteable block, add or update a REPL spec. Product examples should use the default self-hosted HIR/MIR/LLVM compiler. Stage-0-only maintenance examples must be explicitly labeled as recovery or comparison tooling.
