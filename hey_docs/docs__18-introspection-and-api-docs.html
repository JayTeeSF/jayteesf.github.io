Introspection and API documentation plan

Hey's tutorial now documents argument shapes, return shapes, and expected output near complex examples. That is the correct teaching style for the current alpha surface, but it should not remain hand-maintained forever. The language and toolchain need a shared introspection layer so users can ask what a function expects, what it returns, and which fields an object-shaped result exposes.

Why this matters

Many Hey APIs are intentionally small porcelain over ordinary immutable values. That keeps the language approachable, but object-shaped return values can look opaque without nearby contract data:

let listener = net.listener('127.0.0.1', 8080, {concurrency: 8})
let resolved = packages.resolve(registry, packages.dependency('cards', '=1.0.0'))
says resolved['code']

The reader should not have to inspect stdlib source to learn that `listener` is a descriptor object or that `resolved` has a `code` field. The compiler, REPL, docs, tutorial, and future diagnostics should consume the same metadata.

Target user experience

The near-term user-facing commands should converge on this shape:

./bin/hey doc net.listener
./bin/heyc --describe net.listener

REPL parity should follow:

hey> help net.listener
hey> describe packages.resolve
hey> docs jobs.result_to

Expected output shape:

net.listener(host, port, options) -> listener descriptor

Arguments:
  host: string, e.g. '127.0.0.1'
  port: int, e.g. 8080
  options: object, e.g. {concurrency: 8}

Returns:
  object listener descriptor with fields:
    kind
    host
    port
    options

Metadata shape

The first version does not need a full static type system. It needs structured metadata that tools can read:

# conceptual target shape
@doc 'Create a listener descriptor.'
@signature 'net.listener(host: string, port: int, options: object) -> listener'
@returns 'listener fields: kind, host, port, options'
fn listener(host, port, options)
  return {kind: 'net_listener', host: host, port: port, options: options, transport: 'tcp'}
end

Until annotations exist, the checked-in foundation is explicit Hey data in `stdlib/hey_docs.hey`. That keeps the shape testable today and lets later slices move the data closer to each source function.

Implementation order

1. **Docs-first signature catalog.** Add structured metadata for tutorial-facing stdlib functions. Use it from tests and docs, even before parser-level annotations exist.

2. **CLI describe surface.** Add `./bin/hey doc SYMBOL` and `./bin/heyc --describe SYMBOL` over the shared catalog.

3. **REPL help surface.** Route `help SYMBOL`, `describe SYMBOL`, and `docs SYMBOL` to the same metadata.

4. **Shape-aware docs smoke.** Fail docs smoke if the tutorial mentions a signature that the catalog cannot describe, or if a cataloged signature is stale.

5. **Source-near annotations.** Move from centralized catalog entries to source-adjacent metadata once the parser supports doc attributes.

6. **Friendly field access.** Add `value.field` as readable sugar for object/map field access. It can initially lower to `value['field']`.

7. **Contract-aware diagnostics.** Use metadata and lightweight overload descriptors to explain wrong-argument calls, especially overloaded porcelain such as `jobs.result_to`.

8. **Generated API reference.** Generate reference docs from metadata so tutorials, stdlib reference pages, REPL help, and compiler diagnostics do not drift.

9. **Explain/trace mode.** Later, add an explanatory run mode that can show how tutorial examples flow through stdlib calls and return shapes.

Shape contracts to prioritize

The tutorial currently needs these symbols first:

• `jobs.start_default`

• `jobs.perform_with_id`

• `jobs.complete`

• `jobs.result_to`

• `jobs.stop`

• `packages.manifest`

• `packages.source`

• `packages.with_source`

• `packages.registry`

• `packages.dependency`

• `packages.resolve`

• `errors.validation`

• `errors.validation_result`

• `debug.event`

• `debug.log_line`

• `net.listener`

• `net.route`

• `net.http_service`

• `net.service_summary`

• selected `collections.*`, `text.*`, `re.*`, `result.*`, `json.*`, and `http.*` helpers used in tutorial sections

Relationship to LLVM parity

This is not separate from the LLVM plan. Structured introspection forces realistic backend coverage: imports, module-qualified calls, nested calls, arrays, object literals, string fields, boxed values, indexing, object-shaped returns, and future dotted field access. The metadata catalog and doc tool should therefore stay in the LLVM priority proof once the relevant language constructs are supported by the backend.

Current alpha contract

The tutorial must keep manually documenting call shapes until the catalog/CLI/REPL surfaces cover each block. Once a tutorial block depends on cataloged metadata, update both the tutorial and the metadata in the same slice. A tutorial example is not complete unless it says what it does, explains non-obvious calls, and shows expected output.
