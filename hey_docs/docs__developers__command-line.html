Hey developer documentationAudience-specific Hey documentation

Hey command-line guide

Hey keeps the public source-checkout command surface intentionally small. Start

with these commands rather than invoking files under `bin/subcmds` or

`tools/windows` directly:

bin/hey            run programs and select command families
bin/heyc           compile, check, test, and inspect Hey source
bin/hey-repl       interactive shell compatibility launcher
bin/test           repository verification suites
bin/perf           performance and profiling suites
bin/hey-zip-source bootstrap-safe complete source archive
bin/hey-apply-source-zip
                   safely apply a candidate source archive

Run the built-in catalog at any time:

./bin/hey help
./bin/hey help all
./bin/hey help target
./bin/hey help windows
./bin/heyc --help
./bin/test --help
./bin/perf --help

Files below `bin/subcmds` implement those command families. Files below

`tools/windows` are copied into the temporary Windows promotion ZIP. Both are

plumbing, not additional top-level user commands.

Run, build, check, and test programs

./bin/hey hello.hey
./bin/hey hello.hey one two three
./bin/hey repl
./bin/hey build hello.hey -o build/hello

./bin/heyc hello.hey --run
./bin/heyc hello.hey --test
./bin/heyc hello.hey --check
./bin/heyc build hello.hey -o build/hello
./bin/heyc build-dir examples -o build/examples

Application builds use the LLVM-hosted Hey frontend, semantic HIR, ABI MIR,

and direct LLVM consumer. Release products no longer force a native-C-hosted frontend. `--backend native-c` is Stage-0 comparison and

recovery machinery, not a second public application architecture.

`./bin/hey source.hey` compiles on the first run and directly executes the cached native application on unchanged later runs. `./bin/hey build source.hey -o build/app` publishes an explicit executable for services, bundles, containers, signing, and release deployment. See Running cached scripts and precompiled applications.

Compiler inspection commands are primarily for maintainers:

./bin/heyc ir hello.hey
./bin/heyc symbols hello.hey
./bin/heyc build-mir --backend llvm program.mir -o build/program
./bin/heyc llvm-ir-from-mir program.mir
./bin/heyc c-from-mir program.mir -o build/program.c
./bin/heyc toolchain
./bin/heyc --backends

`hey advise` is a read-only advisory for application authors:

./bin/hey advise program.hey

It reads the program's semantic HIR and reports sites where a value stays boxed and flows into a runtime-dispatch operation (today it covers boxed `*` sites; coverage grows with the dispatch doctrine), together with the exact conversion (`x.to_i`, `x.to_s`) that would make it static scalar code. It never builds or modifies anything.

Documentation and learning

./bin/hey doc SYMBOL
./bin/hey tour list
./bin/hey tour run actors
./bin/hey tour verify

./bin/hey challenge list
./bin/hey challenge show run-length
./bin/hey challenge check run-length path/to/solution.hey
./bin/hey challenge verify

./bin/hey learn build --out build/learning-portal
./bin/hey learn verify
./bin/hey learn profile

./bin/hey doc-site \
  --out "$HOME/dev/jayteesf.github.io/hey_docs" \
  --kind all \
  --theme water \
  --jobs 11

`rubyquiz` is a maintainer provenance tool, not a package of copied exercises:

./bin/hey rubyquiz --help

Packages, native extensions, and vendoring

./bin/hey package --help
./bin/hey package init --help
./bin/hey native --help
./bin/hey vendor --help

The command families have separate responsibilities:

  • `package` resolves complete project graphs, creates immutable locks, restores

exactly, verifies installed bytes, audits registry/release/archive identity,

registers package commands, and retains lower-level scaffold/pack/install

authoring primitives;

  • `native` generates and verifies typed native-extension ABI shims;
  • `vendor` copies an already constructed target distribution into an

application repository.

Domain libraries such as database drivers remain independently released

packages. They are not compiler subcommands.

Project package workflow:

./bin/hey package ensure --project . --update-lock
./bin/hey package ensure --project . --locked
./bin/hey package verify --project .
./bin/hey package audit --project .
./bin/hey package commands --project .

`verify` is offline. `audit` intentionally reacquires the registry, release JSON,

and archive and rejects same-version replacement. Unknown top-level commands are

looked up only in the lock-bound package command registry after complete-tree

verification.

Cross-target status matrix

Use one command to inspect every desktop target without confusing a structural

compiler check with native execution or signed publication:

./bin/hey target matrix
./bin/hey target matrix --json
./bin/hey target matrix --verify

The default form is fast and reports prerequisite and evidence status. `--json`

is intended for CI and release dashboards. `--verify` executes the maintained

Linux contained-SDK, Darwin relocated-linker, Windows cross-SDK, and Windows

click-to-run structural gates. A passing structural gate does not manufacture a

native target receipt, and a native target receipt does not imply signing or

publication readiness.

Product target archives

The `target` family creates the **actual self-contained compiler distribution**

for a supported operating-system/architecture pair:

./bin/hey target doctor --target darwin-arm64
./bin/hey target doctor --target linux-x64

./bin/hey target release \
  --target darwin-arm64 \
  --out "$HOME/Desktop/hey-VERSION-darwin-arm64.tar.gz"

./bin/hey target release \
  --target linux-x64 \
  --out "$HOME/Downloads/hey-VERSION-linux-x64.tar.gz"

`release` requires a matching native host and first runs the strict contained

SDK closure. Lower-level archive operations are available for release tooling:

./bin/hey target pack \
  --sdk /path/to/contained-sdk \
  --target linux-x64 \
  --out /path/to/hey-VERSION-linux-x64.tar.gz

./bin/hey target verify \
  --archive /path/to/hey-VERSION-linux-x64.tar.gz \
  --target linux-x64

Supported product targets currently are `linux-x64` and `darwin-arm64`.

`windows-x64` does not yet have a product compiler archive.

Temporary Windows promotion commands

The canonical Windows command while the product archive is unfinished is:

./bin/hey windows runtime-test-bundle \
  --out "$HOME/Desktop/hey-windows-native-test.zip"

`windows bundle` and `windows test-bundle` remain compatibility aliases, but

`runtime-test-bundle` states the artifact's purpose accurately. The ZIP is a

maintainer test harness that compiles and executes Win32 runtime receipts on a

Windows host. It is **not** equivalent to `target release`: it does not contain

the final Hey compiler and it temporarily uses Microsoft Build Tools.

Other Windows maintainer commands are:

./bin/hey windows sdk --out build/windows-cross-sdk
./bin/hey windows azure-create
./bin/hey windows azure-status
./bin/hey windows azure-delete

`windows sdk` proves structural PE/COFF construction from another host. Native

promotion still requires a Windows execution receipt. The Azure commands manage

a temporary clean target host.

Repository verification

Common focused suites:

./bin/test
./bin/test --full
./bin/test --repl
./bin/test --hir-mir
./bin/test --conformance
./bin/test --compiler-contract
./bin/test --dist-gates
./bin/test --docs
./bin/test --zip

Release blocks and the final promotion report are intentionally explicit:

HEY_RELEASE_JOBS=2 ./bin/test --release-c
HEY_RELEASE_JOBS=2 ./bin/test --release-a
HEY_RELEASE_JOBS=2 ./bin/test --release-b

time HEY_BACKEND_WORLD_JOBS=8 ./bin/test --promotion-report

The guarded current-version wrappers are also available through porcelain:

./bin/hey verify --focused
./bin/hey verify --full

Source checkout maintenance

./bin/hey clean
./bin/hey rebuild
./bin/hey selfc --help
./bin/hey zip --downloads

`hey-zip-source` and `hey-apply-source-zip` remain top-level because a receiving

checkout may need them before its installed or rebuilt `hey` command exists.

They are bootstrap exceptions to the normal porcelain rule.

Native iOS application

./bin/hey mobile ios native-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/dev/hey-ios-native-app" \
  --launch

This compiles ordinary Hey through semantic HIR, ABI MIR, and direct LLVM into

a real iOS Mach-O object, links a UIKit navigation/list/form application, and

calls native Hey action functions. `--plan-only` emits portable structural evidence without

claiming a target-Mac link or launch receipt.

Native iOS/iPadOS object foundation

./bin/hey mobile ios doctor
./bin/hey mobile ios sdk --out build/hey-ios-sdk

The SDK command emits actual Mach-O objects for arm64 devices and arm64/x86_64 simulators. It does not yet link an application, run a simulator/device, or produce a signed IPA.

Language-maintainer command workflow

The canonical commands for applying each source ZIP as its own commit, guarded

verification, local Windows bundle generation, vendoring, documentation

publication, and iPhone/iPad work are maintained in

`docs/humans/language-developer-workflow.md`.

Physical-device signing, IPA generation, and App Store publication remain separate gates.

Physical iPhone/iPad development deployment

./bin/hey mobile ios device-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/Desktop/hey-ios-device-app" \
  --device DEVICE_ID

This compiles an arm64 iPhoneOS object, links UIKit, development-signs the app,

and installs/launches it through Xcode `devicectl`. Run the command with

`--help` for provisioning variables and its exact evidence boundary. It does

not generate an App Store IPA or prove TestFlight/App Review.

LLVM-first architecture verification

./bin/hey verify architecture

This fast tier builds only the native bootstrap executable needed to run the

Hey-authored audit. It checks the complete native-source inventory, confines

duplicated semantics to the recorded migration debt, proves release frontend

selection is LLVM-hosted, and avoids the full self-host/release matrix.
