v0.98.96a: decade web-service roadmap

Hey's backend parity work should serve a larger target than toy examples: a small, fast, actor-native web/service framework capable of rebuilding the useful parts of Rails while preserving Hey's immutability, supervision, and compact native/LLVM binary goals.

Rails-class capability target

A realistic Hey web stack needs these stdlib-first layers, each with implementation-independent language specs and backend-world fixtures before it can be treated as production-ready:

1. **HTTP/HTTPS server and client**

• Request/response data model, routing, middleware, streaming bodies, file uploads, cookies, redirects, compression, TLS, WebSocket/SSE, and structured logging.

• Actor-supervised listeners, acceptors, connection workers, and backpressure.

• Secure static-file serving must be built in, not left as an app footgun: root confinement, traversal rejection, dotfile policy, URL decoding, symlink/realpath checks, MIME types, cache headers, range requests, compression, and sendfile/runtime streaming need specs before production claims. v0.99.55a starts this with `stdlib/static_files.hey` path-policy helpers.

• Native-C first, then self-hosted, then LLVM lowering through general socket/event-loop primitives; no benchmark-specific C10K shortcuts.

2. **Routing and controllers**

• Immutable params/session/request contexts, route constraints, nested resources, before/after hooks, error pages, and content negotiation.

• Specs should start with tiny route matching and response-building contracts before larger blog-app examples.

• STARTED v0.99.91a: `stdlib/http.hey` covers the request/response data model, HTTP/1.1 parse/serialize, query params, case-insensitive headers, and `:param` route matching (specs 714/715). Still open: middleware pipeline, constraints, nested resources, hooks, content negotiation, and everything socket-facing.

3. **Templates / ERB-like rendering**

• A template parser, escaping policy, partials/layouts, locals, streaming render, and compile-to-Hey or compile-to-backend-IR path.

• Prefer a small ERB-compatible surface first: text chunks plus expression interpolation and escaped/raw output modes.

• STARTED v0.99.90a: `stdlib/ehy.hey` (EHY = Eval Hey) covers `<% %>`/`<%= %>`/`<%# %>` tags, `-%>` trim, bindings-as-locals, compile-to-Hey (`ehy.src`) and exact-stdout rendering (`ehy.result`), `.ehy` files via `tools/hey_ehy.hey`, with positive/negative specs 712/713. Still open: HTML escaping policy, partials/layouts, streaming render, compile-to-backend-IR.

4. **SQL and migrations**

• SQLite first for local/dev/single-binary workflows, then MySQL, and finally PostgreSQL adapters.

• Connection pools, prepared statements, transactions, schema migrations, query rows as immutable data, and actor-supervised pool workers.

• Backend contracts should be tiny: connect descriptor, prepare, bind, step, row decode, transaction, migration lock.

5. **Models and validation**

• Immutable record data, validations, associations, constraints, callbacks/events where they remain explicit and testable.

• Do not clone Rails magic wholesale; use Rails ergonomics as a benchmark while keeping Hey's data-model and actor supervision visible.

6. **Authentication and authorization**

• Password hashing, sessions, CSRF, SSO, OAuth/OIDC, SAML, MFA/TOTP/WebAuthn, signed tokens, rate limits, audit logs, and account recovery.

• Specs should isolate crypto/token/session contracts before web-controller examples.

7. **Jobs, mail, cache, and real services**

• Jobs are already a core stdlib target; add mailers, queue adapters, cron/scheduler, cache store, pub/sub, and service clients.

• Durable actors and supervised workers should be the common runtime model.

8. **Observability and deployability**

• Metrics, tracing, health checks, structured logs, config/secrets, graceful shutdown, hot restart where possible, container/static binary packaging, and no-toolchain distribution.

Why this belongs on the backend roadmap

Every layer above produces more lowering contracts: strings, bytes, sockets, TLS, parsers, templates, database rows, prepared statements, actors, timers, file I/O, and JSON/HTML output. These are not optional afterthoughts. They are the real-world pressure tests that should drive Native-C, self-hosted, and LLVM toward the same semantics.

Tutorial target

Once the stdlib pieces exist, extend `docs/tutorial.md` with a "recreate DHH's 15-minute blog demo" path:

• create app

• route resources

• generate a model/migration

• create/read/update/delete posts

• render templates and partials

• add auth

• add background jobs

• deploy as a small native binary

Until then, keep the tutorial honest: show the current language and stdlib features first, then point to this roadmap for the Rails-class target.

Concrete blockers surfaced by the first real-app targets

These blockers stand between today's backend parity and the three target projects JayTeeSF named: an actor HTTP client/server, an actor Redis-like server, and a Rack/Sinatra/Go-style middleware server with async Jobs request processing and SQL-backed ActiveRecord-like libraries.

1. **General sockets and TLS.** The current `net` facade is still C10K/probe-shaped. Real HTTP and Redis need accept/connect/read/write/close descriptors, TLS termination or a clean reverse-proxy deployment story, backpressure, timeouts, graceful shutdown, and recurring C10K reliability proof.

2. **HTTP parser/serializer and client.** Need request/response structs, headers, chunking, streaming bodies, uploads, redirects, cookies, compression, and connection reuse. HTTPS can initially be delegated to nginx/Caddy, but the stdlib contract should not prevent native TLS later.

3. **Production static files.** `stdlib/static_files.hey` now pins conservative path-policy basics, but production still needs URL decoding, realpath/symlink confinement, MIME/cache/range/compression/sendfile, hidden-file policy overrides, and HTTP integration.

4. **Middleware and async request lifecycle.** The Rack-like pipeline needs immutable env/request/response values, middleware composition, error boundaries, context propagation, cancellation/timeouts, and Jobs integration that can hand off work without blocking acceptor actors.

5. **Actor scheduler/runtime pressure.** HTTP/Redis servers need reliable high-concurrency actors, mailbox pressure behavior, link/monitor/supervisor semantics, idle connection timers, and direct actor throughput baselines beyond toy ring rows.

6. **Binary-safe protocol data.** Redis and HTTP require bytes-first parsing, efficient buffers, CRLF scanning, streaming parsers, and codecs that do not force every request through JSON/string allocation.

7. **SQLite/PostgreSQL/MySQL adapters.** Need FFI or runtime adapters, connection pools, prepared statements, row decoding, transactions, migrations, schema metadata, parameter binding, and failure/reconnect semantics.

8. **ActiveRecord-like layer.** Need packageable modules for models, validations, associations, query building, migrations, callbacks/events, and explicit transaction boundaries without hiding actor/job behavior.

9. **Package system.** Real community libraries need namespacing, semantic versions, dependency resolution, lockfiles, install layout, registry trust/signing, checksum verification, offline/cache behavior, and compatibility metadata per backend/platform.

10. **No-toolchain distribution.** Users must be able to install Hey and compile/run these services without managing clang/LLVM/C toolchains manually.

11. **Typed IR and canonical symbols.** Backend progress is still too line-scanner-heavy. Real apps require one symbol/import model and typed IR consumed by LLVM/self-hosted so user functions, modules, strings, bytes, actors, jobs, and data-model values lower consistently.

12. **Observability and operations.** Need structured logs, metrics, tracing hooks, config/secrets, health checks, graceful restarts, deployment templates, and platform gates.

These are roadmap blockers, not reasons to stop. Native-C and self-hosted parity show the language can support real projects; LLVM is still a real possibility if it moves through typed IR and general runtime/ABI lowering instead of backend-specific pattern matching.
