Post-Windows learning and application roadmap

The Windows cross-SDK line and generic package/native-extension foundation are implemented, while native Windows execution remains an external promotion receipt. Application and integration packages must remain independently released consumers of Hey rather than becoming compiler-repository subsystems.

Product goal

Make Hey learnable without requiring a local installation for the first experience, while keeping the language reference, executable examples, and real project work connected to the same compiler and documentation sources.

why Hey -> five-minute start -> guided Tour of Hey -> task tutorials
-> language/stdlib reference -> progressively larger projects
-> challenge catalog -> browser playground -> local SDK projects

Reference patterns

The Go learning site is useful because it combines a fast start, guided interactive tour, task-oriented tutorials, training links, and reference material. Hey should use the same layered information architecture while teaching its own actor/worker/Job/stream model.

RubyQuiz is useful because it made learning social and project driven. Hey should preserve provenance separately from maintained Hey-native exercises, review redistribution rights before publishing copied material, and support multiple valid solution strategies.

v0.99.271a — documentation information architecture and executable Tour of Hey — complete

  • Every maintained developer page is linked from one executable learning-path map.
  • The landing page separates tutorials, how-to guides, explanations, and reference pages and offers explicit newcomer, experienced-programmer, systems/concurrency, and application/package-author routes.
  • `why-hey.md`, the local quick start, and the eight-lesson Tour form the first-experience sequence.
  • `./bin/hey tour list|run|verify` connects curated prose to ordinary self-testing source under `examples/tour`.
  • Documentation smoke, generated HTML, full tests, and source archives gate the Tour and the map.
  • Generated API/reference pages remain subordinate to curated learning pages.

v0.99.272a — RubyQuiz preservation and Hey challenge catalog — complete

  • Added a one-time Hey-native RubyQuiz provenance archiver over the public RSS/index and challenge pages.
  • The archiver writes raw third-party pages only to an explicit archive directory and emits a canonical manifest with quiz number, title, source URL, publication date when available, tags, SHA-256, and raw path.
  • Every archived entry is marked `redistribution_status: unreviewed` and `publishable: false`; copied statements or solutions remain unpublished until a human rights review.
  • Added four original maintained Hey challenges with statements, starters, separate correctness fixtures, bounded verification deadlines, advisory memory budgets, and editorial solutions.
  • Added `./bin/hey challenge list|show|verify|check` and deterministic provenance-archive fixture coverage.
  • Preserved multiple valid solution strategies: the editorial solution is evidence and teaching material, not a required implementation shape.

v0.99.273a — server-backed learning prototype — superseded

  • Added fail-fast staged source application and a first integrated learning UI.
  • Its execution lane required a loopback/native compiler server and therefore could not be deployed on the static `jayteesf.github.io` site.
  • The server-specific runner and fixtures are removed in v0.99.276a rather than preserved as a competing product path.

v0.99.276a — static browser Tour integrated into documentation — complete

  • `hey-doc-site` always publishes `learn/` beside the generated documentation.
  • All HTML, CSS, JavaScript, catalog, manifest, and service-worker paths are relative and safe below `https://www.jayteesf.com/hey_docs/`.
  • Added an explicit browser-safe Hey-to-JavaScript backend and a closed Web Worker runtime; no API endpoint or execution server is required.
  • Proved exact output parity for all eight canonical Tour source files: values, functions, actors, workers, Jobs, streams, virtual I/O/Results, and web route dispatch.
  • Added `heyc browser-js SOURCE.hey -o OUTPUT.js` for deterministic backend inspection.
  • Browser file examples use an in-memory virtual filesystem; host filesystem, process, networking, package, environment, and native-library capabilities are absent.
  • Source ZIP output now carries a mandatory safe-installer reminder, and a release contract requires it.
  • Darwin SDK closure now preserves stage-specific diagnostics and uses signed versioned dylibs with unversioned symlink aliases.

Post-native-Windows performance campaign — mandatory conditional milestone

As soon as a full native Windows installation and execution receipt exists, this campaign preempts later application milestones. The goal is not merely to make the test suite faster; it is to determine exactly where ordinary Hey programs diverge from equivalent optimized C and then remove avoidable overhead.

  • Build correctness-gated C reference programs and equivalent ordinary Hey programs for every benchmark.
  • Separate compile time, link time, cold start, warm runtime, peak RSS, allocations, and output validation.
  • Add microbenchmarks for scalar arithmetic, branches, loops, calls, recursion, strings, byte scanning, arrays, objects, dynamic keys, JSON, allocation/release, file I/O, sockets, TLS, native FFI, actors, workers, Jobs, streams, scheduling, and process startup.
  • Add macrobenchmarks for 1BRC, parsers, compression/checksum-style byte workloads, web request handling, concurrent pipelines, and application-shaped persistence/package workloads.
  • Run repeated warmups and measured trials on Darwin, Linux, and Windows; record machine/toolchain metadata, medians, tails, throughput, and ratios to the C reference.
  • Benchmark `heyc test --all` separately so compiler/test-orchestration overhead is not confused with generated-program runtime.
  • Establish explicit ratio budgets per workload class, keep regressions blocking once stabilized, and publish failures rather than averaging them away.
  • Use profiles and generated IR/assembly to drive changes; do not add benchmark-only language intrinsics or special-case source programs.

v0.99.276a — application-capability package proof

Build independently released packages needed for credible application demos without placing their implementations or API manuals in the Hey repository:

  • calendar/recurrence and import/export capabilities;
  • email composition/delivery, attachments, retry/idempotency policy, and local test transport;
  • persistence migrations, pooling, cancellation, and configuration/secrets boundaries;
  • HTTPS/web integration above the maintained standard-library foundations;
  • explicit capability matrices and unsupported behavior rather than framework-shaped placeholders.

Any Hey compiler/runtime change in this milestone requires a reduced package-neutral fixture. Package APIs, target matrices, and release notes remain in package repositories.

v0.99.276a — fifteen-minute EHY/persistence/Jobs blog demonstration

  • Create a small but complete blog using `stdlib:web`, EHY, a locked third-party persistence stack, migrations, forms, validation, static assets, and background Jobs.
  • Keep the initial path teachable as a genuine fifteen-minute walkthrough from an empty directory to a running application.
  • Include tests, development workflow, deployment packaging, and a minimal production configuration.
  • Measure cold start, steady-state RSS, request throughput, latency, and multicore behavior without special compiler/runtime shortcuts.

v0.99.277a — event scheduling application and Rails-class proof

  • Implement the maintained event-scheduling project specification as a real Hey application.
  • Require persistence, calendar invitations/updates, email delivery, HTTPS endpoints, authentication/authorization boundaries, and background Job workflows.
  • Use partitioned Jobs for per-event or per-account serialization, adaptive pools for bursty work, streams for bounded bulk operations, and supervised services for long-lived components.
  • Use independently released, locked packages for local and production persistence rather than repository-owned drivers.
  • Include failure recovery, duplicate-delivery protection, cancellation, observability, backups/migrations, and deployment documentation.
  • Produce a reproducible screen-recorded build with source, commands, tests, deployment, and operational behavior.
  • Compare against Rails only with equivalent features, persistence, TLS, workload, hardware, warmup, concurrency, and correctness checks; publish distributions rather than unsupported marketing claims.

Constraints

  • Do not make the documentation site depend on a proprietary service for ordinary reading.
  • Keep browser syntax and semantics parity-gated against canonical ordinary Hey sources; do not create tutorial-only syntax.
  • Do not put optional integration implementations into the language repository.
  • Keep package-specific documentation, compatibility, and version matrices with each package.
  • Preserve source provenance and licensing boundaries.
  • Keep every performance claim reproducible and correctness-gated.
