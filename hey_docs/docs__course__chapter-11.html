Chapter 11 — Tooling: REPL, Specs, and Verification

A language is more than a compiler. This chapter covers the three

tools that make Hey usable and — more importantly for this course —

*trustworthy*: the interactive prompt, the executable test format, and

the verification gates the project runs before any change lands.

The REPL

A **REPL** ("read–evaluate–print loop") is an interactive prompt: you

type an expression, it prints the result, and definitions accumulate

so you can build up a session incrementally.

Hey's REPL honors the one-pipeline rule from Chapter 1: there is no

separate interpreter hiding behind the prompt. Each input becomes a

**fragment** — a small, complete program assembled from your

accumulated session (bindings, imports, function definitions) plus the

new line — and that fragment goes through the ordinary compile

pipeline. Two engineering tricks make this fast enough to feel

instant:

  • **Warm caching.** Session state and compiled artifacts are reused

between turns, so unchanged material is never recompiled.

  • **In-process machine-code loading.** Compiled fragments are loaded

and executed via LLVM's JIT ("just-in-time") facilities rather than

by writing an executable to disk and launching it.

The visible payoff: anything the compiler accepts, the REPL accepts,

with identical meaning — because it *is* the compiler.

A related, deliberately fenced capability is `Eval` — a standard-library

module that lets a program compile and run Hey source *at runtime*.

Eval is a **restricted capability**: tooling (like the REPL) uses it,

but ordinary programs must opt in explicitly. A language that can

compile itself must decide who else gets that power; Hey's answer is

"only those who ask out loud."

Specs: executable expectations

Hey's test format is the **spec** — a file that states an expectation

and contains the program that must meet it:

spec 'doubling works'
  expect stdout "42\n"
end

program
  says 21 + 21
end

The harness compiles the program, runs it, and compares its actual

output (and exit status) against the `expect` block, character for

character. The repository contains thousands of specs in three

families:

DirectoryWhat it pins
`specs/language/`observable behavior of language features
`specs/compiler/`contracts on the compiler itself — including exact intermediate output
`specs/repl/`session behavior of the interactive prompt

Two properties make specs unusually strong tests. First, they are

*exact*: output is byte-compared, so a formatting regression cannot

hide. Second, they are *self-describing*: the expectation travels with

the program, so a spec file is complete documentation of one promised

behavior.

Regression scripts and gates

Above the specs sit **regression scripts** (in `test/`) that sweep

whole surfaces — for example, compiling every program in `examples/`

and asserting each one took the expected path with the expected

output. A wrapper called a **gate** runs an entire suite and prints a

single machine-readable receipt line; the project's rule is that a

change may only be committed on a green gate line, never on a partial

log.

This receipts habit — every important property gets a one-line,

grep-able proof of itself — pervades the toolchain. Even ordinary

compilation prints receipts naming the route taken, so scripts (and

humans) can *assert* facts about the build rather than assume them.

flowchart LR
    C["proposed change"] --> F["focused spec<br/>(the new behavior)"]
    F --> R["regression scripts<br/>(the affected surface)"]
    R --> G["full contracts gate<br/>(every compiler spec)"]
    G -->|"green receipt line"| M["commit lands"]
    G -->|"any red"| C

Why a language course cares

The verification design is a language-design decision. Because every

stage of the pipeline is pinned by exact-output specs, the team can

make bold internal changes — replace a whole compiler stage, delete a

legacy path — and *know* observable behavior held. Stability is not a

mood; it is a checked property. That is what Chapter 1's "stability

before speed" means operationally, and it is what lets performance

work proceed fearlessly later.

Discussion questions

1. Exact-output tests are brittle on purpose. When is byte-exactness

the right bar, and when would a fuzzier assertion serve better?

2. Hey's REPL compiles every input. Name a convenience a

separate-interpreter REPL could offer more easily, and how you

might recover it within the one-pipeline rule.

3. Why should `Eval` be opt-in for ordinary programs? Sketch an

attack or accident that the restriction prevents.
