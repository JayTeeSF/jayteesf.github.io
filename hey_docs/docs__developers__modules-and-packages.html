Modules, packages, and visibility

Hey imports name the kind of dependency being loaded. Application code should not depend on where the standard library happens to live on disk, and third-party package imports should identify the package version used by the build.

Standard-library imports

Use `stdlib:` for bundled modules:

import 'stdlib:collections'
import 'stdlib:json'
import 'stdlib:jobs'
import 'stdlib:web'

`std:` is a supported short alias:

import 'std:json'

New code and documentation should prefer `stdlib:name`. Checkout-relative paths such as `stdlib/json.hey` remain compatibility forms, not the recommended style.

Local imports

Use an explicit relative path for code owned by the current application or repository:

import './cards/formatting.hey'
import '../shared/slug.hey'

The path is resolved relative to the file containing the import. Imported modules retain their module namespace:

# cards/formatting.hey
module Formatting
  fn title(card)
    return card.front
  end
end

import './cards/formatting.hey'

program
  says Formatting.title({front: 'What is Hey?'})
end

Hey does not currently provide wildcard imports, `from ... import ...`, language-level aliases, or implicit unqualified imports. Prefixing calls with the module name keeps dependencies visible at the call site.

Namespace naming and lexical shadowing

Application and package modules should use `UpperCamelCase` names. This makes a

namespace visually distinct from ordinary lowercase values:

module StudySessions
  fn open(deck)
    return {deck: deck, open: true}
  end
end

program
  let session = StudySessions.open('Biology')
  says session.open
end

Capitalization is a readability convention, not a hidden lookup rule. The

standard library retains its established lowercase namespaces, and lexical

bindings are authoritative in every compiler lane. A local value named

`result` therefore makes `result.ok` a field read; when no local `result` is in

scope, `result.ok(value)` remains the standard-library module call:

import 'stdlib:result'

fn summarize(result)
  return result.ok
end

fn build_result(value)
  return result.ok(value)
end

When one scope genuinely needs both, prefer a domain name such as `outcome` for

the value. Future import aliases may provide an additional explicit namespace

name, but capitalization alone must never change expression semantics.

Third-party package imports

Use `pkg:` with an explicit package version:

import 'pkg:brainscape/cards@1.0.0/main'

The default package cache resolves that import to:

${HEY_PACKAGE_PATH:-.hey/packages}/brainscape/cards/1.0.0/main.hey

Package manifests and lock data should pin versions and SHA-256 checksums so installation and vendored distributions can verify exact source contents. See Authoring a third-party package.

Public and private functions

Module functions are public unless declared private:

module Cards
  private fn normalize_title(title)
    return trim(title)
  end

  fn create(title)
    return {title: Cards.normalize_title(title)}
  end
end

Importers can call `Cards.create(...)`. They cannot call `Cards.normalize_title(...)`.

A top-level helper can also be private:

private fn slugify(value)
  return trim(value)
end

It remains available inside its source file but is not part of the file's imported API.

Private imports

Use `private import` for an implementation dependency that should not become reachable through a downstream import:

private import './internal/slug_helpers.hey'

module Decks
  fn slug(name)
    return SlugHelpers.slugify(name)
  end
end

The current file may call `SlugHelpers.slugify(...)`. A file that imports this module may call `Decks.slug(...)`, but it does not gain access to `SlugHelpers` through the transitive dependency.

Private visibility controls API exposure; it is not a security sandbox. Private functions and imports still compile into the program when public code needs them.

Resolution safety

Standard-library and package import names reject empty segments, absolute paths, `..`, and backslashes. Local imports retain normal relative-path behavior because application source trees need `./` and `../` navigation.

Publishing and discovering packages

Use the package publishing guide to create immutable archives and release metadata. The third-party package foundations explain trust, locking, installation, and native-artifact responsibilities; package discovery and API documentation live outside the Hey repository.

Verified package imports

A `pkg:name@version/module` import is accepted only when the package was installed by `hey package install`, the project lockfile pins the package/version, and the module bytes match the recorded SHA-256. This prevents a mutable cache directory from silently changing compiled application code.

Generated package cache and Git

Project-local installs belong under `.hey/` and should not be committed. Keep

`hey-package.json` and `hey.lock.json`, but ignore generated cache trees with:

.hey/
**/.hey/

A repository that deliberately vendors immutable package payloads should place

them under a separately documented path such as `vendor/packages/`, not by

weakening the default `.hey/` cache rule.

Manifest module aliases

The keys under `hey-package.json` `modules` are the package's stable public

import names. Values are private paths inside the immutable package archive:

{
  "modules": {
    "main": "src/main.hey",
    "format": "src/format.hey"
  }
}

Consumers import the key, not the internal path:

import 'pkg:my_package@1.0.0/main'

The compiler resolves `main` through the installed manifest, then verifies the

resolved `src/main.hey` bytes against installation metadata and the project

lockfile. Direct path imports remain supported for compatibility, but public

documentation should use manifest aliases so a package can reorganize private

source paths in a future version.
