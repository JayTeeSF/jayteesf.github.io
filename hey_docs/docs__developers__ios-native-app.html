Hey developer documentationAudience-specific Hey documentation

Compile an ordinary Hey program into a native iOS app

`hey mobile ios native-app` is the maintained mobile direction. It compiles the

application through the normal product pipeline:

ordinary Hey -> semantic HIR -> ABI MIR -> direct LLVM -> iOS Mach-O object
                                                    -> native UIKit executable

The UIKit process calls exported Hey functions directly. Application logic and

native capability requests remain in the ordinary Hey compilation pipeline.

The source exports two initial native ABI functions:

fn ios_app_manifest()
  return %q({"title":"My App","subtitle":"Native UIKit","primary_title":"Continue"})
end

fn ios_action_code(action_id)
  return action_id + 1000
end

Generate a portable plan on any host with Clang:

./bin/hey mobile ios native-app \
  --source examples/mobile/native_ios_app.hey \
  --out build/hey-ios-native-plan \
  --plan-only

On a Mac with Xcode, link, ad-hoc sign, install, and launch:

./bin/hey mobile ios native-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/dev/hey-ios-native-app" \
  --bundle-id com.example.myapp \
  --display-name 'My App' \
  --launch

The receipt must say:

ordinary_hey_source=true
native_hey_codegen=true
mach_o=true
uikit=true

This is an App-Store-compatible native architecture, but not yet an App Store

submission receipt. Developer-team signing, physical-device execution, archive

creation, entitlements, privacy declarations, and review remain separate gates.

Native view manifest

The no-argument `ios_app_manifest()` function returns a static JSON manifest.

UIKit renders its `screens` as native navigation, list, and form controllers.

Fields with `kind: "secure"` become native secure `UITextField` controls.

Actions call the exported integer function `ios_action_code(action_id)` inside

the Mach-O object generated from Hey.

Opaque native capabilities

The UIKit host interprets selected integer results from `ios_action_code` as

narrow platform capabilities: `6101` ensures a device-only Keychain secret,

`6102` validates the configured HTTPS blind-sync health endpoint, and `6103`

creates complete-protection local storage. Native objects and secret bytes never

enter the Hey value model. The application bundle also contains

`PrivacyInfo.xcprivacy`.

Package boundary

The current UIKit/view/capability host is generic compiler evidence. Reusable

navigation, lifecycle, accessibility, Keychain, biometric, storage, and Apple

framework behavior should move toward independent `hey_mobile` and `hey_ios`

packages after equivalent Android requirements are proven. They are not language

syntax and should not become product policy inside the Hey repository. See

`../humans/native-mobile-plumbing-porcelain.md`.

Run on a physical iPhone or iPad

The physical-device command is a higher evidence level than Simulator launch:

xcrun devicectl list devices

export HEY_IOS_TEAM_ID='YOURTEAMID'
export HEY_IOS_DEVELOPMENT_SIGNING_IDENTITY='Apple Development: Your Name (YOURTEAMID)'
export HEY_IOS_DEVELOPMENT_PROVISIONING_PROFILE="$HOME/Library/MobileDevice/Provisioning Profiles/PROFILE.mobileprovision"

./bin/hey mobile ios device-app \
  --source examples/mobile/native_ios_app.hey \
  --out "$HOME/Desktop/hey-ios-device-app" \
  --bundle-id com.example.hey-native \
  --display-name 'Hey Native' \
  --device DEVICE_ID

Run `./bin/hey mobile ios device-app --help` for the complete setup, options,

and evidence boundary. The command requires full Xcode, a trusted and paired

device with Developer Mode enabled, an Apple Development signing identity, and

a matching development provisioning profile.

`--build-only` proves arm64 iPhoneOS compilation and development signing without

installing. A full run additionally proves installation and launch through

`devicectl`. Neither result is an IPA, TestFlight, App Store validation, or App

Review receipt.

Generated output is disposable

The `--out` directory contains generated LLVM, Mach-O objects, native host

sources, the `.app`, signing material derived from the selected profile, and

machine receipts. Keep application source in its own repository and regenerate

this output; do not treat the output directory as a source repository.
