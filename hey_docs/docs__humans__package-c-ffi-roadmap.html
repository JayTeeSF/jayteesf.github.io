Package-level C FFI and native extension roadmap

Hey needs a package extension boundary before the database/application milestones. The boundary must preserve the language's ownership, cancellation, packaging, and cross-platform guarantees rather than exposing arbitrary host pointers as ordinary values.

Why this is required

SQLite is distributed as a C amalgamation and is best embedded directly into a package. MySQL compatibility can initially use a packaged MariaDB/MySQL Connector/C adapter, while a later pure-Hey wire-protocol client may provide a more naturally asynchronous alternative. Calendar, SMTP, and HTTP do not require C, but database drivers, compression libraries, image codecs, and OS integrations make a stable native-package boundary necessary.

Preferred model: generated package-native shim

A package may contain:

hey.package.json
src/*.hey
native/include/*.h
native/src/*.c
native/ffi.hey

`native/ffi.hey` declares a constrained C ABI. The package compiler generates a C shim, compiles package-owned C sources with the SDK's contained compiler, links declared native libraries, and records every native input in the package lock and distribution manifest.

Proposed declaration shape:

native c module sqlite3_native
  opaque type Database
  opaque type Statement

  fn open(path: c.utf8, out database: c.out_ptr<Database>) -> c.int
    symbol: 'sqlite3_open_v2'
    ownership: caller_closes
    blocking: true
  end

  fn close(database: c.ptr<Database>) -> c.int
    symbol: 'sqlite3_close_v2'
    consumes: database
  end
end

The exact syntax remains a design item. The required semantics do not:

  • fixed-width C integers, sizes, booleans, floating values, byte spans, UTF-8/UTF-16 strings, opaque pointers, and explicit out parameters;
  • no implicit conversion between Hey references and native pointers;
  • no arbitrary pointer arithmetic in the first release;
  • explicit ownership (`borrowed`, `owned`, `consumes`, destructor) and nullable annotations;
  • generated lifetime checks around opaque handles;
  • callbacks only after a separate callback/thread-affinity design is proven;
  • `blocking: true` calls execute through bounded workers/Jobs rather than blocking actor scheduler threads;
  • native errors are translated into ordinary Hey `Result` values by the public package layer;
  • package manifests declare headers, sources, libraries, frameworks, target conditions, licenses, and checksums;
  • builds use SDK-relative compiler, sysroot, and native-library roots and remain reproducible from the lock file;
  • browser/Wasm packages cannot load unrestricted native extensions.

Package ABI

The initial package ABI should be source/shim based, not a permanently frozen in-memory `HeyValue` ABI. Packages compile with the consuming SDK and interact through generated wrappers. A later stable binary plugin ABI may be introduced only after value layout, allocator ownership, runtime initialization, and version negotiation are explicit.

Each generated extension exports a small descriptor containing:

  • ABI version;
  • package name/version/content hash;
  • target triple;
  • initialization and shutdown functions;
  • declared function table;
  • runtime capability requirements;
  • thread-safety and blocking metadata.

The loader rejects ABI, target, checksum, or capability mismatches before executing initialization code.

SQLite plan

  • Vendor the upstream SQLite amalgamation in a separately licensed package or fetch it through a checksum-pinned package source.
  • Compile it with the contained SDK, with thread-safe serialized mode and explicit feature flags.
  • Expose connections, prepared statements, typed bindings, rows, transactions, migrations, cancellation, busy timeout, and backup APIs through safe Hey objects.
  • Use a partitioned Job per connection (or connection identifier) so one connection is never used concurrently while the pool can use multiple connections across cores.
  • Make statement/finalizer ownership deterministic and test leaks under ASan.

MySQL plan

Two compatible implementations may share one high-level database contract:

1. **Connector/C adapter:** fastest path to broad protocol/authentication compatibility through the C FFI. Package the native client or require an explicitly declared system capability; never discover random host libraries silently.

2. **Pure-Hey protocol client:** longer-term path using TCP/TLS, packet framing, prepared statements, authentication plugins, cancellation, and async Jobs without a blocking foreign client library.

The public database API must not leak connector-specific pointers or error objects.

Alternatives and why they are secondary

  • **Write every integration into the core runtime:** easy initially, but turns optional packages into permanent runtime surface and prevents independent releases.
  • **Subprocess adapters:** useful for tools and isolation, but poor for high-throughput transactions and deployment simplicity.
  • **Pure-Hey implementations only:** desirable for network protocols, but inappropriate for replacing mature embedded engines such as SQLite.
  • **Dynamic libffi-style calls:** flexible, but unsafe without compile-time type/ownership checking and difficult to package reproducibly.

Numbered milestone

The ABI/code-generation foundation is completed in **v0.99.266a**. Versioned Unix loading, typed invocation, blocking/thread-safety enforcement, and lifecycle closure are completed in **v0.99.267a**. Static registration/native Windows loading remain portability work; SQLite starts in **v0.99.268a** and MySQL in **v0.99.269a** before the documentation/application demonstration line.

Promotion requires:

  • one checked native package built from source on Linux, Darwin, and Windows;
  • package-lock and archive reproducibility;
  • invalid signature/type/ownership diagnostics;
  • blocking-call Job integration;
  • lifecycle, unload/reload, and sanitizer evidence;
  • SQLite open/create/prepare/bind/step/finalize/transaction tests;
  • no native extension access in the browser capability profile.

v0.99.266a completed foundation

The repository now contains `runtime/hey_native_abi.h`, `tools/hey_native_generate.hey`, and `bin/hey-native`. A checked sample package proves deterministic wrapper and lock generation, host execution, AMD64 COFF compilation, explicit blocking metadata, invalid raw-pointer/ownership diagnostics, and no package-facing `HeyValue`. v0.99.267a now adds versioned dynamic loading and uniform typed invocation without exposing `HeyValue`; static registration and native Windows loading remain unclaimed.

v0.99.267a completed loader boundary

The runtime now validates `HeyNativeExtensionV1`, returns opaque integer capabilities, translates stable ABI values, rejects undeclared/closed handles, rejects blocking functions on `native.call`, serializes functions not marked thread-safe, waits for active calls during close, and unloads remaining extensions during runtime shutdown. A generated math extension returns identical results through the interpreter, Stage-0/native-C comparison, and default LLVM-hosted HIR/MIR/direct-LLVM lanes.

v0.99.268a completed SQLite boundary

SQLite now exists as an ordinary package under `ecosystem/hey_sqlite`, not as a core-runtime special case. It covers open/close, busy timeout, prepared statements, integer/float/text/null binding, row stepping, typed column access, reset/finalize, transactions, change counts, and last-insert identifiers. Native ABI v2 adds machine-readable argument and return tags, allowing opaque handles to round-trip safely and enabling the loader to reject mismatched calls before entering C.

The repository smoke uses a discovered system SQLite development library and reports `mode=system-library`. Promotion still requires a separately licensed, checksum-pinned amalgamation or equivalent contained package source built for Linux, Darwin, and Windows.

v0.99.269a MySQL Connector/C package candidate

`ecosystem/hey_mysql` now adapts MariaDB/MySQL Connector/C through the same generated ABI used by SQLite. The package exposes opaque connection and buffered-result handles, connect/read/write timeouts, explicit TLS modes, connector diagnostics, ping/autocommit, query execution, affected rows, insert IDs, transactions, result metadata, row iteration, null checks, and borrowed UTF-8 column views. Network-touching operations are declared blocking and the Hey layer requires one bounded or partitioned Job lane to own a connection and all derived results.

The maintained package regression discovers Connector/C explicitly through `mariadb_config`, `mysql_config`, or named `pkg-config` packages. It proves generation, compilation, loading, TLS-option validation, deterministic connection refusal, and interpreter/native-C/default-LLVM parity. `test/regression_mysql_live_server.sh` is a separate configured gate for actual server ping, TLS policy, rollback/commit, affected rows, insert IDs, row iteration, nulls, and cleanup. The current environment cannot produce that external receipt, so the adapter is an alpha package candidate rather than a promoted database contract.

The slice also demonstrates why native package work must exercise the compiler: Result-wrapped booleans and predicate calls now have an explicit direct-LLVM contract, and release emitter caches include target/Stage-0 identity so a Darwin binary cannot be reused by Linux container bootstrap.
