LLVM-first Hey toolchain and REPL

**Status:** Accepted architecture; migration in progress

**First implementation slice:** v0.99.381a

**Primary objective:** Eliminate duplicated native-C language implementations and make the canonical Hey → HIR → MIR → LLVM pipeline responsible for all Hey source execution.

1. Architectural principle

Hey owns language meaning and compiler policy.

LLVM owns code generation and optimization.

A small native runtime owns unavoidable OS and ABI services.

There must be exactly one implementation of Hey language semantics.

Hey source
  -> Hey parser
  -> semantic analysis
  -> semantic HIR
  -> ABI MIR
  -> LLVM
  -> native code
  -> minimal Hey runtime

The following surfaces must all use that same pipeline:

  • `heyc build`
  • `heyc --run`
  • `heyc --test`
  • `hey file.hey`
  • `hey-repl`
  • `eval(...)`
  • package compilation
  • compiler self-hosting
  • executable specs
  • platform application builds

The REPL, script runner, tests, and compiler must not contain independent parsers, evaluators, builtin dispatchers, or semantic rules.

---

Current fixed-point constraint

The private native Stage-0 compiler remains a required seed and recovery tool until

verified distributed bootstrap artifacts can build Stage 1 and Stage 2 through LLVM

and prove a fixed point on every supported host. This temporary seed is not a second

product backend. The v0.99.381a slice therefore does two things immediately:

1. stops release products from forcing the native-C-hosted Hey frontend; and

2. makes every native source file and duplicated semantic owner explicit and

mechanically audited.

Removing the Stage-0 seed, the C REPL evaluator, and the native MIR consumer are

separate later slices that require replacement evidence before deletion.

2. Goals

1. Make LLVM the only product backend for Hey applications.

2. Make the REPL execute LLVM-generated code.

3. Remove the C REPL interpreter.

4. Remove the native-C application backend from normal builds.

5. Move MIR-to-LLVM policy out of the native bootstrap compiler.

6. Retain only a narrow, platform-independent native runtime.

7. Make bootstrap dependencies explicit, reproducible, and isolated.

8. Prevent silent fallback from LLVM to native-C.

9. Ensure compiled files, REPL input, and evaluated source have identical semantics.

10. Reduce the number of compiler paths that must be tested and maintained.

---

3. Non-goals

This project does not require eliminating every line of C.

The following may remain native where appropriate:

  • process startup;
  • allocation and boxed-value ownership;
  • reference counting or future garbage collection;
  • pthread or platform thread integration;
  • atomics and synchronization primitives;
  • filesystem and process syscalls;
  • sockets and readiness APIs;
  • TLS library integration;
  • dynamic loading;
  • platform SDK entry points;
  • LLVM itself;
  • target linkers and object-file tooling;
  • narrow native-extension ABI adapters.

These components must not implement Hey syntax or language semantics.

---

4. Current native components

4.1 `runtime/hey_repl.c`

Current responsibilities include an independent implementation of:

  • parsing REPL commands;
  • expression evaluation;
  • function execution;
  • builtin dispatch;
  • actor behavior;
  • session variables;
  • imports;
  • `eval`;
  • source reflection.

This duplicates language behavior already owned by the Hey compiler.

**Target:** Delete the evaluator and replace it with an LLVM-backed REPL session.

Terminal editing and history may remain in a small native UI component until they are conveniently moved into Hey.

4.2 `runtime/heyc.c`

Current responsibilities include:

  • Stage-0 compiler bootstrap;
  • private MIR validation;
  • MIR-to-LLVM emission;
  • object generation;
  • linking;
  • portions of backend selection;
  • legacy native-C generation.

**Target:** Remove product compiler policy from this file.

It may temporarily remain as an explicit Stage-0 recovery tool, but it must not be used by normal application builds.

4.3 `runtime/hey_run_fast.c`

This currently provides a fast launcher/cache path.

**Target:** Either:

1. replace it with a Hey-authored launcher compiled through LLVM; or

2. retain only a very small dispatcher that validates a published cache record and calls `execve`.

It must not:

  • parse Hey source;
  • calculate language semantics;
  • compile source;
  • infer dependencies;
  • publish partially built cache entries;
  • contain application-specific behavior.

4.4 `runtime/hey_runtime.c`

This is a legitimate runtime boundary.

**Target:** Keep it, but split it into explicit components:

runtime/value
  boxed values
  retain/release
  strings
  bytes
  immutable collections

runtime/concurrency
  actors
  workers
  jobs
  mailboxes
  scheduler
  supervision

runtime/io
  files
  processes
  pipes
  sockets
  readiness
  timers
  TLS

runtime/platform
  startup
  dynamic loading
  operating-system adapters

No runtime component may parse or interpret Hey source.

---

5. Canonical compiler driver

Introduce one Hey-authored compiler driver, conceptually:

HeyCompiler.compile(request)

A compilation request contains:

{
  source_files: [...],
  mode: 'build',
  target: Target.host(),
  optimization: 'debug',
  output: 'build/app',
  executable_specs: true,
}

The compiler driver owns:

1. source loading;

2. import graph resolution;

3. parsing;

4. semantic validation;

5. HIR generation;

6. MIR lowering;

7. LLVM generation;

8. object generation;

9. linking;

10. diagnostics;

11. build metadata;

12. cache publication.

Different commands may configure the driver, but they must not reimplement it.

heyc build      -> HeyCompiler.compile(...)
heyc --run      -> HeyCompiler.compile(...) + execute
heyc --test     -> HeyCompiler.compile(test mode) + execute
hey file.hey    -> HeyCompiler.compile(cached script mode) + execute
hey-repl        -> HeyCompiler.compile(repl fragment mode)

---

6. Hey-authored MIR-to-LLVM consumer

Create a canonical Hey module, conceptually:

stdlib:HeyLlvm

Its public boundary should consume validated MIR, not Hey source.

let result = HeyLlvm.emit_module(
  mir,
  target: target,
  optimization: optimization
)

The LLVM consumer must not:

  • reopen source files;
  • parse expressions;
  • infer source-level types;
  • recognize application filenames;
  • reinterpret actor or worker semantics;
  • guess lexical identity from variable names;
  • introduce source-specific compatibility branches.

HIR owns source semantics.

MIR owns:

  • concrete callable ABIs;
  • boxed versus scalar representation;
  • ownership operations;
  • lexical binding identities;
  • actor and worker identities;
  • control-flow blocks;
  • runtime calls;
  • target-facing operations.

LLVM emission must be mechanical.

6.1 First implementation

The fastest first implementation should emit deterministic LLVM textual IR:

validated MIR
  -> Hey-authored LLVM IR emitter
  -> `.ll`
  -> LLVM/Clang toolchain
  -> object
  -> linker

This removes compiler policy from C without initially requiring a complete LLVM C API binding.

6.2 Optional later implementation

After correctness is established, a narrow native bridge may expose:

  • LLVM module creation;
  • verification;
  • optimization pipelines;
  • object emission;
  • ORC JIT operations.

The bridge must expose LLVM mechanics, not Hey semantics.

Conceptually:

let module = Llvm.module_create(name)
Llvm.add_function(module, ...)
Llvm.verify(module)
let object = Llvm.emit_object(module, target)

---

7. LLVM-backed REPL

The existing C evaluator must be replaced.

7.1 REPL architecture

terminal input
  -> complete Hey source fragment
  -> canonical parser
  -> semantic analysis against session state
  -> HIR
  -> MIR
  -> LLVM
  -> loaded native code
  -> execute fragment
  -> transactionally update session state

7.2 Session state

The REPL must maintain a runtime-owned session object containing:

  • bindings;
  • binding IDs;
  • function definitions;
  • function version IDs;
  • module definitions;
  • actor and worker definitions;
  • imported module identities;
  • source metadata;
  • session generation.

Conceptually:

{
  generation: 12,
  bindings: {...},
  functions: {...},
  modules: {...},
  imports: [...],
}

Generated fragments communicate through a narrow ABI:

hey_repl_binding_get(session, binding_id)
hey_repl_binding_set(session, binding_id, value)
hey_repl_define_function(session, function_id, pointer)
hey_repl_find_function(session, function_id)
hey_repl_commit(session, generation)
hey_repl_rollback(session, generation)

7.3 Initial loading strategy

The fastest implementation path is:

REPL fragment
  -> LLVM
  -> temporary object
  -> temporary dylib/shared object
  -> dlopen
  -> execute generated entry function

This already makes the REPL LLVM-backed and reuses the product compiler.

It does not require LLVM ORC before the semantic duplication can be removed.

7.4 ORC JIT follow-up

Once the shared-object implementation is correct, replace its final loading stage with LLVM ORC JIT:

REPL fragment
  -> MIR
  -> LLVM module
  -> ORC JIT
  -> execute

The compilation stages before LLVM remain identical.

7.5 Transactional behavior

A REPL fragment must not modify the visible session until it has:

1. parsed successfully;

2. passed semantic analysis;

3. produced valid HIR;

4. produced valid MIR;

5. passed LLVM verification;

6. loaded successfully;

7. completed required initialization.

On failure, the previous session remains intact.

7.6 Redefinition

Function redefinition must use stable semantic identity and versioned machine symbols.

greet              semantic function identity
greet$generation_4 generated machine symbol
greet$generation_7 generated machine symbol

New calls resolve to the newest version.

Previously compiled functions may retain their original references unless Hey explicitly defines dynamic late binding.

This behavior must be specified and tested rather than accidentally determined by `dlsym`.

7.7 `eval`

`eval(...)` must submit source to the same REPL compiler service.

It must not invoke a separate interpreter.

eval source
  -> restricted capability request
  -> canonical parser
  -> HIR
  -> MIR
  -> LLVM
  -> callable value

Capability restrictions belong in semantic analysis and the compilation request, not in a second evaluator.

---

8. Bootstrap strategy

Normal builds must stop compiling the self-hosted compiler through the native-C application backend.

8.1 Distributed bootstrap artifact

Each supported host SDK may include a verified bootstrap compiler:

bootstrap/darwin-arm64/hey-bootstrap
bootstrap/linux-x64/hey-bootstrap
bootstrap/windows-x64/hey-bootstrap.exe

The bootstrap compiler is used only to compile the canonical Hey compiler through LLVM.

prebuilt bootstrap
  -> compile Hey compiler with LLVM
  -> Stage 1 compiler
  -> compile Hey compiler again
  -> Stage 2 compiler
  -> compare Stage 1 and Stage 2 artifacts

The normal build must not invoke:

--backend native-c

8.2 Reproducibility

A release must record:

  • bootstrap artifact hash;
  • compiler source hash;
  • HIR hash;
  • MIR hash;
  • LLVM IR hash;
  • target configuration;
  • Stage 1 binary hash;
  • Stage 2 binary hash;
  • fixed-point comparison result.

8.3 Recovery path

The old Stage-0 native-C compiler may temporarily remain behind an explicit recovery command:

make bootstrap-recovery

It must not be:

  • part of `make`;
  • part of release packaging;
  • selected automatically;
  • used as a fallback when LLVM compilation fails;
  • advertised as an application backend.

The recovery implementation can be deleted after the distributed bootstrap chain is independently reproducible on all supported hosts.

---

9. Remove the native-C application backend

The product CLI must not expose competing source backends.

Remove or retire:

heyc build --backend native-c
HEY_SELFC_EMITTER_BACKEND=native-c
automatic native-C fallback
native-C application parity as a release requirement

During migration, explicit internal tests may retain a Stage-0 comparison command.

After migration, backend selection becomes target selection:

heyc build app.hey
heyc build app.hey --target darwin-arm64
heyc build app.hey --target linux-x64
heyc build app.hey --target windows-x64

LLVM is implied.

If LLVM or the target toolchain is unavailable, compilation must fail clearly.

It must never switch to another source backend.

---

10. Required semantic invariants

10.1 One parser

The same source must produce the same syntax tree in:

  • compiled applications;
  • tests;
  • REPL fragments;
  • imported REPL files;
  • `eval`;
  • compiler self-hosting.

10.2 One semantic analyzer

Diagnostics must match across every execution surface.

An expression rejected by `heyc build` must also be rejected by the REPL.

10.3 One HIR

HIR must preserve:

  • lexical `BindingId`;
  • callable identity;
  • source locations;
  • inferred value facts;
  • explicit type annotations;
  • actor and worker identity;
  • capability requirements;
  • ownership intent.

10.4 One MIR

MIR must explicitly describe:

  • boxed and scalar operations;
  • calls and calling conventions;
  • ownership transitions;
  • runtime API calls;
  • control flow;
  • object and collection operations;
  • actor turns;
  • worker calls;
  • error propagation.

10.5 No source semantics in LLVM

LLVM emission must not make decisions based on:

  • source strings;
  • source variable names;
  • source filenames;
  • indentation;
  • parser forms;
  • application-specific patterns.

---

11. Migration slices

Slice A: Guardrails and inventory

1. Inventory every use of:

  • `native-c`;
  • `hey_repl.c`;
  • source parsing in C;
  • C builtin dispatch;
  • direct-source LLVM emission;
  • automatic fallback.

2. Classify each occurrence as:

  • runtime ABI;
  • bootstrap;
  • duplicated semantics;
  • launcher;
  • test-only;
  • obsolete.

3. Add a repository check preventing new native-C semantic implementations.

4. Document the canonical pipeline in one authoritative file.

Slice B: Canonical compiler request

1. Create a single compilation-request structure.

2. Route build, run, test, and script execution through it.

3. Remove duplicated command-specific compilation logic.

4. Add deterministic HIR and MIR artifact hashes.

Slice C: Hey MIR-to-LLVM emitter

1. Implement MIR validation in Hey.

2. Implement LLVM textual IR generation in Hey.

3. Compile representative scalar programs.

4. Add boxed values and objects.

5. Add collections and strings.

6. Add functions and closures.

7. Add actors, workers, Jobs, streams, and I/O.

8. Verify all product suites without the native MIR emitter.

Slice D: LLVM REPL foundation

1. Separate terminal input from evaluation.

2. Introduce the persistent REPL session ABI.

3. Compile one expression as an LLVM fragment.

4. Load it as a shared object.

5. Add persistent bindings.

6. Add functions and redefinition.

7. Add imports and modules.

8. Add actors, workers, Jobs, and streams.

9. Route `eval` through the same compiler.

10. Delete the C expression evaluator.

Slice E: Remove native-C application compilation

1. Remove public backend selection.

2. Remove normal-build native-C compilation.

3. Make missing LLVM support a hard error.

4. Remove native-C application tests.

5. Retain only explicit bootstrap-recovery tests.

Slice F: Bootstrap fixed point

1. Publish verified bootstrap binaries.

2. Build Stage 1 with LLVM.

3. Build Stage 2 with Stage 1.

4. compare fixed-point artifacts;

5. remove native-C bootstrap from normal `make`;

6. remove the recovery compiler after platform coverage is complete.

Slice G: Runtime and launcher cleanup

1. Split the runtime by responsibility.

2. Remove semantic helpers from runtime code.

3. Replace or minimize the fast C launcher.

4. Generate runtime ABI declarations from one machine-readable definition.

5. Verify runtime portability independently from compiler semantics.

Slice H: ORC JIT optimization

1. Add a narrow LLVM ORC bridge.

2. Replace temporary shared-object REPL loading.

3. Reuse compiled session modules.

4. cache imported module code;

5. preserve identical HIR and MIR behavior.

6. Measure and reduce interactive latency.

---

12. Verification tiers

Tier 1: Sub-minute architecture checks

Canonical command:

./bin/hey verify architecture
  • no new source parser in C;
  • no new C builtin semantics;
  • no normal-build `native-c`;
  • no automatic backend fallback;
  • deterministic compiler artifact hashes.

Tier 2: Phase-boundary tests

source -> syntax
syntax -> HIR
HIR -> MIR
MIR -> LLVM
LLVM -> runtime behavior

Each boundary must have focused golden tests.

Tier 3: REPL equivalence

For every supported REPL fragment:

compiled file output == REPL output
compiled diagnostics == REPL diagnostics
compiled HIR meaning == REPL HIR meaning
compiled MIR ABI == REPL MIR ABI

Tier 4: Product matrix

  • functions and closures;
  • recursive values;
  • objects and collections;
  • strings and bytes;
  • files and processes;
  • networking and TLS;
  • actors and workers;
  • Jobs and streams;
  • packages;
  • reflection;
  • native extensions;
  • executable specs;
  • supported target platforms.

Tier 5: Bootstrap and release

  • clean checkout bootstrap;
  • Stage 1 build;
  • Stage 2 fixed point;
  • target distributions;
  • package installation;
  • release verification;
  • promotion last.

13. Performance requirements

Correctness and architectural consolidation come first, but the migration must not institutionalize a slow REPL.

Initial shared-object REPL target:

  • no recompilation of unchanged imported modules;
  • no complete compiler bootstrap per fragment;
  • immutable compiler services shared across fragments;
  • isolated per-fragment output artifacts;
  • parallel-safe cache publication;
  • warm trivial-fragment latency below 500 ms on the reference development machine.

ORC JIT target:

  • warm trivial-fragment latency below 100 ms;
  • unchanged function bodies reused;
  • unchanged imported modules reused;
  • no linker subprocess per fragment;
  • bounded memory growth across long sessions.

Benchmarks must report:

  • parse time;
  • semantic-analysis time;
  • HIR time;
  • MIR time;
  • LLVM generation time;
  • optimization time;
  • object/JIT time;
  • load time;
  • execution time;
  • peak memory;
  • accumulated session memory.

---

14. Completion criteria

The migration is complete when:

1. `heyc build` uses the Hey-authored frontend and LLVM consumer.

2. `hey file.hey` uses the same compiler request.

3. `heyc --test` uses the same compiler request.

4. `hey-repl` compiles every fragment through HIR, MIR, and LLVM.

5. `eval` uses the same restricted compiler pipeline.

6. `runtime/hey_repl.c` no longer contains a Hey evaluator.

7. Normal builds contain no `--backend native-c`.

8. Product releases do not ship a native-C application backend.

9. C code contains no Hey parser or source-language semantic implementation.

10. The runtime exposes only documented ABI and platform services.

11. A clean checkout builds from a verified bootstrap artifact.

12. Stage 1 and Stage 2 compiler builds reach a reproducible fixed point.

13. REPL and compiled-source equivalence suites pass.

14. LLVM failures never trigger silent fallback.

15. Documentation describes one compiler, one semantic pipeline, and one product backend.

Final target

                         +----------------------+
Hey source ------------>| Hey parser           |
                         | semantic analysis    |
REPL fragment ---------->| HIR                  |
                         | MIR                  |
eval source ------------>| compiler policy      |
                         +----------+-----------+
                                    |
                                    v
                         +----------------------+
                         | LLVM consumer        |
                         | AOT / shared object  |
                         | ORC JIT              |
                         +----------+-----------+
                                    |
                                    v

                         | OS and ABI services  |

There is one language implementation.

There is one semantic pipeline.

There is one product backend.

Native code remains only where native boundaries are genuinely required.

v0.99.381a implementation receipt

Ordinary product compilation no longer accepts `native-c`, `--native`, or

`build-native`. Native source compilation remains available only through the

explicit `stage0-*` maintainer commands. This keeps the bootstrap/recovery lane

observable without allowing product commands to fall back to it.

Release preservation and release acceptance are separate phases of one command:

source is already committed before `bin/hey-finalize-release` begins; the command

runs the full verifier in a disposable worktree and only then amends the commit

to a `Release v...` subject. Failure never discards the preserved source.

v0.99.381a implementation receipt

`HeyCompileRequest` is the first canonical compiler-policy boundary. The initial

fixed-point representation is deterministic canonical text rather than a native

struct or command-specific object. Compiler and script surfaces construct it,

cache keys include its semantic identity, and compiler receipts publish the

request/HIR/MIR identities required for later build/run/test/REPL equivalence.

v0.99.381a implementation receipt: complete fragments

`hey-repl --llvm-fragment` is the first executable REPL migration boundary. It

runs a complete fragment through the same compiler request, HIR, MIR, LLVM cache,

and native execution path as scripts, without entering the C expression

evaluator. The request implementation is imported as ordinary compiler-owned

Hey source, preserving typed scalar functions in fixed-point HIR/MIR.

This is deliberately not called a persistent LLVM REPL. Session bindings,

transactional generations, imports, function versioning, and Eval still require

the session ABI described above. Warm fragment execution is bounded by a

Hey-timed test and must prove `backend=llvm` with no compiler invocation.

v0.99.382a implementation receipt: true fast dispatch

Canonical request identity is now part of a source-index v2 record with a unique

`request_mode` field. The thin C dispatcher validates that record and executes

the cached LLVM program directly; it does not parse source, compute request

policy, or compile. A maintained benchmark points `HEY_RUN_FULL` at a forbidden

path during warm runs, proving the full compiler launcher is not started.

The benchmark itself is Hey-authored and uses `Time.now_ns`. Source-archive and

script-audit product builds also use the ordinary LLVM compiler. Native-C remains

only behind explicit Stage-0 comparison/recovery surfaces.
