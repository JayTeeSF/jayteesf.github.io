Parallel streams

`std:streams` provides familiar functional chains while executing callable stages through bounded worker-backed Jobs. Streams do not expose channels and do not create a second scheduler.

import 'std:streams'

fn parse_line(line)
  return parse(line)
end

fn valid?(record)
  return record.valid
end

let records = files.lines(path)
  .map(parse_line, workers: 8)
  .filter(valid?, workers: 4)
  .collect()

Operations

  • `.map(callable, workers: N)` replaces every value with the callable result.
  • `.filter(callable, workers: N)` retains values whose callable result is true.
  • `.each(callable, workers: N)` executes the callable and returns the processed item count.
  • `.through(Job)` submits each value to an existing named Job and uses its successful result as the next value.
  • `.unordered()` makes subsequent parallel stages emit successful values in observed completion order.
  • `.resequence` or `.resequence()` restores original input order with a bounded reorder buffer.
  • `.collect()` executes the accumulated stages and returns an array.

`workers:` defaults to `1` when omitted and currently accepts `1..16`. The explicit bound keeps the temporary stage pool predictable; larger or adaptive policies should be represented by an existing Job and `.through(Job)`.

Existing Job stages

worker ParseLine(line)
  return parse(line)
end

program
  let ParseLines = jobs.define(
    'ParseLines',
    ParseLine,
    min_workers: 2,
    max_workers: 16,
    target_queue_latency_ms: 20,
    capacity: 256,
    result_capacity: 64
  )

  let records = files.lines(path)
    .through(ParseLines)
    .filter(valid?, workers: 4)
    .collect()

  ParseLines.stop()
end

`.through(Job)` preserves the Job's retries, idempotency, durability, adaptive scaling, metrics, and lifecycle policy.

Backpressure and bounded memory

Each stage submits at most one bounded window. The window is the smaller of the Job queue capacity and `result_capacity`. When enqueue rejects because the queue is full, the producer waits and retries instead of accumulating an unbounded receipt list. Results are consumed before the next window is submitted.

This matters for Jobs whose retained-result capacity is smaller than their queue:

let Transform = jobs.define(
  'Transform',
  TransformItem,
  count: 4,
  capacity: 64,
  result_capacity: 8
)

let output = input.through(Transform).collect()

The stream keeps no more than eight outstanding results in this example.

Ordering and resequencing

Ordered output remains the default:

let ordered = values
  .map(transform, workers: 8)
  .collect()

Use `.unordered()` when downstream work benefits from receiving whichever item completes first:

let completion_order = values
  .unordered()
  .map(transform, workers: 8)
  .collect()

Restore original input order later with `.resequence`:

let input_order = values
  .unordered()
  .map(transform, workers: 8)
  .filter(valid?, workers: 4)
  .resequence
  .collect()

Every result carries a private input sequence and the identity/limit of its bounded submission window. `.resequence` sorts only one such window at a time; it does not accumulate an unbounded global reorder map. Filtered items simply have no output entry, while remaining entries keep their relative input order.

Failure behavior

A failed, cancelled, timed-out, or unknown stage result stops the stream with that Job error. A stopped Job also terminates an enqueue-backpressure wait. Job retries finish before a value enters completion order. The stream never silently drops a failed item or invents a placeholder for a missing sequence.
