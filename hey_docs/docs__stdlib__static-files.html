Static files stdlib facade

`stdlib:static_files` provides safe path resolution and file reads for web applications. It is designed to sit behind `stdlib:web`, which owns routing and the live HTTP service boundary.

Goal

A Hey app should be able to expose a configured public directory without accidentally serving `../` paths, hidden files, Windows-style backslash traversal, encoded traversal, or arbitrary absolute filesystem paths. HTTP middleware can later put this facade behind actor-supervised listeners, caching, content types, range requests, compression, and sendfile-style runtime primitives.

Current API

import 'stdlib:static_files'

program
  let resolved = static_files.resolve('public', '/assets/app.css?cache=1')
  if resolved['ok']
    says resolved['path']
  else
    says resolved['status']
  end

  let file = static_files.read('public', '/index.html')
  if file['ok']
    says file['body']
  end
end

`static_files.resolve(root, request_path)` returns one of:

{ok: true, status: 200, path: 'public/app.css', leaf: 'app.css'}
{ok: false, status: 403, reason: 'unsafe_segment'}
{ok: false, status: 403, reason: 'forbidden_char'}
{ok: false, status: 404, reason: 'not_found', path: 'public/missing.css'}

`static_files.read(root, request_path)` only reads after `resolve` succeeds and then adds `body`.

Secure defaults

Current defaults reject:

  • `.` and `..` path segments
  • dotfiles such as `.env`
  • backslashes
  • colons
  • percent-encoded input, so callers must URL-decode before this boundary or encoded traversal stays blocked
  • empty request paths

This is intentionally conservative. Future HTTP middleware can add explicit allowlists for safe dot-directories such as `.well-known` if needed.

Remaining production work

The full production static-file server still needs MIME type selection, `index.html` policy, directory listing policy, cache headers, ETags, byte ranges, compression, sendfile/runtime streaming, symlink/root realpath checks, URL decoding, and an HTTP integration layer. These belong on the web/service roadmap and must be spec-backed before being advertised as complete.
