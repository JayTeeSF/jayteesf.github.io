Interactive source evaluation

import 'std:Eval'
let callable = eval(source_text)
let value = Eval.expression('1 + 2')

Evaluation is REPL-only, installs restricted declarations with `<eval>` metadata, and is unavailable in precompiled applications. See Evaluate source in the REPL.

Hey Language Reference

This is a compact current-surface reference. The tutorial should remain the first read for style and motivation.

Identifier policy is defined in Hey reserved words and backend-safe names. In particular, C11 keywords are legal Hey identifiers and are escaped only in generated C.

Function, worker, and actor names may end in one `?` or `!`. The suffix is part of the

callable name, works on module-qualified calls/spawns, and carries convention

rather than automatic semantics. See Callable name suffixes.

Functions and workers use ordinary parameter names. The compiler infers call, representation, and ABI facts from usage without requiring extra declaration metadata.

Writing a function name without parentheses produces a first-class `Callable`. Module-qualified functions behave the same way. Callables may be stored in collections and objects, transported through actor messages and Jobs payloads, and invoked dynamically. Writing an actor name without `spawn` produces an `ActorDefinition`, which can be stored and spawned later. `bind(callable, value)` captures one leading argument in an owned closure environment. See Functions and callable values.

Top-level forms

import 'path/to/file.hey'

module Name
  fn helper(arg)
    return arg
  end

  actor worker
    receive msg
    says msg
  end
end

fn function_name(arg1, arg2)
  return arg1
end

fn predicate?(value)
  return value != nil
end

fn publish!(value)
  return value
end

worker ParseLine(message)
  return parse_line(message)
end

job ParseLines
  worker ParseLine
  count: 4
  capacity: 64
end

actor name(optional, params)
  # statements
end

actor writer!
  receive message
end

spec 'name'
  expect stdout '...'
end

program
  # statements
end

`program` is the executable entry block. `spec` blocks make a file testable with `heyc <file> --test`. A top-level `job Name ... end` declaration must precede `program` and is syntax only over `Jobs.define(...)`.

Scope and imports

Current Hey scope is explicit and conservative:

  • `let` introduces a binding in the current statement/function-style scope.
  • `set` rebinds an existing local binding.
  • Module functions/actors are called with their module prefix, such as `Greeter.hello(user)`.
  • `import 'stdlib:Web'` loads a bundled stdlib module without exposing its checkout path.
  • `import './file.hey'` loads an app-local module relative to the importing file.
  • `import 'pkg:cards@1.2.3/main'` loads an installed package module from `${HEY_PACKAGE_PATH:-.hey/packages}`.
  • Imports do not inject every imported name unqualified into every scope.
  • There is no user-facing mutable global variable scope and no implicit root/main namespace automatically imported into modules.
  • Application/package module names conventionally use `UpperCamelCase`. Capitalization is not semantic: lexical bindings win over same-named module namespaces for field/property resolution in every maintained compiler lane.
  • A future language slice should add `const`/immutable module exports and import aliases/from-imports deliberately, with canonical-symbol collision diagnostics. See `docs/developers/modules-and-packages.md` for the canonical import path contract.

Core maps, arrays, and JSON codecs

Object field syntax and built-in-name precedence are specified in Object field access.

let doc = {name: 'Hey', visits: 2}
says doc.name
set doc = {...doc, visits: doc.visits + 1}
says doc.to_json

let xs = [1, 2.5, true, nil, 'A']
let expanded = [...xs, {name: 'Hey'}]
says expanded.to_json

Arrays, floats, object/map literals, and JSON codec helpers are supported in the current runtime. Primitive helpers such as `put` remain available, but ordinary public examples should prefer spread literals where supported.

I/O helpers and collection helpers

Current public I/O statements include `says`, `print`, `warn`, `write`, `log`, and `read`.

For live resources, prefer the stateful opaque-handle APIs documented in Stateful I/O handles and Processes, pipes, TCP, UDP, and readiness. Files, child-process streams, pipes, and TCP connections share stream capabilities through `stdlib:Io`; UDP remains datagram-oriented.

Use dot access for ordinary fixed object fields, bracket access for array indexes and dynamic/colliding keys, spread literals for immutable updates, and read-only properties for collection/value queries.

Read-only collection and value operations use property syntax:

says cards.len
says cards.length
says cards.size
says cards.count

Properties include `keys`, `values`, `entries`, `type`, `to_json`, and `json_canonical_text`. They may chain, for example `deck.keys.length`. Use spread literals for ordinary immutable updates.

After `import 'stdlib:Collections'`, receiver-oriented collection functions may

use dot syntax: `names.uniq`, `names.includes?('Ada')`, `items.chunk(20)`, and

`records.group_by_key('team')`. Direct module calls such as

`Collections.uniq(names)` remain valid and are not recursively desugared. Receiver chains may begin from literals, call results, or earlier receiver calls; `Jsonl.decode(source).sort(compare).take(2).to_json` recursively lowers to the corresponding imported module calls.

The `stdlib:Json` facade owns JSON decoding and encoding at external boundaries. Ordinary Hey code should keep values structured and render them with `value.to_json` only when text is required.

Truthiness

`false` and `nil` are falsey. Everything else is truthy. Prefix `!` is the concise negation operator; `not` remains a readable word alias. Boolean expressions compose with `or` and `and`, and all maintained compiler/interpreter lanes share the same precedence and short-circuit behavior: `true or right()` skips `right`, while `false and right()` skips `right`. Predicate callables conventionally end in `?`.

Typing

Hey does not require annotations for ordinary functions. Optional annotations are currently an alpha compiler feature; the clean architecture plan keeps source-level types semantic and keeps machine/ABI types inside compiler IR. Generated `.hey/` metadata may cache analysis or module interfaces, but it is never a required semantic input and deleting it must not change program behavior. See Clean compiler architecture.

Plumbing and porcelain rule

Language/runtime primitives should stay small: immutable values, functions, modules, imports, stateful actors, stateless workers, private runtime mailboxes, supervision, links/monitors, I/O targets, scheduler hooks, shape metadata, and allocation/performance introspection. General-purpose channels are not a user-facing primitive.

Higher-level workflows such as JSON codecs, job queues, stream processors, HTTP, and 1BRC helpers belong in the standard library or packages unless missing runtime plumbing blocks them.

Strings

Triple-quoted strings use `"""` delimiters for multi-line text. They preserve newlines and support interpolation. See `docs/developers/multiline-strings.md`.
