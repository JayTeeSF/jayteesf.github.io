Unified I/O handles, streams, mappings, and actor suspension

Corrected v0.99.226a status

The portable **blocking** handle layer is implemented on one opaque,

reference-counted `io_handle` runtime kind. Live operating-system resources are

never represented by JSON descriptors.

Implemented constructors and resource kinds:

  • `files.open(path, mode)` for stateful files;
  • `files.open_mapping(path)` for explicit read-only mapping lifetimes;
  • `files.map_readonly(path)` for automatic-lifetime mapped `Bytes`;
  • `process.spawn(args, options)` with live child/stdin/stdout/stderr handles;
  • `pipes.create(capacity)` with bounded reader/writer handles;
  • `net.listen`, `net.connect`, and `net.accept` for TCP;
  • `net.bind_udp` for datagrams;
  • `io.cancel_token` for readiness wake-up cancellation.

The common operations live in `stdlib:io` and are also available as receiver

methods on compatible handles:

import 'stdlib:files'
import 'stdlib:io'

let file = files.open('events.log', 'read').value
let first = file.read(64.kibibytes)
let line = file.read_line(1.mebibyte)
let framed = file.read_until(10, 1.mebibyte) # newline byte
file.seek(0, 'start')
says file.tell().value
says file.capabilities().value
file.close()

`read`, `read_exact`, `read_until`, and `read_line` return Results whose value

contains `{data, count, eof}` plus operation-specific metadata. Empty bytes are

valid data, not an error sentinel.

Explicit mapping lifetime

let opened = files.open_mapping(path)
let mapping = opened.value
let shard = mapping.mapping_slice(start, finish).value
says mapping.capabilities().value.kind # mapping
mapping.close()

# Existing retained byte views remain valid after close.
consume(shard)

# Close prevents creation of new views.
says mapping.mapping_slice(0, 1).ok # false

A mapping handle owns one retained mapped backing. `close` prevents new views

and releases the handle's reference; physical unmapping occurs after the last

retained `Bytes` view is released. The concise `files.map_readonly` form remains

appropriate when explicit close is unnecessary.

Capabilities

`io.capabilities(handle)` reports the actual resource contract:

  • `readable`;
  • `writable`;
  • `seekable`;
  • `closable`;
  • `pollable`;
  • `deadline_aware`;
  • `closed`;
  • resource `kind`.

Files and mappings are not reactor deadline-aware. TCP, TLS, pipes, child streams, listeners, and cancellation handles are pollable. Blocking `io.select` still occupies its calling OS thread; `io.select_async` and actor-source `await` register with the multiplexed reactor instead.

Process, pipe, TCP, and UDP ownership

Child stdio, independent pipes, files, and TCP connections use the same

`io.read`, `read_exact`, `read_until`, `read_line`, `write`, `write_all`,

`flush`, `set_timeout`, `close`, `eof?`, and `closed?` semantics. Construction

and specialized controls remain in domain modules.

UDP is intentionally datagram-oriented and uses `net.send_to`/`recv_from`

rather than pretending to be a sequential stream.

MIR and LLVM boundary

Source methods lower to ordinary module calls and then to a small typed runtime

ABI. The compiler does not recognize applications or protocols. LLVM cannot

optimize a kernel syscall, but it can optimize the surrounding Hey control flow

when handles, byte views, offsets, lengths, ownership, and Result tags remain

explicit.

The desired long-term MIR vocabulary is:

io.open / io.map / io.spawn / io.connect / io.accept
io.read / io.read_at / io.write / io.seek / io.flush / io.close
io.poll / io.cancel

The current ABI is still represented by stable runtime facade calls; future

slices should consolidate them without changing observable semantics.

Chunk isolation for compiler-world proof

The LLVM/MIR world runner uses one process per bounded source chunk, with shared

content-addressed compiler/parser/runtime caches. Per-source isolation was

rejected because it made cold closure proofs impractically slow. Chunk-level

isolation still bounds retained compiler state around large imported units such

as the honest 1BRC library.

Implemented scheduler and transport slices

Callable and actor values

Functions, bound callable environments, and actor definitions are real owned values. Generated program dispatchers preserve the direct-call fast path while allowing callbacks and actor targets to survive arrays, objects, messages, and Jobs payloads.

Actor-suspending reactor

  • kqueue on Darwin/BSD;
  • epoll on Linux;
  • poll fallback;
  • deadlines, cancellation, and close-safe descriptor duplication;
  • explicit `io.select_async`;
  • actor-native `await handles, timeout_ms, tag`;
  • automatic lightweight worker-pool scheduling for native-C actors containing `await`;
  • mailbox re-entry through tagged `io_completion` Results.

Direct LLVM accepts the same source and MIR semantics but still uses thread-per-actor execution. Moving that lane onto the lightweight pool and keeping a persistent OS registration set are the next scheduler refinements.

Portable TLS and HTTPS

Darwin and Linux use one explicit OpenSSL provider ABI. TLS validates certificate chains and DNS/IP identities by default, participates in readiness including `SSL_pending`, and supports byte-preserving bounded, checksum-aware, atomically published HTTP/HTTPS downloads.

Process capture and pipe pressure

`process.run_argv`, `process.run_env_argv`, and `process.run_argv_timeout` now

share one process-group-aware capture loop. It drains stdout and stderr

concurrently with nonblocking reads while polling child status. Ordinary command

results include both `stdout` and `stderr`; diagnostics are data, not inherited

terminal side effects. This prevents the classic deadlock where a child fills

stderr while the parent blocks reading stdout. Spec 934 and

`test/regression_process_dual_pipe_capture.sh` push both pipes beyond their

normal capacity and keep the regression externally bounded.
